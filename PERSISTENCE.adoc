== Overview

The `smallmind-persistence` module is a layered persistence toolkit. It sits
between an application's domain model and the underlying storage — a JDBC
connection pool, a Hibernate or JPA entity manager, a Throng-based MongoDB
client, a Memcached daemon — and supplies a set of small abstractions that let
the same DAO code be backed by different stores and composed with a distributed
cache.

The module is published as a single artifact, `smallmind-persistence`, but
its dependencies on ORM and cache implementations are declared `optional`. A
consumer pulls in only the technologies it actually uses — Hibernate, JPA,
QueryDSL, Throng (MongoDB), Memcached, Spring — and the unused parts stay
dormant.

It is organised into six top-level packages under `org.smallmind.persistence`:

* `persistence` — the core `Durable` contract, the `Dao` hierarchy, natural
  keys, and the shared exception types.
* `persistence.cache` — the `PersistenceCache` contract, the vector model
  (`DurableVector`, `VectorKey`, `VectoredDao`), cache domains, and the
  Memcached backend.
* `persistence.cache.aop` — AspectJ annotations (`@CacheAs`, `@CachedWith`,
  `@CacheCoherent`) and the aspects that implement them.
* `persistence.orm` — `ORMDao`, `ProxySession`, `ProxyTransaction`, the
  `SessionManager`, and adapters for Hibernate (via JPA), JPA, Throng, and
  QueryDSL.
* `persistence.orm.aop` — AspectJ annotations (`@Transactional`,
  `@NonTransactional`, `@Timed`) and the aspects and thread-local state that
  enforce transactional boundaries.
* `persistence.sql` / `persistence.sql.pool` — a `DriverManagerDataSource`,
  a pooled `DataSource` built on `smallmind-quorum`'s component pool, and
  Spring factory beans to wire everything up.
* `persistence.database` — the `Sequence` abstraction, a static
  `SequenceManager`, an `InMemorySequence`, and a MySQL-specific
  `SimulatedSequence`.

=== Why This Package Exists

The persistence ecosystem in Java is rich but uneven. Hibernate provides
sessions, transactions, and first-level caching; JPA standardises the same
ideas; MongoDB has its own driver; Memcached has its own client. An
application that mixes these — a relational primary store, a document
secondary store, an out-of-process cache in front of both — quickly
accumulates glue code that is neither testable nor portable.

`smallmind-persistence` exists to name and standardise that glue:

. *One DAO contract per durable, regardless of backing store.* `Durable<I>`
  is the one interface every persisted object implements. `ORMDao` is the one
  DAO base class the application writes. Behind it sits either `JPADao`
  (relational), `ThrongDao` (MongoDB), or a future adapter — the application
  code does not change.
. *One transactional boundary mechanism across ORMs.* SmallMind's
  `@Transactional` is not Spring's and not JPA's. It is an AspectJ annotation
  whose aspect manages thread-local stacks of boundary sets, permitting
  explicit control over which data sources a boundary covers, whether it is
  rollback-only, and whether exceptions trigger rollback. It participates
  with `@NonTransactional` to prevent session stealing and to enforce that
  every session opened for a boundary is closed when that boundary unwinds.
. *A cache model that is more than get/put.* A single `PersistenceCache<K,V>`
  is only useful for toy examples. Real applications cache *query results*
  that may become stale when any of several durables are persisted or
  deleted. The `@CacheAs`/`@CachedWith` annotation pair expresses that
  dependency graph at the DAO method level — the aspect then handles cache
  population, invalidation, and fine-grained vector updates mechanically.
. *Portable identifier parsing.* REST and RPC layers need to convert a
  string back into a strongly-typed id. `AbstractManagedDao.getIdFromString`
  handles primitives, wrappers, enums, `String`, and any custom `Identifier`
  implementation with a static `fromString`, so upstream framework code
  never writes type-specific parsing.
. *Sequences that work on MySQL.* MySQL has no native `SEQUENCE`. The
  `database.mysql.SimulatedSequence` simulates one via `LAST_INSERT_ID` on a
  counter table, with optional block allocation to reduce round-trips. The
  `SequenceManager` exposes it behind the same static `nextLong(name)` API
  an Oracle-backed implementation would use.

=== What It Is Not

* It is not a full ORM. Hibernate and JPA remain responsible for mapping,
  lazy loading, and SQL generation. `ORMDao` is a thin wrapper over a
  `ProxySession`, not a replacement for the entity manager.
* It is not a transaction manager. `TransactionalState` manages the
  *boundary stack*; the underlying commit and rollback are delegated to the
  native `Transaction` inside the `ProxySession`.
* It is not an auto-configuration framework. Wiring is explicit: a caller
  declares a `ProxySession`, a `DataSource`, an optional cache, and the
  DAOs that use them. The `*SeekingBeanFactoryPostProcessor` classes help
  Spring XML discover annotated entity classes, but they do not replace the
  XML itself.
* It is not a caching framework in the sense of an on-heap cache. The
  `PersistenceCache` interface models an out-of-process store (Memcached by
  default). Hibernate's first-level cache and any second-level cache remain
  orthogonal.

== Installation

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>smallmind-persistence</artifactId>
  <version>${project.version}</version>
</dependency>
----

Every ORM, cache, and Spring dependency is declared `optional` in the module
POM. A consumer must explicitly pull in the subset it uses.

[cols="1,3",options="header"]
|===
| You want to use... | Also declare

| Throng (MongoDB)
| `mongodb-driver-sync`, `mongodb-driver-core`, `bson`,
  `mongodb-throng`, `spring-beans`

| `memcached-utility` (and its spymemcached transitive)

| The connection pool
| `smallmind-quorum`, `spring-beans`

| `testbench-condition`

The module is AspectJ-woven at build time. Consumers that declare
`@Transactional`, `@NonTransactional`, `@CacheAs`, `@CachedWith`, or
`@CacheCoherent` must themselves be woven — the pom's `aspectj-maven-plugin`
configuration is the canonical reference.

Hibernate entities are bytecode-enhanced at build time by the
`hibernate-enhance-maven-plugin`. Consumers that provide their own entity
classes normally declare the same plugin in their own builds.

[[core-durable]]
== The Durable and DAO Core

=== Durable<I>

`Durable<I extends Serializable & Comparable<I>>` is the one contract every
persisted object implements. Its identifier is both `Serializable` (so it
can survive a trip through a cache) and `Comparable<I>` (so roster-backed
caches can keep a total order).

`AbstractDurable<I, D extends AbstractDurable<I, D>>` provides:

* id-based `compareTo`, with nulls sorting before non-nulls;
* id-based `equals`/`hashCode` when both sides have a non-null id, falling
  back to identity semantics when one side is transient;
* `mirrors(Durable)` — reflective, id-excluding deep equality with an
  optional exclusion list, useful for change detection;
* a thread-local-guarded `toString` that handles cyclic graphs.

A typical domain class extends `AbstractDurable` (or a framework-specific
subclass like `JPADurable`):

[source,java]
----
@Entity
@Table(name = "orders")
public class Order extends JPADurable<Long> {

  private String reference;
  private Long customerId;

  @Id
  @GeneratedValue(strategy = GenerationType.IDENTITY)
  public Long getId () { return super.getId(); }
  public void setId (Long id) { super.setId(id); }

  public String getReference () { return reference; }
  public void setReference (String reference) { this.reference = reference; }

  public Long getCustomerId () { return customerId; }
  public void setCustomerId (Long customerId) { this.customerId = customerId; }
}
----

=== The DAO Hierarchy

The DAO contracts form a layered stack. Each layer adds exactly one concern.

`Dao<I, P>`::
    Minimum contract: `get` and `delete`. Never implemented directly.

`DurableDao<I, D>`::
    Adds CRUD for `Durable`: `get`, `persist`, `delete`, `list` (full,
    bounded, or keyset-paged), and bulk `list(Collection<I>)`. The id
    pagination variant `list(I greaterThan, int fetchSize)` is exclusive
    on the lower bound and ordered ascending.

`ManagedDao<I, D>`::
    Adds metadata: `getMetricSource`, `getManagedClass`, `getIdClass`,
    `getIdFromString`. `AbstractManagedDao` infers the id and durable
    classes from the subclass's generic parameters at runtime.
    `getIdFromString` handles primitives, primitive wrappers, `String`,
    enums, and any `Identifier` subtype with a static `fromString`.

`VectorAwareDao<I, D>`::
    Mixin declaring `getVectoredDao()`. `AbstractVectorAwareManagedDao`
    implements it, returning the delegate only when `isCacheEnabled()` is
    true, so cache can be toggled per session or globally.

`CacheDao<I, D> extends VectoredDao<I, D>`::
    Adds direct access to instance and vector caches. `AbstractCacheDao`
    delegates to a `CacheDomain` for the actual `PersistenceCache`
    instances.

`ORMDao<I, D, F, N> extends AbstractVectorAwareManagedDao<I, D>`::
    The base every application-visible DAO extends. Holds a
    `ProxySession<F, N>`, registers itself with `OrmDaoManager` via
    `register()`, and delegates `get`/`persist`/`delete` to framework
    subclasses (`JPADao`, `ThrongDao`, etc.).

=== Exceptions

All persistence exceptions extend `PersistenceException`, which extends
`FormattedRuntimeException` from `smallmind-nutsnbolts`. They all accept a
`printf`-style message template and arguments.

* `DataIntegrityException` — persisted data violates a domain invariant.
* `AuthorizationException` — the caller lacks authorization for the op.
* `IdentifierNotFoundException` — a looked-up id has no matching record.
* `CacheOperationException` — an underlying cache operation failed.
* `ORMInitializationException` — no session or DAO is registered for a key.
* `ORMOperationException` — a framework-level persist/fetch/delete failed.
* `SessionEnforcementException` — a session was opened outside any active
  boundary and the session is marked `boundaryEnforced`.

[[transactional-annotations]]
== @Transactional And @NonTransactional

=== Why There Are Two Annotations, Not One

A service method either needs a transaction or it does not. In most
frameworks, the absence of `@Transactional` is taken to mean "no transaction".
SmallMind treats both states as *explicit*:

* `@Transactional` opens a transactional boundary. Any
  `ProxySession.getNativeSession()` called inside the boundary joins the
  active transaction for its data source.
* `@NonTransactional` opens a non-transactional session boundary. A session
  is opened on first use, kept open for the duration of the boundary body,
  and closed when the boundary unwinds. No commit is issued.

Why make absence-of-transaction explicit? Because a `ProxySession` that has
`boundaryEnforced=true` will *refuse* to open a session outside any
boundary, throwing `SessionEnforcementException`. This guarantees that
every query either runs inside a transaction (consistent reads, commits on
success) or inside a `@NonTransactional` boundary (read-only, closed on
return). It prevents accidental leaks of `EntityManager` instances and
accidental auto-commit behaviour.

=== @Transactional

[source,java]
----
@Target({ElementType.TYPE, ElementType.CONSTRUCTOR, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface Transactional {

  String[] dataSources () default {};
  boolean implicit () default true;
  boolean rollbackOnly () default false;
  boolean rollbackOnException () default true;
}
----

`dataSources`::
    An explicit list of session source keys this boundary permits, matched
    against each joining session's `getSessionSourceKey()`. Declaring explicit
    sources forces a boundary to be *explicit*, not implicit — set
    `implicit=false` (see below). Left empty, the array adds no source
    constraint; what an empty array permits then depends on `implicit`.

`implicit` (default `true`)::
    When `true`, the boundary is established implicitly from a class-level
    annotation on a DAO and acts as a catch-all: with no `dataSources` it
    permits a session from *any* source. An implicit boundary that also lists
    explicit `dataSources` throws `IllegalArgumentException` at boundary-check
    time, so set `implicit=false` whenever you list explicit sources. An
    explicit boundary (`implicit=false`) with no `dataSources` permits only the
    default unnamed source (a `null` source key); with `dataSources` it permits
    only the listed sources.

`rollbackOnly` (default `false`)::
    Forces every transaction in the boundary to roll back regardless of
    outcome. Used for tests or for methods whose side-effects should be
    observed but not committed.

`rollbackOnException` (default `true`)::
    When an exception escapes the boundary, roll back (`true`) or commit
    (`false`). The default is the common-sense behaviour; `false` is
    useful when an exception is a *signal*, not a failure, and the
    updates made before it should persist.

=== @NonTransactional

[source,java]
----
@Target({ElementType.TYPE, ElementType.CONSTRUCTOR, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface NonTransactional {

  String[] dataSources () default {};
  boolean implicit () default true;
}
----

`dataSources` and `implicit` have the same meaning as on `@Transactional`.
A non-transactional boundary opens a read-only session scope: the session
is created on first use, held for the boundary body, and closed on exit.
Commit and rollback do not apply.

=== Boundary Semantics In Detail

Both aspects use matching stacks of thread-local boundary sets:

`TransactionalState`::
    Thread-local `LinkedList<RollbackAwareBoundarySet<ProxyTransaction<?>>>`.
    Every `@Transactional` method push/pops one entry. Each entry carries
    the annotation's `dataSources` array and `rollbackOnly` flag, and
    holds every `ProxyTransaction` that joined it.

`NonTransactionalState`::
    Thread-local `LinkedList<BoundarySet<ProxySession<?, ?>>>`. Every
    `@NonTransactional` method push/pops one entry. Each entry carries
    the annotation's `dataSources` array and holds every `ProxySession`
    that joined it.

When a DAO asks its `ProxySession` for a native session (e.g.,
`EntityManager`), the session does this:

. Call `TransactionalState.obtainBoundary(this)` — does any active
  transactional boundary permit my source key?
. If yes, begin a transaction and add it to that boundary's set. Also
  check `NonTransactionalState.containsSession(this)` — if a
  non-transactional boundary has already taken the session on this
  thread, raise `StolenTransactionError`. This prevents a transactional
  boundary from *stealing* a session that a non-transactional boundary
  is relying on.
. If no transactional boundary allows the session, call
  `NonTransactionalState.obtainBoundary(this)`.
. If neither allows it, and `isBoundaryEnforced()` is true, raise
  `SessionEnforcementException`. Otherwise the session runs detached
  (rare; `boundaryEnforced=true` is the normal production configuration).

On boundary exit:

* `@Transactional` success → `commitBoundary()`: for every transaction in
  the boundary, commit unless the boundary is `rollbackOnly` or the
  transaction itself is `rollbackOnly`.
* `@Transactional` exception + `rollbackOnException=true` →
  `rollbackBoundary(throwable)`: every transaction rolls back.
* `@Transactional` exception + `rollbackOnException=false` →
  `commitBoundary(throwable)`: commit as if success.
* `@NonTransactional` exit → close every session in the boundary.

If any individual commit or rollback throws, the first such failure is
wrapped in `IncompleteTransactionError`; subsequent failures are silently
absorbed so one bad transaction cannot hide the others from failing.

=== Aspect Precedence

`TransactionalAspect` is declared after
`org.smallmind.nutsnbolts.inject.LazyFieldAspect`:

[source,java]
----
@Aspect
@DeclarePrecedence("org.smallmind.nutsnbolts.inject.LazyFieldAspect, "
                   + "org.smallmind.persistence.orm.aop.TransactionalAspect")
public class TransactionalAspect { ... }
----

This means that when a lazy-injected field is resolved inside a
`@Transactional` method, the lazy field is resolved *before* the
transaction boundary is established, so lazy-injection lookups do not
accidentally open a transaction of their own.

=== Usage Examples

The canonical pattern: annotate a service-layer class with
`@Transactional`. Every method (and constructor) in the class establishes
a boundary unless separately annotated `@NonTransactional`.

[source,java]
----
@Transactional
public class OrderService {

  private final OrderDao orderDao;
  private final AuditDao auditDao;

  public OrderService (OrderDao orderDao, AuditDao auditDao) {
    this.orderDao = orderDao;
    this.auditDao = auditDao;
  }

  public Order placeOrder (Long customerId, String reference) {

    Order order = new Order();
    order.setCustomerId(customerId);
    order.setReference(reference);
    orderDao.persist(order);               <1>

    auditDao.persist(new AuditEntry(order, "placed"));  <2>
    return order;
  }

  @NonTransactional                        <3>
  public Order findById (Long id) {
    return orderDao.get(id);
  }
}
----
<1> Runs inside the implicit boundary opened by the class-level annotation.
<2> Same boundary — commits or rolls back atomically with the order.
<3> Overrides the class-level boundary. `findById` opens a
    non-transactional session, closes it on return, and never commits.

Multiple data sources:

[source,java]
----
public class CrossStoreReport {

  private final ReportingDao relationalDao;
  private final MetricsDao mongoDao;

  @Transactional(implicit = false, dataSources = {"relational", "metrics"})  <1>
  public ReportBundle assemble (LocalDate date) {
    return new ReportBundle(
      relationalDao.findByDate(date),
      mongoDao.findByDate(date)
    );
  }
}
----
<1> `implicit=false` is required whenever `dataSources` is explicit.
    Both sessions (keyed `"relational"` and `"metrics"`) join the same
    boundary, and both commit together or roll back together.

Forcing rollback in an integration test:

[source,java]
----
@Transactional(rollbackOnly = true)       <1>
public void assertCustomerCanBeInserted () {

  Customer customer = new Customer("Test", "test@example.com");
  customerDao.persist(customer);
  assertThat(customer.getId()).isNotNull();
}
----
<1> Writes happen and the id generator fires, but the boundary rolls
    back on return. The test leaves no trace.

Suppressing rollback on a signalling exception:

[source,java]
----
@Transactional(rollbackOnException = false)  <1>
public void runMigrationStep (int step) throws StopMigrationSignal {

  migrationDao.applyStep(step);
  if (step == lastStep()) {
    throw new StopMigrationSignal();
  }
}
----
<1> The `StopMigrationSignal` propagates to the caller *after* the step's
    updates commit. Use sparingly; the default of `true` is safer.

=== Spring Wiring For Boundaries

A `ProxySession` bean's `init-method="register"` places it into the
`SessionManager` under a source key. The `boundaryEnforced` constructor
argument chooses whether the session will refuse to open outside an AOP
boundary.

[source,xml]
----
<bean id="schemaProxySession"
      class="org.smallmind.persistence.orm.hibernate.HibernateProxySession"
      init-method="register">
  <constructor-arg index="0" value="orm"/>           <1>
  <constructor-arg index="1" value="schema"/>        <2>
  <constructor-arg index="2" ref="schemaSessionFactory"/>
  <constructor-arg index="3" value="true"/>          <3>
  <constructor-arg index="4" value="true"/>          <4>
</bean>
----
<1> `dataSourceType` — descriptive, used as the metric tag.
<2> `sessionSourceKey` — the key the aspects will match against
    `@Transactional(dataSources=...)`.
<3> `boundaryEnforced` — raise `SessionEnforcementException` if a session
    is requested outside a boundary. Set to `true` in production.
<4> `cacheEnabled` — default value of `isCacheEnabled()` on this session;
    togglable per-thread via `overrideCacheEnabled`.

[[cache-as]]
== Using @CacheAs

`@CacheAs` is the cache-aside read path. Apply it to a DAO method whose
result is expensive to compute and changes rarely; the next call for the
same key will be served from the cache instead of the ORM.

[source,java]
----
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface CacheAs {

  Vector value ();
  Class<? extends Comparator> comparator () default Comparator.class;
  int max () default 0;
  Time time () default @Time(0);
  boolean ordered () default false;
}
----

=== What The Aspect Does

On invocation, `CacheAsAspect`:

. Builds a `VectorKey` from the `@Vector` descriptor, the method
  arguments, and the `VectorCalculator`.
. Calls `vectoredDao.getVector(vectorKey)`.
. If the vector exists and `isAlive()` returns true, serves from cache.
  For a single-durable method this is `vector.head()`; for an iterable
  method this is `asBestEffortPreFetchedList()` (if the return type is
  `List`) or `asBestEffortLazyList()` (otherwise).
. If the vector is present but stale, deletes it and falls through to
  the ORM call.
. Proceeds to the underlying method. If the result is non-null, wraps
  it in a vector (`createSingularVector` or `createVector`), stores it
  via `persistVector`, and returns.

Any misconfiguration surfaces as a `CacheAutomationError` with a
specific message. In particular:

* A scalar `@CacheAs` method cannot declare `ordered`, `max`, or
  `comparator`.
* An iterable `@CacheAs` method must be parameterised to
  `<? extends Iterable<? extends ManagedDurable>>`; the aspect reads the
  actual type argument through reflection.
* A non-iterable method that declares a comparator throws — the aspect
  will not silently ignore the annotation.

=== Vector, Key, Classifier

`@Vector` describes how the cache key is built. Its attributes:

`namespace` (default: method name when used on `@CacheAs`)::
    The prefix of every key built from this vector.

`value` (required)::
    An ordered `@Key[]` describing each key segment.

`classifier` (default: empty)::
    An optional `@Classifier` appended to the key.

`@Key` attributes:

`value` (default `"id"`)::
    The name of the method parameter (for `@CacheAs`) or the bean
    property on the durable (for `@CachedWith` updates) that provides the
    value.

`alias` (default `""`)::
    Renames the key segment in the generated cache key. Useful when the
    parameter name is verbose but you want a short, stable key.

`constant` (default `false`)::
    When true, `value` is used verbatim as a literal string, not as a
    parameter or field name.

`nullable` (default `false`)::
    Permits null for this segment. Without it, a null value at key
    construction time is an error.

`@Classifier` attributes:

`value` (required)::
    A literal string, or a method parameter name if `asParameter=true`.

`asParameter` (default `false`)::
    When true, treat `value` as a method parameter name and use the
    parameter's runtime value as the classifier.

=== Time

`@Time` configures the cache entry's TTL with optional stampede jitter:

[source,java]
----
@Time(value = 60, stochastic = 10, unit = TimeUnit.SECONDS)
----

`value`::
    Base TTL in `unit` units. Zero means "use the cache domain default".

`stochastic` (default 0)::
    Upper bound of a random delta added to `value`, in the same unit.
    The aspect adds `stochastic * Random.nextDouble() + 1` to the base,
    then converts to seconds. Zero disables jitter.

`unit` (default `TimeUnit.SECONDS`)::
    Unit for both `value` and `stochastic`.

The jitter exists to spread expiry times across callers that populated
the same entry at the same moment. Without it, an entry populated at
startup by thousands of requests would expire simultaneously and all of
them would miss on the next tick — a classic cache-stampede pattern.

=== Examples

A scalar lookup by id:

[source,java]
----
@CachedWith(...)
public class CustomerDao extends JPADao<Long, Customer> {

  @CacheAs(
    value = @Vector(namespace = "customer", value = @Key("id")),
    time = @Time(value = 5, unit = TimeUnit.MINUTES)
  )
  public Customer findById (Long id) {
    return acquire(Customer.class, id);
  }
}
----

A scalar lookup by a non-id parameter, with a renamed key:

[source,java]
----
@CacheAs(
  value = @Vector(
    namespace = "customer.byEmail",
    value = @Key(value = "email", alias = "e")
  ),
  time = @Time(value = 10, unit = TimeUnit.MINUTES)
)
public Customer findByEmail (String email) {
  // JPA query or QueryDSL call
}
----

An ordered iterable with a comparator and a cap:

[source,java]
----
@CacheAs(
  value = @Vector(
    namespace = "orders.byCustomer",
    value = @Key("customerId")
  ),
  comparator = Order.ByReferenceComparator.class,
  ordered = true,
  max = 50,
  time = @Time(value = 1, stochastic = 15, unit = TimeUnit.MINUTES)
)
public List<Order> findAllByCustomer (Long customerId) {
  // ...
}
----

A literal-keyed singleton, useful for cacheable configuration:

[source,java]
----
@CacheAs(
  value = @Vector(
    namespace = "config.currentTaxRate",
    value = @Key(value = "current", constant = true)
  ),
  time = @Time(value = 1, unit = TimeUnit.HOURS)
)
public TaxRate currentTaxRate () { ... }
----

A polymorphic classifier to cache different subtypes under the same
namespace:

[source,java]
----
@CacheAs(
  value = @Vector(
    namespace = "asset.byOwner",
    value = @Key("ownerId"),
    classifier = @Classifier(value = "kind", asParameter = true)
  ),
  time = @Time(value = 5, unit = TimeUnit.MINUTES)
)
public Iterable<Asset> findByOwnerAndKind (Long ownerId, String kind) { ... }
----

=== Graceful Degradation

`@CacheAs` depends on `ORMDao.getVectoredDao()` returning non-null. That
method returns the delegate only when `isCacheEnabled()` is true on the
underlying `ProxySession`. If either is false — the DAO was wired without
a cache, or the session's thread-local cache flag was temporarily turned
off — the aspect simply proceeds to the underlying method and returns
its result unwrapped. The annotation is a hint, not a requirement.

[[cached-with]]
== Using @CachedWith

`@CacheAs` keeps fresh data in the cache. `@CachedWith` keeps the cache
coherent when the underlying data changes. The two annotations are
complements: without `@CachedWith`, a persist or delete will leave
`@CacheAs`-populated vectors stale until their TTL expires. That may be
acceptable for low-churn data; for anything else, `@CachedWith` is
mandatory.

`@CachedWith` is a class-level annotation on a DAO:

[source,java]
----
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface CachedWith {

  Update[] updates () default {};
  Invalidate[] invalidates () default {};
}
----

=== What The Aspect Does

`CachedWithAspect` intercepts:

* any `persist(Durable+)` method on a `@CachedWith` class (or any method
  annotated `@Persist`), and
* any `void delete(..)` method on a `@CachedWith` class (or any method
  annotated `@Delete`).

After the method proceeds, the aspect walks `updates()` and
`invalidates()`. For each one:

. *Filter* — if a `filter` method name is given, call
  `filterMethod.invoke(dao, durable)`. The method must return `boolean`.
  If false, skip this update/invalidate.
. *Finder* — if a `finder.method` is given, call it on the DAO with the
  durable; otherwise use the durable itself (as a `SingleItemIterable`).
  The finder's return type is reconciled with `finder.with` (the
  expected durable type) reflectively.
. *Proxy* — if a `proxy.method` is given, call it to transform the
  durable into the form used for key construction (useful when the key
  is taken from a related entity, not the one being persisted).
. *Compute vector key* from the `@Update`/`@Invalidate` `@Vector`
  descriptor, the (possibly-proxied) durable, and the classifier.
. *Apply the action*:
  - `@Update` + `OnPersist.INSERT` → `vectoredDao.updateInVector(key,
    indexingDurable)`.
  - `@Update` + `OnPersist.REMOVE` → `vectoredDao.removeFromVector(key,
    indexingDurable)`.
  - `@Invalidate` → `vectoredDao.deleteVector(key)`.

On delete, the semantics flip: every `@Update` applies `REMOVE`,
regardless of the configured `onPersist`, because the durable no longer
exists. `@Invalidate` runs unchanged.

=== OnPersist

`@Update.onPersist` is an optional DAO method name that returns
`OnPersist`. When omitted, the aspect defaults to `OnPersist.INSERT`.
The point of making it a method rather than an annotation attribute is
that the action may depend on the durable's state: a "soft delete" flag
flipping to true may mean REMOVE, while any other persist means INSERT.

[source,java]
----
public OnPersist deletedOrActive (Order order) {
  return order.isCancelled() ? OnPersist.REMOVE : OnPersist.INSERT;
}
----

=== Examples

A DAO that caches its own by-id lookup and invalidates it on persist:

[source,java]
----
@CachedWith(
  invalidates = {
    @Invalidate(
      value = @Vector(namespace = "customer", value = @Key("id"))
    )
  }
)
public class CustomerDao extends JPADao<Long, Customer> {

  @CacheAs(
    value = @Vector(namespace = "customer", value = @Key("id")),
    time = @Time(value = 5, unit = TimeUnit.MINUTES)
  )
  public Customer findById (Long id) {
    return acquire(Customer.class, id);
  }
}
----

A DAO whose persist updates a per-customer order list:

[source,java]
----
@CachedWith(
  updates = {
    @Update(
      value = @Vector(
        namespace = "orders.byCustomer",
        value = @Key("customerId")
      ),
      onPersist = "deletedOrActive"
    )
  },
  invalidates = {
    @Invalidate(
      value = @Vector(namespace = "order", value = @Key("id"))
    )
  }
)
public class OrderDao extends JPADao<Long, Order> {

  @CacheAs(
    value = @Vector(
      namespace = "orders.byCustomer",
      value = @Key("customerId")
    ),
    comparator = Order.ByReferenceComparator.class,
    ordered = true,
    max = 50,
    time = @Time(value = 1, stochastic = 15, unit = TimeUnit.MINUTES)
  )
  public List<Order> findAllByCustomer (Long customerId) { ... }

  public OnPersist deletedOrActive (Order order) {
    return order.isCancelled() ? OnPersist.REMOVE : OnPersist.INSERT;
  }
}
----

When a persisted `Order` has `customerId=42`, the aspect:

. Evaluates the filter (none in this example).
. Calls `deletedOrActive(order)` → `INSERT`.
. Uses the durable itself (default finder) as the indexing durable.
. Builds the vector key `Order:orders.byCustomer[customerId=42]`.
. Calls `vectoredDao.updateInVector(key, order)`.
. Builds the key `Order:[id=<order.id>]` from the `@Invalidate` and
  calls `deleteVector`, evicting the scalar entry.

A cross-entity invalidation using `proxy`:

[source,java]
----
@CachedWith(
  invalidates = {
    @Invalidate(
      value = @Vector(
        namespace = "customer.profile",
        value = @Key("id")
      ),
      finder = @Finder(with = Customer.class, method = "customerOf"),  <1>
      proxy = @Proxy(with = Customer.class, method = "toCustomer")     <2>
    )
  }
)
public class OrderDao extends JPADao<Long, Order> {

  public Customer customerOf (Order order) {                          <1>
    return customerDao.get(order.getCustomerId());
  }

  public Customer toCustomer (Customer customer) {                    <2>
    return customer;
  }
}
----
<1> The finder resolves the related customer given the order being
    persisted. The aspect uses that customer as the `indexingDurable`.
<2> The proxy leaves the customer alone, but any transformation could
    be performed here. The (possibly-transformed) durable is then used
    to build the cache key — so the vector key reads the `id` property
    of the customer, not the order.

=== A Note On @Persist And @Delete

The default pointcut matches any `persist(Durable+)` and `void delete(..)`
on a `@CachedWith` class. If a DAO has a persist-style method with a
different name (`saveOrUpdate`, `store`, etc.), tag it with `@Persist`;
for a delete-style method, tag it `@Delete`. The aspect picks up both
forms.

[[cache-coherent]]
== Using @CacheCoherent

`@CacheCoherent` is the smaller cousin of `@CachedWith`. It is a
method-level marker that says "whatever this method returns, make sure
the cache knows about it before you hand it out". Useful when the method
fetches durables from an external system — a remote API, a bulk load —
that should immediately appear in the cache as if they had been
`persist()`ed.

The aspect wraps a single durable in
`vectoredDao.persist(Class, Durable, UpdateMode.SOFT)`. For an
`Iterable<Durable>` result it wraps the result in `CacheCoherentIterable`,
which performs the same `persist` lazily as each element is iterated, so
an unused tail of a very long iterable is not eagerly cached.

Use it alongside a finder that does not itself go through the DAO:

[source,java]
----
@CacheCoherent
public List<Customer> importFromFeed () {
  return externalFeed.readCustomers();
}
----

[[cache-architecture]]
== Cache Architecture

The cache layer sits between `ORMDao` and an underlying
`PersistenceCache`. Everything between those two ends is framework code.

=== PersistenceCache

`PersistenceCache<K, V>` is a minimal get/set/remove store. Its semantics
match Memcached's:

* `get(K)` — single fetch, or null on miss.
* `get(K[])` — bulk fetch, missing keys are omitted from the result map.
* `set(K, V, int timeToLiveSeconds)` — unconditional write.
* `putIfAbsent(K, V, int ttl)` — conditional write. Returns the
  pre-existing value if any, else null.
* `remove(K)` — eviction.

`CASSupportingPersistenceCache<K, V>` extends with `getViaCas` (fetches
a `CASValue<V>` pairing the value with a version token) and `putViaCas`
(attempts an update keyed on the version; succeeds only if the stored
version is unchanged). Used for optimistic concurrency on vector
mutations. Not every backend supports it;
`requiresCopyOnDistributedCASOperation()` allows the backend to ask for a
deep copy before a CAS cycle.

=== CacheDomain

`CacheDomain<I, D>` is the factory from which a DAO obtains its logical caches:

* `getInstanceCache(Class<D>)` — single-durable cache, keyed by id.

* `getVectorCache(Class<D>)` — the backing store for `DurableVector`.

A cache backend such as Memcached ships a concrete `CacheDomain`
(`MemcachedCacheDomain`) plus Spring factory beans
(`MemcachedCacheDomainFactoryBean`, `MemcachedCacheDaoFactoryBean`).

=== DurableVector And Rosters

A `DurableVector<I, D>` is a serialisable, optionally-ordered, size-bounded
collection of durables stored under one cache key. It has a TTL and an
`isAlive()` check.

The concrete vector implementations live in `cache.praxis`. They split
along two axes:

* *Singular vs plural* — one element or many.
* *By-key vs by-reference* — whether the vector stores references to the
  durable (lazy, fetched through `OrmDaoManager` on access) or the
  durables themselves (eager).

The plural vectors are backed by `Roster<T>` — a serialisable `List`
with deque-style `addFirst`/`removeLast` operations. Two roster
implementations:

* `IntrinsicRoster` — doubly-linked-node structure of durables,
  used when the durables themselves are serialised into the cache.
* `ExtrinsicRoster` — same structure but of `DurableKey`s,
  used when only references are serialised.

A DAO picks a roster style by selecting its `CacheDao`
implementation — `ByKeyIntrinsicCacheDao`, `ByReferenceIntrinsicCacheDao`,
or `ByKeyExtrinsicCacheDao`. For most applications a Memcached-backed
extrinsic (key-only) roster is the right starting point: it stores less
data and always returns fresh durables at iteration time.

=== VectorKey

`VectorKey<D>` is the actual cache key the aspects compute:

[source]
----
<ClassName>:<namespace>[index1=value1,index2=value2]<classifier>
----

A `VectorKey` built from `Order`, namespace `orders.byCustomer`, key
`customerId=42`, and an empty classifier looks like:

[source]
----
Order:orders.byCustomer[customerId=42]
----

=== VectoredDao

`VectoredDao<I, D>` is the cache-facing interface `CacheAsAspect` and
`CachedWithAspect` call. It exposes:

* `get(Class<D>, List<DurableKey<I,D>>)` — bulk hydration from instance
  cache.
* `persist(Class<D>, D, UpdateMode)` — write with cache side-effects.
* `updateInVector(VectorKey, D)` and `removeFromVector(VectorKey, D)` —
  fine-grained vector mutations.
* `getVector(VectorKey)`, `persistVector(VectorKey, DurableVector)`,
  `deleteVector(VectorKey)` — whole-vector ops.
* `createSingularVector(VectorKey, D, int ttlSeconds)` and
  `createVector(VectorKey, Iterable<D>, Comparator<D>, int max, int ttl,
  boolean ordered)` — vector constructors called by `CacheAsAspect`.

`AbstractCacheDao` is the usual implementation, delegating to a
`CacheDomain`.

[[sql-pool]]
== SQL And Connection Pooling

=== DriverManagerDataSource

`org.smallmind.persistence.sql.DriverManagerDataSource` is a tiny
`DataSource` that loads a driver class in its constructor and calls
`DriverManager.getConnection(url, user, password)` on every
`getConnection()`. No pooling, no statement caching, no login-timeout
support. Useful for single-connection tools and for Liquibase migrations
(see `schema-hibernate.xml`).

=== Connection Pool

The real pool is `PooledDataSource`, a `DataSource` wrapping a
`ComponentPool<PooledConnection>` from `smallmind-quorum`:

[source,xml]
----
<bean id="schemaConnectionInstanceFactory"
      class="org.smallmind.persistence.sql.pool.DriverManagerComponentInstanceFactory">
  <constructor-arg index="0" value="${jdbc.driver.class_name.pool}"/>
  <constructor-arg index="1" value="${jdbc.url.pool}"/>
  <constructor-arg index="2" value="${jdbc.user.pool}"/>
  <constructor-arg index="3" value="${jdbc.password.pool}"/>
  <property name="validationQuery" value="${jdbc.validation_query.pool}"/>
</bean>

<bean id="schemaDBPool"
      class="org.smallmind.quorum.pool.complex.ComponentPool"
      init-method="startup" destroy-method="shutdown">
  <constructor-arg index="0" value="dbPool"/>
  <constructor-arg index="1" ref="schemaConnectionInstanceFactory"/>
  <property name="complexPoolConfig">
    <bean class="org.smallmind.quorum.pool.complex.ComplexPoolConfig">
      <property name="testOnAcquire"         value="${jdbc.pool.test_on_acquire.pool}"/>
      <property name="initialPoolSize"       value="${jdbc.pool.initial_size.poo}"/>
      <property name="minPoolSize"           value="${jdbc.pool.min_size.poo}"/>
      <property name="maxPoolSize"           value="${jdbc.pool.max_size.poo}"/>
      <property name="acquireWaitTimeMillis" value="750"/>
      <property name="creationTimeoutMillis" value="300"/>
      <property name="maxIdleTimeSeconds"    value="300"/>
      <property name="maxLeaseTimeSeconds"   value="1800"/>
    </bean>
  </property>
</bean>

<bean id="schemaPooledDataSource"
      class="org.smallmind.persistence.sql.pool.PooledDataSource">
  <constructor-arg index="0" value="schema"/>
  <constructor-arg index="1" ref="schemaDBPool"/>
</bean>
----

Key traits:

* Constructor arg 0 of `PooledDataSource` is the *registration key*. A
  call to `register()` places the bean in `DataSourceManager` under that
  key so `DataSourceManager.getDataSource("schema")` finds it.
* The pool is an active component: it has `startup` and `shutdown`
  methods, declared in Spring as `init-method`/`destroy-method`.
* `AbstractPooledDataSource` implements `ConnectionPoolDataSource` and
  exposes the pooled connection as a `javax.sql.PooledConnection`, which
  is what Hibernate and JPA expect.
* For XA, `PooledXADataSource` uses the same pool infrastructure but
  exposes an `XADataSource`.

The `sql.pool.context` package provides "contextual" variants
(`ContextualPooledDataSource`, `ContextualPooledXADataSource`) that pick
a pool by a thread-local key — useful when the same DAO method is
targeted at different shards depending on request context. The
`ContextualPoolNameTranslator` plugs in the name-resolution strategy;
`DefaultContextualPoolNameTranslator` joins the base name and the context
with a separator character, mapping a `null` context to the bare base name
and parsing that form back to `null` so a default, context-less pool
round-trips cleanly.

The `sql.pool.spring` package adds Spring factory beans that build a
`PooledDataSource` (or XA variant) directly:

* `PooledDataSourceFactoryBean` — builds a single pool-backed
  data source.
* `DynamicPooledDataSourceFactoryBean` /
  `DynamicPooledDataSourceInitializingBean` — discover `DatabaseConnection`
  beans by `ContextIndex` and build one pool per context.
* `DataSourceLocator` — application-scoped lookup by key.

=== DataSourceManager

`DataSourceManager` is a static registry analogous to `SessionManager`:
`DataSourceManager.register(key, dataSource)` places a
`CommonDataSource` into the per-application context, and
`DataSourceManager.getDataSource(key)` retrieves it. Used by runtime
components that need a data source but were not given one by
constructor injection.

=== DataSourceAvailableTestCondition

`org.smallmind.persistence.sql.testbench.DataSourceAvailableTestCondition`
implements `TestCondition` from `testbench-condition`. It polls a JDBC
datasource until a connection attempt succeeds, which is the normal
readiness gate when an integration test starts a containerized database.

The constructor accepts the same four arguments as `DriverManagerDataSource`:
`driverClassName`, `jdbcUrl`, `userName`, and `password`. Each `test()` call
attempts `getConnection()` on a `DriverManagerDataSource` built from those
coordinates. Success returns `null`. Failure returns a
`MessageTestConditionFailure` and prints a counter to standard output so
the waiting is visible in build logs.

`testbench-condition` is declared optional in the `persistence` POM.
Consumers that use `DataSourceAvailableTestCondition` must add it explicitly
with `test` scope.

[[sequences]]
== Sequence Generation

=== The Problem

Oracle, PostgreSQL, and most other relational databases provide a native
`SEQUENCE`. MySQL does not — the usual substitutes (`AUTO_INCREMENT` on a
surrogate column, `UUID()`, a separate counter table) each fail in some
meaningful dimension:

* `AUTO_INCREMENT` is column-local and cannot be shared.
* `UUID()` values are not monotonic, interact badly with clustered
  indexes, and cost more storage.
* A hand-written counter table is not atomic without explicit locking.

`smallmind-persistence.database` provides a portable abstraction
(`Sequence`) and a MySQL-specific implementation (`SimulatedSequence`)
that solves the counter-table problem correctly.

=== Sequence And SequenceManager

`Sequence` is abstract:

[source,java]
----
public abstract class Sequence {
  public abstract long nextLong (String name);
  public void register () { SequenceManager.register(this); }
}
----

`SequenceManager` holds exactly one `Sequence` globally, guarded by an
`AtomicReference`. The pattern is: at application startup, construct the
desired `Sequence` implementation, call `register()`, then everywhere
else call `SequenceManager.nextLong(name)` as a static.

`InMemorySequence` is a testing implementation: a
`ConcurrentHashMap<String, AtomicLong>` per name.
`SequenceManager.nextLong("orderNumber")` returns 1, 2, 3, ... within a
JVM run; it does *not* survive restart.

=== SimulatedSequence (MySQL)

`SimulatedSequence` is the production implementation for MySQL. It uses
a dedicated counter table:

[source,sql]
----
CREATE TABLE sequences (
  name     VARCHAR(64) PRIMARY KEY,
  next_val BIGINT NOT NULL
);
----

And atomically advances the counter using MySQL's `LAST_INSERT_ID()`
idiom:

[source,sql]
----
UPDATE sequences
   SET next_val = LAST_INSERT_ID(next_val + <incrementBy>)
 WHERE name = '<name>';

-- then retrieve: SELECT LAST_INSERT_ID();
----

The JDBC driver surfaces the result through `Statement.RETURN_GENERATED_KEYS`.
Because the update is atomic at the storage engine level, two concurrent
JVMs cannot collide.

The constructor takes `incrementBy`. When it is `1`, every `nextLong`
call hits the database. When it is larger, each database round-trip
reserves that many values; subsequent `nextLong` calls draw from the
local block until it is exhausted.

[source,java]
----
@Bean(initMethod = "register")
public SimulatedSequence applicationSequence (DataSource ds) {
  return new SimulatedSequence(ds, "sequences", 100);   <1>
}
----
<1> Block of 100 values per round-trip. A JVM crash loses the unallocated
    remainder of the block; that is acceptable because the block
    reservation has already committed, so no other process will reuse
    the discarded values.

`SimulatedSequenceDisasterException` is thrown if the counter table is
unreachable, if the update returned no key, or if any other SQL fault
surfaced during a sequence call. The exception is unchecked — callers
are expected to let it propagate and fail the request.

=== Usage

[source,java]
----
long orderNumber = SequenceManager.nextLong("orderNumber");
----

=== Aspect Weaving

None of this XML configures the AOP aspects. Aspects are applied at
build time by `aspectj-maven-plugin` in the consumer's POM. The
aspects' `@Aspect` classes live in `smallmind-persistence` and must be
on the classpath both at weave time and at runtime.

== Behavioral Reference

=== Thread And Session Model

`ProxySession` binds native sessions to the current thread through a
`ThreadLocal` held inside the implementation. `JPAProxySession` has one
`ThreadLocal<EntityManager>` and one `ThreadLocal<JPAProxyTransaction>`;
`ThrongProxySession` has a single session because a Throng client is
already thread-safe.

The aspect model assumes one thread per request. A request enters at a
`@Transactional` or `@NonTransactional` boundary, the boundary is pushed
onto the thread-local stack, the request body runs, the boundary pops
and closes its sessions. Asynchronous work that returns results on a
different thread must establish a new boundary on the worker thread —
the parent's boundary does not follow.

=== Session Stealing

If a method runs under `@NonTransactional` and then calls a method
annotated `@Transactional`, the inner call will request a native session
for the same source key. `TransactionalState.obtainBoundary` walks the
transaction stack first; if it finds a matching boundary it will raise
`StolenTransactionError` when the outer non-transactional boundary
already holds a session with the same source key. This is intentional —
silently promoting a non-transactional session to a transactional one
loses the non-transactional boundary's close-on-exit guarantee.

=== Cache Misses Are Never Failures

An unreachable cache, a serialisation mismatch, or a misconfigured
`CacheDomain` manifests as `CacheOperationException` *only* when the
underlying backend actually failed mid-call. A cache that returns null
for every key is a correct cache; `@CacheAs` will proceed to the ORM
and attempt to repopulate.

=== OrmDaoManager Registration Order

`ByKeySingularVector` and similar lazy-hydrating vectors call
`OrmDaoManager.get(Class)` at iteration time. The DAO must have called
`register()` before the first such call. In the canonical Spring wiring
that happens during context refresh, so a by-key vector read during a
servlet request is always safe. For standalone tooling that reads from
the cache outside a Spring context, either call `dao.register()`
manually or avoid by-key vectors entirely.

== Minimal Checklist

Before declaring persistence wired:

. A `DataSource` bean exists, pooled, and its `startup`/`shutdown`
  methods are bound to the Spring lifecycle.
. A `ProxySession` bean exists with `init-method="register"` and
  `boundaryEnforced=true`.
. Every application DAO extends `JPADao`, `ThrongDao`, or another
  `ORMDao` subclass, and calls `register()` (either via Spring
  `init-method="register"` or from a `@PostConstruct`).
. Every transactional code path is reachable only through a
  `@Transactional` or `@NonTransactional` boundary.
. `aspectj-maven-plugin` is present in the consumer's POM.
. If sequences are used, a `Sequence` is registered at startup.
. If the cache is used, a `CacheDomain` and a matching
  `VectoredDao`/`CacheDao` are wired, and the `ProxySession` is
  constructed with `cacheEnabled=true`.

== Limitations And Known Issues

Sessions are per-thread::
    Every `ProxySession` implementation binds to `ThreadLocal`. An
    asynchronous handoff that returns on a different thread must
    establish its own boundary. Virtual threads work if the AOP weave
    covers them, but boundaries still do not cross thread boundaries.

Cache backends must serialise durables::
    `PersistenceCache<String, DurableVector>` requires the vector (and
    its durables, in intrinsic mode) to serialise. Entities with
    non-serialisable fields (open streams, framework proxies) do not
    cache correctly. Prefer `ByKeyExtrinsicCacheDao` when only the
    durable's id needs to survive the trip.

No automatic cache invalidation across JVMs::
    `@CachedWith` updates are push-based: the persisting JVM writes the
    update into the shared cache. If a second JVM persists the same
    durable without going through a `@CachedWith` DAO, the first JVM
    will serve stale data until the TTL expires. Every write path that
    shares the cache must share the annotation.

`@CacheAs` does not batch misses::
    Two concurrent cache misses on the same key will both call the
    underlying method and both write the result; the second write
    overwrites the first. For expensive methods, combine `@CacheAs`
    with the caller's own `putIfAbsent`-style guard.

Simulated sequences lose unallocated blocks on crash::
    `SimulatedSequence` with `incrementBy > 1` reserves a block per
    round-trip. A JVM crash discards the remainder; the values are
    never reused. This is a feature (no reassignment) but a surprise
    if you assume sequence values are dense.

`MySQL` is the only `Simulated*` store::
    Other relational databases have native sequences or `IDENTITY`
    columns. Rolling a new `Sequence` subclass for a non-MySQL store
    is straightforward, but the shipped implementation is MySQL-only.
