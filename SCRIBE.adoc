[[scribe, Scribe]]
= Scribe

[partintro]
Scribe is Smallmind's logging library. It gives libraries a neutral logging API and lets applications choose the runtime endpoint explicitly. The core module is `scribe-pen`; concrete endpoint integrations live in `scribe-ink-*`; optional bridge modules route SLF4J and Apache Commons Logging into the same Scribe configuration.

Scribe is built around a few stable rules:

* reusable libraries depend on the API while applications choose the backend
* applications place exactly one endpoint implementation on the runtime classpath
* logging behavior is configured with templates, appenders, filters, enhancers, and formatters assembled in code or Spring beans
* loggers are selected by name and matched against explicit templates rather than an implicit logger hierarchy
* request metadata is attached explicitly through parameters and can be emitted by formatters, bridges, and structured sinks

Audience: developers wiring logging into reusable libraries or applications without surrendering backend, container, or configuration ownership.
Versioning: examples use `${smallmind.version}` for Smallmind artifacts; keep related Smallmind modules on the same version unless you have a specific reason to split them.

[[scribe-why-use, Why Use Scribe]]
== Why Use Scribe

Use Scribe when you want logging in shared libraries or applications with explicit endpoint selection and template-based configuration.

Scribe is most useful when these constraints matter:

* a reusable library can log while leaving endpoint selection to the application
* logging policy should be assembled explicitly in code or beans rather than hidden in a framework-specific runtime
* different APIs in the same process need to converge on one logging policy
* request metadata should be carried explicitly and emitted by text or structured outputs
* logger matching should be deliberate and reviewable rather than inherited through an implicit tree

Compared with using only SLF4J as a facade:

* Scribe is configured directly in code or Spring beans
* Scribe assembles logger behavior from explicit templates, appenders, filters, enhancers, and formatters
* Scribe can direct the output of many loggers to one template, or a choice of templates, based on logging parameters such as the logger name, so logger selection stays explicit
* Scribe uses `String.format` conventions for message formatting

[[scribe-installation, Installation]]
== Installation

=== Libraries

Reusable libraries should normally depend only on `scribe-pen`.

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>scribe-pen</artifactId>
  <version>${smallmind.version}</version>
</dependency>
----

Reusable libraries usually stop at `scribe-pen` so applications keep endpoint selection.

=== Applications

Applications normally include:

* `scribe-pen`
* exactly one `scribe-ink-*` endpoint integration
* optionally `scribe-slf4j` and/or `scribe-apache` if those APIs must route through Scribe

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>scribe-pen</artifactId>
  <version>${smallmind.version}</version>
</dependency>

<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>scribe-ink-indigenous</artifactId>
  <version>${smallmind.version}</version>
</dependency>
----

If SLF4J traffic should land in Scribe:

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>scribe-slf4j</artifactId>
  <version>${smallmind.version}</version>
</dependency>
----

If Apache Commons Logging traffic should land in Scribe:

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>scribe-apache</artifactId>
  <version>${smallmind.version}</version>
</dependency>
----

[IMPORTANT]
====
There must be exactly one `LoggingBlueprint` implementation on the runtime classpath. In practice that means exactly one `scribe-ink-*` endpoint integration.

If no endpoint is present, `LoggingBlueprintFactory` fails during static initialization. If more than one endpoint is present, it fails and lists the conflicting implementations.
====

=== Common Classpath Mistakes

The most common installation errors are:

* adding `scribe-pen` and bridge modules but no endpoint module
* adding more than one `scribe-ink-*` module
* assuming `scribe-slf4j` or `scribe-apache` replaces the need for an endpoint module
* registering no templates, then expecting custom levels, appenders, or context behavior

[[scribe-quick-start, Quick Start]]
== Quick Start

The smallest useful setup is:

1. add `scribe-pen`
2. add one endpoint module such as `scribe-ink-indigenous`
3. register one or more templates
4. obtain loggers through `LoggerManager`

[source,java]
----
import java.time.format.DateTimeFormatter;

import org.smallmind.scribe.pen.ConsoleAppender;
import org.smallmind.scribe.pen.DateFormatTimestamp;
import org.smallmind.scribe.pen.DefaultTemplate;
import org.smallmind.scribe.pen.Level;
import org.smallmind.scribe.pen.Logger;
import org.smallmind.scribe.pen.LoggerManager;
import org.smallmind.scribe.pen.PatternFormatter;

DateFormatTimestamp timestamp = new DateFormatTimestamp(
    DateTimeFormatter.ofPattern("yyyy-MM-dd'T'HH:mm:ss.SSSZ"));
PatternFormatter formatter = new PatternFormatter(
    timestamp,
    "%d %n %+5l (%.1C.%M:%L) [%T] - %m%!+\n\t!p%!+\n\t!s");
ConsoleAppender consoleAppender = new ConsoleAppender(formatter);

new DefaultTemplate(Level.INFO, true, consoleAppender).register();

Logger logger = LoggerManager.getLogger(MyService.class);

logger.putParameter("requestId", "ab12");
try {
  logger.info("Started %s on port %d", "MyService", 8080);
  logger.warn(new IllegalStateException("example"), "Something happened");
} finally {
  logger.clearParameters();
}
----

What this does:

* installs one baseline template for all loggers
* enables automatic caller-context capture for that template
* writes text output to the console
* emits request-scoped metadata through parameters

For tests, `TestLoggerConfiguration.setup()` installs a DEBUG console template using the same verbose pattern used throughout the module:

[source,java]
----
import org.smallmind.scribe.pen.TestLoggerConfiguration;

TestLoggerConfiguration.setup();
----

[[scribe-core-concepts, Core Model]]
== Core Model

=== How A Log Event Flows

At runtime the normal flow is:

1. `LoggerManager.getLogger(...)` returns a cached logger by name.
2. The logger is associated with the single strongest matching template.
3. A logging call delegates to the active endpoint's `LoggerAdapter`.
4. Logger-level filters may veto the event.
5. Enhancers can mutate the record before publication.
6. Appenders publish the record, optionally applying their own filters and error handlers.

That model matters because template choice is explicit and singular. Scribe does not merge logger behavior through an inherited tree.

=== LoggerManager

`LoggerManager` is the normal entry point for application and library code.

Use one of these:

[source,java]
----
Logger applicationLogger = LoggerManager.getLogger(MyService.class);
Logger namedLogger = LoggerManager.getLogger("com.acme.billing.SqlTracer");
----

`LoggerManager` does three things that direct construction does not:

* caches loggers by name so callers share the same logger instance
* associates loggers with the best matching template
* re-associates already-created loggers when templates are registered or removed later

=== Logger

`Logger` is the public facade used at call sites.

Each level exposes overloads for:

* formatted string plus arguments
* `Throwable`
* `Throwable` plus formatted string plus arguments
* arbitrary `Object`
* `Supplier<String>`
* the generic `log(Level, ...)` variants

Example:

[source,java]
----
Logger logger = LoggerManager.getLogger(OrderService.class);

logger.debug("Loaded %d order rows", rowCount);
logger.info(orderSummary);
logger.warn(new IllegalStateException("unexpected"), "Order %s needs attention", orderId);
logger.error(() -> expensiveDiagnosticMessage());
----

[IMPORTANT]
====
Direct construction with `new Logger(...)` exists, but it does not place that logger in `LoggerManager`'s cache and does not automatically associate it with registered templates.

Use `LoggerManager.getLogger(...)` unless you intentionally want a manually managed logger.
====

=== Message Formatting Semantics

Direct Scribe logging uses `String.format`-style placeholders:

[source,java]
----
logger.info("Started %s on port %d", serviceName, port);
----

That means:

* `%s`, `%d`, and the rest of `String.format` apply to direct Scribe calls
* invalid format strings raise `MessageFormattingException`
* `logger.info(object)` uses `object.toString()`
* `logger.log(null, ...)` falls back to the logger's current level

The SLF4J bridge is different. Bridged SLF4J calls accept `{}` placeholders because `scribe-slf4j` translates them before calling Scribe.

[source,java]
----
logger.info("Started %s", serviceName);         // direct Scribe API
slf4jLogger.info("Started {}", serviceName);    // bridged SLF4J API
----

`Supplier<String>` is useful when building the message itself is expensive. The supplier is evaluated only after the call passes the logger's level threshold. Do not rely on suppliers as a guarantee that every downstream filter or backend path will skip evaluation; exact timing depends on the active endpoint.

=== Levels

Scribe levels are ordered:

[source,text]
----
TRACE < DEBUG < INFO < WARN < ERROR < FATAL < OFF
----

The logger's level acts as a threshold. For example, a logger at `INFO` admits `INFO`, `WARN`, `ERROR`, and `FATAL`, but not `DEBUG` or `TRACE`. `OFF` disables output.

[[scribe-template-model, Template Model]]
== Template Model

Templates are Scribe's configuration unit. A template carries:

* a level
* the `autoFillLoggerContext` setting
* zero or more filters
* zero or more appenders
* zero or more enhancers

When a logger matches a template, that template's settings are pushed into the logger. Only one template wins for a given logger.

=== Matching Rules

The built-in template types are:

[cols="2,4,2",options="header"]
|===
|Template |Matches |Priority Notes

|`DefaultTemplate`
|all logger names
|baseline catch-all

|`ClassNameTemplate`
|dot-notated logger names using Smallmind dot notation
|specific patterns score higher than broader ones

|`PersonalizedTemplate`
|one exact logger name
|highest priority

|`RegExTemplate`
|logger names matching a regular expression
|highest priority
|===

Matching rules that matter in practice:

* the highest match score wins
* only one template applies to a logger at a time
* template changes made after `register()` propagate to already-associated loggers
* equal strongest scores are a bad idea; tied winners depend on internal set iteration order

=== Dot-Notation Patterns

`ClassNameTemplate` uses Smallmind dot notation, which is designed for Java-style logger names. The common patterns are:

* `com.acme.billing.InvoiceService` for one exact class name
* `com.acme.billing.?` for one segment at that position
* `com.acme.billing.*` for a broad package-tree style match

If the pattern is malformed, `ClassNameTemplate` throws `LoggerException`.

=== Typical Template Layout

Most applications end up with:

* one `DefaultTemplate`
* a small number of targeted `ClassNameTemplate` overrides
* an occasional `PersonalizedTemplate` for one noisy or critical logger

[source,java]
----
import org.smallmind.scribe.pen.Appender;
import org.smallmind.scribe.pen.ClassNameTemplate;
import org.smallmind.scribe.pen.ConsoleAppender;
import org.smallmind.scribe.pen.DefaultTemplate;
import org.smallmind.scribe.pen.Level;
import org.smallmind.scribe.pen.PatternFormatter;
import org.smallmind.scribe.pen.PersonalizedTemplate;

Appender consoleAppender = new ConsoleAppender(new PatternFormatter());

new DefaultTemplate(Level.INFO, false, consoleAppender).register();
new ClassNameTemplate(Level.DEBUG, true, "com.acme.checkout.*", consoleAppender).register();

PersonalizedTemplate tracerTemplate = new PersonalizedTemplate(Level.TRACE, true, "com.acme.checkout.SqlTracer");
tracerTemplate.setAppender(consoleAppender);
tracerTemplate.register();
----

That layout means:

* everything logs at `INFO` by default
* `com.acme.checkout.*` logs at `DEBUG` and captures caller context
* `com.acme.checkout.SqlTracer` gets an exact `TRACE` override

[IMPORTANT]
====
Templates replace logger configuration; they do not merge with weaker templates. If a more specific template wins, its filters, appenders, enhancers, level, and context setting become the logger's active configuration.
====

=== Live Template Updates

Once a template is registered, later mutations propagate to currently associated loggers.

[source,java]
----
DefaultTemplate defaultTemplate = new DefaultTemplate(Level.INFO, false, consoleAppender);
defaultTemplate.register();

defaultTemplate.setLevel(Level.WARN);
----

That updates all loggers currently associated with that template.

[[scribe-appenders, Appenders]]
== Appenders

An `Appender` publishes records to a destination. Appenders can also carry their own filters, an error handler, and an active/inactive flag.

[cols="2,3,4",options="header"]
|===
|Appender |Use It For |Notes

|`ConsoleAppender`
|simple console output
|writes to `System.out`

|`FileAppender`
|local file output
|UTF-8 output, parent directories created automatically, optional rollover and cleanup

|`AsynchronousAppender`
|protecting callers from slow sinks
|wraps another appender with a bounded queue and worker threads

|`EmailAppender`
|SMTP alarm or escalation output
|sends each formatted record as an email body

|`SyslogAppender`
|structured syslog output
|requires `syslog4j`; call `afterPropertiesSet()` outside Spring

|`AWSAppender`
|CloudWatch Logs output
|requires AWS SDK modules; keep one instance per JVM per stream

|`FluentBitAppender`
|Fluent Bit forward-protocol output
|requires MessagePack/Jackson; call `afterPropertiesSet()` outside Spring
|===

=== Formatted Appenders Need A Formatter

`ConsoleAppender`, `FileAppender`, and `EmailAppender` extend `AbstractFormattedAppender`. They need a `Formatter` before they can publish anything.

This works:

[source,java]
----
ConsoleAppender consoleAppender = new ConsoleAppender(new PatternFormatter());
----

This compiles but is not usable until a formatter is supplied:

[source,java]
----
ConsoleAppender consoleAppender = new ConsoleAppender();
consoleAppender.setFormatter(new PatternFormatter());
----

If a formatted appender publishes without a formatter, it fails with `LoggerException`.

=== ConsoleAppender

`ConsoleAppender` is the simplest destination:

[source,java]
----
ConsoleAppender consoleAppender = new ConsoleAppender(new PatternFormatter());
----

It writes to `System.out`, not `System.err`.

=== FileAppender

`FileAppender` writes UTF-8 text to a file and creates parent directories automatically.

The simple case:

[source,java]
----
FileAppender fileAppender = new FileAppender("var/log/service.log", new PatternFormatter());
----

The production case is usually a `FileAppender` plus `Rollover` and `Cleanup`; see <<scribe-rollover-cleanup,File Rollover And Cleanup>>.

`FileAppender` holds an open stream. Close it at shutdown.

=== AsynchronousAppender

`AsynchronousAppender` wraps another appender and publishes through a bounded queue.

[source,java]
----
Appender asyncAppender = new AsynchronousAppender(fileAppender, 400, 2);
----

Defaults:

* constructor with one argument uses buffer size `1` and concurrency `1`
* if the queue is full, publication is rejected and the appender's error handling path is used
* `close()` stops workers and then closes the wrapped appender

Use it when the destination is slower than the calling thread should tolerate.

=== EmailAppender

`EmailAppender` sends the formatted output of each record as an email body.

[source,java]
----
EmailAppender emailAppender = new EmailAppender("smtp.acme.internal", 25);
emailAppender.setFrom("alerts@acme.internal");
emailAppender.setTo("ops@acme.internal");
emailAppender.setSubject("Billing Alert");
emailAppender.setFormatter(new PatternFormatter());
----

This is best reserved for low-volume alerting paths, not general application logging.

=== SyslogAppender

`SyslogAppender` emits structured syslog data through `syslog4j`.

[source,java]
----
SyslogAppender syslogAppender = new SyslogAppender();
syslogAppender.setSyslogHost("syslog.acme.internal");
syslogAppender.setSyslogPort(514);
syslogAppender.setFacility("LOCAL7");
syslogAppender.afterPropertiesSet();
----

It includes logger name, timestamp, thread data, context, parameters, and stack trace information in structured fields when present. `TRACE` maps to syslog debug.

If you set `base64EncodeStackTraces` to `true`, stack traces are encoded before publication.

=== AWSAppender

`AWSAppender` publishes formatted text to CloudWatch Logs.

Use it only when you have clear ownership of the CloudWatch stream and sequence-token lifecycle.

[IMPORTANT]
====
Treat `AWSAppender` as a singleton per JVM per CloudWatch log stream. Sequence tokens are serialized inside the appender.

Pass an explicit `streamName`. That is the least surprising way to use the current API.
====

`AWSAppender` does not close the `CloudWatchLogsClient` you pass in. If your application owns that client, close it separately.

=== FluentBitAppender

`FluentBitAppender` batches records and ships them to Fluent Bit over TCP using MessagePack.

[source,java]
----
import java.util.HashMap;

import org.smallmind.scribe.pen.ConsoleAppender;
import org.smallmind.scribe.pen.DefaultErrorHandler;
import org.smallmind.scribe.pen.PatternFormatter;
import org.smallmind.scribe.pen.fluentbit.FluentBitAppender;

HashMap<String, String> additionalEventData = new HashMap<>();
additionalEventData.put("stream", "service_log");
additionalEventData.put("service", "billing");

FluentBitAppender fluentBitAppender = new FluentBitAppender(
    "billing-service",
    new DefaultErrorHandler(new ConsoleAppender(new PatternFormatter())));
fluentBitAppender.setHost("127.0.0.1");
fluentBitAppender.setPort(24224);
fluentBitAppender.setBatchSize(25);
fluentBitAppender.setConcurrencyLimit(2);
fluentBitAppender.setAdditionalEventData(additionalEventData);
fluentBitAppender.afterPropertiesSet();
----

Defaults and behaviors:

* `retryAttempts` defaults to `3`
* `concurrencyLimit` defaults to `1`
* `batchSize` defaults to `1`
* `batchGracePeriodMilliseconds` defaults to `3000`
* `close()` waits for worker completion

Outside Spring, call `afterPropertiesSet()` before publishing.

[[scribe-formatters, Formatters]]
== Formatters

Formatters turn a `Record` into output text. They matter most for `AbstractFormattedAppender` subclasses such as `ConsoleAppender`, `FileAppender`, and `EmailAppender`.

=== PatternFormatter

`PatternFormatter` is the general-purpose text formatter. Every formatted record ends with the platform line separator.

Constructor defaults:

* `new PatternFormatter()` uses `DateFormatTimestamp.getDefaultInstance()`
* `new PatternFormatter()` uses format `%d %n %+5l [%T] - %m`

General field shape:

[source,text]
----
[{header]%[+|-width][.precision][!(+|-)prefix!]conversion[footer}]
----

Supported conversions:

[cols="1,4",options="header"]
|===
|Code |Meaning

|`d`
|formatted date using the configured `Timestamp`

|`t`
|epoch milliseconds

|`n`
|logger name

|`l`
|level

|`m`
|message

|`T`
|thread name

|`C`
|calling class

|`M`
|calling method

|`N`
|native-method flag

|`L`
|line number

|`F`
|file name

|`s`
|stack trace

|`p`
|parameters
|===

Important formatting notes:

* precision on `n` and `C` keeps the right-most dot-notated segments
* precision on `p` limits the number of parameter lines
* `%%` emits a literal percent sign
* `\n` and `\t` are recognized in literal and prefix text
* headers and footers are emitted only when the field has a value
* `s` and `p` are multi-line fields and ignore width/padding

The verbose pattern used throughout this module is:

[source,text]
----
%d %n %+5l (%.1C.%M:%L) [%T] - %m%!+\n\t!p%!+\n\t!s
----

That pattern gives you:

* a timestamp
* the full logger name
* a padded level
* short caller context
* thread name
* message text
* parameters and stack trace on separate indented lines when present

=== XMLFormatter

`XMLFormatter` emits structured XML and lets you choose which `RecordElement` values to include.

Defaults:

* `timestamp` is `DateFormatTimestamp.getDefaultInstance()`
* `newLine` is `System.getProperty("line.separator")`
* `cdata` is `false`
* `indent` is `3`
* `recordElements` defaults to all `RecordElement` values

Example:

[source,java]
----
import java.time.format.DateTimeFormatter;

import org.smallmind.scribe.pen.DateFormatTimestamp;
import org.smallmind.scribe.pen.RecordElement;
import org.smallmind.scribe.pen.XMLFormatter;

XMLFormatter xmlFormatter = new XMLFormatter();
xmlFormatter.setTimestamp(new DateFormatTimestamp(
    DateTimeFormatter.ofPattern("yyyy-MM-dd'T'HH:mm:ss.SSSZ")));
xmlFormatter.setCdata(true);
xmlFormatter.setRecordElements(new RecordElement[] {
    RecordElement.DATE,
    RecordElement.LEVEL,
    RecordElement.MESSAGE,
    RecordElement.PARAMETERS,
    RecordElement.STACK_TRACE
});
----

`RecordElement` values are:

* `DATE`
* `MILLISECONDS`
* `LOGGER_NAME`
* `LEVEL`
* `MESSAGE`
* `THREAD`
* `LOGGER_CONTEXT`
* `PARAMETERS`
* `STACK_TRACE`

=== Timestamp Implementations

[cols="2,4",options="header"]
|===
|Type |Use

|`DateFormatTimestamp`
|default choice; formats with `DateTimeFormatter`, default pattern `yyyy-MM-dd'T'HH:mm:ss.SSSZ`

|`FormatStringTimestamp`
|uses `String.format(...)` conventions for date rendering

|`NullTimestamp`
|returns an empty string; useful when the sink adds its own timestamp
|===

[[scribe-filters, Filters And Enhancers]]
== Filters And Enhancers

Filters decide whether a record should be logged. Enhancers mutate a record before publication.

Filters can appear in two places:

* on a logger, usually through a template
* on an appender

Both are honored. A record must survive every configured filter path to reach the destination.

=== LevelFilter

`LevelFilter` admits records at or above a threshold:

[source,java]
----
LevelFilter warnAndAbove = new LevelFilter(Level.WARN);
----

This is usually more useful on an appender than on a logger when you want one sink to be stricter than another.

=== DotNotatedLoggerNameFilter

`DotNotatedLoggerNameFilter` is useful when you want dynamic debug-category style control across a logger-name tree.

Its model is:

* records at or above `passThroughLevel` always pass
* lower-level records pass only when the logger name matches one of the configured dot-notation patterns

[source,java]
----
import java.util.Arrays;

import org.smallmind.scribe.pen.ConsoleAppender;
import org.smallmind.scribe.pen.DotNotatedLoggerNameFilter;
import org.smallmind.scribe.pen.Level;
import org.smallmind.scribe.pen.PatternFormatter;

DotNotatedLoggerNameFilter filter = new DotNotatedLoggerNameFilter(
    Level.INFO,
    Arrays.asList("com.acme.checkout.*", "com.acme.risk.*"));

ConsoleAppender consoleAppender = new ConsoleAppender(new PatternFormatter());
consoleAppender.addFilter(filter);
----

At runtime you can add or remove patterns with `setDebugCategory(...)`.

=== ExceptionSuppressingLogFilter

`ExceptionSuppressingLogFilter` drops records carrying specific throwable classes.

[source,java]
----
import java.net.SocketTimeoutException;
import java.util.Arrays;

import org.smallmind.scribe.pen.ExceptionSuppressingLogFilter;

ExceptionSuppressingLogFilter.addSuppressedThrowableClasses(
    Arrays.asList(SocketTimeoutException.class));
----

[IMPORTANT]
====
`ExceptionSuppressingLogFilter` matches exact throwable classes, not subclasses.
====

=== Enhancer

Enhancers are a lower-level hook than parameters. Use them when you need to mutate the record before it reaches the appender.

For ordinary request metadata, `Logger.putParameter(...)` is simpler. Use an enhancer when you genuinely need record mutation logic.

Example:

[source,java]
----
import java.util.Arrays;

import org.smallmind.scribe.pen.Enhancer;
import org.smallmind.scribe.pen.Parameter;
import org.smallmind.scribe.pen.ParameterAwareRecord;
import org.smallmind.scribe.pen.Record;

Enhancer serviceEnhancer = new Enhancer() {
  @Override
  public void enhance (Record<?> record) {

    if (record instanceof ParameterAwareRecord<?>) {

      ParameterAwareRecord<?> parameterAwareRecord = (ParameterAwareRecord<?>)record;
      Parameter[] currentParameters = parameterAwareRecord.getParameters();
      Parameter[] enhancedParameters = Arrays.copyOf(currentParameters, currentParameters.length + 1);

      enhancedParameters[currentParameters.length] = new Parameter("service", "billing");
      parameterAwareRecord.setParameters(enhancedParameters);
    }
  }
};
----

[[scribe-parameters, Parameters And Logger Context]]
== Parameters And Logger Context

=== Parameters

Parameters are explicit key/value metadata attached to log records.

Use them for:

* request ids
* tenant ids
* partner ids
* workflow or batch ids
* temporary operational metadata you do not want duplicated in every message string

Example request lifecycle:

[source,java]
----
Logger logger = LoggerManager.getLogger(OrderResource.class);

logger.putParameter("requestId", requestId);
logger.putParameter("tenant", tenantId);
try {
  logger.info("Accepted order %s", orderId);
} finally {
  logger.clearParameters();
}
----

Convenience methods on `Logger`:

* `putParameter(String, Serializable)`
* `removeParameter(String)`
* `clearParameters()`
* `getParameters()`

Technical details:

* parameter storage is provided by `org.smallmind.scribe.pen.adapter.Parameters`
* the storage implementation uses `InheritableThreadLocal`
* SLF4J MDC is backed by the same storage when you use `scribe-slf4j`

[WARNING]
====
Treat thread boundaries as explicit context boundaries. Parameter storage uses Java's default `InheritableThreadLocal` behavior, so worker threads and asynchronous callbacks should clear or re-establish context deliberately instead of assuming a fresh empty parameter map.
====

=== Logger Context

Logger context is the captured caller location: class, method, file, line number, and native-method flag.

It is disabled by default. Enable it by setting `autoFillLoggerContext` on a template or logger:

[source,java]
----
DefaultTemplate defaultTemplate = new DefaultTemplate(Level.INFO, true, consoleAppender);
defaultTemplate.register();
----

When enabled, Scribe uses `DefaultLoggerContext` to inspect the current stack and fill:

* class name
* method name
* file name
* line number
* native-method flag

Pattern conversions that use logger context are `C`, `M`, `N`, `L`, and `F`.

Because stack inspection has cost, keep caller-context autofill targeted when you care about hot-path performance.

[[scribe-rollover-cleanup, File Rollover And Cleanup]]
== File Rollover And Cleanup

`FileAppender` can rotate files through `Rollover` and prune old archives through `Cleanup`.

The normal production shape is:

[source,java]
----
import java.nio.file.Path;
import java.nio.file.Paths;
import java.time.format.DateTimeFormatter;
import java.util.concurrent.TimeUnit;

import org.smallmind.nutsnbolts.time.Stint;
import org.smallmind.scribe.pen.Appender;
import org.smallmind.scribe.pen.AsynchronousAppender;
import org.smallmind.scribe.pen.Cleanup;
import org.smallmind.scribe.pen.ConsoleAppender;
import org.smallmind.scribe.pen.DateFormatTimestamp;
import org.smallmind.scribe.pen.DefaultErrorHandler;
import org.smallmind.scribe.pen.DefaultTemplate;
import org.smallmind.scribe.pen.FileAppender;
import org.smallmind.scribe.pen.FileSizeQuantifier;
import org.smallmind.scribe.pen.FileSizeRolloverRule;
import org.smallmind.scribe.pen.LastModifiedCleanupRule;
import org.smallmind.scribe.pen.Level;
import org.smallmind.scribe.pen.PatternFormatter;
import org.smallmind.scribe.pen.Rollover;
import org.smallmind.scribe.pen.TimestampQuantifier;
import org.smallmind.scribe.pen.TimestampRolloverRule;

Path logPath = Paths.get("var", "log", "billing.log");

DateFormatTimestamp archiveTimestamp = new DateFormatTimestamp(
    DateTimeFormatter.ofPattern("yyyy-MM-dd"));
DateFormatTimestamp lineTimestamp = new DateFormatTimestamp(
    DateTimeFormatter.ofPattern("yyyy-MM-dd'T'HH:mm:ss.SSSZ"));
PatternFormatter formatter = new PatternFormatter(
    lineTimestamp,
    "%d %n %+5l (%.1C.%M:%L) [%T] - %m%!+\n\t!p%!+\n\t!s");

Rollover rollover = new Rollover(
    archiveTimestamp,
    '.',
    new TimestampRolloverRule(TimestampQuantifier.TOP_OF_DAY),
    new FileSizeRolloverRule(250, FileSizeQuantifier.MEGABYTES));
Cleanup cleanup = new Cleanup(
    '.',
    new LastModifiedCleanupRule(new Stint(14, TimeUnit.DAYS)));

FileAppender fileAppender = new FileAppender(
    logPath,
    rollover,
    cleanup,
    formatter,
    new DefaultErrorHandler(new ConsoleAppender(formatter)));
Appender asyncAppender = new AsynchronousAppender(fileAppender, 400, 2);

new DefaultTemplate(Level.INFO, true, asyncAppender).register();
----

With that setup a file such as `billing.log` can roll to names like `billing.2026-04-11.0.log`.

=== Rollover

`Rollover` combines:

* a `Timestamp` used to build archive suffixes
* a separator character
* one or more `RolloverRule` instances

If any configured rule says rollover should happen, the file rotates before the current write.

=== Rollover Rules

[cols="2,4",options="header"]
|===
|Rule |Behavior

|`FileSizeRolloverRule`
|rolls when the current file size plus pending bytes exceeds the configured size threshold

|`TimestampRolloverRule`
|rolls when the current time crosses a configured boundary such as top-of-day or top-of-hour
|===

`FileSizeRolloverRule` defaults to `10` megabytes when constructed with no arguments.

`TimestampRolloverRule` defaults to `TOP_OF_DAY`.

`TimestampQuantifier` values are:

* `TOP_OF_MINUTE`
* `TOP_OF_HOUR`
* `HALF_DAY`
* `TOP_OF_DAY`
* `TOP_OF_WEEK`
* `TOP_OF_MONTH`

=== Cleanup

`Cleanup` examines adjacent rollover files after a rollover happens and deletes the ones matching its rules.

Cleanup is not a background sweeper. It runs only when rollover occurs.

=== Cleanup Rules

[cols="2,4",options="header"]
|===
|Rule |Behavior

|`FileCountCleanupRule`
|retains up to a configured maximum number of archive files

|`LastModifiedCleanupRule`
|deletes archive files older than the configured `Stint`
|===

[IMPORTANT]
====
Use the same separator for `Cleanup` that you use for `Rollover`. Both components identify archive files by the same naming convention.

If the separators do not match, cleanup will not see the archive files you expect it to prune.
====

[[scribe-error-handling, Error Handling]]
== Error Handling

Appenders can have an `ErrorHandler`. If an appender has no error handler, the default behavior is to print the throwable stack trace.

=== DefaultErrorHandler

`DefaultErrorHandler` republishes failures through a backup appender.

Defaults:

* `new DefaultErrorHandler()` uses `new ConsoleAppender(new XMLFormatter())`
* when handling a failure tied to an existing record, it publishes an error record and then republishes the original record through the backup appender

Common pattern:

[source,java]
----
PatternFormatter formatter = new PatternFormatter();
ConsoleAppender consoleFallback = new ConsoleAppender(formatter);
DefaultErrorHandler errorHandler = new DefaultErrorHandler(consoleFallback);
----

This is a good fit for file or network appenders where the failure path should still be visible somewhere obvious.

=== Custom ErrorHandler

Implement `ErrorHandler` when you need an application-specific failure policy:

* emit alerts
* count logging failures
* reroute to a platform-specific fallback
* suppress or aggregate repeated appender failures

[[scribe-integrations, Built-In Endpoint Integrations]]
== Built-In Endpoint Integrations

Choose exactly one endpoint module.

[cols="2,3,4",options="header"]
|===
|Artifact |Use It When |Notes

|`scribe-ink-indigenous`
|you want Scribe to own logging behavior end-to-end
|least surprising choice when your appenders and formatters are mostly Scribe-native

|`scribe-ink-jdk`
|you intentionally want Scribe to participate in `java.util.logging`
|template changes push levels, handlers, and filters into JUL loggers

|`scribe-ink-log4j`
|you intentionally want Scribe to participate in Log4J2
|template changes push levels and appenders into Log4J2 loggers; the adapter sets `additive` to `false`
|===

=== Indigenous Endpoint

`scribe-ink-indigenous` is the simplest endpoint when Scribe is your primary logging model.

Characteristics:

* records are Scribe-native
* parameters come from `Parameters`
* templates directly control the logger's level, appenders, filters, and enhancers

If you want Scribe's appenders, formatters, and template model without backend interop concerns, this is usually the easiest endpoint to reason about.

=== JDK Endpoint

`scribe-ink-jdk` adapts Scribe to `java.util.logging`.

Important consequences:

* template-applied levels are pushed into JUL loggers
* template-applied appenders are wrapped as JUL `Handler` instances
* template-applied filters are checked in the JUL path as well as Scribe's path
* JUL native `Formatter`, `Filter`, and `ErrorManager` types can be adapted through the provided wrapper/adaptor classes

Use this endpoint when JUL interop is the goal, not when you want Scribe isolated from JUL behavior.

=== Log4J2 Endpoint

`scribe-ink-log4j` adapts Scribe to Log4J2 core.

Important consequences:

* template-applied levels are pushed into Log4J2 loggers
* template-applied appenders are wrapped as Log4J2 appenders
* the adapter sets the underlying logger to non-additive mode
* Log4J2 `Layout`, `Filter`, and `ErrorHandler` types can be adapted through the provided wrapper/adaptor classes

This endpoint is appropriate when Scribe calls must join an existing Log4J2-centered system deliberately.

[[scribe-bridges, Logging API Bridges]]
== Logging API Bridges

Bridge modules route other logging APIs into Scribe. They do not replace the need for one actual endpoint module.

=== Apache Commons Logging

Artifact: `org.smallmind:scribe-apache`

The bridge ships `commons-logging.properties` so Commons Logging resolves `org.smallmind.scribe.apache.CommonsLogWrapper`.

Effects:

* Commons Logging logger names are resolved through `LoggerManager`
* Commons Logging levels map directly to Scribe levels
* object and throwable overloads delegate to the corresponding Scribe methods

=== SLF4J

Artifact: `org.smallmind:scribe-slf4j`

This module supplies:

* `org.smallmind.scribe.slf4j.ScribeLoggerFactory`
* `org.smallmind.scribe.slf4j.ScribeLoggerAdapter`
* `org.smallmind.scribe.slf4j.ScribeMDCAdapter`
* `org.slf4j.impl.StaticLoggerBinder`
* `org.slf4j.impl.StaticMDCBinder`

Effects:

* SLF4J logger names map to Scribe logger names of the same value
* `{}` placeholders are translated to `%s` before delegating into Scribe
* SLF4J MDC is backed by `Parameters`
* deque-based MDC operations are supported

Example:

[source,java]
----
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.slf4j.MDC;

Logger logger = LoggerFactory.getLogger(OrderService.class);

MDC.put("requestId", requestId);
try {
  logger.info("Accepted order {}", orderId);
} finally {
  MDC.clear();
}
----

If your Scribe formatter includes `%p`, or if your structured sink includes parameters, the `requestId` value appears there.

[[scribe-spring-support, Spring Support]]
== Spring Support

Spring support lives in `scribe-pen` and is optional. Add `spring-beans` only if you use these helpers.

[cols="2,4",options="header"]
|===
|Type |Purpose

|`DateTimeFormatterFactoryBean`
|builds `DateTimeFormatter` instances from pattern strings in Spring XML

|`TemplateInitializingBean`
|registers a supplied list of templates at initialization time

|`DynamicClassNameTemplateInitializingBean`
|scans Spring properties for `log.pattern.<key>` and `log.level.<key>` pairs and registers `ClassNameTemplate` instances

|`ExceptionSuppressingLogInitializingBean`
|registers suppressed throwable classes for `ExceptionSuppressingLogFilter`

|`ConsoleLoggingPlan`
|creates a console appender plus templates with an asynchronous wrapper

|`FileLoggingPlan`
|creates a rolling file appender plus templates with an asynchronous wrapper

|`FluentBitLoggingPlan`
|creates a configured `FluentBitAppender` plus templates with an asynchronous wrapper
|===

`LoggingPlan` defaults:

* `defaultLogLevel = INFO`
* `logRecordBufferSize = 400`
* `concurrencyLimit = 1`
* all plan-created templates set `autoFillLoggerContext = true`

[IMPORTANT]
====
Set `logPath` explicitly on `FileLoggingPlan`. Its field default is `/var/log`, which is usually a directory and therefore not a valid `FileAppender` target path.
====

=== Example Spring XML

Using `FileLoggingPlan`:

[source,xml]
----
<bean class="org.smallmind.scribe.pen.spring.plan.FileLoggingPlan">
  <property name="logPath" value="/var/log/billing/service.log"/>
  <property name="defaultLogLevel" value="INFO"/>
  <property name="rolloverMegabyteLimit" value="250"/>
  <property name="retentionDays" value="14"/>
  <property name="logs">
    <array>
      <bean class="org.smallmind.scribe.pen.spring.plan.Log">
        <property name="pattern" value="com.acme.billing.sql.*"/>
        <property name="level" value="DEBUG"/>
      </bean>
    </array>
  </property>
</bean>
----

Property-driven `ClassNameTemplate` registration:

[source,properties]
----
log.pattern.billing=com.acme.billing.*
log.level.billing=DEBUG
log.pattern.partner=com.acme.partner.*
log.level.partner=INFO
----

[source,xml]
----
<bean class="org.smallmind.scribe.pen.spring.DynamicClassNameTemplateInitializingBean">
  <property name="appenders">
    <array>
      <ref bean="consoleAppender"/>
    </array>
  </property>
  <property name="autoFillLoggerContext" value="true"/>
</bean>
----

`DynamicClassNameTemplateInitializingBean` depends on the active `SpringPropertyAccessor`. Every `log.pattern.<key>` must have a matching `log.level.<key>` or initialization fails with `LoggerException`.

[[scribe-endpoint-spi, Endpoint SPI]]
== Endpoint SPI

This section is for authors implementing a new Scribe endpoint.

The SPI centers on `LoggingBlueprint`:

[source,java]
----
import org.smallmind.scribe.pen.Record;
import org.smallmind.scribe.pen.adapter.LoggerAdapter;
import org.smallmind.scribe.pen.adapter.LoggingBlueprint;

public class MyLoggingBlueprint extends LoggingBlueprint<MyNativeRecord> {

  @Override
  public LoggerAdapter getLoggingAdapter (String name) {

    return new MyLoggerAdapter(name);
  }

  @Override
  public Record<MyNativeRecord> errorRecord (String loggerName, Throwable throwable, String message, Object... args) {

    return new MyErrorRecord(loggerName, throwable, message, args);
  }
}
----

You also need a service-registration file:

[source,text]
----
META-INF/services/org.smallmind.scribe.pen.adapter.LoggingBlueprint
----

with the implementation class name as its content:

[source,text]
----
com.acme.logging.MyLoggingBlueprint
----

`LoggingBlueprintFactory` loads that SPI through `ServiceLoader` using the current thread context class loader.

Responsibilities of an endpoint implementation:

* implement `LoggingBlueprint`
* implement `LoggerAdapter`
* provide a `Record` view over the native record type
* build a sensible `errorRecord(...)` for appender failure reporting
* register any additional logging-infrastructure package prefixes through `LoggerManager.addLoggingPackagePrefix(...)` if your endpoint introduces new wrapper layers that caller-context capture should skip

The built-in endpoints in `scribe/ink/*` are the best reference implementations.

[[scribe-migration, Migrating From SLF4J And Logback]]
== Migrating From SLF4J And Logback

Scribe is not only a facade replacement. The main shift is from implicit backend wiring to explicit logger templates and one explicit endpoint choice.

[cols="2,4",options="header"]
|===
|Common Habit |Scribe Equivalent

|`LoggerFactory.getLogger(...)`
|`LoggerManager.getLogger(...)`

|root logger plus child logger hierarchy
|one `DefaultTemplate` plus targeted overrides

|logger-specific config in backend XML
|template registration in code or Spring beans

|MDC
|`Logger.putParameter(...)` or bridge-backed MDC mapped to `Parameters`

|backend binding chosen by whatever is on the classpath
|exactly one `scribe-ink-*` module selected intentionally

|Logback async appender
|`AsynchronousAppender`
|===

Practical migration sequence:

1. If you already have SLF4J call sites, add `scribe-slf4j` first and keep those call sites intact.
2. Add exactly one `scribe-ink-*` endpoint and remove conflicting backend bindings.
3. Recreate the baseline logging policy with `DefaultTemplate`.
4. Add targeted `ClassNameTemplate` or `PersonalizedTemplate` overrides.
5. Move MDC lifecycle assumptions to explicit parameter or MDC boundaries.
6. Once the runtime policy is stable, migrate direct call sites to Scribe only if that buys you something.

[[scribe-usage-guidance, Usage Guidance]]
== Usage Guidance

=== For Library Authors

Library code should usually:

* depend only on `scribe-pen`
* obtain loggers through `LoggerManager.getLogger(MyType.class)`
* avoid registering templates or selecting endpoints on behalf of the application
* avoid leaking request parameters across library boundaries

If a library needs logging, it should expose logging as usage, not as ownership of the caller's runtime.

=== For Application Authors

Applications should:

* choose exactly one endpoint
* register templates during startup
* decide where parameters begin and end
* close appenders that hold files, queues, sockets, or worker threads
* keep one owner for backend configuration when using the JUL or Log4J2 endpoints

=== For High-Volume Paths

For noisy or latency-sensitive paths:

* keep `autoFillLoggerContext` off unless you really need caller location
* use shorter `PatternFormatter` patterns
* avoid heavy multi-line output (`%p`, `%s`) on the hottest paths
* wrap slow sinks in `AsynchronousAppender`
* use `Supplier<String>` for expensive message construction, but do not rely on it as a substitute for every downstream filter/backend optimization

[[scribe-threading, Threading And Lifecycle]]
== Threading And Lifecycle

Scribe is a library. Lifecycle ownership stays with the caller.

[cols="2,3,3",options="header"]
|===
|Type |Startup Responsibility |Shutdown Responsibility

|`ConsoleAppender`
|none
|none

|`FileAppender`
|none
|call `close()`

|`AsynchronousAppender`
|none; worker threads start in the constructor
|call `close()` so workers drain and the wrapped appender closes

|`SyslogAppender`
|call `afterPropertiesSet()` if Spring is not managing it
|none in normal use

|`FluentBitAppender`
|call `afterPropertiesSet()` if Spring is not managing it
|call `close()` so worker threads finish

|`AWSAppender`
|construct with the client, formatter, group, and stream
|close the CloudWatch client separately if your application owns it
|===

Lifecycle checklist:

1. build appenders and formatters
2. call `afterPropertiesSet()` for Spring-style appenders when wiring manually
3. register templates
4. run the application
5. close appenders and owned clients at shutdown

If you create backup appenders for `DefaultErrorHandler`, remember that those appenders may have their own lifecycle too.

[[scribe-troubleshooting, Troubleshooting]]
== Troubleshooting

[cols="2,3,4",options="header"]
|===
|Symptom |Likely Cause |What To Check

|`StaticInitializationError` says no provider found for `LoggingBlueprint`
|no endpoint module is present
|add exactly one `scribe-ink-*` dependency

|`StaticInitializationError` lists conflicting implementations
|more than one endpoint module is present
|remove extras until only one `scribe-ink-*` module remains

|logger output ignores your template
|the logger name does not match, or you built the logger directly with `new Logger(...)`
|verify the logger name and obtain the logger through `LoggerManager`

|`{}` appears literally in direct Scribe output
|direct Scribe logging uses `String.format`, not SLF4J placeholders
|use `%s` in direct Scribe calls or route through `scribe-slf4j`

|parameters are not visible in output
|the formatter or sink does not render them
|include `%p`, `RecordElement.PARAMETERS`, or a structured sink that emits parameters

|file retention never runs
|cleanup only runs after rollover, or separators do not match
|verify rollover actually happens and `Cleanup` uses the same separator as `Rollover`

|manual `SyslogAppender` or `FluentBitAppender` seems inert
|initialization hook was not called
|call `afterPropertiesSet()` outside Spring

|`FileLoggingPlan` fails on startup with a file-path error
|`logPath` points at a directory
|set `logPath` to a real file, not `/var/log`
|===

[[scribe-operational-notes, Operational Notes]]
== Operational Notes

* `PatternFormatter` always appends a trailing platform line separator.
* `ConsoleAppender` writes to `System.out`.
* `FileAppender` writes UTF-8 and creates parent directories automatically.
* `FileAppender` cleanup runs only when rollover occurs.
* `DefaultErrorHandler` defaults to `ConsoleAppender(new XMLFormatter())`.
* `ExceptionSuppressingLogFilter` uses exact throwable-class matches.
* `JDK` and `Log4J2` endpoints push template-applied appender and level choices into the underlying backend loggers.
* `LoggerManager.addLoggingPackagePrefix(...)` exists so bridges and custom endpoints can keep caller-context capture focused on application code.

[[scribe-edge-cases, Behavioral Edge Cases]]
== Behavioral Edge Cases

[cols="2,3",options="header"]
|===
|Behavior |Current Semantics

|runtime classpath has no `LoggingBlueprint` provider
|`LoggingBlueprintFactory` throws `StaticInitializationError` during static initialization

|runtime classpath has multiple `LoggingBlueprint` providers
|`LoggingBlueprintFactory` throws `StaticInitializationError` listing conflicting implementations

|a logger is constructed with `new Logger(...)`
|the logger works, but it is not automatically managed by `LoggerManager` template association

|two templates return the same strongest match score
|selection depends on internal template-set iteration order; avoid deliberate ties

|`RegExTemplate` is used before its expression is initialized
|`matchLogger(...)` throws `LoggerRuntimeException`

|`RegExTemplate.setExpression(...)` is called after the expression is already set
|`LoggerRuntimeException` is thrown

|`PersonalizedTemplate.setLoggerName(...)` is called after the logger name is already set
|`LoggerRuntimeException` is thrown

|`AsynchronousAppender.publish(...)` is called when the queue is full
|the record is rejected and routed through appender error handling

|`AsynchronousAppender.publish(...)` is called after close
|publication fails and is routed through error handling

|parameters cross a thread boundary unexpectedly
|storage uses Java's default `InheritableThreadLocal` behavior; reset context explicitly in worker paths
|===

== Minimal Checklist

To stand up Scribe correctly:

== Defaults Reference

This appendix centralizes the constructor and field defaults most likely to matter during integration.

=== Template

* level defaults to `INFO`
* `autoFillLoggerContext` defaults to `false`
* filters default to none
* appenders default to none
* enhancers default to none

=== PatternFormatter

* `timestamp` defaults to `DateFormatTimestamp.getDefaultInstance()` with pattern `yyyy-MM-dd'T'HH:mm:ss.SSSZ`
* default format is `%d %n %+5l [%T] - %m`
* formatted output always ends with the platform line separator

=== XMLFormatter

* `timestamp` defaults to `DateFormatTimestamp.getDefaultInstance()`
* `newLine` defaults to `System.getProperty("line.separator")`
* `cdata` defaults to `false`
* `indent` defaults to `3`
* `recordElements` defaults to all `RecordElement` values

=== FileAppender

* output encoding is UTF-8
* parent directories are created automatically
* no rollover or cleanup policy is present unless you configure one

=== Rollover And Cleanup

* `Rollover.separator` defaults to `'-'`
* `Cleanup.separator` defaults to `'-'`
* `FileSizeRolloverRule()` defaults to `10` megabytes
* `TimestampRolloverRule()` defaults to `TOP_OF_DAY`

=== AsynchronousAppender

* `new AsynchronousAppender(appender)` defaults to buffer size `1`
* `new AsynchronousAppender(appender)` defaults to concurrency `1`

=== FluentBitAppender

* `timestamp` defaults to `DateFormatTimestamp.getDefaultInstance()`
* `recordElements` defaults to all `RecordElement` values
* `newLine` defaults to `System.lineSeparator()`
* `retryAttempts` defaults to `3`
* `concurrencyLimit` defaults to `1`
* `batchSize` defaults to `1`
* `batchGracePeriodMilliseconds` defaults to `3000`

=== SyslogAppender

* `syslogHost = localhost`
* `syslogPort = 514`
* `facility = LOCAL7`
* `base64EncodeStackTraces = false`

=== LoggingPlan

* `defaultLogLevel = INFO`
* `logRecordBufferSize = 400`
* `concurrencyLimit = 1`

=== FileLoggingPlan

* `rolloverPeriod = TOP_OF_DAY`
* `retentionDays = 31`
* `rolloverMegabyteLimit = 100`
* set `logPath` explicitly to a file path before use
