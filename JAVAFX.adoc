[[javafx, JavaFX]]
= JavaFX

[partintro]
Smallmind JavaFX currently exposes one runtime artifact, `org.smallmind:smallmind-javafx-extras`, a focused set of JavaFX controls, dialogs, layout bridges, event-throttling helpers, and instrumentation charts. The module stays deliberately small. It does not bootstrap a JavaFX application, own a stage, manage container policy, or hide the surrounding UI stack behind a framework layer. You construct the pieces you need directly and keep startup, shutdown, and JavaFX thread policy in application code.

Audience: developers building JavaFX desktop tools, operator consoles, or internal applications who want a few reusable widgets and monitoring helpers without adopting a larger UI framework.
Versioning: examples use `${smallmind.version}` for Smallmind artifacts; keep related Smallmind modules on the same version unless you intentionally need to split them.

[[javafx-at-a-glance, At A Glance]]
== At A Glance

[cols="2,4",options="header"]
|===
|Question |Answer

|What do I actually depend on?
|`org.smallmind:smallmind-javafx-extras`. The parent `org.smallmind:smallmind-javafx` module is an aggregator POM only.

|What ships in the current code?
|One jar containing controls, dialogs, layout bridges, text and table helpers, consolidating listener wrappers, time-series charts, JMX-backed charts, three packaged chart stylesheets, and six dialog icon PNGs.

|Who owns JavaFX startup and shutdown?
|The caller. The module does not provide an `Application` subclass, stage bootstrap, Spring integration, or container lifecycle surface.

|Where does background work exist?
|`ConsolidatingChangeListener`, `ConsolidatingEventHandler`, `TimeAxis`, `JMXGaugeChart`, `JMXSigmaChart`, and `JMXVelocityChart` all create daemon-backed background tasks.

|What do I need to remember operationally?
|Call `stop()` on `TimeAxis` users when you discard them, verify required JMX attribute names before using `JMX*Chart`, and preserve the packaged CSS and image resources on the runtime classpath.

|Does the module ship FXML or a JPMS descriptor?
|No current `.fxml` files are packaged, and `smallmind-javafx-extras` does not ship `module-info.java` or an `Automatic-Module-Name` manifest entry.
|===

[[javafx-why-use, Why Use Smallmind JavaFX]]
== Why Use Smallmind JavaFX

Use this module when you want a few explicit JavaFX helpers instead of a larger UI framework or container policy layer.

What it does well:

* stays on ordinary JavaFX types such as `ChoiceBox`, `Dialog`, `TableColumn`, `Region`, `LineChart`, and `StackedAreaChart`
* keeps application bootstrap, scene ownership, JMX wiring, and shutdown policy in caller code
* exposes small entry points that can be adopted independently
* packages its own CSS and dialog images instead of requiring an external theme to make the shipped widgets usable

What it intentionally does not do:

* no Spring beans, XML namespace, or auto-configuration layer
* no application shell, stage manager, or dependency injection model
* no broad widget toolkit abstraction above JavaFX itself

[[javafx-installation, Installation]]
== Installation

Choose the runtime jar directly.
The parent `smallmind-javafx` module is not itself a runtime surface.

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>smallmind-javafx-extras</artifactId>
  <version>${smallmind.version}</version>
</dependency>
----

Direct dependencies declared by `javafx/extras/pom.xml`:

* `org.jfxtras:jfxtras-all`
* `org.openjfx:javafx-base`
* `org.openjfx:javafx-controls`
* `org.openjfx:javafx-graphics`
* `org.smallmind:scribe-pen`
* `org.smallmind:smallmind-nutsnbolts`

Adoption guidance:

* applications can depend on `smallmind-javafx-extras` directly
* reusable libraries should add it only when they intentionally expose JavaFX types in their public API
* JavaFX launcher policy stays application-owned; this module does not provide a `main(...)` method or `Application` subclass
* the repository build currently targets Java `25` through the parent POM's `maven.compiler.release`

[IMPORTANT]
====
`TimeAxis`, `GaugeChart`, `VelocityChart`, `SigmaChart`, `JMXGaugeChart`, `JMXSigmaChart`, and `JMXVelocityChart` do not stop themselves when removed from a scene.
If you create them for short-lived windows or tabs, call `stop()` during your own shutdown path.
====

[NOTE]
====
The jar currently has no `module-info.java` and no `Automatic-Module-Name` manifest entry.
If you run on the module path, treat module naming and access policy as application-owned deployment work and verify the packaged application explicitly.
====

=== No Spring Or Container Layer

The module is direct JavaFX code only.
There are no Spring beans, no XML namespaces, and no discovery or registration hooks beyond standard Java construction and JavaFX scene composition.

=== Resource Packaging

The `extras` build is prepared to package `.css` and `.fxml` files from `src/main/java`, plus resources from `src/main/resources/xsd` and `src/main/resources/image`.
In the current source tree:

* three `.css` files are packaged
* no `.fxml` files are present
* no XSD resources are present
* six PNG dialog icons are present

[[javafx-quick-start, Quick Start]]
== Quick Start

=== 1. Build A Small Enum And Pattern-Validated Form

[source,java]
----
import java.util.regex.Pattern;
import javafx.scene.control.Label;
import javafx.scene.layout.VBox;
import org.smallmind.javafx.extras.choice.EnumChoiceBox;
import org.smallmind.javafx.extras.text.PatternTextField;

enum Severity {
  LOW_PRIORITY, NORMAL_PRIORITY, HIGH_PRIORITY
}

EnumChoiceBox<Severity> severityBox =
    new EnumChoiceBox<Severity>(Severity.class, Severity.NORMAL_PRIORITY);

PatternTextField ticketField =
    new PatternTextField(Pattern.compile("[A-Z]{3}-[0-9]{1,6}"));

VBox form = new VBox(8);
form.getChildren().addAll(
    new Label("Severity"),
    severityBox,
    new Label("Ticket"),
    ticketField
);
----

`EnumChoiceBox` renders enum constants in display case and sorts them alphanumerically by that display text.
`PatternTextField` rejects edits that would leave the whole field outside the regex.

=== 2. Consolidate Noisy Text Changes

[source,java]
----
import java.util.regex.Pattern;
import javafx.beans.value.ChangeListener;
import javafx.beans.value.ObservableValue;
import javafx.scene.control.Label;
import org.smallmind.javafx.extras.ConsolidatingChangeListener;
import org.smallmind.javafx.extras.text.PatternTextField;

PatternTextField searchField =
    new PatternTextField(Pattern.compile("[A-Za-z0-9 _-]*"));

Label preview = new Label();

searchField.textProperty().addListener(
    new ConsolidatingChangeListener<String>(250, new ChangeListener<String>() {

      @Override
      public void changed (ObservableValue<? extends String> observable, String oldValue, String newValue) {

        preview.setText(newValue);
        reloadResults(newValue);
      }
    }));
----

Only the last value seen after `250` milliseconds of quiet time is delivered to the wrapped listener.
Delivery occurs on the JavaFX thread.

=== 3. Lay Out Content With `ParaboxPane`

[source,java]
----
import javafx.geometry.Insets;
import javafx.scene.control.Button;
import javafx.scene.control.TextArea;
import org.smallmind.javafx.extras.layout.ParaboxPane;
import org.smallmind.nutsnbolts.layout.Alignment;
import org.smallmind.nutsnbolts.layout.Constraint;

ParaboxPane root = new ParaboxPane(new Insets(12, 12, 12, 12));
TextArea detailsArea = new TextArea();
Button closeButton = new Button("Close");

root.setHorizontalBox(
    root.parallelBox(Alignment.TRAILING)
        .add(detailsArea, Constraint.stretch())
        .add(closeButton)
);
root.setVerticalBox(
    root.serialBox(10)
        .add(root.parallelBox().add(detailsArea, Constraint.stretch()))
        .add(root.parallelBox(Alignment.TRAILING).add(closeButton))
);
----

`ParaboxPane` is a thin JavaFX bridge over the nutsnbolts parabox layout engine.
Use it when you want explicit serial and parallel box construction rather than one of the stock JavaFX layout panes.

=== 4. Feed A Chart Programmatically

[source,java]
----
import javafx.stage.Stage;
import org.smallmind.javafx.extras.instrument.GaugeChart;
import org.smallmind.javafx.extras.instrument.Measure;

Stage stage = new Stage();

GaugeChart chart = new GaugeChart(10 * 60 * 1000L);
chart.setTitle("Request Rate");
chart.getXAxis().setLabel("Time");
chart.getYAxis().setLabel("Quantity/Second");

chart.addMeasure(System.currentTimeMillis(), new Measure(42.0D, 40.5D, 39.0D, 38.0D));

stage.setOnCloseRequest(event -> chart.stop());
----

`GaugeChart`, `VelocityChart`, and `SigmaChart` lazy-initialize their data series on the first sample.
Calling `setPaused(true)` freezes the time window, but direct `add*` calls still append data.

=== 5. Poll A JMX MBean

[source,java]
----
import java.lang.management.ManagementFactory;
import javax.management.MBeanServerConnection;
import javax.management.ObjectName;
import javafx.stage.Stage;
import org.smallmind.javafx.extras.instrument.JMXGaugeChart;

Stage stage = new Stage();
MBeanServerConnection connection = ManagementFactory.getPlatformMBeanServer();
ObjectName objectName =
    new ObjectName("org.smallmind.example:type=Meter,name=RequestRate");

JMXGaugeChart chart =
    new JMXGaugeChart(15 * 60 * 1000L, connection, objectName);

stage.setOnCloseRequest(event -> chart.stop());
----

Constructing a `JMXGaugeChart`, `JMXSigmaChart`, or `JMXVelocityChart` starts polling immediately.
Each chart pauses itself and shows a `JavaErrorDialog` if a polling cycle throws an exception.

[[javafx-package-reference, Package Reference]]
== Package Reference

[[javafx-core-package, Core Package]]
=== `org.smallmind.javafx.extras`

[cols="2,5",options="header"]
|===
|Type |Contract

|`ConsolidatingChangeListener<T>`
|Wraps a `ChangeListener<T>` and emits only the most recent change seen after the configured quiet period. Pending changes live in a class-wide `ConcurrentSkipListMap`, a daemon worker polls for expirations every `50` milliseconds, and the wrapped listener is invoked through `PlatformUtil.runAndWait(...)`.

|`ConsolidatingEventHandler<T extends Event>`
|The event-handler counterpart to `ConsolidatingChangeListener`. It uses the same class-wide queue model, the same `50` millisecond polling interval, and the same JavaFX-thread handoff.

|`ImageNotFoundException`
|A `FormattedRuntimeException` subtype for missing image resources. The current dialog classes load bundled images directly and do not throw this type themselves.
|===

Important behavior:

* the quiet period is supplied in the constructor in milliseconds
* the latest generation wins; intermediate updates are dropped
* there is no public shutdown hook for the consolidating worker threads
* work done in the wrapped listener or handler still runs on the JavaFX thread, so long-running UI callbacks remain your responsibility

[[javafx-choice-package, Choice Package]]
=== `org.smallmind.javafx.extras.choice`

`EnumChoiceBox<E extends Enum<E>>` extends `ChoiceBox<E>` and provides four construction paths:

* `new EnumChoiceBox<E>(Class<E>)`
* `new EnumChoiceBox<E>(Class<E>, E selectedEnum)`
* `new EnumChoiceBox<E>(Class<E>, List<E> exemptEnums)`
* `new EnumChoiceBox<E>(Class<E>, E selectedEnum, List<E> exemptEnums)`

Behavioral rules:

* entries come from `enumClass.getEnumConstants()`
* `exemptEnums` are removed before display
* remaining entries are sorted by display-cased enum name, not by declaration order
* the visible label is generated with `StringUtility.toDisplayCase(enumerated.name(), '_')`
* if `selectedEnum` is `null`, the first remaining entry becomes the value
* `fromString(...)` maps display text back to an enum constant with `StringUtility.toStaticFieldName(...)`

[[javafx-dialog-package, Dialog Package]]
=== `org.smallmind.javafx.extras.dialog`

[cols="2,5",options="header"]
|===
|Type |Contract

|`ErrorEvent`
|JavaFX `Event` subtype carrying `exceptionSource` and `throwable`. The package exposes `ErrorEvent.ANY` and `ErrorEvent.OCCURRED`.

|`JavaErrorDialog`
|Subclass of `Dialog<ButtonType>` that renders a stack trace inside a `ScrollPane` using `ParaboxPane` and `InsetsPane`. The title is fixed to `Java Error Message...`, the dialog is resizable, the initial size is `600x400`, and hiding the dialog fires `ErrorEvent.OCCURRED` from the dialog pane.

|`WarningAlert`
|Subclass of `Alert` configured as `AlertType.WARNING` with `ButtonType.CLOSE` and a packaged warning icon.

* `JavaErrorDialog.showJavaErrorDialog(Object source, Throwable throwable)`
* `WarningAlert.showWarningAlert(String warningText)`

Both helpers call `showAndWait()` and therefore block until the dialog is dismissed.

[[javafx-image-package, Image Package]]
=== `org.smallmind.javafx.extras.image`

`ImageViewPane` is a `Region` that owns exactly one `ImageView` property.
On every layout pass it:

* sets `fitWidth` to the pane width
* sets `fitHeight` to the pane height
* lays the image out centered in the full available area

Replacing the `ImageView` through `setImageView(...)` removes the old node from the scene graph and adds the new one.

[[javafx-layout-package, Layout Package]]
=== `org.smallmind.javafx.extras.layout`

[cols="2,5",options="header"]
|===
|Type |Contract

|`InsetsPane`
|A `Region` wrapper that adds fixed `Insets` around its managed children. Its minimum, preferred, and maximum measurements are the largest managed child measurement plus the matching inset gutter.

|`ParaboxPane`
|A JavaFX `Region` implementing `ParaboxContainer<Node>`. It delegates sizing and layout to `ParaboxLayout`, exposes `parallelBox(...)` and `serialBox(...)` convenience methods, and allows `remove(...)` and `removeAll()`.

|`JavaFxParaboxElement`
|Bridge from JavaFX `Node` measurement and baseline APIs into the nutsnbolts `ComponentParaboxElement<Node>` contract.

|`JavaFxParaboxPlatform`
|Defines the JavaFX parabox defaults: related gap `5.0`, unrelated gap `10.0`, frame perimeter `10.0` on all four sides, and orientation `Bias.HORIZONTAL` with `Flow.FIRST_TO_LAST`.

|`Cut`
|Internal enum used by `InsetsPane` when selecting minimum, preferred, or maximum measurement logic.
|===

Important `ParaboxPane` details:

* `computeMaxWidth(...)` and `computeMaxHeight(...)` both return `Double.MAX_VALUE`
* content is not laid out until both the horizontal and vertical boxes are defined
* the module does not add a higher-level DSL on top of parabox; you work directly with `Alignment`, `Constraint`, `Gap`, `Justification`, `ParallelBox`, and `SerialBox`

[[javafx-table-package, Table Package]]
=== `org.smallmind.javafx.extras.table`

[cols="2,5",options="header"]
|===
|Type |Contract

|`PropertyTableColumn<S, T>`
|A `TableColumn<S, T>` configured around a JavaBean property name and a `PropertyValueFactory<S, T>`. Default alignment is `Pos.CENTER_LEFT`. If you supply a converter, it is applied in `updateItem(...)` when the cell is not empty and not editing.

* `toString(...)` uses display case, but `fromString(...)` expects the enum constant name, not the display label

[[javafx-text-package, Text Package]]
=== `org.smallmind.javafx.extras.text`

`PatternTextField` extends `TextField` and overrides both `replaceText(...)` and `replaceSelection(...)`.
Before committing an edit it forecasts the full resulting string and accepts the edit only when:

* no pattern has been configured, or
* `pattern.matcher(forecast).matches()` returns `true`

That means pattern matching is whole-value matching on every edit.

[[javafx-instrument-package, Instrument Package]]
=== `org.smallmind.javafx.extras.instrument`

==== Data Records

[cols="2,4",options="header"]
|===
|Type |Meaning

|`Measure`
|Four rate samples: inception average, one-minute average, five-minute average, and fifteen-minute average

|`Blur`

==== `TimeAxis`

`TimeAxis` is a `ValueAxis<Long>` that manages a sliding wall-clock window.

Operational rules:

* construction requires a span in milliseconds
* `setAutoRanging(false)` is applied immediately
* tick labels are rotated `90` degrees
* a shared daemon executor updates each axis once per second
* the lower bound is `now - span`, the upper bound is `now`
* major ticks are placed at `15` second boundaries
* minor tick marks are placed at `5` second boundaries when they do not overlap a major tick
* `pausedProperty()`, `getPaused()`, and `setPaused(...)` control bound updates
* `stop()` cancels the instance's scheduled updater task

==== Base Charts

[cols="2,2,3,2",options="header"]
|===
|Chart |Input Type |Series Names |Style Hook

|`GaugeChart`
|`Measure`
|`Inception`, `15 minute`, `5 minute`, `1 minute`
|Style class `gauge-chart`, stylesheet `GaugeChart.css`

|`VelocityChart`
|`Blur`
|`Inception`, `15 minute`, `5 minute`, `1 minute`
|Style class `velocity-chart`, stylesheet `VelocityChart.css`

|`SigmaChart`
|`Dispersion`
|`Median`, `75th pctl`, `95th pctl`, `98th pctl`, `99th pctl`, `99.9th pctl`
|Style class `sigma-chart`, stylesheet `SigmaChart.css`
|===

Shared chart behavior:

* each chart constructs its own `TimeAxis`
* data series are added lazily on the first sample
* `addMeasure(...)`, `addBlur(...)`, and `addDispersion(...)` marshal the append work to the JavaFX thread through `Platform.runLater(...)`
* samples older than the current lower bound are removed from each series before appending a new sample
* `stop()` delegates to the underlying `TimeAxis`
* `setPaused(true)` pauses the `TimeAxis`; direct `add*` calls still append data in the base chart types

`SigmaChart` stores stacked percentile deltas rather than absolute percentile values for every series after the median:

* series `0`: median
* series `1`: `75th - median`
* series `2`: `95th - 75th`
* series `3`: `98th - 95th`
* series `4`: `99th - 98th`
* series `5`: `99.9th - 99th`

[cols="2,4,3",options="header"]
|===
|Chart |Attributes Read |Other Defaults

|`JMXGaugeChart`
|Reads `RateTimeUnit` once at construction, then polls `AverageRate`, `OneMinuteAvgRate`, `FiveMinuteAvgRate`, and `FifteenMinuteAvgRate`
|Title `Attribute/get`; x-axis label `Time`; y-axis label `Quantity/<display-case singular time unit>`

Shared JMX chart behavior:

* construction starts polling immediately
* polling uses a shared daemon executor
* the first poll runs after `1` second, then every `15` seconds
* on any polling exception the chart calls `setPaused(true)` and opens a `JavaErrorDialog` on the JavaFX thread
* `stop()` cancels the JMX polling future and then stops the inherited `TimeAxis`

[[javafx-threading, Threading And Lifecycle]]
== Threading And Lifecycle

[cols="2,5",options="header"]
|===
|Surface |Lifecycle Rule

|`ConsolidatingChangeListener`, `ConsolidatingEventHandler`
|Each class starts one daemon worker thread when the class is first loaded. There is no public stop or dispose hook. The worker keeps running for the life of the JVM.

|`TimeAxis`
|Each instance schedules one updater task on a shared daemon executor. Call `stop()` when the axis or owning chart is no longer needed.

|`GaugeChart`, `VelocityChart`, `SigmaChart`
|These charts do not own a second scheduler beyond `TimeAxis`, but they do own that `TimeAxis`. Call `stop()` on disposal.

|`JMXGaugeChart`, `JMXSigmaChart`, `JMXVelocityChart`
|Each instance owns both the inherited `TimeAxis` updater and a second scheduled poller. Call `stop()` to cancel both.

|`JavaErrorDialog.showJavaErrorDialog(...)`, `WarningAlert.showWarningAlert(...)`
|Both helpers call `showAndWait()` and block until the user dismisses the dialog.
|===

Additional rules worth keeping explicit:

* the base chart `add*` methods are safe to call from background threads after the JavaFX toolkit has started because they hand off to `Platform.runLater(...)`
* the consolidating listener wrappers deliver their inner callback through `PlatformUtil.runAndWait(...)`, so the wrapped callback still executes synchronously on the JavaFX thread
* pausing a base chart stops the axis window from advancing but does not block explicit `add*` calls
* pausing a JMX chart also suppresses future collection because each collector checks `isPaused()` before polling

[[javafx-edge-cases, Behavioral Edge Cases]]
== Behavioral Edge Cases

[cols="3,5",options="header"]
|===
|Behavior |Why It Matters

|`EnumChoiceBox` sorts by display text instead of enum declaration order
|If enum order is domain-significant, you need a different control or a custom wrapper

|Excluding every enum constant from `EnumChoiceBox`
|The constructor still tries to select the first remaining value, so a fully excluded enum set fails during construction

|`PatternTextField` uses `matcher(...).matches()`
|Partial edits are rejected unless the entire forecast string matches your regex

|`PropertyTableColumn` applies its converter only when the cell is not editing
|Display formatting and edit parsing are different concerns in the current implementation

|Base chart pruning happens only inside `addMeasure(...)`, `addBlur(...)`, and `addDispersion(...)`
|Old points are removed when new samples arrive; there is no independent pruning task

|Base chart pause freezes the axis, not data acceptance
|If you call `add*` while paused, the sample is still appended

|JMX chart polling failure pauses the chart and shows a dialog
|One bad poll stops further collection until you unpause or reconstruct the chart

|`JMXVelocityChart` currently requests `OneMinuteAvgVelocity` for its last three series
|The five-minute and fifteen-minute lines are not independently sourced in the current implementation

|`TimeAxis` tick labels should be verified in a real UI
|The current label formatting path uses the raw `Long` tick value directly, so exact wall-clock rendering should be checked in the application that adopts it

|`ImageNotFoundException` is shipped but not used by the packaged dialog classes
|Do not rely on the exception as proof that dialog icon lookup failures are surfaced through that type
|===

[[javafx-troubleshooting, Troubleshooting]]
== Troubleshooting

[cols="2,3,4",options="header"]
|===
|Problem |Likely Cause |What To Check

|`NoClassDefFoundError` or `ClassNotFoundException` for `javafx.*` or `jfxtras.*`
|The JavaFX or JFXtras runtime is not present in the launched application
|Verify the final application packaging, module path or classpath setup, and platform-specific JavaFX runtime layout

|`IllegalStateException` from `JMXGaugeChart` construction saying it cannot determine the velocity unit of time
|The target MBean does not expose `RateTimeUnit`, or the connection failed during construction
|Confirm the `ObjectName`, `MBeanServerConnection`, and `RateTimeUnit` attribute before constructing the chart

|A chart continues scheduling work after the window closes
|`stop()` was never called
|Add an explicit shutdown hook on the owning stage, tab, or controller

|A JMX chart freezes and a Java error dialog appears
|An exception occurred during `getAttributes(...)`
|Check connectivity, the MBean name, the exact attribute names, and the data types returned by the MBean

|Table cells display formatted values but editing does not round-trip correctly
|`PropertyTableColumn` uses the converter for display only in the current implementation
|If you need custom parse behavior, supply your own editing cell strategy instead of relying on the display converter alone

|A `PatternTextField` rejects keystrokes that look locally valid
|The regex must match the entire resulting field value on every edit
|Test the exact full-string pattern, including empty-string handling and intermediate edit states

|`EnumChoiceBox` fails during construction
|Every enum value may have been excluded, or the selected value may not belong to the intended enum class
|Check `exemptEnums` and the selected value before construction

|Chart time labels do not match the wall-clock values you expected
|The current `TimeAxis` label formatting should not be assumed without verification
|Render the chart in a focused smoke test and verify the labels before relying on them in operator-facing views

|Warning or error dialogs have missing icons
|The packaged PNG resources were not preserved on the final classpath
|Confirm the packaged resource paths and any shading or relocation rules

|The velocity chart's one-minute, five-minute, and fifteen-minute JMX lines look identical
|The current `JMXVelocityChart` implementation reads `OneMinuteAvgVelocity` for the last three series
|Treat that as current behavior and verify whether you need a code change before using the chart for operations work
|===

[[javafx-verification, Verification Checklist]]
== Verification Checklist

Use this checklist when adopting the module or validating an upgrade:

1. Confirm you depend on `org.smallmind:smallmind-javafx-extras`, not only the parent `smallmind-javafx` POM.
2. Start a minimal JavaFX application and instantiate one control from each package family you plan to use.
3. If you use `PatternTextField`, test both valid and invalid intermediate edits, not only final values.
4. If you use `EnumChoiceBox`, verify the displayed order and the exclusion list against the enum values you actually want visible.
5. If you use `PropertyTableColumn`, cover both display and edit behavior in a focused UI test.
6. If you use `ConsolidatingChangeListener` or `ConsolidatingEventHandler`, exercise burst input and confirm that only the last quiet-period value is delivered.
7. If you use charts, verify `pause`, `resume`, and `stop()` behavior, including window teardown.
8. If you use JMX charts, confirm the exact MBean attribute names and returned value types before wiring the chart into a production view.
9. Verify that the packaged CSS files and dialog PNG resources survive your application packaging process.
10. Because the current module has no focused `src/test` suite, keep a small application-level smoke test or UI harness for the exact widgets you adopt.
