== Overview

The `smallmind-schedule` module is a thin, opinionated adaptation of
https://www.quartz-scheduler.org/[Quartz] for services that already run inside
a Spring application context. It does not rewrite Quartz. It does not hide it.
It provides the small, repeatable piece of plumbing that every Quartz-on-Spring
integration ends up hand-writing: a Quartz `Job` base class that records how
long it ran and what it produced, a `JobFactory` that resolves job beans from
the Spring context, a `SchedulerFactory` that ties scheduler lifecycle to
context lifecycle, a pair of `InitializingBean` implementations that install
and retire `CronTrigger` schedules declaratively, and a JMX facade for
operators who need to pause triggers without redeploying.

It is organized as two independently publishable artifacts:

* **`schedule-base`** — a framework-neutral contract (`ProxyJob`) and nothing
  else. No Quartz dependency, no Spring dependency. Safe to import from a
  service interface module, a shared model jar, or any library that wants to
  *describe* a schedulable unit of work without pulling in the runtime that
  executes it.
* **`schedule-quartz`** — the Quartz implementation. Implements `ProxyJob` as
  an abstract `InterruptableJob` base class, wires a Quartz `Scheduler` into a
  Spring `ApplicationContext`, installs and retires cron schedules from bean
  definitions, and exposes scheduler lifecycle over JMX. Ships a ready-to-import
  Spring XML configuration and the Liquibase changelog that references the
  `QRTZ_*` metadata tables.

The module is small by design: one interface in `schedule-base`, eleven classes
plus three Spring XML files and a Liquibase changelog reference in
`schedule-quartz`. It exists to turn Quartz into something a Spring service
can adopt in an afternoon — without learning the Quartz bean graph, without
writing a `JobFactory` for Spring integration, without hand-coding trigger
installation logic that has to decide between "add" and "replace," and without
coupling the service's public API to Quartz.

=== Why this package exists

Quartz is a capable scheduler with a long track record, but it has a shape
problem for services whose scheduling work is a minor part of a larger
application. A production Quartz-on-Spring setup needs, at minimum:

1. A `SchedulerFactory` — typically the JDBC-backed `StdSchedulerFactory` —
   configured from a `Properties` instance with twenty-plus entries covering
   the instance name, thread pool, job store, datasource, clustering, and
   RMI exposure.
2. A bridge between Spring and Quartz job instantiation. Out of the box,
   Quartz instantiates `Job` classes via `Class.newInstance()`; a Spring
   service wants dependency-injected `@Autowired` fields and prototype-scoped
   beans, which requires a custom `org.quartz.spi.JobFactory`.
3. Lifecycle coordination. The Quartz scheduler is started, paused, and shut
   down explicitly. A Spring service wants the scheduler to come up when the
   context is ready and shut down cleanly — waiting for in-flight jobs — when
   the context closes.
4. Declarative schedule management. Every Quartz job is a `JobDetail` paired
   with one or more `Trigger` instances, keyed by `name` and `group`. A
   service wants to declare "these jobs on these cron expressions" in a
   configuration file, have missing jobs added, changed jobs replaced, and
   obsolete jobs deleted — all at startup, without bespoke bootstrap code.
5. Bookkeeping that survives a crash. Every job firing produces a start time,
   a stop time, an outcome (success / failure / interrupted), a processed
   count, and zero or more exceptions. A service wants that information in
   its log stream, formatted consistently, without every job author
   reimplementing it.
6. An operator control surface. At 03:00 in production, an on-call engineer
   may need to suspend trigger firing without redeploying the service. A
   service wants that control exposed through an existing operational
   channel — JMX — rather than through a new one.
7. A database schema. The `JobStoreTX` job store requires the `QRTZ_*`
   metadata tables. A service wants those created and migrated as part of
   the same startup that initializes the scheduler.

`smallmind-schedule` assembles all seven, once, as an importable Spring XML
file plus a handful of collaborating beans, and exposes one abstract
`QuartzProxyJob` class that subclasses implement in terms of `proceed()` and
`cleanup()` — nothing else.

=== What it does, concretely

* **Defines a neutral job contract** (`ProxyJob` in `schedule-base`) so
  shared libraries can describe a schedulable unit of work without
  referencing Quartz.
* **Implements `ProxyJob` over Quartz's `InterruptableJob`**
  (`QuartzProxyJob`), recording start and stop timestamps, tracking a
  processed-item count, accumulating exceptions, and emitting a single
  structured log entry at the end of each firing.
* **Resolves job instances from a Spring context** (`SpringJobFactory`),
  enforcing that every job bean is prototype-scoped so each firing gets a
  fresh instance with freshly injected collaborators.
* **Ties scheduler lifecycle to Spring context lifecycle**
  (`SpringSchedulerFactory`), exposing an `activeMode` toggle that decides
  whether the scheduler starts firing triggers immediately or comes up in
  standby, and shutting the scheduler down cleanly on `ContextClosedEvent`.
* **Installs `JobDetail`/`CronTrigger` pairs declaratively**
  (`CronJobInitializingBean`), adding missing jobs, replacing changed jobs,
  scheduling missing triggers, and rescheduling changed cron expressions.
* **Retires obsolete schedules declaratively**
  (`JobDeletionInitializingBean`), deleting jobs by `name`/`group` at
  startup.
* **Assembles cron jobs by allowlist** (`CronJob` + `CronJobMapFactoryBean`),
  so the set of jobs actually scheduled is the intersection of what the
  context defines and what the deployment explicitly permits.
* **Exposes scheduler lifecycle over JMX** (`SchedulerMXBean` /
  `SchedulerMonitor` / `SchedulerStatus`), giving operators a start,
  standby, and status surface without adding a new control channel.
* **References the `QRTZ_*` schema as Liquibase**
  (`Quartz.changelog.xml`), wired through `smallmind-liquibase` so the
  metadata tables are created or migrated as part of the same application
  startup that initializes the scheduler.
* **Ships the Quartz-authored DDL** for every supported database
  (`tables_postgres.sql`, `tables_mysql_innodb.sql`, `tables_oracle.sql`,
  and twenty more) under `src/main/resources/quartz/` so a deployment that
  manages its own migrations can load the schema directly.

=== What it intentionally does not do

* It does not define jobs, triggers, or schedules. Those remain ordinary
  Quartz beans that the caller authors. This module starts the scheduler,
  resolves job beans, installs and retires trigger pairs, and logs firings;
  it does not describe the work.
* It does not replace Quartz's `StdSchedulerFactory`. `SpringSchedulerFactory`
  extends it, substituting a Spring-aware `JobFactory` and adding
  Spring-lifecycle hooks.
* It does not ship a REST or web admin console. `SchedulerMonitor` is a
  standard MBean; wrapping it in a transport is the caller's decision.
* It does not support non-cron triggers in its declarative-install path.
  `CronJobInitializingBean` is specifically for `CronTrigger`. Simple
  triggers and calendar-based triggers can still be registered against
  the resolved `Scheduler` bean directly; the declarative helpers just
  don't cover them.
* It does not attempt cross-node orchestration itself. Clustering is a
  Quartz feature configured via `org.quartz.jobStore.isClustered`; the
  shipped XML sets that flag to `true`, but the coordination is Quartz's,
  not this module's.

[[installation]]
== Installation

`smallmind-schedule` publishes two artifacts. Depend on the smallest one that
fits your use.

=== `schedule-base`

Depend on this from shared libraries, interface modules, or any caller that
needs to name the job contract without pulling in Quartz or Spring.

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>schedule-base</artifactId>
  <version>${project.version}</version>
</dependency>
----

This artifact brings in `smallmind-nutsnbolts` (for `SuccessOrFailure`) and
nothing else.

=== `schedule-quartz`

Depend on this from the application or service that actually runs the
scheduler.

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>schedule-quartz</artifactId>
  <version>${project.version}</version>
</dependency>
----

Brings in `schedule-base`, `smallmind-nutsnbolts`, `smallmind-scribe-pen`
(for structured logging), Quartz (`org.quartz-scheduler:quartz`), and the
Spring beans and context artifacts. The runtime classpath must also
include a JDBC driver for the database holding the `QRTZ_*` tables and —
if you want Liquibase migration — `smallmind-liquibase` plus the beans it
requires.

[IMPORTANT]
====
`schedule-quartz` is built expecting to find its resources on the
classpath at `spring/org/smallmind/schedule/quartz/quartz.xml` and
`liquibase/org/smallmind/schedule/quartz/Quartz.changelog.xml`. The module's
`pom.xml` flattens its two resource directories (`src/main/resources/spring`
and `src/main/resources/liquibase`) into the jar root, so these classpath
locations resolve without any further configuration on the consumer's side.
====

[[conceptual-model]]
== Conceptual Model

The module ships three layered abstractions. Each layer is responsible for
exactly one thing; understanding the split is the shortest path to using the
module correctly.

`ProxyJob`::
    The *contract*. A schedulable unit of work as seen by code that has no
    reason to know about Quartz. Declares `proceed()` and `cleanup()`, the
    bookkeeping state (start time, stop time, count, status, throwables),
    and the two policy predicates (`isEnabled()`, `logOnZeroCount()`). Lives
    in `schedule-base`.

`QuartzProxyJob`::
    The *Quartz adapter*. An abstract class that implements `ProxyJob` on
    top of Quartz's `InterruptableJob`. Owns the `execute()` method — the
    one Quartz calls when a trigger fires — and implements it as a fixed
    lifecycle: capture start time, register the running thread, call
    `proceed()`, catch exceptions, capture stop time, emit a structured log
    line, call `cleanup()`. Subclasses override `proceed()` and optionally
    `cleanup()`; everything else is fixed.

`SpringSchedulerFactory` + `SpringJobFactory` + `CronJobInitializingBean`::
    The *Spring integration*. Three collaborating beans, each responsible
    for a different slice of integration: factory wiring and lifecycle, job
    bean resolution at firing time, and declarative trigger installation.

Bookkeeping invariants::
    `QuartzProxyJob` initializes `getJobStatus()` to
    `SuccessOrFailure.SUCCESS`. Any uncaught `Exception` thrown from
    `proceed()` (other than `InterruptedException`) flips the status to
    `FAILURE` through `setThrowable(throwable)`. An `InterruptedException`
    specifically flips the status to `INTERRUPTED`. Subclasses may call
    `setThrowable(throwable, false)` to record a non-fatal exception
    without changing the status.

Prototype invariant::
    Every Spring bean resolved by `SpringJobFactory` must be prototype-scoped.
    This is enforced at firing time; a singleton job bean throws
    `FormattedSchedulerException`. The reason is that Quartz reuses `Job`
    instances across firings only when the factory returns the same object,
    and the bookkeeping state on `QuartzProxyJob` — count, start time, stop
    time, throwables — is per-firing mutable state.

[[schedule-base-reference]]
== Schedule-Base Reference

`schedule-base` contains exactly one public type. It is intentionally small.

=== `ProxyJob`

Framework-neutral interface describing a schedulable unit of work. Methods:

`isEnabled()`::
    Returns `true` if the job should be allowed to run on this firing. The
    host scheduler evaluates this before every firing and silently skips
    execution when `false` is returned. Useful for config-driven toggles.

`logOnZeroCount()`::
    Returns `true` if a completion log line should be emitted even when
    `getCount()` is zero. Jobs where an empty run is normal (and not worth
    logging) should return `false`; jobs where an empty run is noteworthy
    should return `true`.

`getJobStatus()`::
    Returns a `SuccessOrFailure` value reflecting the current or most
    recently recorded outcome. Three states: `SUCCESS`, `FAILURE`,
    `INTERRUPTED`.

`getCount()` / `incCount()`::
    A running count of items processed during the current or most recent
    execution. Incremented by the job body.

`getStartTime()` / `getStopTime()`::
    `LocalDateTime` timestamps for the most recent firing, or `null` if
    the job has not yet run.

`getThrowables()`::
    Errors accumulated during the current or most recent execution, or
    `null` if none were recorded.

`setThrowable(Throwable)`::
    Records an error. The default contract is that this also marks the
    job as failed, but subclasses of the Quartz adapter have a two-argument
    variant that disables the failure transition.

`proceed()`::
    Executes the job's work. Throws `Exception` if it cannot continue.

`cleanup()`::
    Post-execution housekeeping. Called after `proceed()` returns, whether
    or not it threw.

`ProxyJob` is written to be implementable without any scheduling framework.
In practice, services implement the Quartz-integrated abstract subclass
(`QuartzProxyJob`), not the bare interface.

[[schedule-quartz-reference]]
== Schedule-Quartz Reference

=== `QuartzProxyJob`

Abstract class that implements `ProxyJob` and Quartz's `InterruptableJob`.
Subclasses override:

`proceed()` (*required*)::
    The job body. Do the work here. Call `incCount()` or `addToCount(int)`
    to record progress. Throw exceptions if work cannot continue — uncaught
    non-`InterruptedException` throws become `FAILURE`;
    `InterruptedException` becomes `INTERRUPTED`.

`cleanup()` (*optional*)::
    Inherited `ProxyJob.cleanup()` declaration; default Quartz adapter does
    not override it, so subclasses supply a concrete implementation (or
    an empty one). Called unconditionally in a `finally` block after
    `proceed()`. Exceptions raised here are logged but do not propagate.

`isEnabled()` (*optional override*)::
    Defaults to `true`. Override to gate firings on external configuration
    or runtime conditions. When `false`, the firing is skipped entirely —
    no start time, no log entry, no `cleanup()`.

`logOnZeroCount()` (*required — abstract on `ProxyJob`*)::
    Return `true` if the completion log line should still be emitted when
    `getCount()` is zero. Jobs where an empty run is normal should return
    `false`.

Method contract highlights:

`execute(JobExecutionContext)` (final in practice — do not override)::
    Called by Quartz when a trigger fires. Records `startTime`, registers
    `Thread.currentThread()` on `threadRef` (so `interrupt()` can reach it),
    calls `proceed()`, catches exceptions, clears `threadRef`, records
    `stopTime`, emits a structured completion log line, and calls
    `cleanup()` in a `finally` block.

`interrupt()`::
    Standard Quartz interrupt hook. Calls `Thread.interrupt()` on the
    thread currently running the job, if one is registered.

`incCount()` / `addToCount(int)`::
    Synchronized increment of the per-firing counter. Use `addToCount` to
    record batched progress in a single operation.

`setThrowable(Throwable)` / `setThrowable(Throwable, boolean)`::
    Records an error. The one-argument form is a `ProxyJob` contract
    method and always marks the job `FAILURE`. The two-argument form is
    a `QuartzProxyJob` extension: pass `false` as the second argument to
    record an error for logging without changing the outcome.

=== `SpringJobFactory`

Quartz `JobFactory` that sources job instances from a Spring
`ApplicationContext`. Constructed by `SpringSchedulerFactory` when the
context is injected.

Resolution algorithm (for a firing whose `JobDetail` points at job class
`J` and has key `group.name`):

1. `getBeansOfType(J)` — fetch every bean assignable to `J`.
2. If the result has exactly one entry, that is the resolved bean.
3. If the result has multiple entries, look for one whose bean name equals
   `jobKey.toString()` (i.e. `group.name`). That is the resolved bean.
4. If the result is empty, or multiple entries exist with no exact name
   match, throw `FormattedSchedulerException` with a diagnostic message
   naming the job key, the job class, and the situation (no candidates /
   ambiguous candidates).
5. In either of the first two cases, the resolved bean must be
   prototype-scoped; if it is not, throw `FormattedSchedulerException`.

The factory also implements `Closeable`: when `SpringSchedulerFactory`
receives `ContextClosedEvent`, it closes the scheduler and then closes
the factory, which in turn closes the underlying `ApplicationContext` if
the context is itself closeable.

=== `SpringSchedulerFactory`

Extends `org.quartz.impl.StdSchedulerFactory`. Responsibilities:

Spring lifecycle integration::
    Implements `ApplicationContextAware` and
    `ApplicationListener<ContextClosedEvent>`. On context injection, it
    constructs a `SpringJobFactory` bound to the context. On context
    close, it calls `scheduler.shutdown(true)` (wait for executing jobs
    to finish) and then closes the job factory.

`JobFactory` installation::
    `getScheduler()` is overridden to install the `SpringJobFactory` on
    the returned `Scheduler` before returning it.

Active-mode control::
    A setter `setActiveMode(OnOrOff)` controls whether the scheduler is
    started (`OnOrOff.ON`) or placed in standby (`OnOrOff.OFF`) when
    first obtained. Default is `ON`. Standby means triggers do not fire
    even though the scheduler is otherwise configured and ready; the JMX
    surface can later move it to `STARTED`.

Constructor::
    Takes a `Properties` instance forwarded to the `StdSchedulerFactory`
    constructor. This is where all Quartz configuration — instance name,
    thread pool, job store, cluster settings, datasource — lives.

=== `CronJob`

Value class pairing a `JobDetail` with a `CronTrigger`. A `CronJob` bean
is what deployments declare; `CronJobMapFactoryBean` collects the allowed
ones into a map and `CronJobInitializingBean` installs that map into the
scheduler.

=== `CronJobMapFactoryBean`

A Spring `FactoryBean<Map<JobDetail, List<CronTrigger>>>` that is also a
`BeanPostProcessor`. Two responsibilities:

* `postProcessAfterInitialization` scans every initialized bean. If a bean
  is a `CronJob`, its job key is compared (as `group.name`) against the
  configured `allowedJobIds` list. Matching `CronJob` beans have their
  `JobDetail` / `CronTrigger` pair added to the map. Non-matching beans
  are returned unchanged.
* `getObject()` returns the accumulated map.

The allowlist pattern exists so that a deployment can ship a large,
shared catalog of `CronJob` beans and enable only the subset relevant to
each deployment environment (production, staging, cluster member, batch
node) by configuration.

=== `CronJobInitializingBean`

Spring `InitializingBean` that synchronizes a `Map<JobDetail,
List<CronTrigger>>` with a Quartz `Scheduler` in `afterPropertiesSet()`.
For each entry:

* If the scheduler has no `JobDetail` with the target key, `addJob` is
  called. If it does have one, the two `JobDetail` objects are compared
  field-by-field (durability, recovery, concurrency, persistence, and
  the complete `JobDataMap`); if they differ, `addJob(jobDetail, true)`
  replaces the installed version.
* For each `CronTrigger` in the entry's list, if the scheduler has no
  trigger with that key, `scheduleJob(trigger)` is called. If it does
  have one, the cron expressions are compared; if they differ,
  `rescheduleJob` replaces the trigger.

Errors on individual entries are logged (via `smallmind-scribe-pen`)
and do not abort processing of the remaining entries. This is
deliberate: a misconfigured trigger in one bean definition should not
prevent the rest of the schedule from installing.

=== `JobDeletionInitializingBean`

Spring `InitializingBean` that deletes jobs from the scheduler in
`afterPropertiesSet()`. Takes a `List<JobIdentifier>` and calls
`scheduler.deleteJob(new JobKey(name, group))` for each entry.

Typical use: a job has been removed from the codebase, but the clustered
job store still has its `JobDetail` and `CronTrigger` rows. Adding the
job's name and group to a `JobDeletionInitializingBean` drops the stale
state on the next application startup. Once the cluster is fully restarted,
the deletion entry can be removed.

=== `JobIdentifier`

Value object with `name` and `group` properties. Used as list entries for
`JobDeletionInitializingBean`. Nothing more.

=== `FormattedSchedulerException`

Extends `org.quartz.SchedulerException` with printf-style formatting. Four
constructors: no-args; format + args; throwable + format + args; throwable
alone. Used throughout `SpringJobFactory` so diagnostic messages carry the
offending job key, class name, or bean id inline.

=== `SchedulerMXBean` / `SchedulerMonitor` / `SchedulerStatus`

A standard JMX MBean surface.

`SchedulerMXBean`::
    Interface declaring `start()`, `standby()`, and `status()`. The
    name ends in `MXBean` so JMX treats it as a
    `javax.management.MXBean`-compliant management interface.

`SchedulerMonitor`::
    `StandardMBean` implementation wrapping a Quartz `Scheduler`. `start()`
    and `standby()` delegate to the matching Quartz methods. `status()`
    inspects the scheduler's `isInStandbyMode`, `isStarted`, and
    `isShutdown` flags, in that order of precedence, and returns the
    first matching `SchedulerStatus` value.

`SchedulerStatus`::
    Enum with four constants: `STARTED`, `STANDBY`, `SHUTDOWN`, `UNKNOWN`.
    `UNKNOWN` is returned when the scheduler's state cannot be mapped to
    any of the other three.

The MBean is declared as a bean (`schedulerMonitor`) in the shipped
`quartz.xml`; registering it with the platform `MBeanServer` is the
caller's responsibility (typically through Spring's `MBeanExporter`).

[[spring-wiring]]
== Spring Wiring

The module ships three Spring XML files under
`src/main/resources/spring/org/smallmind/schedule/quartz/`. They are flattened
into the jar root by `schedule-quartz/pom.xml` so they resolve on the
classpath at `spring/org/smallmind/schedule/quartz/*.xml`.

=== `quartz.xml` — the main wiring file

Importable as:

[source,xml]
----
<import resource="classpath:spring/org/smallmind/schedule/quartz/quartz.xml"/>
----

Defines three beans:

`schedulerFactory` (`SpringSchedulerFactory`)::
    The factory. Its constructor takes a `<props>` block with the full
    Quartz configuration: instance name, thread pool (defaults to
    `SimpleThreadPool`, four threads), RMI export enabled, job store
    `org.quartz.impl.jdbcjobstore.JobStoreTX` with `StdJDBCDelegate`, a
    `QRTZ_` table prefix, `useProperties=true`, clustering enabled with
    a 20-second check-in interval, and a datasource block keyed off
    `${jdbc.driver.classname}`, `${jdbc.url.quartz}`,
    `${jdbc.username.quartz}`, and `${jdbc.password.quartz}`. The factory
    depends on `quartzLiquibase` so the `QRTZ_*` tables are guaranteed
    to exist before the factory tries to touch them. The `interruptJobsOnShutdown`
    and `interruptJobsOnShutdownWithWait` flags are both `true`, meaning
    `shutdown(true)` interrupts running jobs and waits for them to return.

`scheduler`::
    Obtained from `schedulerFactory.getScheduler()`. Registered with
    `destroy-method="shutdown"` so it shuts down on context close even if
    `SpringSchedulerFactory`'s `ContextClosedEvent` listener does not fire
    first.

`schedulerMonitor` (`SchedulerMonitor`)::
    Constructed with the `scheduler` bean. Ready for export under an
    `ObjectName` of the caller's choosing.

[NOTE]
====
The shipped `quartz.xml` is a *template*, not a hardened configuration. Most
deployments will need to override the thread pool size, the RMI export flag,
or the clustering interval. The normal pattern is to copy `quartz.xml` into
the consuming application and modify it in place rather than import it
verbatim. The `import` form above is fine for a first integration.
====

=== `quartz-liquibase.xml` — schema creation

Imported transitively by `quartz.xml`. Defines two beans:

`quartzDataSource` (`org.smallmind.persistence.sql.DriverManagerDataSource`)::
    A standalone datasource dedicated to the schema migration. Keyed off
    the same `${jdbc.*.quartz}` property names as the Quartz datasource
    in `quartz.xml`.

`quartzLiquibase` (`org.smallmind.liquibase.spring.SpringLiquibase`)::
    A `smallmind-liquibase` migrator that runs `Quartz.changelog.xml`
    against `quartzDataSource` with `goal=UPDATE`. This is the bean
    `schedulerFactory` depends on.

=== `job.xml` — example template

`job.xml` is a *template*, not an importable configuration; it uses
placeholder strings (`group`, `nameJob`, `jobClass`) as literal values.
Use it as a copy-paste skeleton for defining a scheduled job in your own
Spring configuration. The four beans it declares are the full pattern:

. A `JobDetailImpl` bean describing the job class and metadata.
. A `CronTriggerImpl` bean bound to that job detail with a cron
  expression.
. The job bean itself (the subclass of `QuartzProxyJob`), declared as
  `scope="prototype"`.
. A `CronJobInitializingBean` populated with a map whose entry ties the
  job detail to a list of triggers.

[[jmx-control]]
== JMX Control

The shipped `schedulerMonitor` bean implements `SchedulerMXBean`:

[source,java]
----
public interface SchedulerMXBean {
  void start ()   throws Exception;
  void standby () throws Exception;
  SchedulerStatus status () throws Exception;
}
----

Expose it through Spring's `MBeanExporter`, jolokia, or any other
mechanism that registers MBeans with the platform `MBeanServer`. Once
registered, an operator with JMX access can:

* Call `status()` to inspect whether the scheduler is `STARTED`,
  `STANDBY`, `SHUTDOWN`, or `UNKNOWN`.
* Call `standby()` to halt all trigger firing without shutting down the
  scheduler or redeploying the service. Jobs already executing continue
  to run until they return; no new firings are scheduled. Existing
  `JobDetail` / `CronTrigger` registrations are preserved.
* Call `start()` to return a standby scheduler to active firing.

[[scheduler-states]]
=== Scheduler States

[cols="1,3",options="header"]
|===
| Status
| Meaning

| `STARTED`
| The scheduler is running and triggers are firing.

| `STANDBY`
| The scheduler is paused; trigger firings are suppressed but all job
  and trigger registrations are preserved.

| `SHUTDOWN`
| The scheduler has been stopped and cannot be restarted. Obtain a new
  scheduler from the factory to resume firing.

| `UNKNOWN`
| The scheduler's state did not match `isInStandbyMode`, `isStarted`, or
  `isShutdown`. Typically indicates the scheduler has been constructed
  but not yet started.
|===

[[quick-start]]
== Quick Start

This section walks through adding one scheduled job to a Spring service
that imports the shipped `quartz.xml`.

[[writing-a-job]]
=== Writing A Job

A job is a subclass of `QuartzProxyJob` that overrides `proceed()` and
`cleanup()`. Below is a minimal, realistic example: a job that advances a
counter of records processed from a queue.

[source,java]
----
package com.example.service.schedule;

import java.util.List;
import org.quartz.DisallowConcurrentExecution;
import org.smallmind.schedule.quartz.QuartzProxyJob;

@DisallowConcurrentExecution                                  // <1>
public class QueueDrainJob extends QuartzProxyJob {

  private final QueueClient queueClient;

  public QueueDrainJob (QueueClient queueClient) {            // <2>

    this.queueClient = queueClient;
  }

  @Override
  public boolean logOnZeroCount () {                          // <3>

    return false;
  }

  @Override
  public void proceed ()
    throws Exception {

    List<Message> batch;

    while (!(batch = queueClient.poll(100)).isEmpty()) {
      for (Message message : batch) {
        try {
          queueClient.process(message);
          incCount();                                         // <4>
        } catch (TransientException transientException) {
          setThrowable(transientException, false);            // <5>
        }
      }
    }
  }

  @Override
  public void cleanup () {
  }
}
----
<1> Quartz's `@DisallowConcurrentExecution` prevents two firings of the
    same job from overlapping. Combined with the prototype-scoped bean
    requirement, this is the standard way to get per-firing isolation.
<2> Constructor injection — ordinary Spring. The bean will be
    prototype-scoped in `applicationContext.xml`.
<3> Return `false` because an idle queue is a normal state and does not
    warrant a log line.
<4> Record each processed message. `getCount()` will report this value
    in the completion log line and through JMX if the subclass is
    exposed there.
<5> Record the exception in the per-firing list *without* flipping the
    outcome to `FAILURE`, because a transient failure on one message is
    expected and should not mark the whole firing as failed.

=== Declaring The Job Bean

The job bean must be prototype-scoped:

[source,xml]
----
<bean id="scheduled.queueDrainJob" class="com.example.service.schedule.QueueDrainJob" scope="prototype">
  <constructor-arg ref="queueClient"/>
</bean>
----

The bean id matches the job key (`group.name`) so that `SpringJobFactory`
resolves it unambiguously when multiple jobs share the same class — see
<<job-bean-resolution,Job Bean Resolution>>.

[[declaring-a-cron-schedule]]
=== Declaring A Cron Schedule

Declare a `JobDetail` and a `CronTrigger` and hand them to
`CronJobInitializingBean`.

[source,xml]
----
<bean id="scheduled.queueDrainDetail" class="org.quartz.impl.JobDetailImpl">
  <property name="description" value="Drains the pending-message queue"/>
  <property name="group" value="scheduled"/>
  <property name="name" value="queueDrainJob"/>
  <property name="jobClass" value="com.example.service.schedule.QueueDrainJob"/>
  <property name="durability" value="true"/>
  <property name="requestsRecovery" value="false"/>                     <!--1-->
</bean>

<bean id="scheduled.queueDrainTrigger" class="org.quartz.impl.triggers.CronTriggerImpl">
  <property name="description" value="Every five minutes"/>
  <property name="group" value="scheduled"/>
  <property name="name" value="queueDrainTrigger"/>
  <property name="jobGroup" value="scheduled"/>
  <property name="jobName" value="queueDrainJob"/>
  <property name="cronExpression" value="0 0/5 * * * ?"/>
</bean>

<bean class="org.smallmind.schedule.quartz.spring.CronJobInitializingBean">
  <property name="scheduler" ref="scheduler"/>
  <property name="jobMap">
    <map>
      <entry key-ref="scheduled.queueDrainDetail">
        <list>
          <ref bean="scheduled.queueDrainTrigger"/>
        </list>
      </entry>
    </map>
  </property>
</bean>
----
<1> Set to `true` only if the clustered deployment should attempt to
    recover a failed firing on another node. For most scheduled batch
    work — where the next firing will do the remaining work — leave
    this `false`.

On context startup, `CronJobInitializingBean` inspects the scheduler's
job store: if no `scheduled.queueDrainJob` is present, it adds one; if
the installed `JobDetail` differs, it replaces it; if the trigger is
missing, it schedules it; if the cron expression has changed, it
reschedules.

=== Using The Allowlist Pattern

For services that ship a catalog of `CronJob` beans and enable only a
subset per environment, the allowlist pattern is:

[source,xml]
----
<bean id="scheduled.queueDrainCronJob" class="org.smallmind.schedule.quartz.spring.CronJob">
  <property name="jobDetail" ref="scheduled.queueDrainDetail"/>
  <property name="cronTrigger" ref="scheduled.queueDrainTrigger"/>
</bean>

<bean id="scheduled.otherCronJob" class="org.smallmind.schedule.quartz.spring.CronJob">
  <property name="jobDetail" ref="scheduled.otherDetail"/>
  <property name="cronTrigger" ref="scheduled.otherTrigger"/>
</bean>

<bean id="cronJobMap" class="org.smallmind.schedule.quartz.spring.CronJobMapFactoryBean">
  <property name="allowedJobIds">
    <list>
      <value>scheduled.queueDrainJob</value>                       <!--1-->
    </list>
  </property>
</bean>

<bean class="org.smallmind.schedule.quartz.spring.CronJobInitializingBean">
  <property name="scheduler" ref="scheduler"/>
  <property name="jobMap" ref="cronJobMap"/>
</bean>
----
<1> Only the `scheduled.queueDrainJob` key is admitted. The other
    `CronJob` bean is instantiated — it is a Spring bean in the context —
    but its job / trigger pair is silently dropped from the map and
    never reaches the scheduler.

The allowlist entries match `JobDetail` keys (`group.name`), not
bean ids. Check `CronJob.getJobDetail().getKey()` when in doubt.

[[retiring-obsolete-jobs]]
=== Retiring Obsolete Jobs

A clustered job store remembers jobs across deployments. Deleting a
`CronJob` bean from Spring XML does not remove the row — the scheduler
will still fire it. Retire the job explicitly:

[source,xml]
----
<bean class="org.smallmind.schedule.quartz.spring.JobDeletionInitializingBean">
  <property name="scheduler" ref="scheduler"/>
  <property name="jobIdentifierList">
    <list>
      <bean class="org.smallmind.schedule.quartz.spring.JobIdentifier">
        <property name="group" value="scheduled"/>
        <property name="name" value="legacyExportJob"/>
      </bean>
    </list>
  </property>
</bean>
----

Once every cluster member has deployed the configuration with the
deletion entry and run at least once, the entry can be removed.

[[behavioral-reference]]
== Behavioral Reference

This section documents edge cases, invariants, and runtime details that
readers will look for by name.

[[completion-log-rules]]
=== Completion Log Rules

`QuartzProxyJob.execute()` emits at most one structured log line per
firing. The decision tree is:

. If `isEnabled()` returned `false`, nothing is logged. The firing was
  silently skipped.
. If the job was interrupted (`InterruptedException` caught in
  `proceed()`), an `INFO` line is emitted through the subclass's
  `scribe-pen` logger with the template
  `Job(<simpleName>) start(<start>) has been interrupted`.
. Otherwise, if the outcome was `FAILURE`, or `getCount() > 0`, or
  `logOnZeroCount()` returned `true`, a line is emitted with the
  template
  `Job(<simpleName>) start(<start>) stop(<stop>) count(<count>) state(<status>)`.
  The log level is `WARN` when the status is `FAILURE`, `INFO` otherwise.
. Otherwise (status `SUCCESS`, count zero, `logOnZeroCount()` returned
  `false`) no line is emitted. An idle job runs silently.

The logger is obtained from `LoggerManager.getLogger(this.getClass())`,
so each job class has its own logger name for filtering.

=== Exception Handling

Inside `QuartzProxyJob.execute()`:

* `InterruptedException` transitions status to `INTERRUPTED`. It does
  *not* go through `setThrowable`; no entry is added to the throwable
  list.
* Any other `Exception` is passed to `setThrowable(exception)`, which
  appends it to the throwable list, sets status to `FAILURE`, and
  logs it immediately at `ERROR` level.
* `cleanup()` is invoked in a `finally` block. Any exception from
  `cleanup()` is caught, logged at `ERROR` level, and swallowed — it
  does not propagate and does not change the job's status.

A subclass that wants to record an error without marking the firing as
failed must call `setThrowable(throwable, false)` directly rather than
let an exception bubble out of `proceed()`.

[[job-bean-resolution]]
=== Job Bean Resolution

`SpringJobFactory.newJob` is called by Quartz each time a trigger fires.
Its resolution rules, in order:

. `applicationContext.getBeansOfType(jobClass)` returns zero beans →
  `FormattedSchedulerException` "No job(&lt;key&gt;) of type(&lt;class&gt;) is
  present in the application context".
. Returns exactly one bean → that bean is used, after verifying it is
  prototype-scoped.
. Returns multiple beans → the factory searches for one whose bean name
  equals `jobDetail.getKey().toString()` (i.e. `group.name`). If found,
  that one is used after verifying it is prototype-scoped. If not,
  `FormattedSchedulerException` "Multiple jobs of type(&lt;class&gt;) are
  present in the application context, but none with an exact
  id(&lt;key&gt;)".
. The resolved bean must be prototype-scoped;
  `applicationContext.isPrototype(beanName)` must return `true`. A
  singleton or request-scoped bean throws `FormattedSchedulerException`
  "The matching job(&lt;key&gt;) with id(&lt;beanId&gt;) in the application
  context is not a prototype bean".

The practical consequence is the naming convention: declare job beans
with `id="group.name"` matching the `JobKey` of the corresponding
`JobDetail`. If every job class has exactly one bean in the context,
resolution succeeds on step 2 and the bean id is unconstrained. If
multiple jobs share a class, resolution succeeds on step 3 only when
the bean ids are `group.name`.

=== Thread Registration And Interrupt Semantics

On entry to `execute()`, the calling thread (the Quartz worker) is
stored in `threadRef` (an `AtomicReference<Thread>`). On exit, whether
normal or exceptional, `threadRef` is cleared. `interrupt()` reads
`threadRef` and calls `Thread.interrupt()` on it if non-null.

Consequences:

* Calling `interrupt()` between firings is a no-op.
* Calling `interrupt()` concurrently with a firing interrupts the
  worker thread. `proceed()` must either cooperate with interruption
  (check `Thread.interrupted()` in loops, propagate
  `InterruptedException` from blocking calls) or the interrupt is
  silently lost.
* `scheduler.shutdown(true)` (as configured by
  `interruptJobsOnShutdownWithWait=true` in the shipped
  `quartz.xml`) reaches the worker thread through Quartz's own
  interrupt machinery, which in turn calls `interrupt()` on the
  job.

=== Schedule Synchronization Semantics

`CronJobInitializingBean.afterPropertiesSet()` is idempotent within a
single startup but is *not* safe to run concurrently on multiple
cluster members. The decision to add, replace, schedule, or reschedule
is based on reading the installed state, deciding, and writing — a
classic check-then-act race.

In practice, Quartz's clustered job store serializes writes, so the
worst case is that two cluster members both see no installed trigger,
both call `scheduleJob`, and one of them gets a Quartz duplicate-key
error. The error is logged and the rest of the configuration proceeds.
The net effect is correct: exactly one copy of each trigger ends up in
the store.

If you want deterministic single-writer behavior, stagger cluster member
startups or keep the bean on a single "leader" deployment only.

=== JobDetail Comparison

When `CronJobInitializingBean` finds an installed `JobDetail` with the
same key, it uses `isSame(desired, installed)` to decide whether to
replace. The comparison covers:

* `isDurable`
* `requestsRecovery`
* `isConcurrentExecutionDisallowed`
* `isPersistJobDataAfterExecution`
* The full `JobDataMap`: identical set of keys, identical values under
  each key (via `Object.equals`).

Any difference triggers `addJob(jobDetail, true)`, which replaces the
stored definition. The comparison does *not* include description or
job class — change the class, and the installed `JobDetail` stays
wrong. This is rare in practice because the class is what caused the
change in the first place; either evict the job explicitly through
`JobDeletionInitializingBean` first, or bump a `JobDataMap` entry to
force a replace.

=== Trigger Comparison

Only the cron expression is compared. Changes to misfire instructions,
time zone, priority, or description do not trigger a reschedule. Work
around this by appending a version suffix to the cron expression when
you need to force a reschedule, or by evicting the trigger first (delete
the job and re-add it on the next deployment).

[[operational-notes]]
== Operational Notes

=== Dependencies

`schedule-quartz` brings in Quartz, Spring beans, Spring context, and
`smallmind-scribe-pen`. If you use the Liquibase wiring, you additionally
need `smallmind-liquibase` and its transitive Liquibase core jar on the
classpath. The `quartzDataSource` bean in `quartz-liquibase.xml` uses
`org.smallmind.persistence.sql.DriverManagerDataSource`, which requires
`smallmind-persistence` to be present.

=== Startup Ordering

`SpringSchedulerFactory` is declared with `depends-on="quartzLiquibase"`
so the `QRTZ_*` tables exist before Quartz tries to query them. If you
replace `quartzLiquibase` with your own migration tool (Flyway, a
standalone Liquibase runner, hand-managed DDL), you must preserve the
ordering: the scheduler cannot be constructed before its tables exist.

=== Shutdown Ordering

Two mechanisms wait for executing jobs to finish:

. The `scheduler` bean is declared with `destroy-method="shutdown"`,
  which calls `scheduler.shutdown()` (no wait).
. `SpringSchedulerFactory.onApplicationEvent(ContextClosedEvent)`
  calls `scheduler.shutdown(true)` (wait).

Both paths execute on context close. The `true` argument in path 2
dominates: `shutdown(true)` waits for executing jobs, and Quartz
tolerates a second `shutdown()` call as a no-op. The
`interruptJobsOnShutdown` properties in `quartz.xml` additionally cause
Quartz to call `interrupt()` on `InterruptableJob` instances before
waiting, so well-behaved jobs return quickly.

=== Logging

All module-level logging goes through `smallmind-scribe-pen`. The
logger name is the job class's fully-qualified name for job execution,
and the class name of the bean (`CronJobInitializingBean`,
`SpringSchedulerFactory`) for infrastructure events. Configure the
filter level for these names explicitly in your `scribe` configuration
if you want to suppress the one-line-per-firing completion output.

=== JMX Registration

The shipped `schedulerMonitor` bean is constructed but not registered.
Add a Spring `MBeanExporter` bean to expose it; the MBean's
`ObjectName` is the caller's choice. A common pattern:

[source,xml]
----
<bean class="org.springframework.jmx.export.MBeanExporter">
  <property name="beans">
    <map>
      <entry key="com.example:type=Scheduler,name=Default" value-ref="schedulerMonitor"/>
    </map>
  </property>
  <property name="server" ref="platformMBeanServer"/>
</bean>

<bean id="platformMBeanServer" class="org.springframework.jmx.support.MBeanServerFactoryBean">
  <property name="locateExistingServerIfPossible" value="true"/>
</bean>
----

[[limitations-and-known-issues]]
== Limitations And Known Issues

Declarative install covers cron triggers only::
    `CronJobInitializingBean` is specifically a cron-trigger installer.
    `SimpleTrigger`, `DailyTimeIntervalTrigger`, and calendar-interval
    triggers must be installed against the resolved `Scheduler` bean
    directly. This is a narrowing choice — most recurring service work
    is naturally cron-shaped.

Trigger comparison is cron-expression-only::
    `CronJobInitializingBean` replaces a trigger only when its cron
    expression has changed. Changes to misfire instructions, time zone,
    or priority are not detected. Work around by forcing a cron-expression
    change (e.g. append a no-op space normalization) or by deleting and
    re-adding the trigger.

JobDetail comparison ignores class and description::
    Durability, recovery, concurrency, persistence, and the full
    `JobDataMap` are compared; the `Class<? extends Job>` and
    description are not. A change to the job class alone does not
    trigger a replace. Change a `JobDataMap` entry to force a replace.

Allowlist matches on `group.name`, not bean id::
    `CronJobMapFactoryBean` compares against the `JobDetail.getKey()`
    of the `CronJob` bean, not its Spring bean id. If the two differ,
    the allowlist entry must be the job key, not the bean id.

Prototype scope is enforced at firing time, not at startup::
    A singleton job bean surfaces only when a trigger actually fires.
    Include every scheduled job in a smoke test that forces at least
    one firing if you want catch this misconfiguration during
    deployment validation.

Single active-mode enum::
    `SpringSchedulerFactory.setActiveMode(OnOrOff)` has exactly two
    values. There is no third "leave whatever state Quartz's own
    startup produced" option. If you want deployment-specific nuance
    (e.g. "start on the primary, standby on replicas"), compute the
    value externally and inject it.

`SchedulerMonitor` exposes three operations, not nine::
    The JMX surface is intentionally narrow: start, standby, status.
    There is no pause-trigger-by-name, no trigger-now, no remove-job,
    and no last-fired timestamp. Operations that target individual
    jobs are available through the Quartz `Scheduler` bean programmatically;
    wrap them in your own MBean if you need them over JMX.

`ProxyJob.setThrowable(Throwable)` contract differs from the two-argument
form::
    The one-argument form, as implemented by `QuartzProxyJob`, always
    marks the job `FAILURE`. Only the two-argument
    `setThrowable(Throwable, boolean)` extension — which is not on the
    `ProxyJob` interface — allows recording an exception without
    changing the outcome.
