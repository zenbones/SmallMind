[[nutsnbolts, NutsnBolts]]
= NutsnBolts

[partintro]
NutsnBolts is Smallmind's shared utility module. It collects reusable glue-code families in one place: command parsing, property expansion, resource resolution, CSV, file-tree operations, archives, reflection helpers, security primitives, Spring property loading, time utilities, validation constraints, and XML support.

The module stays library-safe. You adopt the packages you need, and startup, shutdown, dependency selection, and application behavior remain in caller code.

Audience: library authors and application developers who need explicit utility APIs without hidden runtime ownership.
Versioning: examples use `${smallmind.version}` for Smallmind artifacts; keep related modules on the same version unless you intentionally need to split them.

[[nutsnbolts-why-use, Why Use NutsnBolts]]
== Why Use NutsnBolts

Use NutsnBolts when you want a small amount of reusable infrastructure without letting a utility library take ownership of your runtime.

What it does well:

* keeps APIs explicit and composable instead of inventing a module-wide framework
* makes optional integrations stay optional, especially around Spring, Jakarta APIs, FreeMarker, BouncyCastle, and SnakeYAML
* gives you narrow adapters around common seams such as resource loading, placeholder expansion, archive handling, password storage, and XML parsing
* keeps caller policy visible: retry counts, startup order, logging, transport behavior, key management, and container choice stay in your code

It works best as a selective toolkit: pull in the packages that match your integration seam and leave the rest of the runtime model untouched.

=== Command, Configuration, And Resource Selection

* <<nutsnbolts-pkg-apt,`apt`>>: compile-time helpers for annotation processors that need to inspect mirrors, defaults, and `TypeMirror` lists cleanly

* <<nutsnbolts-pkg-command-template,`command.template`>>: explicit grammar model built from `Template`, `Option`, and argument types
* <<nutsnbolts-pkg-command-sax,`command.sax`>>: load a command grammar from a sidecar `.arguments.xml`
* <<nutsnbolts-pkg-property,`property`>>: nested placeholder expansion with optional system and environment lookup and optional decryption
* <<nutsnbolts-pkg-resource,`resource`>>: classpath, file, jar, and URL resource resolution through one `Resource` abstraction
* <<nutsnbolts-pkg-spring,`spring`>>: explicit Spring property loading, placeholder resolution, extension loading, and property access
* <<nutsnbolts-pkg-spring-property,`spring.property`>>: normalize `.properties` and YAML files into `PropertyEntry` streams

=== Files, Streams, Data Formats, And Transport Helpers

* <<nutsnbolts-pkg-csv,`csv`>>: streaming CSV read, write, and callback parsing
* <<nutsnbolts-pkg-io,`io`>>: recursive copy and delete, path filtering, circular buffers, and stream helpers
* <<nutsnbolts-pkg-zip,`zip`>>: zip and jar compression, walking, and extraction
* <<nutsnbolts-pkg-freemarker,`freemarker`>>: FreeMarker loaders that understand Smallmind resource schemes
* <<nutsnbolts-pkg-email,`email`>>: `Mail` plus `Postman` for SMTP or SMTPS delivery and optional FreeMarker interpolation
* <<nutsnbolts-pkg-http,`http`>>: thin `HttpURLConnection` helpers and codec utilities
* <<nutsnbolts-pkg-json,`json`>>: JAXB-friendly adapters for binary data, dates, enums, and HTTP method values
* <<nutsnbolts-pkg-xml,`xml`>>: DOM rendering plus classpath-aware protocol, entity, and URI resolution
* <<nutsnbolts-pkg-xml-sax,`xml.sax`>>: pluggable SAX parsing built around document and element extenders

* <<nutsnbolts-pkg-context,`context`>>: typed thread-local context stacks with export and import support

[[nutsnbolts-quick-start, Example Catalog]]
== Example Catalog

These examples are intentionally small. They are meant to show how the APIs are supposed to be entered, not to provide a framework around them.

=== Command Parsing With An Explicit Grammar

[source,java]
----
import org.smallmind.nutsnbolts.command.CommandLineParser;
import org.smallmind.nutsnbolts.command.OptionSet;
import org.smallmind.nutsnbolts.command.template.NoneArgument;
import org.smallmind.nutsnbolts.command.template.Option;
import org.smallmind.nutsnbolts.command.template.SingleArgument;
import org.smallmind.nutsnbolts.command.template.Template;

Template template = new Template(
    "example",
    new Option("verbose", 'v', false, NoneArgument.instance()),
    new Option("config", 'c', true, new SingleArgument("path"))
);

OptionSet optionSet = CommandLineParser.parseCommands(
    template,
    new String[] {"--config", "app.yml", "-v"}
);

boolean verbose = optionSet.containsOption("verbose");
String configPath = optionSet.getArgument("config");
----

`command` parses; `command.template` declares the grammar. If you want to allow extra trailing arguments, call `parseCommands(template, args, true)` and read them back through `OptionSet.getRemaining()`.

=== Nested Property Expansion

[source,java]
----
import java.util.HashMap;
import java.util.Map;
import org.smallmind.nutsnbolts.property.PropertyExpander;

PropertyExpander expander = new PropertyExpander();
Map<String, Object> values = new HashMap<>();
values.put("app.name", "smallmind");
values.put("welcome", "hello ${app.name}");

String expanded = expander.expand("${welcome}", values);
----

`property` supports nested placeholders and circular-reference detection. If you need encrypted values such as `!{secret.key}`, construct the expander with `new PropertyClosure(decryptor)`.

=== Resolving Classpath Or File Resources

[source,java]
----
import java.io.InputStream;
import org.smallmind.nutsnbolts.resource.Resource;
import org.smallmind.nutsnbolts.resource.ResourceParser;
import org.smallmind.nutsnbolts.resource.ResourceTypeResourceGenerator;

ResourceParser resourceParser = new ResourceParser(new ResourceTypeResourceGenerator());
Resource resource = resourceParser.parseResource("classpath:/schema/example.xsd");

try (InputStream inputStream = resource.getInputStream()) {
  byte[] bytes = inputStream.readAllBytes();
}
----

The built-in schemes come from `ResourceTypeResourceGenerator`. If no scheme is supplied, `resource` assumes `file:`.

=== CSV Round Trip

[source,java]
----
import java.io.ByteArrayInputStream;
import java.io.ByteArrayOutputStream;
import org.smallmind.nutsnbolts.csv.CSVReader;
import org.smallmind.nutsnbolts.csv.CSVWriter;

ByteArrayOutputStream out = new ByteArrayOutputStream();
try (CSVWriter writer = new CSVWriter(out, "id", "name")) {
  writer.write("1", "Ada");
}

try (CSVReader reader = new CSVReader(new ByteArrayInputStream(out.toByteArray()), true)) {
  String[] headers = reader.getHeaders();
  String[] row = reader.readLine();
}
----

`CSVReader` supports multiline quoted fields and optional headers. `CSVWriter` enforces a stable field count per row.

=== Copying And Cleaning A Tree

[source,java]
----
import java.nio.file.Path;
import org.smallmind.nutsnbolts.io.FileUtility;

FileUtility.copyBuilder(Path.of("source"), Path.of("target"))
    .includeSourceDirectory(true)
    .build();

FileUtility.deleteBuilder(Path.of("target"))
    .includeTargetDirectory(true)
    .build();
----

The include flags matter. `includeSourceDirectory(true)` copies the source directory itself under the destination. `includeTargetDirectory(true)` deletes the root directory as well as its contents.

=== Building And Exploding A Zip

[source,java]
----
import java.nio.file.Path;
import org.smallmind.nutsnbolts.zip.CompressionType;

CompressionType.ZIP.compress(Path.of("build/output"), Path.of("build/output.zip"));
CompressionType.ZIP.explode(Path.of("build/output.zip"), Path.of("build/unpacked"));
----

Use `CompressionType.JAR` instead when you want jar semantics and optional manifest support.

=== Password Hashing

[source,java]
----
import org.smallmind.nutsnbolts.security.password.PBKDF2PasswordEngine;
import org.smallmind.nutsnbolts.security.password.PasswordEngine;

PasswordEngine passwordEngine = new PBKDF2PasswordEngine();
String stored = passwordEngine.encrypt("p@ssw0rd");
boolean ok = passwordEngine.match("p@ssw0rd", stored);
----

The returned string is the stored credential. `security.password` is for password verification, not for general-purpose encryption.

=== Sending Templated Mail

[source,java]
----
import java.util.HashMap;
import org.smallmind.nutsnbolts.email.Authentication;
import org.smallmind.nutsnbolts.email.Mail;
import org.smallmind.nutsnbolts.email.Postman;

Postman postman = new Postman("smtp.example.net", 587, Authentication.NONE, true);
Mail mail = new Mail(
    "noreply@example.net",
    "ops@example.net",
    "Deployment notice",
    "Build ${build} is live."
);

HashMap<String, Object> interpolationMap = new HashMap<>();
interpolationMap.put("build", "2026.04.10.1");

postman.send(mail, interpolationMap);
----

If `freemarker` is on the classpath, `Postman` can treat the message body as a template. SMTP credentials, TLS policy, and retry behavior remain caller-owned.

=== Reading Nested Bean Properties By Path

[source,java]
----
import org.smallmind.nutsnbolts.reflection.bean.BeanUtility;

String city = (String)BeanUtility.executeGet(order, "customer.address.city", false);
BeanUtility.executeSet(order, "customer.address.city", "Oakland");
----

`reflection.bean` is useful when property names are runtime data, especially in mapping, admin, or expression-style code.

=== Time Parsing, Spread Expansion, And Sortable IDs

[source,java]
----
import java.time.ZonedDateTime;
import org.smallmind.nutsnbolts.time.TimeUtility;
import org.smallmind.nutsnbolts.util.SnowflakeId;
import org.smallmind.nutsnbolts.util.Spread;

ZonedDateTime cutoff = TimeUtility.parse("2026-04-10T09:30:00-07:00");
String correlationId = SnowflakeId.newInstance().generateDottedString();
String[] shards = Spread.calculate("1..3,7");
----

`time` and `util` work well together in operational code: parse user input, expand concise ranges, and stamp events with sortable IDs.

=== Spring Placeholder Loading

[source,xml]
----
<bean class="org.smallmind.nutsnbolts.spring.PropertyPlaceholderConfigurer">
  <property name="locations">
    <list>
      <value>classpath:/application.properties</value>
      <value>file:/etc/myapp/override.yaml</value>
    </list>
  </property>
  <property name="systemPropertyMode" value="OVERRIDE"/>
  <property name="ignoreResourceNotFound" value="true"/>
</bean>
----

This is the Spring integration point for `.properties` and YAML loading. It keeps the model explicit: you list resource locations, optionally add a decryptor, and choose how system properties or environment variables interact with file-backed values.

=== Command, Configuration, And Resource Packages

[[nutsnbolts-pkg-apt]]
==== `apt`

Use `apt` inside annotation processors. It exists to keep Smallmind processors from repeating the same annotation-mirror and default-value boilerplate.

Start with: `AptUtility`.

Know this:

* it is compile-time support only; runtime code should not depend on it
* it is useful for direct annotations, meta-annotations, defaulted values, and `TypeMirror` to `TypeElement` conversion
* it stays deliberately thin and does not impose a code-generation model

[[nutsnbolts-pkg-command]]
==== `command`

Use `command` when you want a declared command grammar instead of ad hoc string checks.

Start with: `CommandLineParser.parseCommands(...)` and `OptionSet`.

Know this:

* GNU-style long options (`--name`) and single-character flags (`-n`) are both supported
* repeated option use is only accepted when the option argument type is `MULTIPLE`
* quoted values may span tokens and retain their surrounding quote characters

[[nutsnbolts-pkg-command-template]]
==== `command.template`

`command.template` is where the grammar lives. It is the package you use to declare what options are legal before `command` ever parses anything.

Start with: `Template`, `Option`, `SingleArgument`, `ListArgument`, `MultipleArgument`, `EnumeratedArgument`, and `NoneArgument.instance()`.

Know this:

* option names and flags must be unique within the whole template
* child options are explicit dependencies; parsing fails if a child is supplied without its parent
* `Template.toString()` renders a readable usage-style description of the option tree

[[nutsnbolts-pkg-command-sax]]
==== `command.sax`

Use `command.sax` when the command grammar belongs in XML rather than in Java constructors.

Start with: `Template.createTemplate(entryClass)` and the sidecar `.arguments.xml` file.

Know this:

* the XML file must live on the classpath under the entry class name with the suffix `.arguments.xml`
* the loader uses `ExtensibleSAXParser` and `SmallMindProtocolResolver` under the hood
* this is a declaration format, not a second parser; the resulting `Template` still feeds `CommandLineParser`

[[nutsnbolts-pkg-property]]
==== `property`

Use `property` for nested placeholder expansion in strings or maps where you want caller-controlled lookup order.

Start with: `PropertyExpander`, `PropertyClosure`, and `SystemPropertyMode`.

Know this:

* default placeholders are `${...}`
* if you supply `new PropertyClosure(decryptor)`, encrypted placeholders with the default `!{...}` prefix are also supported
* unresolved-key behavior is controlled by `ignoreUnresolvableProperties`, and circular references are detected explicitly

[[nutsnbolts-pkg-resource]]
==== `resource`

Use `resource` when you want one abstraction over `classpath:`, `file:`, `jar:`, and `url:` inputs.

Start with: `ResourceParser`, `Resource`, and `ResourceTypeResourceGenerator`.

Know this:

* the parser needs at least one registered `ResourceGenerator`; the built-in generator is `new ResourceTypeResourceGenerator()`
* if the identifier has no scheme prefix, parsing falls back to `file`
* the package is intentionally read-oriented; it normalizes access, not storage policy

[[nutsnbolts-pkg-spring]]
==== `spring`

Use `spring` when you already have a Spring container and want explicit property loading or a few assembly helpers without moving to a convention-heavy model.

Start with: `PropertyPlaceholderConfigurer` and `SpringPropertyAccessor`.

Know this:

* `PropertyPlaceholderConfigurer` accepts explicit resource locations and can also discover extra locations from `PropertyConfiguration` beans
* placeholder resolution order is controlled by `SystemPropertyMode` plus `searchSystemEnvironment`
* property loading and debug output are printed to standard out by the configurer

[[nutsnbolts-pkg-spring-property]]
==== `spring.property`

Use `spring.property` when the job is just file-format handling for property sources.

Start with: `PropertyFileType`, `PropertyHandler`, `PropertyEntry`, and `YamlPropertyHandler`.

Know this:

* `.properties` and YAML are both normalized into iterable property entries
* YAML lists are flattened with numeric path segments such as `servers.0.host`
* precedence, placeholder expansion, and decryption stay with `spring` and `property`, not this package

=== Files, Streams, Data Formats, And Transport Helpers

[[nutsnbolts-pkg-csv]]
==== `csv`

Use `csv` when you want a small streaming CSV reader or writer without bringing in a larger data-binding library.

Start with: `CSVReader`, `CSVWriter`, or `CSVParser` plus `CSVLineHandler`.

Know this:

* `CSVReader` supports multiline quoted fields and optional header capture
* `CSVWriter` enforces a fixed field count for each row written by one writer instance
* malformed quoted input fails early instead of trying to guess intent

[[nutsnbolts-pkg-io]]
==== `io`

`io` is the file-system and stream utility package.

Start with: `FileUtility.copyBuilder(...)`, `FileUtility.deleteBuilder(...)`, and `CircularBufferIOStream`.

Know this:

* recursive copy and delete are explicit about whether the root directory itself is included
* path filters are opt-in and caller-defined
* the circular-buffer types are useful when you need an `InputStream` or `OutputStream` bridge between producer and consumer code

[[nutsnbolts-pkg-zip]]
==== `zip`

Use `zip` when you need zip or jar creation and extraction with the same surface area.

Start with: `CompressionType.ZIP` or `CompressionType.JAR`.

Know this:

* `compress(...)`, `explode(...)`, and `walk(...)` are the core operations
* jar manifests matter only for `CompressionType.JAR`
* the package is archive-oriented, not a virtual filesystem abstraction

[[nutsnbolts-pkg-freemarker]]
==== `freemarker`

Use `freemarker` only if you already use FreeMarker and want template names to resolve through Smallmind resources.

Start with: `ResourceTemplateLoader`.

Know this:

* template names can point at `classpath:`, `file:`, `jar:`, or `url:` resources
* this is a loader integration, not a replacement configuration model for FreeMarker itself
* it is a good fit when template lookup needs to follow the same rules as the rest of your resource loading

[[nutsnbolts-pkg-email]]
==== `email`

Use `email` for explicit mail composition and delivery when Jakarta Mail is already acceptable in your stack.

[[nutsnbolts-runtime-notes, Runtime Notes]]
== Runtime Notes

=== Threading And State

* `ContextFactory` uses `InheritableThreadLocal`, so child threads inherit copies of current context stacks.
* `PerApplicationContext` uses its own `InheritableThreadLocal` map and offers both `ContextCarrier` and `wrapThreadFactory(...)` for propagation.
* `Retry.execute(...)` creates a new thread for each attempt and blocks until that thread finishes.
* `CircularBuffer` and related stream bridge types are synchronized and interruption-aware; caller code still decides retry and shutdown behavior.

=== Parsing, Expansion, And Data Format Details

* `CommandLineParser` preserves the surrounding quote characters on quote-delimited arguments.
* Only `MULTIPLE` arguments may repeat. Repeating a `NONE`, `SINGLE`, `LIST`, or `ENUMERATED` option is an error.
* `PropertyExpander` supports nested placeholders and circular-reference detection. Unresolved-key behavior is entirely controlled by `ignoreUnresolvableProperties`.
* `ResourceParser` assumes `file` when no scheme prefix is present, but only if a registered generator can handle that scheme.
* `CSVReader` supports multiline quoted fields and throws `CSVParseException` if the stream ends before a quoted field closes.
* `YamlPropertyHandler` flattens lists into numeric dot segments such as `servers.0.host`.

=== Files, Archives, And Resource Behavior

* `FileUtility` copy and delete operations are explicit about whether the root itself is included. That flag changes results in meaningful ways.
* `CompressionType` exposes the same surface area for zip and jar handling, but manifest support only matters for jars.
* `Resource` is a read abstraction. If you need write policy, choose it in caller code.

=== Security And TLS Notes

* `EncryptionUtility` is intentionally thin. Algorithm selection, provider selection, key storage, and rotation policy stay outside the module.
* Password engines return self-contained stored values. Migration and rehash strategy remain caller-owned.
* `KeyStoreUtility.construct(...)` writes a JKS file into the current user's home directory and fills in defaults when alias, name, or password are omitted.
* Permissive SSL helpers in `ssl` are deliberately unsafe for production and belong only in controlled test or local-dev flows.

=== Spring Notes

* `PropertyPlaceholderConfigurer` can receive locations directly or gather them from `PropertyConfiguration` beans.
* `.properties` and YAML are both supported through `spring.property`.
* Missing-resource behavior is controlled by `ignoreResourceNotFound`.
* `PropertyPlaceholderConfigurer` prints property-loading and optional debug banners to standard out.
* `RMIDaemonInitializingBean` depends on the external `rmid` executable being available on `PATH`.

[[nutsnbolts-troubleshooting, Troubleshooting]]
== Troubleshooting

=== `ResourceParser` Says No `ResourceFactory` Or Cannot Handle A Scheme

Typical cause:

* the parser was created without any registered generator
* the scheme is not one of the built-in `classpath`, `file`, `jar`, or `url` schemes

Fix:

* use `new ResourceParser(new ResourceTypeResourceGenerator())` for the built-in schemes
* register your own `ResourceGenerator` if you need another scheme

=== Placeholder Expansion Fails On An Unresolved Key

Typical cause:

* the supplied map does not contain the key
* the `SystemPropertyMode` is not what you expected
* unresolved placeholders are configured to fail instead of being ignored

Fix:

* verify the source map and system properties or environment variables
* check whether `OVERRIDE`, `FALLBACK`, or another `SystemPropertyMode` is correct for your use case
* set `ignoreUnresolvableProperties` only if leaving unresolved content in place is acceptable

=== `CSVReader` Fails With An Open Quoted Field

Typical cause:

* the CSV source has a multiline field that never closes its quote

Fix:

* repair the source data rather than trying to work around it
* if the data is valid multiline CSV, ensure the full stream is being read and not truncated upstream

=== Spring Property Loading Is Noisier Than Expected

Typical cause:

* `PropertyPlaceholderConfigurer` logs loading and optional debug information to standard out by design

Fix:

* treat that output as expected module behavior
* avoid using the debug-key feature if you need quieter startup output

=== `KeyStoreUtility` Wrote A File Somewhere Unexpected

Typical cause:

* omitted keystore name means the default path is `${user.home}/keystore.jks`

Fix:

* pass an explicit name when constructing the keystore
* remember that omitted alias defaults to `mykeystore` and omitted password defaults to `changeit`

[[nutsnbolts-verification, Verification Checklist]]
== Verification Checklist

Use the smallest realistic validation pass for the packages you adopt.

1. Command parsing: parse valid and invalid templates, including required-option and parent-child failures.
2. Property expansion: verify nested substitutions, encrypted placeholder handling if used, unresolved-key policy, and circular-reference detection.
3. Resource loading: resolve one `classpath:` and one `file:` resource through the same `ResourceParser`.
4. CSV: round-trip a small dataset and confirm malformed quoted input fails as expected.
5. File and archive operations: copy, delete, compress, and explode a temporary tree with the exact include flags you intend to use.
6. Password engine: encrypt and then match both success and failure cases.
7. Spring property loading, if used: load both `.properties` and YAML and verify placeholder precedence.
8. XML SAX, if used: parse one representative document through your `DocumentExtender` and element extender chain.
