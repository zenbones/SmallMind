== Overview

The `smallmind-kafka` module is a thin, opinionated facade over the stock
https://kafka.apache.org/[Apache Kafka] `kafka-clients` library. It is
organized as a single publishable artifact today:

* **`kafka-utility`** — four small classes that solve the recurring integration
  chores that the `kafka-clients` API expects every caller to redo: assembling
  a bootstrap server string from a set of broker descriptors, spinning up a
  short-lived `AdminClient` for a single operation, creating producers and
  consumers whose defaults match this project's telemetry patterns, gating
  application startup on broker reachability, and wiring broker address lists
  from a compact Spring property.

The module is small by design: one value object (`KafkaServer`), one Spring
`FactoryBean` that expands a pattern into an array of those value objects
(`KafkaServerFactoryBean`), one factory for client objects (`KafkaConnector`),
one formatted exception (`KafkaConnectionException`), and one reference-only
class that documents recommended broker-side settings as a comment block
(`KafkaConfiguration`). It exists to turn "create a producer, create a
consumer, check the cluster is up" into a handful of lines of calling code
without dragging every caller through the `Properties` / `ProducerConfig` /
`ConsumerConfig` vocabulary by hand.

=== Why This Package Exists

`kafka-clients` is a complete, transport-level API. Every service that uses
it ends up writing the same glue:

1. Turn a list of broker hostnames or `host:port` pairs into the single
   comma-separated `bootstrap.servers` string that every client constructor
   accepts.
2. Assemble a `Properties` object with the right serializer / deserializer
   classes, the right reliability settings (`acks`, `retries`, idempotence),
   the right timeouts (`delivery.timeout.ms`, `request.timeout.ms`,
   `max.block.ms`), and the right batching knobs (`linger.ms`, `batch.size`).
3. Open an `AdminClient` just long enough to describe the cluster, then close
   it cleanly. Retry until the cluster is reachable so the rest of the
   application doesn't start against a phantom broker list.
4. In a Spring-driven application, turn `kafka-1.internal`, `kafka-2.internal`,
   `kafka-3.internal` into three `KafkaServer` beans without writing three
   near-identical XML entries or maintaining a parallel spreadsheet of
   broker DNS names.
5. Agree somewhere in the project on the *broker* side settings (partition
   count, replication factor, retention, compression) that the service's
   producers and consumers implicitly rely on, so the tuning conversation
   does not get re-litigated in each team's design review.

`smallmind-kafka` assembles (1) through (4) once, as a four-class Java
facade, and solves (5) with a reference class whose body is a commented
broker config template. A caller that needs a producer writes:

[source,java]
----
Producer<Long, byte[]> producer =
    new KafkaConnector(servers).check(30).createProducer("order-ingest");
----

…and never touches `Properties`, `ProducerConfig`, or
`AdminClient#describeCluster` directly.

=== What It Does, Concretely

* **Models a broker as a value object.** `KafkaServer` is a
  `(host, port)` pair, default port `9094`, `Comparable` for predictable
  ordering in sets and log lines. It is the unit of wiring that Spring XML or
  programmatic setup produces.
* **Expands a broker pattern into an array.** `KafkaServerFactoryBean` takes
  a template like `kafka-#.internal:9094` and a spread like `1-3` and emits
  three `KafkaServer` beans, so adding a node is a property edit rather than
  a bean-definition edit.
* **Builds a bootstrap server string once.** `KafkaConnector` accepts a
  `KafkaServer...` vararg and caches the comma-separated
  `host1:port1,host2:port2,…` string that every client constructor needs.
* **Proves reachability before startup continues.** `KafkaConnector.check`
  opens an `AdminClient`, calls `describeCluster().nodes().get()`, and retries
  once a second until either a non-empty node list comes back or the supplied
  grace period elapses — in which case it throws `KafkaConnectionException`
  rather than letting the application start against a cluster that might be
  uncontactable.
* **Vends pre-configured producers and consumers** whose defaults match a
  specific use profile: low-latency `Long`-keyed telemetry with `byte[]`
  payloads, fire-and-forget reliability on the producer side, manual offset
  commits on the consumer side.
* **Runs one-off admin operations** through `invokeAdminClient`, a
  function-style helper that opens an `AdminClient`, passes it to a caller
  lambda, and closes it on return.
* **Documents broker-side configuration** in `KafkaConfiguration` as a
  commented reference the operations team can paste into `server.properties`
  or a Kubernetes `ConfigMap`. There is no runtime code in `KafkaConfiguration`;
  it is deliberately a pinned location for the tuning decisions the producer
  and consumer defaults assume.

=== What It Intentionally Does Not Do

* It does not wrap `KafkaProducer` or `KafkaConsumer`. The types the factory
  returns are the stock `org.apache.kafka.clients.producer.Producer` and
  `org.apache.kafka.clients.consumer.Consumer` interfaces. Callers use them
  directly.
* It does not own lifecycle for those clients. Every `createProducer` /
  `createConsumer` call returns a freshly constructed client; closing it is
  the caller's responsibility.
* It does not provide serializers beyond `LongSerializer` / `LongDeserializer`
  for keys and `ByteArraySerializer` / `ByteArrayDeserializer` for values.
  A service that wants JSON, Avro, or Protobuf payloads encodes them *into*
  the `byte[]` before calling `producer.send` and decodes on the consumer
  side; the framework does not negotiate that choice.
* It does not manage topics, ACLs, quotas, or schemas. `invokeAdminClient`
  makes it easy to run a single admin operation, but the module does not
  maintain any topic catalog, migration, or schema registry.
* It does not pull in ZooKeeper, Kafka Streams, Kafka Connect, or any
  managed-service SDK. The only Kafka dependency is `kafka-clients` itself.

== Package Structure

[cols="1,3",options="header"]
|===
| Class | Role

| `KafkaServer`
| Immutable-by-convention value object carrying a broker's host name or IP
  address and listener port. Has a no-arg constructor so Spring can build it
  by property injection and a `(host, port)` constructor for programmatic
  use. Default port is `9094`. Implements `Comparable<KafkaServer>` by host
  first, port second.

| `KafkaServerFactoryBean`
| Spring `FactoryBean<KafkaServer[]>`. Parses a `serverPattern` such as
  `kafka-#.internal:9094` and, when the pattern contains `#`, expands it
  using a `serverSpread` expression handed to
  `org.smallmind.nutsnbolts.util.Spread.calculate`. Emits one `KafkaServer`
  per expansion value. For patterns with no `#`, the default port is `9092`
  when no `:port` suffix is supplied.

| `KafkaConnector`
| Factory for all client objects that share a single bootstrap server
  string. Supplies `check`, `invokeAdminClient`, `createProducer`, and
  `createConsumer`. Holds no client state; each call constructs a new
  client.

| `KafkaConnectionException`
| Checked exception thrown by `KafkaConnector.check` when no broker can be
  confirmed reachable within the allowed grace period. Extends
  `org.smallmind.nutsnbolts.lang.FormattedException` so throw sites can
  embed format arguments without a separate `String.format` call.

| `KafkaConfiguration`
| Documentation-only class. Its body is a multi-line comment that lists the
  broker-side `server.properties` values the module's client defaults
  assume. Contains no fields, methods, or runtime behavior.
|===

== Conceptual Model

A few ideas recur throughout the module and are worth naming up front.

Bootstrap server list::
    The comma-separated `host:port,host:port,…` string that every Kafka
    client constructor accepts as `bootstrap.servers`. `KafkaConnector`
    builds it once in its constructor; every client it subsequently vends
    uses the same string. A bootstrap list does not need to enumerate
    every broker — one or two reachable hosts are enough for the client to
    discover the rest — but listing every known broker makes the client
    robust against any single entry being down during startup.

Fire-and-forget telemetry::
    The reliability profile baked into `createProducer`: `acks=0` (the
    producer does not wait for any broker acknowledgement),
    `retries=0` (a send that fails is simply dropped), and
    `delivery.timeout.ms=500` (any send that has not reached the broker
    in half a second is abandoned). This is the right setting for
    high-volume, loss-tolerant streams — click events, heartbeats,
    metrics — and the wrong setting for data whose loss is observable to
    a user. The profile is intentionally *not* configurable through
    `KafkaConnector`; a service that needs durable semantics should build
    its own `Properties` rather than subverting a factory whose name
    implies a specific profile.

Manual offset commit::
    The consumer profile baked into `createConsumer`:
    `enable.auto.commit=false` combined with `auto.offset.reset=latest`.
    The consumer does not advance its committed position automatically;
    the caller commits explicitly (usually via `commitSync()` after a
    batch is processed) so that offset advancement lines up with durable
    side effects. A brand-new consumer group starts at the tail of the
    log, not the head — it reads what is produced after it joins, not
    the backlog.

Startup grace period::
    The maximum number of seconds `KafkaConnector.check` will keep
    retrying `describeCluster().nodes()` before giving up. Inside that
    window the call sleeps one second between attempts. Pass a value
    generous enough to cover the worst-case broker restart on your
    deployment (30–60 seconds is typical); pass `0` to fail immediately
    if the first probe does not come back.

Spread expansion::
    The mechanism `KafkaServerFactoryBean` uses to turn one pattern
    into many `KafkaServer` instances. A `#` in the pattern is replaced
    by each value produced by `Spread.calculate(serverSpread)`. The
    spread expression accepts numeric ranges (`1-3`), alpha ranges
    (`a-c`), and comma-separated lists. An `a-c` spread against pattern
    `kafka-#.internal` yields `kafka-a.internal`, `kafka-b.internal`,
    `kafka-c.internal`.

== Installation

The utility artifact is published under `org.smallmind`:

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>kafka-utility</artifactId>
  <version>${project.version}</version>
</dependency>
----

This pulls in `org.apache.kafka:kafka-clients` at the version declared by
the parent POM, `org.smallmind:scribe-pen` for logging, and
`org.springframework:spring-beans` for the `FactoryBean` interface that
`KafkaServerFactoryBean` implements.

=== Runtime prerequisites

* A running Kafka cluster reachable from the application process. The
  module assumes PLAINTEXT listeners on the port carried by each
  `KafkaServer`. SSL, SASL, mTLS, or any other authentication scheme
  requires extra `Properties` entries that `KafkaConnector` does not
  supply — see <<adding-security>> below for the escape hatch.
* A classpath where exactly one version of `kafka-clients` is present.
  Mixing client versions across modules is a common source of
  `NoSuchMethodError` at `KafkaProducer` construction time.

=== Broker-side configuration

The client defaults in `KafkaConnector` assume a broker cluster tuned for
short-retention, high-throughput telemetry: message TTL around 28 minutes,
LZ4 compression, three-way replication when `acks=all` is in use,
rack-aware replica selection when the deployment spans data centres. The
full recommended `server.properties` snippet lives in
`KafkaConfiguration`'s class-level comment; the most important values are
reproduced in <<broker-configuration-reference>> below.

== Quick Start

The fastest path is to define a `KafkaServer` array through Spring, hand it
to a `KafkaConnector`, and pull a producer or consumer off it.

=== Spring wiring

[source,xml]
----
<bean id="kafkaServers"
      class="org.smallmind.kafka.utility.KafkaServerFactoryBean">
  <property name="serverPattern" value="kafka-#.internal:9094"/>  <1>
  <property name="serverSpread" value="1-3"/>                    <2>
</bean>

<bean id="kafkaConnector"
      class="org.smallmind.kafka.utility.KafkaConnector">
  <constructor-arg ref="kafkaServers"/>                          <3>
</bean>
----
<1> The `#` is the expansion point. The literal suffix `:9094` overrides the
    factory's own default of `9092`, so the three produced brokers all listen
    on port 9094 — the port `KafkaServer`'s no-arg constructor also uses as
    its default.
<2> The spread `1-3` produces the strings `1`, `2`, `3`, giving three brokers.
    `a-c`, `1,2,5`, or any other expression `Spread.calculate` understands is
    equally valid.
<3> `KafkaConnector` takes the broker array as a single constructor argument
    through Spring's vararg-to-array binding.

=== Programmatic wiring

[source,java]
----
import org.smallmind.kafka.utility.KafkaConnector;
import org.smallmind.kafka.utility.KafkaServer;

KafkaConnector connector = new KafkaConnector(
    new KafkaServer("kafka-1.internal", 9094),
    new KafkaServer("kafka-2.internal", 9094),
    new KafkaServer("kafka-3.internal", 9094));
----

=== Producing a single record

[source,java]
----
import org.apache.kafka.clients.producer.Producer;
import org.apache.kafka.clients.producer.ProducerRecord;
import org.smallmind.kafka.utility.KafkaConnector;

public class HeartbeatEmitter {

  private final KafkaConnector connector;

  public HeartbeatEmitter (KafkaConnector connector) {

    this.connector = connector;
  }

  public void emit (long nodeId, byte[] payload) {

    try (Producer<Long, byte[]> producer = connector.createProducer("heartbeat")) {   <1>
      producer.send(new ProducerRecord<>("service.heartbeat", nodeId, payload));      <2>
    }
  }
}
----
<1> `"heartbeat"` becomes the `client.id` the broker will report in its JMX
    metrics. Pick a name that identifies the sender, not the topic.
<2> No `get()` on the returned `Future`: with `acks=0` there is nothing to
    wait for. The send either makes it onto the wire before the producer is
    closed or it does not.

[IMPORTANT]
====
Opening a new `KafkaProducer` for each record is expensive and defeats the
connection pooling Kafka provides. In production, keep one `Producer`
instance per logical emitter and close it during application shutdown; the
example above uses try-with-resources for illustrative brevity, not as a
recommended lifecycle.
====

=== Consuming a stream

[source,java]
----
import java.time.Duration;
import java.util.Map;
import org.apache.kafka.clients.consumer.Consumer;
import org.apache.kafka.clients.consumer.ConsumerRecord;
import org.apache.kafka.clients.consumer.ConsumerRecords;
import org.apache.kafka.clients.consumer.OffsetAndMetadata;
import org.apache.kafka.common.TopicPartition;
import org.smallmind.kafka.utility.KafkaConnector;

public class HeartbeatSink {

  public void run (KafkaConnector connector) {

    try (Consumer<Long, byte[]> consumer =
             connector.createConsumer("heartbeat-reader", "heartbeat-sink",     <1>
                                      "service.heartbeat")) {                   <2>

      while (!Thread.currentThread().isInterrupted()) {
        ConsumerRecords<Long, byte[]> batch = consumer.poll(Duration.ofSeconds(1));

        for (ConsumerRecord<Long, byte[]> record : batch) {
          process(record.key(), record.value());
        }

        if (!batch.isEmpty()) {
          consumer.commitSync();                                                <3>
        }
      }
    }
  }

  private void process (Long nodeId, byte[] payload) { /* ... */ }
}
----
<1> `client.id` (reported to the broker) and `group.id` (the consumer
    group). Two consumers sharing a `group.id` partition the topic between
    themselves; a second `group.id` receives its own independent stream of
    every record.
<2> The vararg topic list subscribes immediately. Passing zero topics here
    is legal and leaves the consumer unsubscribed; callers then use
    `consumer.assign(...)` for manual partition control.
<3> Explicit commit. Because the factory sets `enable.auto.commit=false`,
    failing to commit here means the next `poll` after a restart will
    re-deliver the same batch — which is exactly the point.

=== Gating startup on broker reachability

[source,java]
----
import org.smallmind.kafka.utility.KafkaConnectionException;
import org.smallmind.kafka.utility.KafkaConnector;

KafkaConnector connector =
    new KafkaConnector(servers).check(30);      <1>
----
<1> Blocks for up to 30 seconds, probing once a second. If any attempt
    returns a non-empty node list, the method returns immediately. If the
    full 30 seconds elapse with no success, a `KafkaConnectionException`
    is thrown. The method returns `this` so it can be chained in-line.

== Usage Examples

=== Describing the cluster with a one-off admin client

`invokeAdminClient` is the right tool when an application needs exactly
one admin operation — list topics, describe a consumer group, look up
broker metadata — and does not want to own an `AdminClient` lifecycle.

[source,java]
----
import java.util.Set;
import org.apache.kafka.clients.admin.ListTopicsResult;
import org.smallmind.kafka.utility.KafkaConnector;

public Set<String> listTopics (KafkaConnector connector)
  throws Exception {

  return connector.invokeAdminClient(admin -> {
    try {
      return admin.listTopics().names().get();
    } catch (Exception e) {
      throw new RuntimeException(e);
    }
  });
}
----

The admin client is opened inside `invokeAdminClient`, the lambda runs
with it, and the client is closed on return even if the lambda threw.
`AdminClient` operations return `KafkaFuture`, so lambdas that need a
synchronous result call `.get()`; callers that want the raw future back
can return it from the lambda and await it outside.

=== Creating a topic

[source,java]
----
import java.util.Collections;
import org.apache.kafka.clients.admin.NewTopic;
import org.smallmind.kafka.utility.KafkaConnector;

public void ensureTopic (KafkaConnector connector,
                         String name, int partitions, short replication) {

  connector.invokeAdminClient(admin -> {
    try {
      admin.createTopics(Collections.singleton(
          new NewTopic(name, partitions, replication))).all().get();
    } catch (Exception ignore) {
      // TopicExistsException is the common, safe case
    }
    return null;
  });
}
----

This pattern is useful in integration tests and in one-shot provisioning
jobs. In production, topic creation is typically delegated to
`auto.create.topics.enable=true` on the broker (see
<<broker-configuration-reference>>) or to an out-of-band operations
process.

=== Reusing one connector across many clients

A `KafkaConnector` caches its bootstrap server string; creating many
producers and consumers from the same connector is cheap.

[source,java]
----
KafkaConnector connector = new KafkaConnector(servers).check(30);

Producer<Long, byte[]> orders      = connector.createProducer("order-ingest");
Producer<Long, byte[]> metrics     = connector.createProducer("metrics-ingest");
Consumer<Long, byte[]> orderReader = connector.createConsumer("order-reader", "order-service", "orders");
----

Each call returns a fresh, independent client. They do not share any
underlying network resource — every `KafkaProducer` and `KafkaConsumer`
opens its own connection pool — but they do share the bootstrap list,
which is the only state the connector holds.

[[adding-security]]
=== Layering SASL / SSL on top

`KafkaConnector` does not accept extra `Properties`; the factory methods
commit to a specific profile and deliberately hide the knobs that
contradict it. When a deployment needs authentication, the cleanest
escape hatch is to treat the factory methods as a starting point and
stop using them when the deployment's requirements diverge:

[source,java]
----
Properties props = new Properties();

// Start from the connector-supplied bootstrap list:
props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, connector.getBoostrapServers());

// Apply the rest yourself:
props.put(ProducerConfig.CLIENT_ID_CONFIG, "secure-ingest");
props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG,   LongSerializer.class.getName());
props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, ByteArraySerializer.class.getName());
props.put("security.protocol", "SASL_SSL");
props.put("sasl.mechanism",    "PLAIN");
props.put("sasl.jaas.config",  "org.apache.kafka.common.security.plain.PlainLoginModule required username=\"...\" password=\"...\";");
props.put(ProducerConfig.ACKS_CONFIG,    "all");
props.put(ProducerConfig.RETRIES_CONFIG, Integer.MAX_VALUE);

Producer<Long, byte[]> producer = new KafkaProducer<>(props);
----

[NOTE]
====
The accessor on `KafkaConnector` is named `getBoostrapServers` (note the
typo: `boostrap`, not `bootstrap`). Both the field and the method
preserve the misspelling for source compatibility; callers reference the
method exactly as written.
====

=== Polling with a deadline

The built-in consumer has no notion of "read for N seconds and stop,"
but the factory's defaults make it easy to compose one.

[source,java]
----
import java.time.Duration;
import java.time.Instant;

public void drain (Consumer<Long, byte[]> consumer, Duration budget) {

  Instant deadline = Instant.now().plus(budget);

  while (Instant.now().isBefore(deadline)) {
    ConsumerRecords<Long, byte[]> records =
        consumer.poll(Duration.ofMillis(500));                         <1>

    records.forEach(r -> process(r));
    consumer.commitSync();
  }
}
----
<1> `FETCH_MAX_WAIT_MS_CONFIG` on the consumer is `500ms` by default, so
    asking for a 500ms poll gives the broker room to wait for a full
    batch before returning, at the cost of 500ms of tail latency on a
    quiet topic. Shorten the poll to respond faster; lengthen it to
    trade latency for throughput.

== Behavioral Reference

=== Bootstrap server assembly

`KafkaConnector` concatenates each broker's host and port with a comma
separator in the order they were passed to the constructor. The string
is cached; it does not regenerate per call. Calling
`getBoostrapServers()` after construction always returns the same
instance.

Edge cases:

* Passing zero brokers yields an empty string. Any subsequent client
  construction will fail with `kafka-clients`' own
  `ConfigException("No bootstrap servers")`.
* Passing the same broker twice produces a bootstrap string with
  duplicates. `kafka-clients` tolerates this.
* Host names are used verbatim; no DNS resolution or trimming occurs.

=== `check` semantics

* The first iteration runs immediately, with no sleep.
* On each unsuccessful iteration, the thread sleeps one second before
  retrying.
* The loop exits successfully when `describeCluster().nodes().get()`
  returns a non-null, non-empty collection.
* The loop exits unsuccessfully when `(System.currentTimeMillis() -
  startTimestamp) >= startupGracePeriodSeconds * 1000L` on an error
  iteration — i.e., the timeout is evaluated *after* the failure, so
  the call may take up to one second longer than the grace period.
* An `InterruptedException` during the inner `Thread.sleep` is logged
  and causes the method to return failure immediately, without throwing.
  The caller's interrupt flag is preserved by the underlying
  `Thread.sleep` contract for callers that want to observe it.
* A successful `check` returns the connector itself, so
  `new KafkaConnector(servers).check(30).createProducer("...")` is a
  legal one-liner.

=== Producer defaults

`createProducer` sets exactly these values:

[cols="2,1,3",options="header"]
|===
| Key | Value | Rationale

| `bootstrap.servers`
| connector's cached string
| Every client must carry it.

| `client.id`
| caller-supplied
| Reported in broker JMX; make it identify the sender.

| `key.serializer`
| `LongSerializer`
| Numeric keys hash evenly across partitions without
  `null`-key round-robin.

| `value.serializer`
| `ByteArraySerializer`
| Payload framing is the caller's responsibility; the framework stays
  out of encoding decisions.

| `acks`
| `0`
| Fire-and-forget. The producer does not wait for broker
  acknowledgement.

| `retries`
| `0`
| A failed send is dropped. Combined with `acks=0`, there is nothing
  to retry anyway.

| `retry.backoff.ms`
| `100`
| Relevant only if `retries` is later overridden; set conservatively.

| `max.in.flight.requests.per.connection`
| `5`
| Standard high-throughput value. Safe to raise only if ordering
  concerns allow.

| `delivery.timeout.ms`
| `500`
| Half a second from `send` to success-or-giveup. Must be at least
  `request.timeout.ms + linger.ms`.

| `request.timeout.ms`
| `500`
| Must exceed `replica.lag.time.max.ms` on the broker side.

| `linger.ms`
| `0`
| Send as soon as a batch is eligible. Trade off against throughput on
  high-volume topics by raising to 5–20ms.

| `max.request.size`
| `1048576` (1 MB)
| Matches broker's `message.max.bytes` recommendation.

| `batch.size`
| `16384` (16 KB)
| Standard default. The producer will flush earlier if `linger.ms` or
  the deadline forces it.

| `max.block.ms`
| `1000`
| How long `send` blocks when the send buffer is full. One second
  protects callers from head-of-line blocking on a choked broker.

| `enable.idempotence`
| `false`
| Incompatible with `acks=0`; explicitly disabled.
|===

A caller that needs durable semantics (`acks=all`, retries, idempotence)
must build its own `Properties`; see <<adding-security>> for the
pattern.

=== Consumer defaults

`createConsumer` sets exactly these values:

[cols="2,1,3",options="header"]
|===
| Key | Value | Rationale

| `bootstrap.servers`
| connector's cached string
| Every client must carry it.

| `client.id`
| caller-supplied
| Reported in broker JMX.

| `group.id`
| caller-supplied
| Defines the consumer group this instance joins.

| `group.protocol`
| `consumer`
| KIP-848 "new" consumer group protocol. Supersedes the classic
  client-side rebalance protocol. This value also forbids setting
  `heartbeat.interval.ms` and `session.timeout.ms` from the client
  side — those are broker-side settings under the new protocol.

| `key.deserializer`
| `LongDeserializer`
| Mirror of the producer's key serializer.

| `value.deserializer`
| `ByteArrayDeserializer`
| Mirror of the producer's value serializer.

| `max.poll.interval.ms`
| `300000` (5 min)
| Maximum gap between `poll` calls before the broker removes the
  consumer from the group.

| `max.poll.records`
| `500`
| Records returned in a single `poll`.

| `fetch.min.bytes`
| `1`
| Return as soon as any data is available; do not wait for the batch
  to fill.

| `fetch.max.wait.ms`
| `500`
| Broker-side wait when `fetch.min.bytes` is not yet met.

| `fetch.max.bytes`
| `52428800` (50 MB)
| Per-fetch ceiling across all partitions. Large to accommodate bursty
  traffic.

| `max.partition.fetch.bytes`
| `1048576` (1 MB)
| Per-partition ceiling. Matches broker `message.max.bytes`.

| `enable.auto.commit`
| `false`
| Caller commits explicitly. Auto-commit is disabled to prevent
  offsets from advancing before the records are durably processed.

| `auto.commit.interval.ms`
| `500`
| Inert while `enable.auto.commit=false`; left for completeness in
  case a caller overrides the flag.

| `auto.offset.reset`
| `latest`
| A brand-new group starts at the tail. A group that already has
  committed offsets continues from them; the reset only applies when
  there is no committed offset for the group.
|===

=== Topic subscription behavior

`createConsumer`'s vararg `topics` parameter has three cases:

1. `null`: the consumer is created unsubscribed. The caller is expected
   to call `subscribe(...)` or `assign(...)` before `poll`.
2. Empty array (zero varargs): same as `null`.
3. One or more names: `subscribe(Arrays.asList(topics))` is invoked
   before the consumer is returned.

There is no regex subscription variant in the factory; callers that
need one use `consumer.subscribe(Pattern.compile(...))` after receiving
the unsubscribed consumer.

=== `KafkaServer` ordering

`compareTo` first compares hosts lexicographically. If hosts are equal,
it returns `this.port - other.port`. There is no defense against
integer overflow because broker ports fit in 16 bits by definition.
Hosts are compared with `String.compareTo`; a `null` host on either
side throws `NullPointerException`, which is why the no-arg constructor
is documented as being "for bean-wiring frameworks that set properties
after construction" — direct use of the default-constructed instance
before `setHost` is illegal.

=== `KafkaServerFactoryBean` parsing

The factory applies a straightforward literal substitution, not a
regular-expression match:

* Find the first `:` — call its position `colonPos`. If absent, there
  is no port suffix.
* Find the first `#` — call its position `poundPos`. If absent, the
  pattern is a single-broker pattern.

Then:

* **No `#`, no `:`:** single broker, host = pattern, port = `9092`.
* **No `#`, with `:`:** single broker, host = `pattern.substring(0,
  colonPos)`, port = integer parse of the remainder.
* **With `#`, no `:`:** for each spread value `v`, host =
  `pattern.substring(0, poundPos) + v + pattern.substring(poundPos + 1)`,
  port = `9092`.
* **With `#` and `:`:** for each spread value `v`, host =
  `pattern.substring(0, poundPos) + v + pattern.substring(poundPos + 1, colonPos)`,
  port = integer parse of `pattern.substring(colonPos + 1)`.

Quirks worth knowing:

* The default port when parsed from a pattern is `9092`. The default
  port on `new KafkaServer()` is `9094`. This asymmetry exists because
  `KafkaServer`'s default matches the port most often used for
  *external* (non-broker-to-broker) traffic in this project, while the
  pattern's default matches Kafka's own documented default. If
  consistency matters, always include an explicit `:port` suffix in
  the pattern.
* A `#` placed *after* the first `:` will be treated as part of the
  port string and fail with `NumberFormatException`. Keep the `#` in
  the host portion.
* `serverPattern == null` or `""` leaves `serverArray` at `null`; the
  returned object from `getObject()` will be `null`. Callers that
  require a non-null result must set `serverPattern` explicitly.
* `serverSpread == null` with a `#`-bearing pattern produces the
  `SpreadParserException` thrown by `Spread.calculate(null)` — but in
  fact `Spread.calculate(null)` returns an empty array, yielding a
  zero-length `KafkaServer[]`. Set both properties together.

=== Thread safety

* `KafkaConnector` is safe for concurrent use. It holds only an
  immutable string; all client construction happens inside synchronized
  constructors in `kafka-clients`.
* `KafkaServer` is effectively immutable once constructed with the
  `(host, port)` constructor. The no-arg variant plus setters is
  intended for single-threaded bean assembly; do not mutate a
  `KafkaServer` that is already in circulation.
* `KafkaServerFactoryBean` is populated once by Spring in
  `afterPropertiesSet` and is read-only thereafter. `getObject`
  returns the same array instance every time.
* Clients returned from `createProducer` / `createConsumer` carry
  whatever thread-safety `kafka-clients` documents for them: the
  `KafkaProducer` is thread-safe and may be shared across threads;
  the `KafkaConsumer` is *not* thread-safe and must be used from one
  thread at a time (or with `wakeup`-based coordination).

=== Logging

`KafkaConnector` logs through `scribe-pen`:

* An `InterruptedException` inside the `check` sleep is logged at
  `ERROR` and the call returns failure.
* A terminal `ExecutionException` or `InterruptedException` from
  `describeCluster` (after the grace period has elapsed) is logged at
  `ERROR` before the method returns failure — which the caller turns
  into a `KafkaConnectionException`.

No other log output is produced by this module. Producers and
consumers log through `kafka-clients`' own SLF4J binding; configure
your SLF4J provider separately if that output is wanted.

== Error Model

The exceptions callers will see fall into three groups.

Startup reachability::
    `KafkaConnectionException` is raised by `KafkaConnector.check` when
    the grace period elapses without a successful `describeCluster`.
    The message includes the bootstrap server string so the failure is
    diagnosable from the log line alone. The type extends
    `org.smallmind.nutsnbolts.lang.FormattedException`, so it is a
    checked exception; methods that call `check` must declare it or
    catch it.

Client construction::
    Failures in `createProducer` / `createConsumer` come from
    `kafka-clients` itself, typically `ConfigException` for bad
    arguments (including an empty bootstrap string). The factory does
    not wrap these; they propagate unchanged.

Client operation::
    Once a producer or consumer has been returned, any further
    exceptions — `TimeoutException`, `OffsetOutOfRangeException`,
    `WakeupException`, `CommitFailedException`, and so on — are the
    standard `kafka-clients` exceptions raised from the caller's own
    use of `send`, `poll`, or `commitSync`. The factory has no
    visibility into them.

== Lifecycle

Every object the module creates is owned by the caller. There is no
registry, pool, or shutdown hook:

* `KafkaConnector` holds no closeable state. Discarding the reference
  is enough.
* A `Producer` returned from `createProducer` must be closed by the
  caller, typically in an application shutdown hook or a
  try-with-resources block scoped to a request.
* A `Consumer` returned from `createConsumer` must likewise be closed
  by the caller. Because the Kafka consumer is not thread-safe, the
  close is conventionally performed on the same thread that runs
  `poll`, often in response to a `wakeup` called from another thread.
* `AdminClient` instances opened by `invokeAdminClient` are closed
  automatically by that method's try-with-resources.

=== Rack awareness

When the deployment spans data centres or availability zones:

* `rack.id` must be set on each broker to its zone identifier.
* `replica.selector.class` must be set to
  `org.apache.kafka.common.replica.RackAwareReplicaSelector`.
* Consumers should additionally set `client.rack` to the consumer's
  zone identifier so the broker can serve fetches from a same-zone
  replica. The factory leaves this unset by default; override in a
  custom `Properties` block when a same-zone read is worth more than
  fewer `Properties` entries.

=== Transaction and offsets topics

These are internal to Kafka; the comment block in `KafkaConfiguration`
lists them but marks them "in general, do not override":

* `transaction.state.log.replication.factor=3`
* `transaction.state.log.min.isr=2`
* `offsets.topic.num.partitions=50`
* `offsets.topic.replication.factor=3`
* `offsets.commit.required.acks=-1`

Lowering any of these is only appropriate on single-broker test
clusters.

== Troubleshooting

`KafkaConnectionException` during startup::
    `check` exhausted its grace period without seeing a non-empty node
    list. Confirm: (a) bootstrap hosts resolve from the application
    pod or host; (b) the broker's advertised listener matches the host
    and port the connector is using (an `advertised.listeners` entry
    pointing at an internal DNS name that clients can't reach is the
    most common cause); (c) no firewall or security group blocks the
    listener port. The message includes the full bootstrap string for
    diagnosis.

`ConfigException: No bootstrap servers` at client construction::
    The `KafkaConnector` was built with zero brokers, or the
    `KafkaServerFactoryBean` emitted an empty array (typically because
    `serverPattern` was empty or the spread expression evaluated to
    zero values). Log `connector.getBoostrapServers()` to confirm.

Producer `send` succeeds but no data appears on the topic::
    Expected when `acks=0`. A broker that rejected the record or was
    unreachable at the moment of the send simply dropped it. If this
    is not acceptable, the factory's profile is the wrong fit; build
    a `Properties` with `acks=all` and one or more retries instead.

Consumer starts reading from the end of the topic::
    Expected when `auto.offset.reset=latest` and the group has no
    committed offset yet. Either commit at least once on a
    predecessor run, or override the profile to `earliest` for one-off
    replay consumers (again, by constructing the consumer directly
    from a `Properties` object rather than through the factory).

`CommitFailedException` on `commitSync`::
    The consumer missed a re-balance deadline, usually because
    `poll` was called more than `max.poll.interval.ms` (five minutes)
    after the previous poll. Reduce per-batch work, raise the
    interval, or reduce `max.poll.records`.

`UnsupportedVersionException` on consumer start::
    The broker does not understand the new consumer group protocol
    (`group.protocol=consumer`, KIP-848). Upgrade the broker, or
    override `group.protocol` in a custom `Properties` block — the
    factory hard-codes `consumer` and does not expose a toggle.

`NumberFormatException` from `KafkaServerFactoryBean`::
    The pattern's text after the first `:` is not parseable as an
    integer. The most common cause is a `#` placed after the `:` —
    the parser only substitutes `#` in the host portion. Move the `#`
    left of the `:` or split the pattern in two.

`NullPointerException` from `KafkaServer.compareTo`::
    The no-arg constructor left `host` null. Always call `setHost`
    before inserting a default-constructed `KafkaServer` into a
    sorted collection, or use the `(host, port)` constructor.

== Limitations and Known Issues

No alternate reliability profile::
    The factory commits to `acks=0`, `retries=0`, and
    `enable.idempotence=false`. A caller that needs durable or
    transactional producers must build its own `Properties`. The
    bootstrap string remains reusable through
    `KafkaConnector.getBoostrapServers()`.

No alternate serializer profile::
    Keys are always `Long`; values are always `byte[]`. Serializing
    structured payloads is the caller's responsibility. This is a
    deliberate simplification: it decouples the framework from the
    encoding choice (JSON, Avro, Protobuf) and keeps the types
    uniform in the generic parameter signature.

No security configuration::
    `SSL`, `SASL`, and `mTLS` properties are not exposed. See
    <<adding-security>> for the recommended escape hatch.

No metrics export::
    The factory does not register any metrics or expose
    `KafkaProducer.metrics()` / `KafkaConsumer.metrics()`. Integrate
    with your observability layer directly off the returned client.

Typo preserved in the API::
    `KafkaConnector.getBoostrapServers` is the actual method name.
    The typo dates from the original revision and is preserved for
    source compatibility.

Inconsistent default port::
    `KafkaServer`'s no-arg default is `9094`; the pattern parser's
    default is `9092`. Explicit `:port` suffixes in patterns avoid
    the ambiguity.

No alternate group protocol::
    The consumer hard-codes `group.protocol=consumer` (KIP-848). Older
    brokers that expect the classic protocol will reject the client
    with `UnsupportedVersionException`. The factory does not offer a
    classic-mode toggle.

No connection pooling or sharing::
    Each `createProducer` / `createConsumer` call constructs a fresh
    client with its own connection pool. Callers that want to share a
    single producer across many emitters must hold the returned
    instance themselves.

== Minimal Checklist

Before shipping an integration:

1. Confirm the Spring XML (or programmatic construction) produces a
   non-empty `KafkaServer[]` and log the resulting bootstrap string at
   startup.
2. Call `check(n)` with a grace period that covers the worst realistic
   broker restart window in your deployment.
3. Hold each `Producer` / `Consumer` instance for its intended
   lifetime — not per record — and close it in an orderly shutdown.
4. For consumers, commit explicitly after side effects are durable;
   never rely on `enable.auto.commit` (it is off by default in this
   factory anyway).
5. Ensure the broker cluster's `server.properties` is consistent with
   the values documented in `KafkaConfiguration`, or document any
   intentional divergence where your service's maintainers will find
   it.
6. If the deployment spans availability zones, set `client.rack` on
   the consumer and rack-aware selector on the brokers — the factory
   does not set `client.rack`.
7. If the deployment uses authentication, build `Properties` directly
   using `connector.getBoostrapServers()` as the starting point, and
   stop using the factory methods for that client.

== Summary

`smallmind-kafka` turns the rote parts of using `kafka-clients` — the
bootstrap string, the reachability probe, the properties incantation,
the address spread — into a four-class facade that stays out of the
caller's way. Build a `KafkaConnector` from a `KafkaServer[]`, call
`check` to gate startup, and pull a producer or consumer off the
connector when the service needs one. The `Properties` choices are
opinionated toward high-volume, loss-tolerant telemetry; a caller
whose workload is durable, transactional, or authenticated uses
`getBoostrapServers` as a starting point and builds its own client.
Broker-side tuning assumptions are pinned in `KafkaConfiguration` so
the two sides of the conversation stay in sync.
