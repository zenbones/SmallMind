[[foundation, Foundation]]
= Foundation

[partintro]
Foundation is not a library module. It is a small bundle of example Spring XML and example property files kept in a Maven module so the files can be packaged and inspected. Read it as template material: one historical approach to early selector setup and layered property loading. Copy the pattern if it is useful, but do not treat the shipped files as a supported API or as the source of truth for other Smallmind modules.

Audience: developers looking for concrete Spring configuration examples or a starting template for their own property-loading layout.
Versioning: examples use `${smallmind.version}` for Smallmind artifacts; the resources here are examples only and do not define compatibility requirements for other modules.

[[foundation-files, What Ships]]
== What Ships

The module currently ships these example resources:

* `org/smallmind/foundation/foundation.xml`
* `org/smallmind/foundation/global.properties`
* `org/smallmind/foundation/database-mysql.properties`

There is no Java API in `foundation/src/main/java`.

`foundation.xml` shows one explicit Spring wiring pattern built from `SystemPropertyInitializingBean` and `PropertyPlaceholderConfigurer`.
The two property files show one historical set of keys for logging, server paths, messaging, metrics, ORM, Quartz, and JDBC pooling.

[[foundation-quick-start, Using It As A Template]]
== Using It As A Template

If the pattern is useful, copy it into your own namespace instead of importing it unchanged.

The ideas worth borrowing are simple:

1. seed a small set of selector properties early in Spring startup
2. load property files from broad defaults toward more specific overrides
3. let system properties and environment variables win over file-backed values if that fits your operational model
4. leave an optional final local override file outside the application jar

When adapting it:

* rename the selector properties to match your application vocabulary
* replace the example file paths with paths your application actually owns
* keep only the key families you intend to support
* verify every `${...}` placeholder against the XML that will actually consume it

If you need the exact historical behavior, read `foundation/src/main/resources/spring/org/smallmind/foundation/foundation.xml` and the two packaged property files directly. This chapter intentionally does not restate every path and key.

[[foundation-compatibility, Boundaries]]
== Boundaries

Foundation is not a contract for other modules.

Keep these limits in mind:

* the packaged property names are historical examples, not repository-wide standards
* some names no longer match current module XML exactly; for example, the example uses `rabbit.username` where current `phalanx` XML expects `rabbitmq.username`, and `jdbc.user.quartz` where current Quartz XML expects `jdbc.username.quartz`
* importing `foundation.xml` unchanged means inheriting the exact behavior and runtime dependencies of the classes it references
* the authoritative documentation for any real module remains that module's own chapter and source
