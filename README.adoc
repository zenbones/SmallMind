= Smallmind
:doctype: book
:toc: auto
:toclevels: 3
:icons: font
:sectanchors:
Explicit Java glue code

[preface]
Smallmind is a multi-module Java library collection for the integration code that surrounds business logic: transport adapters, logging, scheduling, packaging, persistence helpers, caching, monitoring, and other operational seams. The project is deliberately library-safe. It favors narrow module surfaces, explicit APIs, optional integrations, and caller-owned runtime policy.

Direct programmatic wiring stays first-class. Some modules also provide Spring beans or XML helpers as assembly conveniences, and the same APIs remain usable from manual wiring or other IOC containers.

== What This Book Covers

This book assembles the Smallmind module guides that currently live in the repository root. It focuses on the modules that currently have standalone root-level chapters, while the Maven build also contains additional supporting modules and internal layers. Use it when you are:

* choosing a module for a new integration
* wiring a Smallmind library into an application or another library
* checking exact behavior, defaults, or operational constraints after initial adoption

== Design Priorities

Smallmind is built around a few stable rules:

* modules stay narrow enough to adopt selectively
* configuration, lifecycle, and runtime policy stay application-owned
* direct programmatic APIs remain first-class even when Spring or XML helpers exist
* dependencies are kept optional where practical
* defaults, lifecycle, and failure behavior are treated as part of the module contract
* wrappers stay thin over third-party APIs instead of hiding them behind framework conventions

== Versioning And Dependencies

All published artifacts use the `org.smallmind` group id. Dependency snippets in this guide use `${smallmind.version}` as a placeholder.
When you combine multiple Smallmind modules, keep them on the same version unless you have a specific reason to split them.
Optional dependencies are used deliberately so applications keep control of the surrounding runtime stack. If a Smallmind module bridges to another library, expect to declare that integration dependency explicitly when you adopt it.

== Configuration Model

Each module keeps configuration close to its integration surface.
Direct programmatic wiring is always part of the intended model.
Some modules provide Spring beans or XML helpers as assembly conveniences, and the same APIs remain usable from manual wiring or other IOC containers.
Choose the smallest surface that fits your application and supply the dependencies the selected module expects.

== Reading This Guide

Each module guide opens with `Quick Navigation`, `Reading Paths`, and `At A Glance`.
If you are new to a module, start with `At A Glance`, then follow `Reading Paths` into installation and quick start.
If you already know the problem space, jump directly to the API index, defaults, troubleshooting, lifecycle, and operational notes sections.
