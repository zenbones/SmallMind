== The Problem This Solves

Two JVM processes need to talk, and the call site wants to look like an
ordinary Java method invocation. The obvious options all require
boilerplate the call site should not be writing:

* hand-written producer/consumer code that serialises arguments, routes by
  string tag, correlates replies, and materialises exceptions;
* a REST client generated from an external schema, with its own stub
  generation and transport coupling;
* a service mesh or gRPC runtime that commits the service topology and
  wire format before the first call is placed.

`smallmind-phalanx` is the piece in the middle. A Java interface plus a
handful of annotations becomes a JDK dynamic proxy on the client side and a
message-broker endpoint on the server side. One argument-named payload is
produced per call, one correlated reply is consumed, and server-side
exceptions arrive back at the caller as throwables.

What the module owns:

* encoding each call as an `InvocationSignal`, dispatching it by voice
  (load-balanced, directed, or broadcast), and waiting for the correlated
  `ResultSignal`;
* reflecting the target interface to dispatch the inbound signal;
* carrying thread-local `WireContext` values across the wire and
  re-establishing them on the server thread;
* broker wiring for RabbitMQ, Kafka, JMS, and an in-process mock router.

What stays with the caller:

* the actual `ConnectionFactory` (RabbitMQ), `KafkaServer[]` list, or JMS
  connection factory;
* choosing a `SignalCodec` — `JsonSignalCodec` (Jackson 3) or
  `JavaSerializationSignalCodec`;
* choosing a concurrency limit and default timeout on each transport pair;
* supplying a `ParameterExtractor` for the service group — and, for
  `@Whisper` interfaces, one for the target instance id.

== At A Glance

|===
| Concern | Entry point

| Declare a remotely callable method | `@InOut(timeoutSeconds = N)`, `@InOnly`, `@Whisper(timeoutSeconds = N)`, or `@Shout` on the method; `@Argument("name")` on every parameter
| Rename a method on the wire | `@CallAs("wire-name")`

| Unit-test without a broker | `MockMessageRouter` + `MockRequestTransport` + `MockResponseTransport`
| Run over RabbitMQ | `RabbitMQConnector`, `RabbitMQRequestTransport`, `RabbitMQResponseTransport`, a `QueueContractor` pair, `NameConfiguration`
| Run over Kafka | `KafkaRequestTransport`, `KafkaResponseTransport`, `KafkaConnector`, `KafkaServer...`, `KafkaGroupProtocol`
| Run over JMS (Jakarta Messaging 3.x) | `JmsRequestTransport`, `JmsResponseTransport`, `RoutingFactories`, `MessagePolicy`, `ReconnectionPolicy`

| Reuse the worker pool outside the wire layer | `org.smallmind.phalanx.worker.WorkManager` with `TransferringWorkQueue` or `BlockingWorkQueue`
|===

== Conceptual Model

The wire layer uses a small vocabulary that the rest of the doc depends on.

**Service interface**::
    Any Java interface whose methods carry a dispatch annotation
    (`@InOut`, `@InOnly`, `@Whisper`, `@Shout`) and whose parameters all
    carry `@Argument`. It does not extend any marker interface.
    `WireInvocationHandler` validates both constraints when the proxy is
    created.

**Signal**::
    The serialised form of one request (`InvocationSignal`) or one reply
    (`ResultSignal`). A `SignalCodec` owns the byte encoding. Two codecs
    ship: `JsonSignalCodec` (Jackson 3) and `JavaSerializationSignalCodec`.

**Route**::
    The `(version, serviceName, function)` tuple inside every invocation
    signal. Version and name come from the proxy configuration. The
    function carries the Java method name (or the `@CallAs` override), its
    parameter types, and its return type.

**Voice**::
    The routing-plus-conversation pair for a single call. A `Voice` combines
    a `VocalMode` — `TALK` (load-balanced within a service group), `WHISPER`
    (directed at one instance), or `SHOUT` (broadcast to every instance) —
    with a `Conversation` (`OneWayConversation` for fire-and-forget,
    `TwoWayConversation` for request/reply with a timeout). The three
    concrete voices are `Talking`, `Whispering`, and `Shouting`.
    `WireInvocationHandler` picks the right one from the method's
    annotations.

**Wire context**::
    A subclass of abstract `WireContext` (which extends the nutsnbolts
    `Context`). Pushed onto the thread-local `ContextFactory` stack by the
    caller, attached to every outbound invocation signal, re-pushed on the
    server thread for the duration of the dispatch, and visible to the
    service implementation via `ContextFactory.getContext`.

**Wired service**::
    The server-side wrapper around a service implementation.
    `WiredService.getServiceName()` and `getVersion()` supply the routing
    key, and `setResponseTransport(...)` gives the wrapper a back-channel
    to reply. `ResponseTransport.register(Class, WiredService)` binds the
    wrapper to an interface.

**Transport pair**::
    `RequestTransport` on the client, `ResponseTransport` on the server.
    Both sides share a `SignalCodec` and matching configuration (service
    group, `NameConfiguration` for RabbitMQ, or `nodeName`/`serviceGroup`
    for Kafka). Each pair is independent; a JVM can host many.

== Installation

The module is a single artifact; the transport-client dependencies it
declares are `optional`, so you only drag in the broker client for the
transport you actually use.

[source,xml]
----
<dependency>
    <groupId>org.smallmind</groupId>
    <artifactId>smallmind-phalanx</artifactId>
    <version>${project.version}</version>
</dependency>
----

Add the matching runtime dependencies:

[cols="1,2",options="header"]
|===
| Transport | Required runtime dependency

| RabbitMQ | `com.rabbitmq:amqp-client`
| Kafka    | `org.apache.kafka:kafka-clients` (and `org.smallmind:kafka-utility` which the module already declares as `optional`)
| JMS      | any Jakarta Messaging 3.x provider; the API is compiled against `jakarta.jms:jakarta.jms-api`
| Mock     | nothing; in-process only
|===

Runtime environment:

* Java 25 — the parent pom sets `maven.compiler.release` to `25`.
* Spring Beans 7.x — used by the classes under `wire/spring/` and
  `transport/amqp/rabbitmq/spring/`; the rest of the module is Spring-free.
* Jackson 3.x — required by `JsonSignalCodec`.
* Jakarta 3.x APIs (`jakarta.jms`, `jakarta.xml.bind`, `jakarta.ws.rs`).

Logging is routed through `scribe-pen`; `LoggerManager.getLogger(...)` is
the internal entry point.

== Quick Start

A complete end-to-end example against the mock router. No broker is
needed, nothing is thread-local except the `WireContext` stack, and the
example fits in one file.

Declare the interface:

[source,java]
----
public interface CalculatorService {

    @InOut(timeoutSeconds = 5)
    Integer add (@Argument("a") int a, @Argument("b") int b);

    @InOnly
    void logCalculation (@Argument("operation") String operation);
}
----

Implement it, including the `WiredService` metadata that
`ResponseTransport.register` requires:

[source,java]
----
public class CalculatorServiceImpl implements CalculatorService, WiredService {

    public Integer add (int a, int b) { return a + b; }

    public void logCalculation (String operation) {
        System.out.println("Performed: " + operation);
    }

    @Override public String getServiceName ()          { return "CalculatorService"; }
    @Override public int    getVersion     ()          { return 1; }
    @Override public void   setResponseTransport (ResponseTransport transport) { }
}
----

Wire both sides through a shared `MockMessageRouter`:

[source,java]
----
MockMessageRouter router = new MockMessageRouter();
JsonSignalCodec   codec  = new JsonSignalCodec();

MockResponseTransport server = new MockResponseTransport(router, codec);
server.register(CalculatorService.class, new CalculatorServiceImpl());      <1>

MockRequestTransport client = new MockRequestTransport(router, codec, 30);  <2>

CalculatorService calculator = (CalculatorService)WireProxyFactory.generateProxy(
    client,
    1,                                           <3>
    "CalculatorService",                         <4>
    CalculatorService.class,
    new StaticParameterExtractor<>("default"),   <5>
    null,                                        <6>
    null                                         <7>
);

Integer sum = calculator.add(5, 3);              // returns 8
calculator.logCalculation("5 + 3");              // returns immediately
----
<1> `register` returns the response transport's instance id — hold on to it
    if the client proxy will issue `@Whisper` calls targeted at this node.
<2> `30` is the default `@InOut` timeout in seconds for methods that don't
    supply one.
<3> Protocol version written into every outbound `Route`.
<4> Logical service name; must match `WiredService.getServiceName()`.
<5> The service-group extractor is required. `StaticParameterExtractor`
    returns the same constant for every call.
<6> Instance-id extractor; required only if the interface contains any
    `@Whisper` method, `null` otherwise.
<7> Per-call timeout extractor; optional. When it returns non-null, it
    overrides both the annotation and the transport default.

== Defining A Service Interface

Each method picks exactly one dispatch annotation. `WireInvocationHandler`
resolves them in the order `@Shout`, `@Whisper`, `@InOnly`, `@InOut` — the
first annotation found wins, so combining them (for example
`@Whisper` + `@InOut`) has no extra effect.

**`@InOut(timeoutSeconds = N)` — request/reply (load-balanced)**::
The caller blocks until the remote implementation returns or the effective
timeout elapses. `timeoutSeconds` is a required attribute; values ≤ 0 fall
back to the request transport's default timeout.

**`@InOnly` — fire-and-forget**::
The method must return `void` and must not declare a `throws` list;
`WireInvocationHandler` rejects both at invocation time. Server-side
exceptions are logged and swallowed — the caller has no way to observe
them.

**`@Whisper(timeoutSeconds = N)` — directed request/reply**::
Delivered to a single instance whose id is supplied by the proxy's
instance-id extractor. Whisper is itself request/reply; do not also mark
the method `@InOut`. `timeoutSeconds` defaults to `0` ("use the transport
default"). If the interface contains any `@Whisper` methods and no
instance-id extractor was supplied to `WireProxyFactory.generateProxy`,
proxy creation fails with `ServiceDefinitionException`.

**`@Shout` — broadcast**::
Delivered to every subscriber of the service group. Shouts collect no
responses; a return value, if declared, is discarded. Use `void` to make
the intent obvious.

**`@Argument("name")` — per-parameter, required on every parameter**::
The name must be unique within the method. Missing annotations cause a
`ServiceDefinitionException` at proxy-creation time. An optional
`type()` attribute overrides the automatically inferred type tag; leave it
empty for normal use.

**`@CallAs("wire-name")` — method-level, optional**::
Overrides the function name carried in the wire signal. The Java method
name on the caller side is unchanged.

Example:

[source,java]
----
public interface OrderService {

    @InOut(timeoutSeconds = 30)
    Order createOrder (@Argument("order") Order order) throws InvalidOrderException;  <1>

    @CallAs("cancel-order")
    @InOut(timeoutSeconds = 5)
    void cancelOrder (@Argument("orderId") String orderId);

    @Whisper(timeoutSeconds = 5)
    Status probe (@Argument("component") String component);

    @Shout
    void clearCache (@Argument("namespace") String namespace);

    @InOnly
    void notifyWarehouse (@Argument("orderId") String orderId);  <2>
}
----
<1> Checked exceptions declared on an `@InOut` or `@Whisper` method are
    materialised on the caller side; see <<error-propagation>>.
<2> `@InOnly` methods can declare no exceptions.

== Creating The Proxy

`WireProxyFactory.generateProxy` builds a JDK dynamic `Proxy` whose calls
are dispatched by `WireInvocationHandler`. The return value is typed as
`java.lang.reflect.Proxy`; cast it to the service interface.

[source,java]
----
public static Proxy generateProxy (
    RequestTransport             transport,
    int                          version,
    String                       serviceName,
    Class<?>                     serviceInterface,
    ParameterExtractor<String>   serviceGroupExtractor,    // required
    ParameterExtractor<String>   instanceIdExtractor,      // required if @Whisper
    ParameterExtractor<Long>     timeoutExtractor          // optional
) throws Exception
----

A `ParameterExtractor<T>` receives the `Method`, the resolved argument map,
and any attached `WireContext` instances; it returns a single value. Two
typical implementations:

* `StaticParameterExtractor<>(value)` — always returns the same constant,
  suitable for fixed service groups.
* A custom extractor that reads a `WireContext` (for example, a
  `TenantContext` that picks the service group dynamically based on tenant).

== Registering An Implementation

On the server side, the implementation must implement `WiredService`:

[source,java]
----
public interface WiredService {
    int    getVersion     ();                                // routing discriminator
    String getServiceName ();                                // routing key
    void   setResponseTransport (ResponseTransport transport) throws Exception;
}
----

Register the implementation with the response transport. The transport
stores a `(version, serviceName) -> MethodInvoker` mapping; subsequent
calls to `register` with the same `(version, serviceName)` are no-ops.

[source,java]
----
String instanceId = responseTransport.register(OrderService.class, new OrderServiceImpl());
----

`register` returns the response transport's instance id — the value a
client-side `instanceIdExtractor` must resolve for `@Whisper` calls
targeted at this node.

Lifecycle control on `ResponseTransport`:

[cols="1,3",options="header"]
|===
| Method | Effect

| `play()`  | Resume request consumption (if previously `pause()`d).
| `pause()` | Suspend request consumption; already-dispatched work continues. Talk queue only; whisper/shout continue uninterrupted.
| `close()` | Stop the transport, shut the worker pool down, release broker resources.
| `getState()` | Returns the current `TransportState` (`PLAYING`, `PAUSED`, `CLOSED`).
|===

== Transports

Each transport ships a matched `RequestTransport` and `ResponseTransport`.
Construct and configure the pair explicitly — nothing is auto-wired.

=== Mock (in-process)

`org.smallmind.phalanx.wire.transport.mock`. Message routing happens inside
a single `MockMessageRouter`; the mock server dispatches inline on the
listener thread, so there is no worker pool.

[source,java]
----
MockMessageRouter     router = new MockMessageRouter();
JsonSignalCodec       codec  = new JsonSignalCodec();
MockRequestTransport  client = new MockRequestTransport(router, codec, 30);
MockResponseTransport server = new MockResponseTransport(router, codec);
----

The third argument to `MockRequestTransport` is the default `@InOut`
timeout in seconds.

=== RabbitMQ

`org.smallmind.phalanx.wire.transport.amqp.rabbitmq`.

`RabbitMQConnector` wraps a RabbitMQ `ConnectionFactory` and a list of
broker `Address` objects. It has no setters — configure authentication,
virtual host, and automatic-recovery flags on the `ConnectionFactory`
before handing it over.

[source,java]
----
ConnectionFactory factory = new ConnectionFactory();
factory.setUsername("guest");
factory.setPassword("guest");
factory.setVirtualHost("/");
factory.setAutomaticRecoveryEnabled(true);

RabbitMQConnector connector = new RabbitMQConnector(
    factory,
    new Address("localhost", 5672)
);
----

A `QueueContractor` declares the AMQP queues the transport uses.
`ClassicQueueContractor` and `QuorumQueueContractor` are the two built-in
implementations — pick one per queue role (the response transport takes
two: an `enduringQueueContractor` for talk queues and an
`ephemeralQueueContractor` for shout and whisper queues).

`NameConfiguration` supplies the suffix tokens used when composing queue
and exchange names. The out-of-the-box suffixes are `requestExchange`,
`responseExchange`, `responseQueue`, `shoutQueue`, `talkQueue`, and
`whisperQueue`; both sides of a deployment must use identical names.

[source,java]
----
NameConfiguration names      = new NameConfiguration();
QueueContractor  ephemeral   = new ClassicQueueContractor();
QueueContractor  enduring    = new QuorumQueueContractor();

RabbitMQRequestTransport client = new RabbitMQRequestTransport(
    connector, ephemeral, names, codec,
    /* clusterSize              */ 4,
    /* concurrencyLimit         */ 8,
    /* defaultTimeoutSeconds    */ 30L,
    /* messageTTLSeconds        */ 60,
    /* autoAcknowledge          */ true,
    /* publisherConfirmationHandler */ null
);

RabbitMQResponseTransport server = new RabbitMQResponseTransport(
    connector, enduring, ephemeral, names,
    InvocationWorker.class, codec,
    /* serviceGroup             */ "default",
    /* clusterSize              */ 4,
    /* concurrencyLimit         */ 8,
    /* messageTTLSeconds        */ 60,
    /* autoAcknowledge          */ true,
    /* publisherConfirmationHandler */ null
);
----

`clusterSize` sets how many distinct `RequestMessageRouter` (or
`ResponseMessageRouter`) instances are created — each owns its own channel
— while `concurrencyLimit` sets how many handler slots the pool round-
robins through. The pool always has `max(clusterSize, concurrencyLimit)`
slots.

=== Kafka

`org.smallmind.phalanx.wire.transport.kafka`. There is no `Properties`-
based configuration surface; brokers are passed as `KafkaServer` instances
and `KafkaConnector` (from `smallmind-kafka-utility`) handles producer,
consumer, and startup-check mechanics.

Topic names are derived by `TopicNames` from a fixed prefix of `"wire"`:

[cols="1,2",options="header"]
|===
| Role | Topic

| shout       | `wire-shout-<serviceGroup>`
| talk        | `wire-talk-<serviceGroup>`
| whisper     | `wire-whisper-<serviceGroup>-<instanceId>`
| response    | `wire-response-<callerId>`
|===

The whisper topic is per-instance and is deleted from the broker when the
response transport is closed; the other three must exist at startup, so
enable topic auto-creation or pre-create them.

[source,java]
----
KafkaServer[] servers = { new KafkaServer("localhost", 9092) };

KafkaRequestTransport client = new KafkaRequestTransport(
    /* nodeName                 */ "orders-api",
    codec,
    /* concurrencyLimit         */ 8,
    /* defaultTimeoutSeconds    */ 30L,
    /* startupGracePeriodSeconds */ 15,
    /* groupProtocol            */ KafkaGroupProtocol.CLASSIC,
    servers
);

KafkaResponseTransport server = new KafkaResponseTransport(
    /* nodeName                 */ "orders-worker",
    /* serviceGroup             */ "default",
    InvocationWorker.class,
    codec,
    /* concurrencyLimit         */ 8,
    /* startupGracePeriodSeconds */ 15,
    /* groupProtocol            */ KafkaGroupProtocol.CLASSIC,
    servers
);
----

`groupProtocol` is the Kafka consumer group protocol applied to every worker
consumer: `KafkaGroupProtocol.CLASSIC` (the classic client-side rebalance
protocol) or `KafkaGroupProtocol.CONSUMER` (the KIP-848 broker-coordinated
protocol). It comes from `smallmind-kafka-utility`.

`startupGracePeriodSeconds` bounds two checks during construction. First
`KafkaConnector` retries the broker connectivity check for up to that many
seconds, throwing `KafkaConnectionException` if no broker becomes reachable.
The transport then waits — within the same budget — for every consumer
ingester to be assigned its partitions and resolve a fetch position
(`awaitConsumerAssignment`); if that readiness is not confirmed before the
budget is exhausted, construction fails with a `KafkaConnectionException`
reporting that consumer readiness could not be confirmed. Because the topics
use `auto.offset.reset=latest`, this readiness wait is what guarantees the
first request published after construction is not silently dropped.

=== JMS

`org.smallmind.phalanx.wire.transport.jms`. The transport targets Jakarta
Messaging 3.x (`jakarta.jms`). Configuration is split across three
collaborators:

* `RoutingFactories` — `ManagedObjectFactory` instances for the request
  queue, request topic, and response topic. Each factory provides a JMS
  `ConnectionFactory` and a `Destination`. The `jndi/` subpackage contains
  JNDI-backed implementations.
* `MessagePolicy` — delivery mode, priority, TTL, and related
  producer-side settings; combined with `DeliveryMode` and
  `JmsAcknowledgeMode` enums.
* `ReconnectionPolicy` — reconnection timing and attempt limits applied by
  the `ConnectionManager` on broker failure.

[source,java]
----
JmsRequestTransport client = new JmsRequestTransport(
    routingFactories, messagePolicy, reconnectionPolicy, codec,
    /* clusterSize           */ 4,
    /* concurrencyLimit      */ 8,
    /* maximumMessageLength  */ 1 << 20,
    /* defaultTimeoutSeconds */ 30L
);

JmsResponseTransport server = new JmsResponseTransport(
    routingFactories, messagePolicy, reconnectionPolicy, codec,
    InvocationWorker.class,
    /* serviceGroup           */ "default",
    /* clusterSize            */ 4,
    /* concurrencyLimit       */ 8,
    /* maximumMessageLength   */ 1 << 20,
    /* transactedSession      */ false,
    /* acknowledgeMode        */ JmsAcknowledgeMode.AUTO_ACKNOWLEDGE
);
----

A JNDI-less Spring assembly is available via the `transport/jms/spring/`
helper beans.

== Wire Contexts

Wire contexts are thread-local values that must follow every call into
remote services without polluting method signatures. Typical examples are
a tenant id, an authenticated principal, a trace/correlation id, or a
request-scoped feature-flag snapshot.

=== Defining A Context Type

Subclass `WireContext` and attach a `@Wire(adapter = ...)` annotation
naming the adapter that serialises it:

[source,java]
----
@Wire(adapter = TenantContextAdapter.class)
public class TenantContext extends WireContext {

    private final String tenantId;

    public TenantContext (String tenantId) { this.tenantId = tenantId; }

    public String getTenantId () { return tenantId; }
}
----

=== Defining The Adapter

`WireAdapter<V, B>` maps a wire value `V` to a business type `B`. The
abstract methods are `getValueType()`, `marshal(B) -> V`, and
`unmarshal(V) -> B`.

[source,java]
----
public class TenantContextAdapter extends WireAdapter<String, TenantContext> {

    @Override public Class<String>   getValueType ()                         { return String.class; }
    @Override public String          marshal       (TenantContext context)   { return context.getTenantId(); }
    @Override public TenantContext   unmarshal     (String tenantId)         { return new TenantContext(tenantId); }
}
----

=== Registering And Using

Register the handle at application startup, on both sides of the wire:

[source,java]
----
WireContextManager.register("tenant", TenantContext.class);
----

Push before calling, pop in `finally`:

[source,java]
----
ContextFactory.pushContext(new TenantContext("customer-123"));
try {
    orderService.createOrder(order);
} finally {
    ContextFactory.popContext(TenantContext.class);
}
----

Read on the server side:

[source,java]
----
TenantContext tenant = ContextFactory.getContext(TenantContext.class);
String tenantId      = tenant.getTenantId();
----

`MethodInvoker` pushes the invocation's contexts before calling the
service method and pops them immediately afterward, so nested outbound
calls inside the same service method see the same contexts.

== Error Propagation

Server-side exceptions are handled inside `WireInvocationCircuit.handle`:

* **Request/reply calls (`@InOut`, `@Whisper`).** The exception is wrapped
  into a `Fault` (from `web-json-scaffold`) and sent back in a
  `ResultSignal` with `error = true`. The request transport unwraps the
  fault on the caller side and the declared checked exception (or a
  `FaultWrappingException` for unchecked ones) escapes the caller's proxy
  method. `TransportException` is reserved for transport-layer problems —
  notably, results that are not `Serializable` are rejected with a
  `TransportException` before transmission.

* **Fire-and-forget calls (`@InOnly`, `@Shout`).** The exception is logged
  via `LoggerManager` and no response is transmitted. The caller cannot
  observe server-side failure.

Framework-thrown conditions worth catching at the call site:

`TransportTimeoutException`::
    Request/reply did not complete within the effective timeout.

`TransportException`::
    A transport-level or serialisation-level failure. Most commonly
    returned when the server-side result is not `Serializable`.

`ServiceDefinitionException`::
    Raised at proxy-creation or registration time when the interface is
    malformed: missing `@Argument`, `@InOnly` with a non-void return or a
    declared `throws`, `@Whisper` without an instance-id extractor, or a
    service group/instance id that the registered wrapper does not match.

`MissingInvocationException` / `MismatchedArgumentException`::
    The method being invoked is not part of the registered interface, or
    the argument shape disagrees with what the interface advertised.

== Timeouts

Timeouts apply to request/reply voices only: `@InOut` and `@Whisper`. For
a single call, the effective timeout is resolved in this order:

1. `timeoutExtractor.getParameter(...)` on the proxy, if it returns
   non-null.
2. The annotation's `timeoutSeconds`, if > 0.
3. The request transport's `defaultTimeoutSeconds` constructor argument.

There is no cancellation. On timeout the framework fails the caller's
callback and discards any late response it later receives for that
correlation id. The server-side invocation continues to completion.
Design services to be idempotent when the caller might retry.

== Threading And Lifecycle

**Client side.** Calls are synchronous. The calling thread blocks inside
`RequestTransport.transmit` until the correlated `ResultSignal` arrives or
the timeout fires, for request/reply voices; for fire-and-forget voices
the method returns as soon as the signal is queued for dispatch. The
request transport allocates no thread pool of its own.

**Server side.** The RabbitMQ and Kafka response transports extend
`WorkManager<InvocationWorker, ...>`. The `concurrencyLimit` constructor
argument sets the number of worker threads; there is no default. Workers
drain a shared `WorkQueue` and dispatch through the
`WireInvocationCircuit`. The mock response transport dispatches inline on
the listener thread and has no pool.

**Context propagation across threads.** A `WireContext` lives on the
pushing thread's `ContextFactory` stack. On the server side
`MethodInvoker` pushes the call's contexts, invokes the method, and pops
them immediately afterward; service code running in the same thread can
therefore rely on `ContextFactory.getContext(...)`. Contexts do *not*
propagate into arbitrary threads spawned from service code — spawn-and-
propagate is the caller's responsibility.

**Shutdown.** Call `close()` on each transport. The response transport
runs `WorkManager.shutDown()` internally and each worker's `stop()`
method is called in sequence (exceptions are logged, not re-thrown). To
drain cleanly: stop issuing requests on the client, close the request
transport, let the response transport's workers complete in-flight work,
then close the response transport.

== Worker Pool Primitives

`org.smallmind.phalanx.worker` is a small, self-contained worker-pool
abstraction. The RabbitMQ and Kafka response transports build on it;
nothing in the package depends on the wire layer, so it can be reused on
its own.

[cols="1,3",options="header"]
|===
| Type | Role

| `WorkManager<W, T>`       | Owns the worker pool's full lifecycle: creates workers via a `WorkerFactory`, wraps them in daemon threads, dispatches work through `execute(T)`, stops them via `shutDown()`. Concurrent `startUp`/`shutDown` calls spin-wait for the transition in flight. Claxon records acquire-worker latency.
| `Worker<T>`               | `Runnable` subclass that drains the queue and processes one item at a time.
| `WorkerFactory<W, T>`     | Produces `Worker` instances bound to the shared queue.
| `WorkQueue<T>`            | Handoff between producer and workers.
| `TransferringWorkQueue`   | `SynchronousQueue`-style handoff — the producer blocks until a worker takes the item. Used as the default by `WorkManager`'s no-queue constructor.
| `BlockingWorkQueue`       | Bounded-buffer handoff for cases where some producer burstiness is expected.
| `WorkManagerException`    | Thrown by `execute(...)` when the manager is not in the `STARTED` state.
|===

== Troubleshooting

**Every call times out.**
Verify `responseTransport.register(Interface.class, wiredService)` was
called, that `wiredService.getServiceName()` and `getVersion()` exactly
match the arguments passed to `WireProxyFactory.generateProxy`, and that
both sides share the same `NameConfiguration` (RabbitMQ) or
node-name/service-group (Kafka). Enable debug logging on
`org.smallmind.phalanx` to see outbound and inbound routes.

**`ServiceDefinitionException` thrown at proxy creation.**
The interface has a method whose parameters are not all annotated with
`@Argument`, an `@InOnly` method that returns non-`void` or declares a
`throws` list, or the interface contains `@Whisper` methods but no
instance-id extractor was supplied to `WireProxyFactory.generateProxy`.

**`TransportException` on an otherwise normal call.**
Most often: the server-side return value is not `Serializable`.
`WireInvocationCircuit` rejects non-`Serializable` results before sending
the reply. Make the type serializable or return a serializable projection.

**Wire contexts are missing on the server.**
The handle must be registered on both sides with
`WireContextManager.register("handle", Class)`; a mismatch causes the
server to drop the context silently. Also verify the context was pushed
before the call and popped in a `finally` block, and that the context
class's no-arg constructor or adapter is reachable from the server
classloader.

**Kafka calls hang after a topic rename or service-group change.**
Phalanx uses `wire-*-<serviceGroup>` topic names. Renaming the group
without restarting both sides leaves the server subscribed to the old
topic while the client publishes to the new one. Coordinate the rollover.

== Limitations

No request cancellation::
    A transmitted request runs to completion on the server even when the
    caller has already timed out. Design retries with idempotency in mind.

Single reply per request/reply call::
    Streaming replies are not supported; return a collection-valued type
    if the caller needs several values.

Shout has no collection or consensus::
    `@Shout` fans out but does not collect responses and does not
    coordinate subscribers. It is not a distributed-transaction primitive.

Results must be `Serializable`::
    `WireInvocationCircuit` rejects non-`Serializable` returns with a
    `TransportException`, independent of which `SignalCodec` is in use.

`@InOnly` methods cannot declare checked exceptions::
    Fire-and-forget contracts cannot participate in checked-exception
    propagation. Move the exception-declaring signature to `@InOut` or
    `@Whisper`.

No automatic retries::
    The transport does not retry failed sends. Wrap the call at the
    application layer if retries are required.

  Jakarta API surfaces compiled against.

  (`WireProxyFactoryBean`, `WireContextInitializingBean`,
