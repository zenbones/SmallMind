[[phalanx, Phalanx]]
= Phalanx

[partintro]
Phalanx is Smallmind's explicit wire and worker module for transport-neutral Java invocation and small background worker pools.
It gives you five practical surfaces:

* interface-first wire contracts and proxy generation in `org.smallmind.phalanx.wire`
* serializable invocation and result payloads plus codecs in `org.smallmind.phalanx.wire.signal`
* transport SPI plus mock, JMS, RabbitMQ, and Kafka integrations
* optional Spring bean helpers and a thin JMX monitor
* reusable worker queues and lifecycle primitives in `org.smallmind.phalanx.worker`

Phalanx stays aligned with Smallmind's library-safe design priorities:

* explicit service registration and bootstrap
* caller-owned transport topology, timeouts, and dependency choice
* thin broker integrations over the native client libraries
* optional Spring usage that complements manual assembly
* reusable worker primitives that stay independent of the wire stack

Audience: developers who want Java-to-Java remote invocation over explicit transports, or who want a small worker abstraction without adopting a larger execution framework.
Versioning: examples use `${smallmind.version}` for Smallmind artifacts; keep related Smallmind modules on the same version unless you intentionally need to split them.

[[phalanx-at-a-glance, At A Glance]]
== At A Glance

=== Integration Model In One Minute

1. Define a Java service interface and annotate every transported parameter with `@Argument`.
2. Let unannotated methods behave as ordinary request/response talk calls, or use `@InOnly`, `@Whisper`, or `@Shout` when the routing pattern differs.
3. Implement the service as a `WiredService` and register it with a `ResponseTransport`.
4. Choose a `SignalCodec`: `JsonSignalCodec` for normal use, `JavaSerializationSignalCodec` only for trusted Java-only boundaries.
5. Generate a proxy with `WireProxyFactory` or `WireProxyFactoryBean`.
6. Supply a required service-group extractor, and optionally instance-id and timeout extractors.
7. If you need ambient metadata, define a `WireContext`, register its handle, and push it into `ContextFactory` before invoking the proxy.
8. Close request transports, response transports, and worker managers explicitly.

[[phalanx-why-use, Why Use Phalanx]]
== Why Use Phalanx

Use Phalanx when you want shared Java contracts and explicit transport control without handing runtime ownership to a framework.

What it does well:

* keeps remote method shape in ordinary Java interfaces instead of generated stubs
* uses named arguments, so wire contracts are more stable than position-only payloads
* keeps service version, service name, service group, and instance targeting explicit
* lets the same contract and proxy model work across mock, JMS, RabbitMQ, and Kafka transports
* integrates with Nutsnbolts `ContextFactory` when you need contextual metadata on the wire
* provides a small worker abstraction that is useful even when you never use the wire stack

Phalanx is a strong fit when:

* caller and callee already share Java code or compatible Java contracts
* you want broker choice to stay separate from contract design
* you need request/response, fire-and-forget, targeted instance calls, and group broadcast in one model
* you need a library-safe worker pool instead of a framework-owned executor stack

Phalanx is especially effective for message-oriented transports and Java-to-Java contracts that stay explicit across broker choices.

=== Direct Integration Dependencies Used By The Module

The `phalanx/pom.xml` shows that the module itself depends directly on Jackson, Scribe, Claxon, Nutsnbolts, Quorum, Spring Beans, and Smallmind's web JSON scaffold.
For most users that means:

* JSON serialization is already wired through `JsonSignalCodec`
* wire faults and non-native error payloads use the Smallmind web JSON fault model
* metrics are emitted through Claxon
* logging uses Scribe's `LoggerManager`
* JMS JNDI helpers can reuse Quorum pooling abstractions

RabbitMQ and Kafka client libraries remain optional so applications keep control of those runtime choices.

[[phalanx-first-run, First Run In 10 Minutes]]
== First Run In 10 Minutes

The safest first integration is:

1. start with the mock transport
2. keep one shared service interface and annotate every parameter with `@Argument`
3. use `JsonSignalCodec`
4. use a static service-group extractor first
5. add whisper only after you have a real instance id from `ResponseTransport.register(...)`
6. close both transports explicitly when the owning component shuts down

That path proves the contract, serialization, context propagation, exception flow, and proxy behavior before you add broker-specific concerns.

[[phalanx-quick-start, Quick Start]]
== Quick Start

The example below is the smallest realistic end-to-end path.
It uses the mock transport so the same proxy and dispatch stack is exercised without external infrastructure.

=== 1. Define The Shared Service Interface

[source,java]
----
import org.smallmind.phalanx.wire.Argument;
import org.smallmind.phalanx.wire.InOnly;
import org.smallmind.phalanx.wire.Whisper;

public interface GreetingService {

  String echo (@Argument("name") String name);

  @InOnly
  void warmCache (@Argument("name") String name);

  @Whisper(timeoutSeconds = 5)
  String identify (@Argument("probe") String probe);
}
----

`echo(...)` is an ordinary talk request/response method because it has no conversation annotation.
`warmCache(...)` is one-way.
`identify(...)` targets a specific instance id.

=== 2. Implement The Service As A `WiredService`

[source,java]
----
import org.smallmind.phalanx.wire.transport.ResponseTransport;
import org.smallmind.phalanx.wire.transport.WiredService;

public class GreetingEndpoint implements GreetingService, WiredService {

  @Override
  public int getVersion () {

    return 1;
  }

  @Override
  public String getServiceName () {

    return "GreetingService";
  }

  @Override
  public void setResponseTransport (ResponseTransport responseTransport) {

    // Built-in transports do not call this hook automatically.
  }

  @Override
  public String echo (String name) {

    return "hello " + name;
  }

  @Override
  public void warmCache (String name) {

  }

  @Override
  public String identify (String probe) {

    return "probe=" + probe;
  }
}
----

=== 3. Wire The Mock Transport, Register The Endpoint, And Create A Proxy

[source,java]
----
import org.smallmind.phalanx.wire.StaticParameterExtractor;
import org.smallmind.phalanx.wire.signal.JsonSignalCodec;
import org.smallmind.phalanx.wire.signal.SignalCodec;
import org.smallmind.phalanx.wire.spring.WireProxyFactory;
import org.smallmind.phalanx.wire.transport.mock.MockMessageRouter;
import org.smallmind.phalanx.wire.transport.mock.MockRequestTransport;
import org.smallmind.phalanx.wire.transport.mock.MockResponseTransport;

MockMessageRouter messageRouter = new MockMessageRouter();
SignalCodec signalCodec = new JsonSignalCodec();

MockRequestTransport requestTransport =
  new MockRequestTransport(messageRouter, signalCodec, 15);
MockResponseTransport responseTransport =
  new MockResponseTransport(messageRouter, signalCodec);

String instanceId =
  responseTransport.register(GreetingService.class, new GreetingEndpoint());

GreetingService greetingService =
  (GreetingService)WireProxyFactory.generateProxy(
    requestTransport,
    1,
    "GreetingService",
    GreetingService.class,
    new StaticParameterExtractor<>("default"),
    new StaticParameterExtractor<>(instanceId),
    null
  );

try {
  String reply = greetingService.echo("Ada");
  greetingService.warmCache("Ada");
  String targetedReply = greetingService.identify("health");
} finally {
  requestTransport.close();
  responseTransport.close();
}
----

That example covers the contract rules you need most often:

* the proxy is generated from the interface, not the implementation
* service version and service name are supplied explicitly
* service-group routing is required even in the mock transport
* whisper calls need an instance-id extractor
* request and response transports are ordinary objects with explicit lifecycle

[[phalanx-call-flow, Call Flow]]
== Call Flow

The normal request/response path has eight steps:

1. The caller invokes a method on a proxy created by `WireProxyFactory`.
2. `WireInvocationHandler` validates interface metadata, turns arguments into a named map, collects matching `WireContext` instances from `ContextFactory`, and constructs a `Voice`.
3. The `RequestTransport` encodes an `InvocationSignal` and publishes transport-specific envelope metadata such as service group, caller id, message id, and correlation id.
4. The `ResponseTransport` receives the request and delegates to `WireInvocationCircuit`.
5. `WireInvocationCircuit` locates a registered service by `(version, serviceName)`, resolves the target `Function`, and asks `MethodInvoker` to build the ordered argument array.
6. `MethodInvoker` pushes incoming contexts into `ContextFactory`, invokes the target method, and pops those contexts in a `finally` block.
7. The result or fault is wrapped as a `ResultSignal` and sent back through a `ResponseTransmitter` unless the call is one-way.
8. The request side callback unwraps the `ResultSignal`, converts the payload back to the declared return type, rethrows the underlying Java exception when possible, or throws `FaultWrappingException` when only a neutral fault payload is available.

=== Signal Versus Envelope

One design detail matters when you are debugging or implementing a custom transport:

* `InvocationSignal` carries `inOnly`, `Route`, named arguments, and contexts.
* Service group, instance id, caller id, message id, correlation id, clock, and content type live in transport-specific envelope metadata.

That is why broker transports use headers, selectors, routing keys, or topic names for voice-based routing instead of putting everything inside the serialized signal.

[IMPORTANT]
====
If you build a custom transport, you must preserve at least these transport-side values:

* caller id for routing responses back to the originating request transport
* message id plus correlation id for request/response matching
* service group for talk, shout, and whisper routing
* instance id for whisper routing
====

[[phalanx-contracts, Wire Contract Model]]
== Wire Contract Model

=== Core Terms

[cols="1,4",options="header"]
|===
|Term |Meaning

|Service name
|Logical service identifier carried in `Route.service`

|Version
|Integer protocol version carried in `Route.version`

|Function
|Method descriptor made up of wire name, result signature, native return type, and parameter signature

|Service group
|Transport-side grouping key used to choose the queue, topic, or routing path for a call

|Instance id
|Transport-side target identifier used by whisper calls

|Talk
|The ordinary call path; unannotated methods use it by default

|Whisper
|A targeted call to one specific response transport instance

|Shout
|A one-way broadcast-oriented call to a service group
|===

=== Annotation Reference

[cols="1,3,4",options="header"]
|===
|Annotation |Where it applies |Effect

|`@Argument("name")`
|Method parameters on the shared service interface
|Required for every transported parameter; the value becomes the key in `InvocationSignal.arguments`

|`@Argument(type = "Hint")`
|Method parameters
|Overrides the neutral signature label for the parameter; useful when simple type names would collide

|`@CallAs("wireName")`
|Method
|Changes the wire-level function name without renaming the Java method

|`@Result("Hint")`
|Method
|Overrides the neutral result signature label used for signature matching; it does not change actual serialization

|`@InOnly`
|Method
|Marks the call as one-way talk traffic; the method must return `void` and declare no checked exceptions

|`@InOut(timeoutSeconds = ...)`
|Method
|Marks a talk call as request/response and can express a timeout override in the contract

|`@Whisper(timeoutSeconds = ...)`
|Method
|Targets a specific instance id and can express a timeout override

|`@Shout`
|Method
|Broadcast-oriented one-way invocation to the whole service group

|`@Wire(adapter = ...)`
|Type
|Causes arguments of that type to be marshalled and unmarshalled through a `WireAdapter`
|===

[IMPORTANT]
====
Put the parameter annotations on the interface used for proxy generation and service registration.
The built-in proxy and dispatch path does not infer parameter names from implementation classes or Java compiler metadata.
====

=== Default Semantics

If a method has no conversation annotation:

* it is treated as talk traffic
* it is request/response (`IN_OUT`)
* it uses the request transport's default timeout unless a timeout extractor overrides it

That means `@InOut` is optional unless you want the timeout choice to be visible on the method itself.

=== Service Metadata Contracts

Two interfaces exist in the module:

* `Service`: metadata-only contract exposing `getServiceName()` and `getVersion()`
* `WiredService`: responder contract exposing the same metadata plus `setResponseTransport(...)`

Built-in response transports require `WiredService` in `register(...)`.

[NOTE]
====
The built-in `register(...)` implementations use `getVersion()` and `getServiceName()`, but they do not call `setResponseTransport(...)`.
If your endpoint relies on that hook, call it yourself during assembly before or after registration.
====

=== Proxy-Time Extractors

`WireInvocationHandler` depends on up to three extractor strategies:

[cols="1,2,4",options="header"]
|===
|Extractor |Required |Purpose

|`ParameterExtractor<String>` for service group
|Yes
|Resolves the destination group for every call

|`ParameterExtractor<String>` for instance id
|Only for whisper methods
|Resolves the target instance id at invocation time

|`ParameterExtractor<Long>` for timeout
|No
|Overrides method annotation or request-transport default when it returns a positive or non-null value
|===

Timeout precedence is:

1. timeout extractor result, if non-null
2. `@Whisper(timeoutSeconds = ...)` or `@InOut(timeoutSeconds = ...)`, if present and positive
3. request transport default timeout

[[phalanx-proxies, Client Proxies And Extractors]]
== Client Proxies And Extractors

`WireProxyFactory.generateProxy(...)` gives you a JDK dynamic proxy, so the normal request-side assembly is explicit and small.
The method parameters are:

* request transport
* service version
* service name
* service interface
* service-group extractor
* instance-id extractor
* timeout extractor

=== Example: Service Group Derived From An Argument

[source,java]
----
import java.lang.reflect.Method;
import java.util.HashMap;
import org.smallmind.phalanx.wire.Argument;
import org.smallmind.phalanx.wire.MissingInstanceIdException;
import org.smallmind.phalanx.wire.ParameterExtractor;
import org.smallmind.phalanx.wire.Whisper;
import org.smallmind.phalanx.wire.signal.WireContext;

public interface QuoteService {

  Quote price (
    @Argument("tenant") String tenant,
    @Argument("sku") String sku
  );

  @Whisper(timeoutSeconds = 3)
  HealthStatus health (
    @Argument("tenant") String tenant,
    @Argument("instanceHint") String instanceHint
  );
}

public class TenantServiceGroupExtractor implements ParameterExtractor<String> {

  @Override
  public String getParameter (Method method, HashMap<String, Object> argumentMap, WireContext... wireContexts) {

    return "pricing-" + argumentMap.get("tenant");
  }
}

public class ArgumentInstanceIdExtractor implements ParameterExtractor<String> {

  @Override
  public String getParameter (Method method, HashMap<String, Object> argumentMap, WireContext... wireContexts)
    throws MissingInstanceIdException {

    String instanceId = (String)argumentMap.get("instanceHint");

    if ((instanceId == null) || instanceId.isEmpty()) {
      throw new MissingInstanceIdException("No instance id supplied for %s", method.getName());
    }

    return instanceId;
  }
}
----

That is the intended shape of extractor logic: ordinary code inspecting the method, the named argument map, and any contexts attached to the call.
There is no registry, expression language, or container-owned routing layer in front of it.

=== Object Methods On Proxies

`toString()`, `hashCode()`, and `equals(Object)` are part of the proxy invocation map even when they are not declared on the service interface.
Calling them on a live wire proxy is therefore a remote invocation, not local proxy-only behavior.

[[phalanx-serialization, Serialization, Typing, And Adapters]]
== Serialization, Typing, And Adapters

=== Choosing A Codec

Two codecs ship with the module:

[cols="1,4",options="header"]
|===
|Codec |Use it when

|`JsonSignalCodec`
|You want the normal default: portable JSON payloads, Jackson-based conversion, and optional verbose logging

|`JavaSerializationSignalCodec`
|You control both sides, trust the boundary, and deliberately want built-in Java serialization semantics
|===

`JsonSignalCodec` is the usual choice.
It handles object conversion through the Smallmind JSON scaffold and keeps payloads inspectable when debugging.

[WARNING]
====
`JavaSerializationSignalCodec` should be treated as a trusted Java-only option, not a general integration default.
It depends on Java object serialization on both sides of the boundary.
====

=== Example: Explicit Wire Names And Type Hints

[source,java]
----
import org.smallmind.phalanx.wire.Argument;
import org.smallmind.phalanx.wire.CallAs;
import org.smallmind.phalanx.wire.Result;

public interface ReservationService {

  @CallAs("reserve")
  @Result("ReservationReceipt")
  Receipt reserve (
    @Argument(value = "request", type = "ReservationRequest") ReservationRequest request
  );
}
----

Use those hints when wire-level compatibility matters more than the current Java type name.
This is especially useful when multiple modules define classes with the same simple name.

=== Example: `@Wire` Adapter For Parameter Marshalling

`@Wire` adapters apply to arguments.
They let you transport a domain type through a simpler serializable value without hard-wiring that decision into the service contract.

[source,java]
----
import org.smallmind.phalanx.wire.Argument;
import org.smallmind.phalanx.wire.Wire;
import org.smallmind.phalanx.wire.WireAdapter;

@Wire(adapter = MoneyWireAdapter.class)
public class Money {

  private final String currency;
  private final long minorUnits;

  public Money (String currency, long minorUnits) {

    this.currency = currency;
    this.minorUnits = minorUnits;
  }

  public String getCurrency () {

    return currency;
  }

  public long getMinorUnits () {

    return minorUnits;
  }

  public static Money parse (String value) {

    String[] parts = value.split(":", 2);

    return new Money(parts[0], Long.parseLong(parts[1]));
  }

  @Override
  public String toString () {

    return currency + ":" + minorUnits;
  }
}

public class MoneyWireAdapter extends WireAdapter<String, Money> {

  @Override
  public Class<String> getValueType () {

    return String.class;
  }

  @Override
  public Money unmarshal (String obj) {

    return Money.parse(obj);
  }

  @Override
  public String marshal (Money obj) {

    return obj.toString();
  }
}

public interface PricingAdminService {

  void updatePrice (
    @Argument("sku") String sku,
    @Argument("price") Money price
  );
}
----

Adapter rules worth knowing:

* the adapter class must have a public no-argument constructor
* the adapter's value type must be serializable
* adapter instances are cached per adapter class
* adapters are used for arguments, not for return values

=== Result And Fault Serialization Rules

Results still have to be serializable or convertible by the chosen codec.
On the error path, the client receives:

* the original Java throwable when the remote side could preserve it as a native Java payload
* `FaultWrappingException` when only a neutral Smallmind fault payload is available

[[phalanx-contexts, Wire Context Propagation]]
== Wire Context Propagation

Wire contexts are the explicit way to move ambient metadata across the invocation boundary.
They integrate with Nutsnbolts `ContextFactory`, so request code pushes a context locally and service code reads it locally after transport.

=== Setup Rules

1. Ensure a `PerApplicationContext` exists before you register context handles.
2. Define a context type that extends `WireContext`.
3. Give it a stable handle by using `@XmlRootElement(name = "...")` or by deliberately relying on the simple class name.
4. Register the handle/class pair with `WireContextManager.register(...)`.
5. Push the context on the caller side with `ContextFactory.pushContext(...)`.
6. Read it from `ContextFactory` inside the service implementation.

[IMPORTANT]
====
`WireContextManager` stores registrations in `PerApplicationContext`.
If you call `WireContextManager.register(...)` before initializing `PerApplicationContext`, registration fails.
====

=== Example: Defining And Registering A Context

[source,java]
----
import jakarta.xml.bind.annotation.XmlElement;
import jakarta.xml.bind.annotation.XmlRootElement;
import org.smallmind.nutsnbolts.lang.PerApplicationContext;
import org.smallmind.phalanx.wire.WireContextManager;
import org.smallmind.phalanx.wire.signal.WireContext;

@XmlRootElement(name = "requestScope")
public class RequestScopeContext extends WireContext {

  private String requestId;

  public RequestScopeContext () {

  }

  public RequestScopeContext (String requestId) {

    this.requestId = requestId;
  }

  @XmlElement(name = "requestId", required = true)
  public String getRequestId () {

    return requestId;
  }

  public void setRequestId (String requestId) {

    this.requestId = requestId;
  }
}

new PerApplicationContext();
WireContextManager.register("requestScope", RequestScopeContext.class);
----

The handle passed to `register(...)` should match the context name that will appear on the wire.
With the example above, `requestScope` matches the `@XmlRootElement` name.

=== Example: Push On The Caller, Read On The Service Side

[source,java]
----
import org.smallmind.nutsnbolts.context.ContextFactory;

ContextFactory.pushContext(new RequestScopeContext("req-101"));
try {
  greetingService.echo("Ada");
} finally {
  ContextFactory.popContext(RequestScopeContext.class);
}

public class GreetingEndpoint implements GreetingService, WiredService {

  @Override
  public String echo (String name) {

    RequestScopeContext requestScope =
      ContextFactory.getContext(RequestScopeContext.class);

    return requestScope.getRequestId() + ":hello " + name;
  }
}
----

=== Unknown Context Handles

If the receiving side has not registered a context handle, the payload is preserved as `ProtoWireContext`.
That means the data is not lost, but typed lookup through `ContextFactory.getContext(...)` will not work until the receiving side knows the handle.

== Transport SPI And Custom Transport Notes

Phalanx is transport-neutral because the core wire layer is built around small transport contracts.

=== Main SPI Types

[cols="1,4",options="header"]
|===
|Type |Role

|`RequestTransport`
|Sends an invocation and completes callbacks when a `ResultSignal` arrives

|`ResponseTransport`
|Registers services, exposes instance id and lifecycle, and receives incoming requests

|`ResponseTransmitter`
|Sends a `ResultSignal` back to the caller after invocation completes

|`WireInvocationCircuit`
|Routes an `InvocationSignal` to the correct registered service and method

|`MethodInvoker`
|Converts named arguments back into ordered Java arguments and manages `WireContext` push/pop
|===

=== Reuse `AbstractRequestTransport` When Building A New Request Transport

`AbstractRequestTransport` already handles the tricky part of synchronous request/response behavior:

* timeout selection
* callback correlation
* early-arriving responses that show up before the waiting callback is installed

If you are implementing a new request transport, extending `AbstractRequestTransport` is usually the right starting point.

=== Example: Skeleton For A Custom Request Transport

[source,java]
----
import java.util.Map;
import org.smallmind.phalanx.wire.ConversationType;
import org.smallmind.phalanx.wire.Voice;
import org.smallmind.phalanx.wire.signal.InvocationSignal;
import org.smallmind.phalanx.wire.signal.Route;
import org.smallmind.phalanx.wire.signal.SignalCodec;
import org.smallmind.phalanx.wire.signal.WireContext;
import org.smallmind.phalanx.wire.transport.AbstractRequestTransport;

public class CustomRequestTransport extends AbstractRequestTransport {

  private final String callerId = "caller-1";
  private final SignalCodec signalCodec;

  public CustomRequestTransport (SignalCodec signalCodec, long defaultTimeoutSeconds) {

    super(defaultTimeoutSeconds);

    this.signalCodec = signalCodec;
  }

  @Override
  public String getCallerId () {

    return callerId;
  }

  @Override
  public Object transmit (Voice<?, ?> voice, Route route, Map<String, Object> arguments, WireContext... contexts)
    throws Throwable {

    String messageId = nextMessageId();
    boolean inOnly =
      voice.getConversation().getConversationType().equals(ConversationType.IN_ONLY);

    byte[] payload =
      signalCodec.encode(new InvocationSignal(inOnly, route, arguments, contexts));

    publish(messageId, voice, payload);

    return acquireResult(signalCodec, route, voice, messageId, inOnly);
  }

  @Override
  public void close () {

  }

  private String nextMessageId () {

    return Long.toString(System.nanoTime());
  }

  private void publish (String messageId, Voice<?, ?> voice, byte[] payload) {

  }
}
----

The point is not the dummy ids.
The point is the division of responsibility: Phalanx owns the signal model and callback semantics, while your transport owns the actual envelope and delivery.

=== Example: Response-Side Dispatch Hook

On the responder side, the hand-off normally looks like this:

[source,java]
----
invocationCircuit.handle(
  this,
  signalCodec,
  callerId,
  messageId,
  invocationSignal
);
----

That one call handles service lookup, argument reconstruction, method invocation, context propagation, fault wrapping, and `ResultSignal` transmission through the supplied `ResponseTransmitter`.

[[phalanx-jms, JMS Transport]]
== JMS Transport

Use the JMS transport when you already have a JMS provider, own your destinations, and want Phalanx to stay a thin adapter over that setup.

=== What You Supply

The JMS transport expects three explicit destination/connection factories wrapped in `RoutingFactories`:

* request queue factory for talk traffic
* request topic factory for shout and whisper traffic
* response topic factory for replies

You also supply:

* `MessagePolicy` for producer settings such as delivery mode, TTL, and priority
* `ReconnectionPolicy` for reconnection delay and attempt limits
* cluster size, concurrency limit, maximum message length, and default timeout

=== Example: Core JMS Assembly

[source,java]
----
import org.smallmind.phalanx.wire.signal.JsonSignalCodec;
import org.smallmind.phalanx.wire.signal.SignalCodec;
import org.smallmind.phalanx.wire.transport.jms.DeliveryMode;
import org.smallmind.phalanx.wire.transport.jms.JmsRequestTransport;
import org.smallmind.phalanx.wire.transport.jms.JmsResponseTransport;
import org.smallmind.phalanx.wire.transport.jms.MessagePolicy;
import org.smallmind.phalanx.wire.transport.jms.ReconnectionPolicy;
import org.smallmind.phalanx.wire.transport.jms.RoutingFactories;

SignalCodec signalCodec = new JsonSignalCodec();

RoutingFactories routingFactories =
  new RoutingFactories(
    requestQueueFactory,
    requestTopicFactory,
    responseTopicFactory
  );

MessagePolicy messagePolicy = new MessagePolicy();
messagePolicy.setDeliveryMode(DeliveryMode.PERSISTENT);
messagePolicy.setTimeToLiveSeconds(30);

ReconnectionPolicy reconnectionPolicy = new ReconnectionPolicy();
reconnectionPolicy.setReconnectionDelayMilliseconds(1000);
reconnectionPolicy.setReconnectionAttempts(-1);

JmsRequestTransport requestTransport =
  new JmsRequestTransport(
    routingFactories,
    messagePolicy,
    reconnectionPolicy,
    signalCodec,
    2,
    8,
    1024 * 1024,
    15
  );

JmsResponseTransport responseTransport =
  new JmsResponseTransport(
    routingFactories,
    messagePolicy,
    reconnectionPolicy,
    signalCodec,
    "default",
    2,
    8,
    1024 * 1024
  );
----

=== Common JMS Wiring Pattern

A common choice is to build each `ManagedObjectFactory` from `JndiManagedObjectFactory` plus `JmsConnectionDetails`.
That keeps destination lookup explicit instead of burying it in the transport itself.

=== JMS-Specific Notes

* talk traffic consumes from the request queue selected by `serviceGroup`
* shout and whisper traffic consume from the request topic selected by `serviceGroup`, with whisper adding an `instanceId` selector
* responses are published to the response topic and filtered back to the request side by `callerId`
* `maximumMessageLength` must be large enough for the largest encoded request or reply you expect
* `clusterSize` controls how many listener/connection sets are created; `concurrencyLimit` controls worker concurrency for invocation handling

[[phalanx-rabbitmq, RabbitMQ Transport]]
== RabbitMQ Transport

Use the RabbitMQ transport when exchange names, queue names, queue types, and broker topology are part of your operational design.

=== What You Supply

RabbitMQ assembly is still intentionally explicit.
You choose:

* broker addresses and credentials through `RabbitMQConnector`
* queue naming through `NameConfiguration`
* queue durability and type through `QueueContractor`
* message TTL, auto-acknowledgement, publisher-confirmation handling, cluster size, and concurrency

=== Example: Explicit RabbitMQ Assembly

[source,java]
----
import com.rabbitmq.client.Address;
import com.rabbitmq.client.ConnectionFactory;
import org.smallmind.phalanx.wire.signal.JsonSignalCodec;
import org.smallmind.phalanx.wire.signal.SignalCodec;
import org.smallmind.phalanx.wire.transport.amqp.rabbitmq.ClassicQueueContractor;
import org.smallmind.phalanx.wire.transport.amqp.rabbitmq.InvocationWorker;
import org.smallmind.phalanx.wire.transport.amqp.rabbitmq.NameConfiguration;
import org.smallmind.phalanx.wire.transport.amqp.rabbitmq.QueueContractor;
import org.smallmind.phalanx.wire.transport.amqp.rabbitmq.QuorumQueueContractor;
import org.smallmind.phalanx.wire.transport.amqp.rabbitmq.RabbitMQConnector;
import org.smallmind.phalanx.wire.transport.amqp.rabbitmq.RabbitMQRequestTransport;
import org.smallmind.phalanx.wire.transport.amqp.rabbitmq.RabbitMQResponseTransport;

ConnectionFactory connectionFactory = new ConnectionFactory();
connectionFactory.setUsername("app");
connectionFactory.setPassword("secret");

RabbitMQConnector connector =
  new RabbitMQConnector(
    connectionFactory,
    new Address("rabbit-1.example.net", 5672),
    new Address("rabbit-2.example.net", 5672)
  );

NameConfiguration nameConfiguration = new NameConfiguration();
QueueContractor enduringQueueContractor = new QuorumQueueContractor(3);
QueueContractor ephemeralQueueContractor = new ClassicQueueContractor();
SignalCodec signalCodec = new JsonSignalCodec();

RabbitMQRequestTransport requestTransport =
  new RabbitMQRequestTransport(
    connector,
    ephemeralQueueContractor,
    nameConfiguration,
    signalCodec,
    2,
    8,
    15,
    30,
    false,
    null
  );

RabbitMQResponseTransport responseTransport =
  new RabbitMQResponseTransport(
    connector,
    enduringQueueContractor,
    ephemeralQueueContractor,
    nameConfiguration,
    InvocationWorker.class,
    signalCodec,
    "default",
    2,
    8,
    30,
    false,
    null
  );
----

=== RabbitMQ-Specific Notes

* `NameConfiguration` lets you rename the request exchange, response exchange, and queue name bases instead of accepting the shipped defaults
* `ClassicQueueContractor` declares classic queues; `QuorumQueueContractor` declares durable quorum queues with an explicit replication count
* request transports use an ephemeral queue contractor because request-side reply handling is caller specific
* response transports use separate enduring and ephemeral queue contractors so you can make service queues durable without forcing the same choice on response plumbing
* `register(...)` returns the responder's instance id for whisper routing

== Kafka Transport

Use the Kafka transport when you want topic-based routing and you are comfortable treating response paths and whisper paths as named Kafka topics rather than hidden callback infrastructure.

=== Example: Topic-Based Assembly

[source,java]
----

import org.smallmind.phalanx.wire.signal.JsonSignalCodec;

=== Derived Topic Names

`KafkaRequestTransport` and `KafkaResponseTransport` both use `TopicNames("wire")`, so the main topic shapes are:

* `wire-talk-<serviceGroup>`
* `wire-shout-<serviceGroup>`
* `wire-whisper-<serviceGroup>-<instanceId>`
* `wire-response-<callerId>`

=== Kafka-Specific Notes

[[phalanx-worker, Worker Package]]
== Worker Package

The worker package is useful even if you never touch the wire layer.
It gives you a small, explicit worker-pool abstraction with caller-owned queue choice and lifecycle.

=== Main Types

* `WorkManager<W extends Worker<T>, T>`
* `Worker<T>`
* `WorkerFactory<W, T>`
* `WorkQueue<T>`
* `BlockingWorkQueue<T>`
* `TransferringWorkQueue<T>`
* `WorkManagerException`

=== Model

`WorkManager` owns:

* a worker class
* a concurrency limit
* a `WorkQueue`

On `startUp(...)`, it creates daemon threads and asks a `WorkerFactory` to build one worker per slot.
On `execute(...)`, it keeps offering work until the queue accepts it.
On `shutDown()`, it calls `stop()` on each worker and waits for them to exit.

=== Queue Choices

[cols="1,4",options="header"]
|===
|Queue |Behavior

|`TransferringWorkQueue`
|Backed by `LinkedTransferQueue`; favors direct hand-off to waiting workers and is the default when no queue is supplied

|`BlockingWorkQueue`
|Backed by `LinkedBlockingQueue`; use it when buffering is preferable to direct hand-off
|===

=== Example: Standalone Worker Pool

[source,java]
----
import org.smallmind.phalanx.worker.BlockingWorkQueue;
import org.smallmind.phalanx.worker.WorkManager;
import org.smallmind.phalanx.worker.WorkQueue;
import org.smallmind.phalanx.worker.Worker;

public class AuditWorker extends Worker<String> {

  public AuditWorker (WorkQueue<String> workQueue) {

    super(workQueue);
  }

  @Override
  public void engageWork (String transfer) {

    System.out.println(transfer);
  }

  @Override
  public void close () {

  }
}

WorkManager<AuditWorker, String> workManager =
  new WorkManager<>(AuditWorker.class, 4, new BlockingWorkQueue<>(1000));

workManager.startUp(AuditWorker::new);
try {
  workManager.execute("audit:order:101");
} finally {
  workManager.shutDown();
}
----

=== Worker Runtime Notes

* worker threads are daemons
* `execute(...)` blocks until a queue accepts the work item
* workers poll in one-second intervals
* exceptions from `engageWork(...)` are logged and do not kill the worker loop
* `stop()` calls `close()`, interrupts the worker thread, and waits for exit

[[phalanx-edge-cases, Behavioral Edge Cases]]
== Behavioral Edge Cases

These are the details most likely to surprise an integrator reading only the type names.

* Every transported parameter on the service interface must have `@Argument`; implementation-only annotations do not rescue an under-annotated interface.
* `@InOnly` is validated strictly: the method must return `void` and declare no exception list.
* `@Shout` is one-way in the transport model, but the proxy layer does not reject a non-void signature; callers should still expect no response.
* `@Wire` adapters apply to arguments, not to return values.
* Built-in `register(...)` implementations do not call `WiredService.setResponseTransport(...)`.
* `toString()`, `hashCode()`, and `equals(Object)` on a wire proxy are remote invocations.
* Custom type neutral signatures use simple names by default, so distinct classes with the same simple name can collide unless you supply type hints.
* Unknown context handles become `ProtoWireContext` instead of failing deserialization immediately.
* Results still need to be serializable or codec-convertible.
* When the remote side cannot preserve a native Java throwable, the caller receives `FaultWrappingException` rather than the original exception type.
* `AbstractRequestTransport` supports early-arriving responses by storing a synchronous callback placeholder before the waiting code starts.
* In the mock transport, shout shares the talk queue path and pause/play only gates talk traffic.
* `WireContextManager.register(...)` depends on `PerApplicationContext`; without it, registration fails before any message is sent.

[[phalanx-troubleshooting, Troubleshooting]]
== Troubleshooting

[cols="2,3,4",options="header"]
|===
|Symptom |Likely cause |What to check

|`ServiceDefinitionException` mentioning `@Argument`
|A parameter on the service interface lacks `@Argument`
|Check the interface used by proxy generation and registration, not only the implementation class

|`ServiceDefinitionException` on a whisper method
|No instance-id extractor was supplied
|Pass a non-null instance-id extractor when whisper methods are possible

|`TransportTimeoutException`
|No matching responder, wrong service metadata, wrong routing, or response delivery failure
|Check service name, version, service group, instance id, timeout policy, and whether the method is intentionally one-way

|`MissingInvocationException`
|Route or function metadata does not match a registered service method
|Check `@CallAs`, service version, service name, and signature changes across caller and responder

|`MismatchedArgumentException`
|Incoming argument names no longer line up with the registered method
|Check renamed `@Argument` values and mixed client/server versions

|`FaultWrappingException`
|The remote side returned a neutral fault instead of a native Java throwable payload
|Inspect the remote exception and codec choice; native throwable replay is not guaranteed across every boundary

|`TransportException` about a non-serializable argument or result
|An argument or return value cannot be encoded by the chosen codec and has no valid adapter path
|Check argument classes, result classes, and any `@Wire` adapter value types

|`MissingPerApplicationContextException` during context registration
|`PerApplicationContext` was not initialized before `WireContextManager.register(...)`
|Create `new PerApplicationContext()` before registering wire-context handles

|Context not visible inside the service
|Context was not pushed on the caller side or the handle was not registered on the responder side
|Check `WireContextManager.register(...)`, `ContextFactory.pushContext(...)`, and whether the handle matches the serialized context name

|JMS message-length failure
|`maximumMessageLength` is too small
|Increase the configured maximum for request and response listeners

|Kafka `AlreadyClosedException`
|A Kafka transport was used after close
|Check shutdown ordering and whether the transport is shared across components

|Mock tests pass but broker runtime behaves differently
|The mock transport intentionally simplifies broker behavior
|Validate shout fan-out, pause/play behavior, selectors, and durability with the real transport
|===

[[phalanx-minimal-checklist, Minimal Checklist]]
== Minimal Checklist

* annotate every transported parameter on the shared service interface with `@Argument`
* keep `serviceName` and `version` identical between proxy and responder
* always supply a service-group extractor
* supply an instance-id extractor whenever whisper methods exist
* choose `JsonSignalCodec` unless you have a deliberate reason to use Java serialization
* initialize `PerApplicationContext` before registering `WireContext` handles
* register every `WireContext` handle on the receiving side before traffic starts
* make sure arguments and results are serializable or intentionally adapted
* close request transports, response transports, and worker managers explicitly

[[phalanx-verification, Verification Checklist]]
== Verification Checklist

For a focused integration check, cover these cases:

1. basic talk request/response with primitive and object arguments
2. one-way method using `@InOnly`
3. targeted whisper call using the instance id returned by `register(...)`
4. exception propagation, including what happens when only a neutral fault payload is available
5. context propagation through a `WireContext`, if your service depends on it
6. shout behavior, if your contract uses `@Shout`
7. `pause()` and `play()` behavior on the chosen response transport
8. explicit close behavior of both request and response transports
9. any transport-specific configuration such as JMS selectors, RabbitMQ queue type, or Kafka topic naming

For the smallest realistic test pattern, the shipped `MockWireTest` is the right starting point.
