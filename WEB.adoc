== Overview

`smallmind-web` is the parent POM for the project's HTTP-side concerns. It does
not ship as a single artifact. Instead it is a curated set of seven sibling
modules that can be pulled in independently, each focused on one seam of the
web stack:

* **`web-grizzly`** — an embedded Grizzly 2.4 HTTP/HTTPS/HTTP2/WebSocket server,
  wired through Spring, that hosts servlets, JAX-RS resources, JAX-WS SOAP
  endpoints, and static resources in one or more named contexts.
* **`web-jetty`** — an embedded Jetty 12 (EE10) server, assembled the same way
  by Spring, for deployments that prefer Jetty's connector model and HTTP/1.1
  semantics.
* **`web-jersey`** — the Jersey 3 (JAX-RS) companion library: JSON provider,
  exception mapper, CORS filter, pagination filter, multipart feature, an
  AspectJ-driven parameter-injection layer, and a dynamic JSON-over-HTTP client
  proxy generator.
* **`web-http`** — a pooled, asynchronous, blocking-friendly HTTP client built
  on Apache HttpComponents 5.
* **`web-json-query`** — a JSON/JAXB-serialisable query-expression language
  with translators for JPA Criteria, QueryDSL, and MongoDB/Throng.
* **`web-json-scaffold`** — the JSON codec, fault model, polymorphic Jackson
  module, reflection helpers, and paging primitives that the rest of the
  `web-*` modules build on.
* **`web-json-doppelganger`** — an annotation processor that generates
  direction- and purpose-aware view classes from domain entities. Because the
  module is a separate compile-time tool with its own conceptual model, it has
  its own reference: see <<_doppelganger,Doppelganger>> for a cross reference,
  and xref:DOPPELGANGER.adoc[DOPPELGANGER.adoc] for the full document.
* **`schema-validation-maven-plugin`** (directory: `web/schema`) — a Maven
  plugin that fails the build when a shipped XSD or XML instance file is
  malformed. Despite living under `web/schema`, it has no relationship to
  JSON-schema generation.

=== Why this package exists

Plenty of frameworks answer "how do I bring up an HTTP server?" but they tend
to couple the server, the resource layer, the serialisation layer, and the
wire-format conventions together. `smallmind-web` is organised around the
opposite principle: each module has a single job and a deliberate dependency
arrow.

. **Pick your container.** `web-grizzly` and `web-jetty` solve the same
  problem — host servlets, JAX-RS, JAX-WS, and WebSockets — with different
  engines. They have near-identical Spring configuration surface (`SSLInfo`,
  `WebApplicationOption`, `FilterInstaller`, `ServletInstaller`,
  `ListenerInstaller`, `WebServiceInstaller`, `@ServicePath`). Swapping
  containers is a dependency swap plus a bean-id rename.
. **Pick your resource style.** `web-jersey` decorates a Jersey `ResourceConfig`
  with the project's opinions (JSON, CORS, exception mapping, paging, entity
  parameter injection). The container modules accept any `ResourceConfig`, so
  the Jersey extensions are optional even when the container is not.
. **Pick your data layer.** `web-json-query` describes predicates, sorts, and
  paging in a way that survives JSON round-trips and translates to multiple
  backends — JPA, QueryDSL, or MongoDB — on the server side.
. **Generate your API views.** `web-json-doppelganger` eliminates the
  hand-written DTOs that usually live between a domain entity and its JSON
  representation, and makes the "create vs. update vs. admin vs. public"
  projection distinction first-class instead of a convention.

A lean service typically pulls in `web-grizzly` (or `web-jetty`) + `web-jersey`
+ `web-json-scaffold`, and declares one or two `WebApplicationOption` beans.
Richer services add `web-json-query` and `web-json-doppelganger`.

=== What it intentionally does not do

* It does not bundle security. There is no built-in authentication, RBAC, or
  principal model; authentication is expected to be driven by standard servlet
  filters, Spring Security, or a JAX-RS `ContainerRequestFilter` that the
  application contributes.
* It does not prescribe a persistence layer. `web-json-query` integrates with
  JPA, QueryDSL, and MongoDB/Throng, but none of those dependencies are
  required to use the module — each is an optional dependency that activates
  one of its translator classes.
* It does not abstract over containers. The two container modules share a
  vocabulary but not an SPI; choose one or the other per deployment.

First time through::
    Read <<_web_grizzly>> (or <<_web_jetty>>), then <<_web_jersey>>, then
    <<_web_json_scaffold>>. That is the minimum to stand up a JSON REST service.

    Read <<_web_json_query>>, then pick one of the JPA/QueryDSL/Throng
    integration sub-sections.

    Read the _Client Proxies_ subsection of <<_web_jersey>> (`JsonTarget`,

    Read <<_web_jwt>>.

Schema validation as a build step::
    Read <<_schema_validation_maven_plugin>>.

Generated DTOs::
    Read xref:DOPPELGANGER.adoc[DOPPELGANGER.adoc].

== Versioning

All examples in this document use `7.0.0-SNAPSHOT`. Substitute the project's
current version or `${project.version}` as appropriate.

== Installation

Every module is a plain Maven artifact under `org.smallmind`:

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>web-grizzly</artifactId>
  <version>7.0.0-SNAPSHOT</version>
</dependency>
----

The container modules (`web-grizzly`, `web-jetty`) transitively pull in
`web-jersey`, which pulls in `web-json-scaffold`. Declare only what you
actually use. Optional integrations in `web-json-query` (JPA API, QueryDSL,
`mongodb-throng`) are expressed as compile dependencies of that POM; if you do
not use a backend, exclude the optional translator class from your classpath
by keeping its runtime dependency out — the class is loaded lazily only when
its utility is called.

[[_web_grizzly]]
== web-grizzly

=== What it does

`web-grizzly` brings up one embedded Grizzly `HttpServer` and, per named
context path, registers everything that should live inside it: servlets,
filters, listeners, a Jersey JAX-RS entry point, JAX-WS SOAP endpoints, a
Tyrus WebSocket container, and one or more static-resource handlers. The
Spring container is the system of record: you declare a
`GrizzlyInitializingBean` with some configuration and let Spring's
`ContextRefreshedEvent` drive the startup.

=== Key types

`GrizzlyInitializingBean`::
    The root bean. Implements `InitializingBean`, `DisposableBean`,
    `ApplicationContextAware`, and `ApplicationListener<ContextRefreshedEvent>`.
    `afterPropertiesSet()` validates the configured
    `WebApplicationOption[]`; `onApplicationEvent(...)` constructs and starts
    the server when the Spring context finishes refreshing; `destroy()` stops
    the server and its WebSocket containers.

`WebApplicationOption`::
    A value object describing one context path. Holds a
    `ClassLoaderResourceOption`, `DocumentRootOption`, `JaxRSOption`,
    `SpringSupportOption`, and `WebSocketOption`, plus `contextPath` (default
    `/context`) and `soapPath` (default `/soap`).

`SSLInfo`::
    Aggregates the HTTPS listener configuration: port (default 443),
    `keySecureStore`, `trustSecureStore`, `requireClientAuth`, and `proxyMode`
    (trusts upstream `SSL_CLIENT_*` and `X-Forwarded-For` headers).

`@ServicePath`::
    Class-level annotation declaring a SOAP endpoint and the context it should
    live in. Detected automatically by `GrizzlyPostProcessor`.

`GrizzlyPostProcessor`::
    A Spring `BeanPostProcessor` that, for each `FilterInstaller`,
    `ServletInstaller`, `ListenerInstaller`, `WebServiceInstaller`,
    `WebSocketExtensionInstaller`, or `@ServicePath`-annotated bean, routes it
    to the `GrizzlyWebAppState` bucket for its declared context path.

`GrizzlyWebAppState` / `GrizzlyWebAppStateLocator`::
    Per-context accumulator. `GrizzlyInitializingBean` implements the locator
    interface so `GrizzlyPostProcessor` can reach the state for a given
    `contextPath`.

`ClientAuthProxyFilter`::
    A Grizzly network filter that either repopulates `SSL_CLIENT_S_DN`,
    `SSL_CLIENT_I_DN`, `SSL_CLIENT_VERIFY`, and `X-Forwarded-For` from the
    peer TLS certificate, or (when `SSLInfo.proxyMode=true`) trusts the
    upstream proxy's versions of those headers.

`GrizzlyRequestContextListener`::
    A `ServletRequestListener` that binds Spring's `RequestContextHolder` and
    `LocaleContextHolder` for each request. Registered only when the context
    sets `SpringSupportOption`.

=== Installers

All installers subclass the `GrizzlyInstaller` abstract base, which carries
the target `contextPath`. The concrete subclasses and their `getInstallerType`
return values are:

[cols="1,1,2",options="header"]
|===
| Installer | Type | Role

| `ServletInstaller`           | `SERVLET`               | Servlet class or instance, URL pattern, init parameters, load-on-startup, async flag.
| `FilterInstaller`            | `FILTER`                | Filter class or instance, URL pattern, init parameters, async flag.
| `ListenerInstaller`          | `LISTENER`              | Servlet context/request/session listener class or instance, context init parameters.
| `WebServiceInstaller`        | (not in enum)           | JAX-WS endpoint — `path`, `service` instance, `asyncSupported`.
| `WebSocketExtensionInstaller`| `WEB_SOCKET_EXTENSION`  | A Tyrus `Extension` to install on one or more endpoints.
|===

Each installer is a plain Spring bean. `GrizzlyPostProcessor` picks them up by
type.

=== Starting up

[source,xml]
----
<bean id="grizzlyServer"
      class="org.smallmind.web.grizzly.GrizzlyInitializingBean">
  <property name="host" value="0.0.0.0"/>
  <property name="port" value="8080"/>
  <property name="sslInfo" ref="sslInfo"/>                        <!--1-->
  <property name="resourceConfig">                                <!--2-->
    <bean class="org.smallmind.web.jersey.spring.JerseyPostProcessor"/>
  </property>
  <property name="webApplicationOptions">
    <list>
      <bean class="org.smallmind.web.grizzly.option.WebApplicationOption">
        <property name="contextPath" value="/"/>
        <property name="jaxRSOption">                             <!--3-->
          <bean class="org.smallmind.web.grizzly.option.JaxRSOption">
            <property name="restPath" value="/rest"/>
          </bean>
        </property>
        <property name="springSupportOption">
          <bean class="org.smallmind.web.grizzly.option.SpringSupportOption"/>
        </property>
      </bean>
    </list>
  </property>
</bean>

<bean id="grizzlyPostProcessor"
      class="org.smallmind.web.grizzly.GrizzlyPostProcessor"/>

<bean class="org.smallmind.web.grizzly.installer.FilterInstaller">
  <property name="contextPath" value="/"/>
  <property name="urlPattern" value="/*"/>
  <property name="filterClass" value="com.example.AuthFilter"/>
</bean>
----
<1> Optional. Omit to serve HTTP only. `allowInsecure=true` is the default on
    the initializing bean.
<2> The Jersey `ResourceConfig`. `JerseyPostProcessor` scans the Spring
    context for `@Path` beans and `ResourceConfigExtension` beans.
<3> Declaring `JaxRSOption` causes the Jersey servlet to be bound at
    `contextPath + restPath + "/*"`.

=== HTTPS

`SSLInfo` expects `SecureStore` values for the keystore and optional
truststore. The initializing bean converts these into a Grizzly
`SSLEngineConfigurator` and attaches an HTTP/2 add-on and a
`ClientAuthProxyFilter` to the secure listener:

[source,java]
----
SSLInfo sslInfo = new SSLInfo();
sslInfo.setPort(8443);
sslInfo.setKeySecureStore(keyStore);
sslInfo.setTrustSecureStore(trustStore);
sslInfo.setRequireClientAuth(true);
sslInfo.setProxyMode(false);
----

When `proxyMode=true`, the filter does not enforce its own certificate
extraction — it assumes an upstream proxy has already populated the
`SSL_CLIENT_*` headers and leaves them in place.

=== WebSockets

`WebSocketOption` is a marker that causes the server to build a
`TyrusGrizzlyServerContainer` per context and (optionally, when
`includeWsadlSupport=true`) mount a `WsadlHttpHandler` that serves a WSADL
descriptor. Register endpoints as ordinary Spring beans annotated with
`@ServerEndpoint` and they will be discovered automatically.

=== SOAP

Annotate a JAX-WS service class with `@ServicePath(value = "/echo",
contextPath = "/")` and the `GrizzlyPostProcessor` will wrap it in a
`WebServiceInstaller`. The endpoint is published at
`contextPath + soapPath + servicePath` (default `/soap/echo`).

[[_web_jetty]]
== web-jetty

=== What it does

`web-jetty` solves the same problem as `web-grizzly` with Eclipse Jetty 12
(EE10 variant). A single `JettyInitializingBean` builds a
`ContextHandlerCollection`, one `ServletContextHandler` per
`WebApplicationOption`, attaches filters/servlets/listeners through the
installer pattern, hooks up Jersey, publishes JAX-WS services via
`JettyHttpServer`, and configures Jakarta WebSocket via
`JakartaWebSocketServletContainerInitializer`.

=== When to choose Jetty over Grizzly

Both modules are production-ready; the choice is driven by runtime
requirements.

* Choose **Grizzly** if you want HTTP/2 out of the box and a Tyrus-based
  WebSocket stack, and you are comfortable with Grizzly's filter-chain model.
* Choose **Jetty** if you want Jetty's thread-pool model, familiar handler
  chain, or integration with tooling that expects a Jetty server (Jetty
  runner, Jetty plugins, Jetty-specific instrumentation).

Both accept the same installer POJOs with different package names.

=== Key types

`JettyInitializingBean`::
    Spring lifecycle bean. Validates `WebApplicationOption[]` in
    `afterPropertiesSet()`, builds the server in `onApplicationEvent(...)`,
    stops it in `destroy()`. Configurable surface: `host`, `port`,
    `allowInsecure`, `sslInfo`, `resourceConfig`, `webApplicationOptions`,
    `initialWorkerPoolSize`, `maximumWorkerPoolSize`, `maxHttpHeaderSize`,
    `suppressConnectionClosedException`, and `debug` (dumps SOAP traffic).

`WebApplicationOption` (`org.smallmind.web.jetty.option`)::
    Mirrors the Grizzly version. Holds `ClassLoaderResourceOption`,
    `DocumentRootOption`, `JaxRSOption`, `SpringSupportOption`, and
    `WebSocketOption`, plus `contextPath` and `soapPath`.

`WebSocketOption`::
    `maxSessionIdleTimeout` in ms, default five minutes.

`SSLInfo` + `SSLStore`::
    Jetty's `SSLStore` references a resource (`classpath:keystore.p12`,
    `file:/etc/ssl/…`) and a password. `SSLStore.getBytes()` reads the raw
    bytes which `JettyInitializingBean` wraps in `ByteArrayResource` (an
    in-memory `org.eclipse.jetty.util.resource.Resource` implementation) and
    hands to `SslContextFactory.Server`.

`JettyPostProcessor` + `JettyWebAppState` + `JettyWebAppStateLocator`::
    Same split-by-context-path pattern as the Grizzly version.

`@ServicePath(value, context)`::
    Marks a SOAP service bean; `JettyPostProcessor` routes it to the correct
    context.

`JettyRequestContextListener`::
    Binds Spring's request context and locale for each request when
    `SpringSupportOption` is set.

`ByteArrayResource`::
    Minimal Jetty `Resource` backed by a byte array; used for in-memory
    keystore/truststore bytes.

=== Installers

[cols="1,1,2",options="header"]
|===
| Installer | Type | Role

| `ServletInstaller`    | `SERVLET`  | Same payload as Grizzly's version.
| `FilterInstaller`     | `FILTER`   | Includes `matchAfter` flag.
| `ListenerInstaller`   | `LISTENER` | Servlet event listener plus optional context init parameters.
| `WebServiceInstaller` | (not in enum) | `(path, service)` constructor; optional `asyncSupported`.
|===

=== Wiring sketch

[source,xml]
----
<bean id="jettyServer"
      class="org.smallmind.web.jetty.JettyInitializingBean">

  <property name="allowInsecure" value="true"/>
  <property name="sslInfo">                                      <!--1-->
    <bean class="org.smallmind.web.jetty.SSLInfo">
      <property name="port" value="8443"/>

          <property name="resource" value="classpath:server.p12"/>

=== Session handling

The Jetty module currently creates each `ServletContextHandler` with
`NO_SESSIONS`. Request-scoped state is expected to come from Spring's
`RequestContextHolder` via `SpringSupportOption`, not from HTTP sessions.

[[_web_jersey]]
== web-jersey

=== What it does

`web-jersey` is the Jersey 3 layer's batteries-included assembly. The core
building blocks are:

* `JerseyPostProcessor` — a `ResourceConfig` that is also a Spring
  `BeanPostProcessor`. It self-registers `@Path`-annotated Spring beans, then
  applies every `ResourceConfigExtension` bean in the context.
* `ResourceConfigExtension` (abstract) — the SPI that every optional feature
  uses to plug itself in.
* A set of concrete extensions that register JSON serialisation, exception
  mapping, CORS, multipart, paging, entity-parameter injection, and JAXB
  parameter conversion.

=== Package tour

`org.smallmind.web.jersey.spring`::
    `JerseyPostProcessor`, `ResourceConfigExtension`,
    `PrioritizedResourceConfigExtension`, `ExposedApplicationContext` (static
    hook so Jersey code can reach the Spring context).

`org.smallmind.web.jersey.json`::
    `JsonProvider` (a `MessageBodyReader` + `MessageBodyWriter` for
    `application/json` at `@Priority(1)`, delegating to
    `JsonCodec` from `web-json-scaffold`), `ThrowableExceptionMapper` (catch-all
    mapper that tries a list of `ConcreteExceptionMapper` delegates then falls
    back to a JSON `Fault` with HTTP 500), `XmlAdapterParamConverterProvider`
    (makes JAXB adapters work for path and query parameters), and a
    `FaultTranslatingClientResponseFilter` for the client side. Each has a
    sibling `…Extension` class that registers it.

`org.smallmind.web.jersey.cors`::
    `CorsFilter` — a `@PreMatching` container request/response filter that
    handles OPTIONS preflight and stamps CORS headers on every response. The
    `CorsExtension` Spring bean wires it with configured allowed/exposed
    header lists.

`org.smallmind.web.jersey.multipart`::
    `MultiPartFile` (filename, content type, stream, `readAllBytes()`),
    `MultiPartFileIterator`, `MultiPartFileException`,
    `MultiPartContentType`, `MultiPartFeatureExtension` (registers Jersey's
    multipart feature).

`org.smallmind.web.jersey.page`::
    `@PageRange` name-binding annotation + `PageRangeResponseFilter`. The
    filter recognises `org.smallmind.web.json.scaffold.util.Page` return
    values and sets `Content-Range` plus HTTP 200/206/416 appropriately.

`org.smallmind.web.jersey.aop`::
    The `@ResourceMethod` / `@EntityParam` / `@Validated` aspect layer. See
    _Entity parameter injection_ below.

`org.smallmind.web.jersey.proxy`::
    `JsonTarget` (fluent blocking JSON client), `JsonTargetFactory`,
    `JsonEntityResourceProxyFactory` (generates a dynamic proxy that turns
    interface calls into JSON-over-HTTP), `JsonHeaderInjector`.

`org.smallmind.web.jersey.ssl`::
    `ClientUtility.clientBuilder()` returns a Jersey `ClientBuilder` with a
    permissive, all-accepting `SSLContext` and hostname verifier — intended
    for development and test scenarios.

=== JSON serialisation

`JsonProvider` is `@Priority(1)` so it wins over any default providers.
Serialisation and deserialisation go through `JsonCodec`, which uses
Jackson's Jakarta XML-Bind annotation introspector and the scaffold module's
`PolymorphicModule`. A resource method can return any JAXB-annotated POJO or
`web-json-doppelganger`-generated view and it round-trips as JSON without
further configuration.

=== Exception handling

`ThrowableExceptionMapper` is the project's conventional catch-all. It takes
a varargs of `ConcreteExceptionMapper` delegates and:

. returns `WebApplicationException` responses unchanged,
. tries each delegate (first match wins),
. otherwise emits a JSON-serialised `Fault` with HTTP 500.

Register it through `ThrowableExceptionExtension` or by adding a
`ThrowableExceptionMapper(…)` instance to your `ResourceConfig`.

=== CORS

[source,xml]
----
<bean class="org.smallmind.web.jersey.cors.spring.CorsExtension">
  <constructor-arg value="Content-Type, Authorization, X-Request-Id"/>
  <constructor-arg value="Location, X-Trace-Id"/>
</bean>
----

`CorsFilter` sets `Access-Control-Allow-Origin`, `Access-Control-Allow-Headers`,
`Access-Control-Expose-Headers`, `Access-Control-Allow-Methods`, and
`Access-Control-Allow-Credentials`, and short-circuits OPTIONS preflight
requests with a 200 response carrying the appropriate headers.

=== Paging

Annotate a resource method with `@PageRange` and return
`org.smallmind.web.json.scaffold.util.Page<T>`:

[source,java]
----
@GET
@Path("/users")
@PageRange
public Page<UserOut> listUsers(@QueryParam("firstResult") long firstResult,
                               @QueryParam("maxResults") int maxResults) {
  return userService.list(firstResult, maxResults);
}
----

`PageRangeResponseFilter` inspects the returned `Page` and stamps a
`Content-Range: items firstResult-last/totalResults` header, converts the
body to the raw values array, and picks status 206 when the response covers a
proper subrange, 200 when it is complete, and 416 when the requested range
is out of bounds.

=== Entity parameter injection

`@ResourceMethod` (with optional `validate=true`) and `@EntityParam` let a
JAX-RS method declare its parameters individually while still accepting a
single JSON object as the request body:

[source,java]
----
@POST
@Path("/users")
@ResourceMethod(value = UserCreate.class, validate = true)
public UserOut create(@EntityParam("username") String username,
                      @EntityParam("password") String password) {
  ...
}
----

The `ResourceMethodFilter` JAX-RS request filter deserialises the body into a
`JsonEntity`, stashes it in a thread-local, and
`EntityParamResolver.EntityParamValueParamProvider` extracts each named
parameter from the entity when Jersey builds the argument array. When
`validate=true`, `ResourceMethodAspect` (AspectJ) runs Jakarta Bean Validation
on parameters and return value, throwing `EntityValidationException` on a
failed constraint.

`@Validated` applies the same parameter/return-value validation without
requiring `@ResourceMethod`.

=== Client proxies

`JsonTarget` is a small fluent builder over Apache HttpComponents, pinned to a
single host:

[source,java]
----
JsonTarget target = JsonTargetFactory.manufacture(
  HttpProtocol.HTTPS, "api.example.com", 443, "/");
UserOut user = target.path("v1/users/{id}").pathParameter("id", 42).get(UserOut.class);
----

`JsonEntityResourceProxyFactory.generateProxy(...)` creates a `java.lang.reflect.Proxy`
for a service interface. Each method call is encoded as a JSON `Envelope`
(named `Argument[]`) and POSTed to a URL derived from
`{versionPrefix}/{serviceVersion}/{serviceName}/{methodName}`. Header
injectors (`JsonHeaderInjector`) can add per-request headers such as
authorization tokens. The Spring factory beans
`JsonEntityResourceProxyFactoryBean` and `JsonTargetFactoryBean` make the
same proxies declarable in XML.

[[_web_http]]
== web-http

=== What it does

`web-http` is a tiny companion library exposing a shared, asynchronous,
pooled Apache HttpComponents 5 client with a blocking-friendly API. The whole
module is four classes under `org.smallmind.web.http.apache`.

=== Key types

`HttpClient`::
    Singleton holder of a `CloseableHttpAsyncClient` with a
    `PoolingAsyncClientConnectionManager` sized at 300 connections per route
    and 300 total, `IOReactorConfig` with a 5-second socket timeout,
    `HttpVersionPolicy.FORCE_HTTP_1`, and a permissive `SSLContext` that
    trusts any certificate. The client is built in a static initializer;
    construction failure surfaces as a `StaticInitializationError`.
    The public surface is a single static method:
+
[source,java]
----
HttpClient.execute(SimpleHttpRequest request,
                   HttpCallback<SimpleHttpResponse> callback,
                   int timeoutSeconds);
----

`HttpCallback<V>`::
    Abstract `FutureCallback` subclass that uses a `CountDownLatch` so a
    caller can `await(timeout, TimeUnit)` for the callback (not just the
    future) to finish. Subclasses provide `onCompleted`, `onFailed`, and
    `onCancelled`.

`SimpleCallback`::
    Concrete default callback that stashes the response or exception in
    `AtomicReference`s. After `execute(...)` returns, call `getResponse()` to
    retrieve the response or rethrow the captured exception.

`HttpRequestCancelledException`::
    Thrown by `SimpleCallback.onCancelled()` so callers can distinguish a
    cancellation from a timeout or network error.

=== Usage

[source,java]
----
SimpleHttpRequest request = SimpleHttpRequest.create(
  "GET", URI.create("https://example.com/health"));

SimpleCallback callback = new SimpleCallback();
HttpClient.execute(request, callback, 10);
SimpleHttpResponse response = callback.getResponse();
----

=== When to reach for it vs. `JsonTarget`

`JsonTarget` (in `web-jersey`) is a JAX-RS-style builder that understands
JSON serialisation. `web-http` is the raw transport — use it when you need to
talk to a non-JSON service, when you want direct control over the request,
or when you need the shared connection pool from multiple modules.

=== Tuning

The pool size, socket timeout, and permissive TLS trust are hard-coded. If
you need different defaults, fork `HttpClient`; the module is deliberately
small.

[[_web_json_scaffold]]
== web-json-scaffold

=== What it does

`web-json-scaffold` is the JSON/XML foundation the rest of the `web-*`
modules link against. It owns the configured `ObjectMapper`, the project's
transportable `Fault` model, a polymorphic Jackson module, `Page<T>`, a
bean-reflection helper, and a small versioning SPI.

=== Key types

`JsonCodec` (in `org.smallmind.web.json.scaffold.util`)::
    Static facade over the project `ObjectMapper`. The mapper is configured
    with `JakartaXmlBindAnnotationModule` (respects `@XmlElement`,
    `@XmlRootElement`, `@XmlJavaTypeAdapter`), `PolymorphicModule`, and
    `MapperFeature.USE_WRAPPER_NAME_AS_PROPERTY_NAME`.
    Primary methods:
+
* `readAsJsonNode(byte[]|String|InputStream)`
* `readAsObject(source, Class<T>)`
* `writeAsBytes(Object)`
* `jsonConvert(Object, Class<T>)` — round-trip re-bind via Jackson.

`Fault` + `FaultElement` + `Informed`::
    `Fault` is the transportable shape of a server-side exception. Built from
    a `Throwable`, it captures the throwable type, message, `FaultElement[]`
    stack frames, and a nested cause `Fault`. `Informed.getInformation()`
    exposes an `ObjectNode` for free-form context. This is the type
    `ThrowableExceptionMapper` produces on unhandled exceptions.

`PolymorphicModule` + `@XmlPolymorphicSubClasses`::
    A Jackson module that reads a class-level annotation listing the known
    subtypes and wires a discriminator-based deserializer. Paired with
    `PolymorphicXmlAdapter` / `AttributedPolymorphicXmlAdapter` for the JAXB
    side.

`Page<T>`::
    The project's paging primitive. Holds `values`, `firstResult`,
    `maxResults`, `resultSize`, `totalResults`. Methods: `iterator()`,
    `mutate(Class<U>, Mutation<T,U>)` to transform elements,
    `jsonConvert(Class<T>)` to re-bind via Jackson, and a static
    `Page.empty(componentClass)`.

`BeanReflector`::
    Walks dotted paths with array subscripts (`a.b[0].c`) to `get` or `set`
    nested bean properties. Uses `ComponentParser` + `PathComponent`
    internally.

`Version` / `Versioned<V>`::
    A two-interface SPI for DTOs that declare a schema version via an enum
    — the interfaces are opt-in and carry no behaviour other than the type
    marker.

=== Fault subtypes

`NativeObject`, `NativeLanguage`, `NativeObjectException`::
    Represent throwables raised in a non-Java context (for example, a
    database procedure) so the original diagnostic is preserved in the JSON
    payload.

`FaultWrappingException`, `ResourceInvocationException`,
`ObjectInstantiationException`, `FormatteJacksonException`::
    Exception types used across the codebase to tag specific failure
    categories.

=== Usage

[source,java]
----
User user = JsonCodec.readAsObject(requestBytes, User.class);
byte[] payload = JsonCodec.writeAsBytes(new Page<>(users, 0, 20, 135L));
String city = BeanReflector.get(order, "billingAddress.city");
BeanReflector.set(order, "items[2].quantity", 5);
----

[[_web_json_query]]
== web-json-query

=== What it does

`web-json-query` defines a compact predicate/sort/pagination vocabulary that
serialises to JSON and XML. The same query object, after deserialisation on
the server, can be translated into a JPA `Predicate`, a QueryDSL `Predicate`,
or a MongoDB criterion document. The module lives under
`org.smallmind.web.json.query`.

=== The predicate tree

`Where`::
    Root container. Holds a single `WhereConjunction` (`getRootConjunction()`)
    and exposes `getTargetSet()` to walk all referenced field names.

`WhereCriterion` (abstract) → `WhereConjunction` | `WhereField`::
    Polymorphic tree.

`WhereConjunction` (abstract) → `AndWhereConjunction` | `OrWhereConjunction`::
    Logical group of criteria. Holds `List<WhereCriterion>`; `size()`,
    `isEmpty()`, `addCriterion(...)`.

`WhereField(name, operator, operand)` or
`WhereField(entity, name, operator, operand)`::
    Leaf comparison. `entity` names the alias for scope-qualified references.

`WhereOperator`::
    `LT`, `LE`, `EQ`, `NE`, `GE`, `GT`, `EXISTS`, `LIKE`, `UNLIKE`, `IN`.
    Includes an `isTrue(op1, op2)` method used for in-memory evaluation.
    `LIKE` accepts leading and/or trailing `*` wildcards.

`WhereOperand<I>` (abstract)::
    Typed right-hand side of a comparison. Concrete subclasses cover every
    primitive-like type (`String`, `Integer`, `Long`, `Double`, `Float`,
    `Byte`, `Short`, `Boolean`, `Character`, `Date` (`LocalDateTime`), `Enum`,
    `Array`, `Null`). Discriminated on the wire via
    `WhereOperandPolymorphicXmlAdapter`.

`Sort` + `SortField` + `SortDirection` (`ASC`, `DESC`)::
    Ordered list of field sorts, same naming model as `WhereField`.
    `SortDirection` and its companion `SortDirectionEnumXmlAdapter` live
    in `org.smallmind.nutsnbolts.json` so the enum and its lowercase JAXB
    marshaling can be shared across query and persistence layers.

`WhereVisitor`, `WhereUtility.walk(visitor, where)`::
    Visitor infrastructure for tree traversal.

`WhereValidator` + `WherePermit` implementations (`AllowedWherePermit`,
`RequiredWherePermit`, `ExcludedWherePermit`, `DependentWherePermit`,
`TargetWherePermit`)::
    Validate a `Where` against a field whitelist/blacklist — the server can
    reject queries that reference unsupported or sensitive fields.

=== Backend translators

Each translator is in its own sub-package so the backend dependency stays
optional.

[cols="1,2",options="header"]
|===
| Sub-package | Translator

| `…query.jpa`
| `JPAQueryUtility.apply(CriteriaBuilder, Where, WhereFieldTransformer,
  allowNonTerminalWildcards)` → a `Product<Root<?>, Predicate>` wrapping the
  JPA roots and a combined `Predicate`. `applySort(CriteriaBuilder, Sort,
  WhereFieldTransformer)` returns `Order[]`.

| `…query.querydsl`
| `QueryDslQueryUtility.apply(WhereFieldTransformer, Where)` → a QueryDSL
  `Predicate`.

| `…query.throng`
| `ThrongQueryUtility.apply(WhereFieldTransformer, Where)` → a
  mongodb-throng criterion.
|===

The `WhereFieldTransformer` supplied by the caller resolves an
`(entityName, fieldName)` pair to the backend-specific path. For JPA that is
a `Path<?>` into a `Root`; for QueryDSL, an `Expression`; for Throng, a field
key.

=== Usage

[source,java]
----
Where where = Where.instance(
  new AndWhereConjunction(
    new WhereField("age", WhereOperator.GE, new IntegerWhereOperand(18)),
    new WhereField("status", WhereOperator.EQ, new StringWhereOperand("active"))
  )
);

Sort sort = Sort.instance(
  SortField.instance("createdAt", SortDirection.DESC));
----

On the server, apply it to a JPA query:

[source,java]
----
CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Customer> cq = cb.createQuery(Customer.class);
Root<Customer> root = cq.from(Customer.class);
Product<Root<?>, Predicate> p = JPAQueryUtility.apply(
  cb, where,
  (entity, field) -> root.get(field),
  true);
cq.where(p.getProduct());
cq.orderBy(JPAQueryUtility.applySort(cb, sort,
  (entity, field) -> root.get(field)));
----

=== UI metadata

`Hint`, `HintType`, `EnumHint`, `ComponentType`, `ComponentHint`,
and their polymorphic XML adapters let a server publish type metadata (for
example, "this field is an enum; here are its choices") alongside a schema so
a client form generator can render the right widget.

[[_web_jwt]]
== web-jwt

=== What it does

`web-jwt` is a thin, typed wrapper over
https://bitbucket.org/b_c/jose4j[jose4j] focused on the JWS case:
sign compact-serialised JWTs, decode and verify them, and (when needed)
unwrap nested JWE/JWS structures.

=== Key types

`JWTCodec`::
    Static encode/decode façade.
+
* `encode(Object claims, JWTKeyMaster key)` → `String`
* `encode(Object claims, JWTKeyMaster key, String keyId)` → `String`
* `decode(String token, JWTKeyMaster key, Class<T> claimsClass)` → `T`
* `decipher(String token, Class<T> claimsClass)` → `T` (no signature check —
  for inspection and tests only).

`JWTKeyMaster`::
    Pair of `(JWTEncryptionAlgorithm, java.security.Key)`.

`JWTEncryptionAlgorithm`::
    Enum of supported signing algorithms. Values: `HS256` (HMAC-SHA-256),
    `RS256` (RSA with SHA-256), `EDDSA` (Ed25519). `getCode()` returns the
    JWA identifier; `sign(PrivateKey, String)` and
    `verify(PublicKey, String[], boolean)` delegate to the signing algorithm
    implementations in `smallmind-nutsnbolts`.

`SymmetricJWTKeyMaster(String secret)`::
    Builds an HMAC-SHA-256 key from a UTF-8 secret.

`AsymmetricJWTKeyMaster(Key key)` throws `UnknownAlgorithmException`::
    Inspects `key.getAlgorithm()` and maps to `RS256` or `EDDSA`; throws
    `UnknownAlgorithmException` for anything else.

`JWTToken`::
    Optional interface; if your claims class implements it, `getSub()` and
    `getExp()` become part of the contract.

=== Reading tokens

`JWTConsumer` (in `org.smallmind.web.jwt.jose4j`) is the configurable reader:

[source,java]
----
JWTConsumer consumer = new JWTConsumer()
  .setRequireSignature(true)
  .setRequireEncryption(false)
  .setRequireIntegrity(false);

MyClaims claims = consumer.process(token, verificationKey, MyClaims.class);
----

It unwraps nested JOSE layers (`cty: "jwt"` content-types), verifies and/or
decrypts each, and finally deserialises the plaintext JSON to the claim type
via `JsonCodec`. Policy violations surface as `InvalidJWTException`; signature
failures as `InvalidJWTSignatureException`.

=== Caveats

* Expiration is not checked automatically. If your `JWTToken.getExp()` should
  enforce expiry, do the comparison yourself.
* There is no JWK machinery. Keys come in as `java.security.Key` instances —
  whatever the caller has loaded from a keystore, PEM, or config.
* `JWTCodec.encode` always produces a JWS (signed token). JWE is supported on
  the read side by jose4j when `JWTConsumer` is used directly.

[[_schema_validation_maven_plugin]]
== schema-validation-maven-plugin

=== What it does

The module in `web/schema` ships a Maven plugin — `schema-validation-maven-plugin`
— that validates XML Schema Definition (XSD) files and XML instance documents
at build time. It has no runtime artifact and is unrelated to JSON Schema or
to the rest of the `web-*` stack; it lives here because XSDs are frequently
shipped alongside REST resources and the same projects tend to use them.

=== The mojo

`XMLValidationMojo` (`@Mojo(name = "validate-xml", defaultPhase = VALIDATE)`)::
    Validates every configured XSD against the W3C XML Schema meta-schema,
    then validates every declared XML instance against its owning XSD. A
    single SAX `ValidationErrorHandler` rethrows fatal errors, errors, and
    warnings so the build fails on the first diagnostic.

`XSD`::
    Configuration POJO. `path` points to an XSD in the project resources;
    `impls` is the list of XML files to validate against it.

=== Usage

[source,xml]
----
<plugin>
  <groupId>org.smallmind</groupId>
  <artifactId>schema-validation-maven-plugin</artifactId>
  <version>7.0.0-SNAPSHOT</version>
  <executions>
    <execution>
      <goals>
        <goal>validate-xml</goal>
      </goals>
      <configuration>
        <w3c>xsd/XMLSchema.xsd</w3c>                <!--1-->
        <schemas>
          <schema>
            <path>xsd/orders.xsd</path>
            <impls>
              <impl>samples/orders-valid.xml</impl>
              <impl>samples/orders-partial.xml</impl>
            </impls>
          </schema>
        </schemas>
      </configuration>
    </execution>
  </executions>
</plugin>
----
<1> Path to a W3C XML Schema meta-schema file in the project's resource
    directories.

[[_doppelganger]]
== Doppelganger

The `web-json-doppelganger` module is a compile-time annotation processor for
generating directional, purpose-aware view classes from an annotated domain
entity. Because it has its own conceptual vocabulary (purposes, idioms,
visibilities, pledges, polymorphic adapters, real vs. virtual properties)
and its own usage model (the processor runs at javac time and writes
sources into `target/generated-sources/annotations`), its reference is a
separate document:

xref:DOPPELGANGER.adoc[DOPPELGANGER.adoc]

A typical service that builds its API contract with Doppelganger depends on
`web-json-doppelganger` at compile time and on `web-json-scaffold` and
`web-jersey` at runtime. The generated views are plain Jackson/JAXB POJOs
and need no other runtime plumbing.

== Troubleshooting

Jersey doesn't see my `@Path` beans::
    Make sure a `JerseyPostProcessor` bean exists in the same Spring context
    and that the container's `resourceConfig` property refers to it. The
    post-processor both collects `@Path` beans and applies
    `ResourceConfigExtension` beans; without it, Jersey starts with an empty
    `ResourceConfig`.

CORS preflight still fails::
    Confirm `CorsExtension` is registered and that the filter's allowed
    headers include the headers your client actually sends (typically
    `Content-Type`, `Authorization`). `@PreMatching` ensures the filter runs
    before resource method matching.

`@PageRange` returns the wrong status code::
    `PageRangeResponseFilter` picks 206 when the response covers a proper
    subrange, 200 when it is complete, and 416 when the requested range is
    out of bounds. Check that the `Page` you return has the correct
    `firstResult`, `resultSize`, and `totalResults` values.

Grizzly or Jetty starts but nothing answers on the port::
    Both initializing beans defer actual startup to
    `ContextRefreshedEvent`. If the Spring context is not refreshed (for
    example, in an embedded test that uses a plain `BeanFactory`), the
    server is never built. Use `ClassPathXmlApplicationContext` or an
    equivalent full application context.

`AsymmetricJWTKeyMaster` rejects my key::
    Only `RSA` and `Ed25519` keys are recognised. Other key types raise
    `UnknownAlgorithmException`. Convert or wrap the key appropriately
    before constructing the key master.

JPA query from `web-json-query` throws on non-terminal wildcards::
    Pass `allowNonTerminalWildcards=true` to `JPAQueryUtility.apply(...)` if
    your schema permits `*foo*` patterns. The default is restrictive to
    avoid accidental full-scan queries.

`HttpClient.execute` hangs past the timeout::
    The socket timeout is 5 seconds (fixed), but `execute(...)` waits for
    both the future and the callback. If a custom subclass of
    `HttpCallback` blocks inside `onCompleted`, `await` will not return
    until it finishes. Keep callback implementations cheap.
