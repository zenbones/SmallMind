[[spark, Spark]]
= Spark

[partintro]
Spark is Smallmind's build-time packaging module for Java services. It gives you two focused delivery models:

* `singularity`, a single executable jar that keeps runtime dependencies nested as jars
* `tanukisoft`, a Tanuki Java Service Wrapper distribution with launch scripts, native wrapper binaries, configuration, and an optional publishable application archive

Spark stays library-safe. It keeps IOC choice, runtime configuration, and process model aligned with the package format you select.

Audience: developers and build engineers who package Java services and want explicit control over artifact shape, startup wiring, and publication.
Versioning: examples use `${smallmind.version}` for Smallmind artifacts; keep related Smallmind modules on the same version unless you intentionally need to split them.

[[spark-why-use, Why Use Spark]]
== Why Use Spark

Use Spark when you want explicit Maven packaging for service artifacts without handing your runtime model over to a framework packager.

Spark is most useful when these constraints matter:

* you want build-time packaging with explicit runtime bootstrapping
* you need either a single executable jar or a wrapper-managed service distribution
* you want artifact layout and startup wiring to stay visible in Maven configuration
* you want to preserve dependency boundaries instead of shading everything into one flat namespace
* you need installable or publishable artifacts without inventing a parallel release process

Compared with framework-owned packaging, Spark keeps these advantages visible:

* packaging lives in normal Maven `package`, `install`, and `deploy` phases
* main-class, listener, and application wiring stay explicit in plugin configuration
* packaging format can change without reshaping the rest of the application architecture
* runtime ownership stays with the application and the selected package format

[[spark-choose-packaging, Choosing Singularity Or Tanukisoft]]
== Choosing Singularity Or Tanukisoft

[cols="2,3,3",options="header"]
|===
|Topic |`singularity` |`tanukisoft`

|Primary output
|single executable jar
|application directory plus optional `-app` archive

|How you start it
|`java -jar ...`
|wrapper launcher script or executable in `bin/`

|Dependency layout
|nested jars inside one jar
|plain jars in `lib/`

|Best fit
|plain JVM process deployment, artifact repositories, simple launch surfaces
|OS service style lifecycle, wrapper-native monitoring, service install and removal commands

|Publication unit
|the generated jar
|the generated `-app` archive

|Operational tradeoff
|simpler artifact, custom class loader requirements
|larger distribution, simpler flat `lib/` layout plus wrapper scripts and native binaries
|===

As a practical rule:

* choose `singularity` when you want one file and your operations model is already comfortable with `java -jar`
* choose `tanukisoft` when you need the wrapper's service lifecycle, native integration, or script-driven installation

[[spark-packaging-model, Maven Packaging Model]]
== Maven Packaging Model

Both Spark formats are Maven packaging extensions. The packaging type changes the lifecycle bindings for `package`, `install`, and `deploy`.

[cols="2,4",options="header"]
|===
|Packaging |Lifecycle Mapping

|`singularity`
|`package` runs `generate-singularity`; `install` runs `install-singularity`; `deploy` runs `deploy-singularity`

|`tanukisoft`
|`package` runs `maven-jar-plugin:jar` and then `generate-wrapper`; `install` runs `install-wrapper`; `deploy` runs `deploy-wrapper`
|===

Shared behavior:

* normal Maven resource processing, compilation, and test phases still run before packaging
* you must declare the Spark plugin with `<extensions>true</extensions>` so Maven recognizes the custom packaging type
* install and deploy goals publish files that the package phase already created; they do not rebuild them independently

[IMPORTANT]
====
The intended model is `packaging=singularity` or `packaging=tanukisoft` with plugin extensions enabled. The individual goals can still be invoked explicitly, but that is not the primary usage pattern.
====

[[spark-singularity, Singularity]]
== Singularity

Singularity builds one executable jar for your service. Your compiled classes and resources are copied into the jar root, runtime dependencies are copied as nested jars, and a Smallmind boot layer launches your actual main class through a custom class loader.

That model is a good fit when:

* you want one distributable file
* you want to preserve dependency jars as jars instead of flattening them
* you can run the service as a plain JVM process
* you are willing to let the packaged application load nested content through Spark's class loader rather than the system class loader

[[spark-singularity-install, Singularity Installation]]
=== Installation

A minimal Singularity build needs three things:

* `<packaging>singularity</packaging>`
* the `spark-singularity-maven-plugin` declared as an extension
* `mainClass`, the application entry point Spark should invoke at runtime

[source,xml]
----
<project>
  <modelVersion>4.0.0</modelVersion>

  <groupId>com.mycompany</groupId>
  <artifactId>orders-service</artifactId>
  <version>1.0.0</version>
  <packaging>singularity</packaging>

  <build>
    <plugins>
      <plugin>
        <groupId>org.smallmind</groupId>
        <artifactId>spark-singularity-maven-plugin</artifactId>
        <version>${smallmind.version}</version>
        <extensions>true</extensions>
        <configuration>
          <mainClass>com.mycompany.orders.Main</mainClass>
        </configuration>
      </plugin>
    </plugins>
  </build>
</project>
----

That is enough for a basic executable jar.

[[spark-singularity-quick-start, Singularity Quick Start]]
=== Quick Start

1. Declare `packaging` as `singularity` and configure `mainClass`.
2. Build the project.
3. Run the generated jar exactly as you would any ordinary executable jar.

[source,text]
----
mvn clean package
java -jar target/orders-service-1.0.0.jar
----

The generated jar contains all of these pieces:

[cols="2,4",options="header"]
|===
|Path or Manifest Entry |Meaning

|`Main-Class=org.smallmind.spark.singularity.boot.SingularityEntryPoint`
|The actual JVM entry point that installs the Spark class loader and delegates to your application

|`Singularity-Class=com.mycompany.orders.Main`
|The application main class you configured in Maven

|`META-INF/singularity/lib/*.jar`
|Runtime dependencies copied intact as nested jars

|`META-INF/singularity/index/singularity.idx`
|Serialized lookup index used by `SingularityClassLoader`
|===

[[spark-singularity-examples, Singularity Examples]]
=== Practical Examples

==== Excluding A Container-Provided Dependency

Use `exclusions` when one runtime dependency should not be nested into the final jar.

[source,xml]
----
<plugin>
  <groupId>org.smallmind</groupId>
  <artifactId>spark-singularity-maven-plugin</artifactId>
  <version>${smallmind.version}</version>
  <extensions>true</extensions>
  <configuration>
    <mainClass>com.mycompany.orders.Main</mainClass>
    <exclusions>
      <exclusion>
        <groupId>org.slf4j</groupId>
        <artifactId>slf4j-simple</artifactId>
      </exclusion>
    </exclusions>
  </configuration>
</plugin>
----

Matches are exact `groupId` and `artifactId` pairs. There is no wildcard, scope rule, or pattern language.

==== Loading Resources Correctly From A Singularity Bundle

Nested dependency jars are visible through Spark's class loader, not through APIs that assume a flat file-based classpath. Use the thread context class loader for runtime discovery.

[source,java]
----
ClassLoader classLoader = Thread.currentThread().getContextClassLoader();

try (InputStream inputStream = classLoader.getResourceAsStream("banner.txt")) {
  if (inputStream == null) {
    throw new IllegalStateException("Missing banner.txt");
  }
}

ServiceLoader<MyPlugin> plugins = ServiceLoader.load(MyPlugin.class, classLoader);
----

Do not assume `ClassLoader.getSystemClassLoader()` or `ClassLoader.getSystemResourceAsStream(...)` can resolve nested Singularity content.

==== Changing The Temporary Assembly Directory

If you need a different staging directory under `target/`, set `singularityBuildDir`.

[source,xml]
----
<configuration>
  <mainClass>com.mycompany.orders.Main</mainClass>
  <singularityBuildDir>packaging/singularity</singularityBuildDir>
</configuration>
----

Spark still writes the final jar to `${project.build.directory}`.

[[spark-singularity-runtime, Singularity Runtime Model]]
=== Runtime Model

At package time, `generate-singularity` does the following:

1. creates a temporary assembly directory under `${project.build.directory}/${singularityBuildDir}`
2. copies Spark boot classes from `spark-singularity-boot` into the jar root
3. copies every non-excluded runtime artifact into `META-INF/singularity/lib/`
4. copies your compiled classes and resources from `${project.build.directory}/classes` into the jar root
5. builds a `SingularityIndex` that maps direct files and nested jar entries
6. writes the final jar manifest and compresses the assembled directory into `target/<artifactId>-<version>[-<classifier>].jar`

At runtime, `SingularityEntryPoint` does the following:

1. opens the enclosing jar and reads the manifest
2. creates a `SingularityClassLoader`
3. installs that loader as the thread context class loader
4. reads `Singularity-Class` from the manifest
5. reflectively invokes your `main(String[])`

`SingularityClassLoader` resolves two categories of content:

* application classes and resources copied into the jar root, exposed as ordinary `jar:` URLs
* nested dependency entries indexed from `META-INF/singularity/lib/*.jar`, exposed through Spark's `singularity:` URL scheme

[IMPORTANT]
====
If your code does runtime discovery of classes or resources, use `Thread.currentThread().getContextClassLoader()`. That is the loader Spark prepares for nested-jar access.
====

[NOTE]
====
`SingularityClassLoader` deliberately refuses to define classes in `jakarta.xml.*`, `org.xml.*`, and `org.w3c.*`, except `jakarta.xml.bind.*`. Those namespaces fall back to parent or system loading.
====

[[spark-singularity-output, Singularity Output Layout]]
=== Output Layout

A typical Singularity jar looks like this:

[source,text]
----
target/orders-service-1.0.0.jar
  org/smallmind/spark/singularity/boot/...
  com/mycompany/orders/...
  META-INF/MANIFEST.MF
  META-INF/singularity/index/singularity.idx
  META-INF/singularity/lib/<runtime-dependency>.jar
----

The nested dependency jars remain jars. Spark does not unpack them into one flattened namespace.

[[spark-singularity-goals, Singularity Goals]]
=== Goals

==== `generate-singularity` (default phase: `package`)

Builds the executable Singularity jar.

Behavior:

* requires runtime dependency resolution
* copies boot classes, runtime jars, and compiled output into the staging directory
* writes the serialized Singularity index
* writes the manifest with `Main-Class` and `Singularity-Class`
* produces `target/<artifactId>-<version>[-<classifier>].jar`

==== `install-singularity` (default phase: `install`)

Installs the generated jar into the local Maven repository.

Behavior:

* looks for the jar in `target/<artifactId>-<version>[-<classifier>].jar`
* preserves an existing classifier on the installed artifact
* if `target/<jar>.asc` exists beside the jar, attaches it as artifact metadata and installs it too

==== `deploy-singularity` (lifecycle-bound to `deploy`)

Deploys the generated jar to the configured distribution repository.

Behavior:

* uses the same file naming and classifier rules as `install-singularity`
* deploys optional `.asc` metadata if present beside the jar
* defaults `deploymentRepository` from Maven distribution management

=== Operational Notes

* `exclusions` removes the entire artifact from the nested dependency set. Spark does not partially filter entries from a jar.
* `install-singularity` and `deploy-singularity` assume the package phase already produced the jar in the standard `target/` location.
* If the project artifact has a classifier, Spark carries that classifier into the jar name and published coordinates.
* Spark installs and deploys the final jar file, not the temporary assembly directory.

=== Troubleshooting

* `Unable to locate the boot class dependencies`
Ensure `spark-singularity-maven-plugin` is declared with `<extensions>true</extensions>` so Maven loads the custom packaging extension and plugin artifacts.

* `Missing singularity index`
The packaged jar is incomplete or corrupted. Rebuild from a clean `target/` directory.

* The application starts but cannot load a resource or SPI implementation
Use the thread context class loader and confirm you did not exclude the dependency that contains the missing resource.

* Install or deploy cannot find the jar you expected
Run the `package` phase first and confirm the output file name still matches `${artifactId}-${version}[-${classifier}].jar`.

[[spark-tanukisoft, Tanukisoft]]
== Tanukisoft

Tanukisoft packaging builds a wrapper-managed application distribution. Instead of one nested executable jar, you get an application directory with scripts, native wrapper binaries, configuration, and jars in `lib/`.

That model is a good fit when:

* you need wrapper-managed start, stop, restart, install, and remove commands
* you want a conventional `bin/`, `conf/`, `lib/` directory layout
* you need Tanuki Java Service Wrapper behavior around the JVM process
* your operational environment prefers a service distribution to a single jar

[[spark-tanukisoft-install, Tanukisoft Installation]]
=== Installation

A minimal Tanukisoft build needs these pieces:

* `<packaging>tanukisoft</packaging>`
* the `spark-tanukisoft-maven-plugin` declared as an extension
* `operatingSystem`, one of Spark's supported `OSType` values
* `wrapperListener`, the class Tanuki should start as `wrapper.java.mainclass`

If you use Spark's built-in listener base classes or Spring helpers, add `spark-tanukisoft-integration` as an ordinary dependency.

[source,xml]
----
<project>
  <modelVersion>4.0.0</modelVersion>

  <groupId>com.mycompany</groupId>
  <artifactId>orders-service</artifactId>
  <version>1.0.0</version>
  <packaging>tanukisoft</packaging>

  <dependencies>
    <dependency>
      <groupId>org.smallmind</groupId>
      <artifactId>spark-tanukisoft-integration</artifactId>
      <version>${smallmind.version}</version>
    </dependency>
  </dependencies>

  <build>
    <plugins>
      <plugin>
        <groupId>org.smallmind</groupId>
        <artifactId>spark-tanukisoft-maven-plugin</artifactId>
        <version>${smallmind.version}</version>
        <extensions>true</extensions>
        <configuration>
          <operatingSystem>LINUX_X86_64</operatingSystem>
          <wrapperListener>org.smallmind.spark.tanukisoft.integration.spring.ClassPathXmlSpringContextWrapperListener</wrapperListener>
          <appParameters>
            <appParameter>classpath:orders-context.xml</appParameter>
          </appParameters>
        </configuration>
      </plugin>
    </plugins>
  </build>
</project>
----

[[spark-tanukisoft-quick-start, Tanukisoft Quick Start]]
=== Quick Start

1. Choose an `operatingSystem` value and configure a `wrapperListener`.
2. Build the project.
3. Run the generated launcher script or executable from `target/application/.../bin`.

[source,text]
----
mvn clean package

# On Unix-like targets
./target/application/orders-service-1.0.0/bin/orders-service.sh console

# On Windows targets
target\application\orders-service-1.0.0\bin\orders-service.bat
----

The package phase produces two related outputs:

* an unpacked application directory under `target/application/`
* when `createArtifact=true`, a publishable `target/<applicationName>-<version>[-<classifier>]-app.<ext>` archive

[[spark-tanukisoft-examples, Tanukisoft Examples]]
=== Practical Examples

==== Wrapping A Spring XML Application

`ClassPathXmlSpringContextWrapperListener` is the shortest path if your service starts from Spring XML resources.

[source,xml]
----
<dependencies>
  <dependency>
    <groupId>org.smallmind</groupId>
    <artifactId>spark-tanukisoft-integration</artifactId>
    <version>${smallmind.version}</version>
  </dependency>
</dependencies>

<build>
  <plugins>
    <plugin>
      <groupId>org.smallmind</groupId>
      <artifactId>spark-tanukisoft-maven-plugin</artifactId>
      <version>${smallmind.version}</version>
      <extensions>true</extensions>
      <configuration>
        <operatingSystem>LINUX_X86_64</operatingSystem>
        <wrapperListener>org.smallmind.spark.tanukisoft.integration.spring.ClassPathXmlSpringContextWrapperListener</wrapperListener>
        <appParameters>
          <appParameter>classpath:orders-context.xml</appParameter>
          <appParameter>classpath:infrastructure-context.xml</appParameter>
        </appParameters>
        <jvmMaxMemoryMB>1024</jvmMaxMemoryMB>
        <jvmArgs>
          <jvmArg>-Dspring.profiles.active=prod</jvmArg>
        </jvmArgs>
      </configuration>
    </plugin>
  </plugins>
</build>
----

The values under `appParameters` are the Spring XML locations that `ClassPathXmlSpringContextWrapperListener` passes to `ClassPathXmlApplicationContext`.

==== Writing A Custom Wrapper Listener

Use `AbstractWrapperListener` when you want explicit startup and shutdown logic without Spring-specific assumptions.

[source,java]
----
package com.mycompany.orders;

import org.smallmind.spark.tanukisoft.integration.AbstractWrapperListener;

public class OrdersWrapperListener extends AbstractWrapperListener {

  private OrdersService ordersService;

  public void startup (String[] args)
    throws Exception {

    String configurationLocation = args[0];

    ordersService = new OrdersService(configurationLocation);
    ordersService.start();
  }

  public void shutdown ()
    throws Exception {

    if (ordersService != null) {
      ordersService.stop();
    }
  }
}
----

[source,xml]
----
<configuration>
  <operatingSystem>LINUX_X86_64</operatingSystem>
  <wrapperListener>com.mycompany.orders.OrdersWrapperListener</wrapperListener>
  <appParameters>
    <appParameter>../conf/orders.properties</appParameter>
  </appParameters>
</configuration>
----

Inside `startup(String[] args)`, `args` is exactly the `appParameters` array. Spark consumes the listener class name before your startup method runs.

==== Copying Configuration And License Files

Use `configurations` and `licenseFile` to place external files into `conf/`.

[source,xml]
----
<configuration>
  <operatingSystem>LINUX_X86_64</operatingSystem>
  <wrapperListener>com.mycompany.orders.OrdersWrapperListener</wrapperListener>
  <licenseFile>${project.basedir}/src/packaging/wrapper-license.conf</licenseFile>
  <configurations>
    <configuration>${project.basedir}/src/packaging/log4j2.xml</configuration>
    <configuration>${project.basedir}/src/packaging/orders.properties</configuration>
  </configurations>
</configuration>
----

Spark copies each file using its existing basename. To let the default `wrapper.conf` include directives find the Tanuki license automatically, package it as `wrapper-license.conf` or `wrapper-license-%WRAPPER_HOST_NAME%.conf`.

==== Adding Environment Variables And Unix Source Files

`envArgs` writes `set.NAME=VALUE` entries into `conf/wrapper.conf`. `toSourceFiles` tells the generated Unix launcher to source shell files before executing commands.

[source,xml]
----
<configuration>
  <operatingSystem>LINUX_X86_64</operatingSystem>
  <wrapperListener>com.mycompany.orders.OrdersWrapperListener</wrapperListener>
  <envArgs>
    <envArg>
      <name>APP_ENV</name>
      <value>production</value>
    </envArg>
    <envArg>
      <name>JAVA_TOOL_OPTIONS</name>
      <value>-Dfile.encoding=UTF-8</value>
    </envArg>
  </envArgs>
  <toSourceFiles>
    <toSourceFile>../conf/orders-env.sh</toSourceFile>
  </toSourceFiles>
</configuration>
----

Relative `toSourceFiles` paths are resolved from the generated `bin/` directory, so `../conf/orders-env.sh` is the usual form for a file you also package into `conf/`.

==== Pulling A Provided Dependency Into `lib/`

The `dependencies` parameter does not resolve new coordinates. It selects already-declared project artifacts, including `provided` or `system` scope entries, and copies them into `lib/`.

[source,xml]
----
<dependencies>
  <dependency>
    <groupId>com.oracle.database.jdbc</groupId>
    <artifactId>ojdbc11</artifactId>
    <version>23.5.0.24.07</version>
    <scope>provided</scope>
  </dependency>
</dependencies>

<build>
  <plugins>
    <plugin>
      <groupId>org.smallmind</groupId>
      <artifactId>spark-tanukisoft-maven-plugin</artifactId>
      <version>${smallmind.version}</version>
      <extensions>true</extensions>
      <configuration>
        <operatingSystem>LINUX_X86_64</operatingSystem>
        <wrapperListener>com.mycompany.orders.OrdersWrapperListener</wrapperListener>
        <compactClasspath>false</compactClasspath>
        <dependencies>
          <dependency>
            <groupId>com.oracle.database.jdbc</groupId>
            <artifactId>ojdbc11</artifactId>
          </dependency>
        </dependencies>
      </configuration>
    </plugin>
  </plugins>
</build>
----

With `compactClasspath=false`, Spark lists every jar explicitly in `wrapper.conf` instead of emitting a wildcard `../lib/*` entry.

==== Using A Property-Driven Operating System Selection

The template POM in the Spark source uses Maven profiles to choose `operatingSystem`. The same pattern works well in application builds.

[source,xml]
----
<properties>
  <tanukisoft.wrapper.os>LINUX_X86_64</tanukisoft.wrapper.os>
</properties>

==== Controlling Distribution Naming

`applicationName`, `classifier`, `compression`, and `includeVersion` control the unpacked directory name and published archive naming.

[source,xml]
----
<configuration>
  <operatingSystem>LINUX_X86_64</operatingSystem>
  <wrapperListener>com.mycompany.orders.OrdersWrapperListener</wrapperListener>
  <applicationName>orders</applicationName>
  <classifier>linux-x86-64</classifier>
  <compression>zip</compression>
  <includeVersion>false</includeVersion>
</configuration>
----

With that configuration, Spark produces:

[source,text]
----
target/application/orders-linux-x86-64/
target/orders-1.0.0-linux-x86-64-app.zip
----

`includeVersion` changes the unpacked directory name only. The `-app` archive name always includes `${project.version}`.

[[spark-tanukisoft-lifecycle, Tanukisoft Startup And Lifecycle Model]]
=== Startup And Lifecycle Model

Tanukisoft packaging assembles the distribution and wires the wrapper startup path.

At package time, `generate-wrapper` does the following:

1. creates `bin/`, `conf/`, and `lib/` under `${project.build.directory}/${applicationDir}/<applicationName>[-<version>][-<classifier>]`
2. copies configuration and license files into `conf/`
3. copies runtime dependencies into `lib/`
4. copies selected extra `provided` or `system` artifacts from `dependencies` into `lib/`
5. copies the application jar into `lib/`; if the project artifact type is not `jar`, Spark first creates a jar from `${project.build.outputDirectory}`
6. copies the wrapper executable into `bin/`
7. copies the platform-specific native library and a generic alias (`libwrapper.so` or `wrapper.dll`) into `lib/`
8. renders `conf/wrapper.conf` from a Freemarker template
9. renders the Unix launcher script or copies Windows batch files
10. optionally compresses the whole application directory into `target/<applicationName>-<version>[-<classifier>]-app.<ext>`

At runtime, the built-in integration classes behave like this:

1. Tanuki starts the configured `wrapperListener` as `wrapper.java.mainclass`
2. Spark prepends the same listener class to `wrapper.app.parameter.1` so `AbstractWrapperListener.main(...)` can instantiate it reflectively
3. `AbstractWrapperListener.main(...)` validates `wrapper.startup.timeout` if present, defaults it to 30 seconds if absent, expands properties in arguments, and calls `WrapperManager.start(...)`
4. `start(String[] args)` launches a signaling worker that repeatedly calls `WrapperManager.signalStarting(...)` until startup completes
5. your `startup(String[] args)` method runs with only the configured application arguments
6. on shutdown, `stop(int exitCode)` delegates to your `shutdown()` implementation

[IMPORTANT]
====
`waitAfterStartup` is not the same thing as Tanuki's `wrapper.startup.timeout`. `waitAfterStartup` only delays how long the Unix launcher waits before returning from `start`. Startup timeout validation happens in `AbstractWrapperListener` and applies to `wrapper.startup.timeout`, which must be at least 10 seconds if you set it.
====

Built-in listener helpers:

[cols="2,4",options="header"]
|===
|Type |Role

|`AbstractWrapperListener`
|Base class for explicit startup and shutdown logic using Tanuki's `WrapperListener` contract

|`SpringContextWrapperListener`
|Abstract helper that loads a `ConfigurableApplicationContext`, refreshes it on startup, and closes it on shutdown

|`ClassPathXmlSpringContextWrapperListener`
|Concrete Spring helper that builds a `ClassPathXmlApplicationContext` from `appParameters`
|===

[[spark-tanukisoft-goals, Tanukisoft Goals]]
=== Goals

==== `generate-wrapper` (default phase: `package`)

Builds the wrapper distribution and, by default, creates a publishable aggregate archive.

Behavior:

* requires runtime dependency resolution
* copies classpath dependencies whose artifact handlers are marked `addedToClasspath=true` and whose scope is `compile` or `runtime`
* can additionally copy matching `compile`, `provided`, or `system` artifacts from the `dependencies` selector list
* writes `wrapper.conf` and launcher scripts from templates
* creates an aggregate archive when `createArtifact=true`

==== `install-wrapper` (default phase: `install`)

Installs the aggregate `-app` archive into the local Maven repository.

Behavior:

* resolves compression from `compression`
* publishes the archive with classifier `app` by default, or `${classifier}-app` when `classifier` is set
* looks for the archive in `target/<applicationName>-<version>[-<classifier>]-app.<ext>`

==== `deploy-wrapper` (default phase: `deploy`)

Deploys the aggregate `-app` archive to the configured distribution repository.

Behavior:

* uses the same source file naming rules as `install-wrapper`
* defaults `deploymentRepository` from Maven distribution management
* publishes classifier `app` or `${classifier}-app` exactly like `install-wrapper`

=== Operational Notes

* `createArtifact=false` suppresses the `-app` archive. `install-wrapper` and `deploy-wrapper` then have nothing to publish unless you create a matching archive some other way.
* `serviceDependencies` only affects Windows service metadata in `wrapper.conf`.
* relative `toSourceFiles` entries are interpreted from the generated `bin/` directory.
* the generated Unix launcher auto-detects `systemd`, `upstart`, `init.d`, and on AIX `src` at install and start time.
* if `bin/<applicationName>.service` exists beside the Unix launcher, `install` copies it into `/etc/systemd/system/<applicationName>.service` instead of generating a default one
* if `bin/<applicationName>.install` exists beside the Unix launcher, `install` copies it into `/etc/init/<applicationName>.conf` instead of generating a default upstart file
* on AIX, the generated Tanuki script validates that `applicationName` is shorter than 14 characters when installing through SRC tooling
* if `runAs` is set on Unix, the generated launcher expects enough privilege to switch users and will require root for the relevant operations

=== Troubleshooting

* `Unknown operating system type(...)`
Use one of the exact `OSType` names listed above.

* `Unknown compression type(...)`
Use `zip` or `jar`.

* The wrapper starts, but your application logic never runs
Confirm `wrapperListener` names a real class in the packaged `lib/` set and that `appParameters` contain the arguments your `startup(String[])` implementation expects.

* Install or deploy cannot find the `-app` archive
Run the package phase first, keep `applicationName`, `classifier`, and `compression` consistent across goals, and remember that `createArtifact=false` suppresses archive creation.

* The packaged service is missing a provided dependency
Declare the dependency in the project, then add a matching selector under the plugin's `dependencies` parameter.

* The Unix launcher returns before the service is really ready
Increase Tanuki `wrapper.startup.timeout` if startup is genuinely long. Do not expect `waitAfterStartup` to change wrapper startup timeout handling.

[[spark-known-constraints, Known Constraints]]
== Known Constraints

* Spark packaging is Maven-oriented. The guide assumes custom Maven packaging types and plugin extensions.
* Singularity relies on Spark's thread context class loader for nested-jar access. Code that hard-codes the system class loader can fail at runtime.
* Singularity does not shade or relocate dependencies. Package conflicts remain visible.
* Tanukisoft validates `operatingSystem` against a fixed enum even though the plugin resources contain additional wrapper binaries.
* Tanukisoft packaging has a few sharp edges: `dependencies` only selects already-resolved project artifacts, `includeVersion` only affects the unpacked directory name, and `useUpstart` and `useSystemD` are exposed parameters that the shipped Unix launcher does not read.
* Spark install and deploy goals publish package-phase outputs. If the corresponding jar or archive is missing, install and deploy fail rather than rebuilding it.

[[spark-verification-checklist, Verification Checklist]]
== Verification Checklist

For `singularity`:

1. Run `mvn clean package`.
2. Confirm `target/<artifactId>-<version>[-<classifier>].jar` exists.
3. Inspect the jar and confirm `META-INF/singularity/index/singularity.idx` exists.
4. Run `java -jar target/<artifactId>-<version>[-<classifier>].jar`.
5. If runtime discovery fails, test the same code path with `Thread.currentThread().getContextClassLoader()`.

For `tanukisoft`:

1. Run `mvn clean package`.
2. Confirm `target/application/<applicationName>.../bin`, `conf`, and `lib` exist.
3. Confirm `conf/wrapper.conf` contains the JVM args, app parameters, and classpath shape you expect.
4. Start the application through the generated launcher, not by invoking the jar in `lib/` directly.
5. If `createArtifact=true`, confirm `target/<applicationName>-<version>[-<classifier>]-app.<ext>` exists before running `install` or `deploy`.
