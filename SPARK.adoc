== Overview

The `smallmind-spark` module is the project's answer to a single, recurring
question: *how do you ship a Java application as something a human or an
operating system can actually run?* A Maven build produces jars and a
handful of transitive dependency jars; the gap between that artifact set
and "a service my ops team can start on boot and stop on command" is
wide enough that every project ends up re-inventing the same pair of
tools. `smallmind-spark` collects those two tools into the tree and gives
each a Maven packaging type:

* **`singularity`** — bundles the project's classes and every runtime
  dependency jar into a *single self-executing jar*. The bundle ships a
  small boot library and a custom `ClassLoader` that reads nested jars
  in place, so `java -jar my-app.jar` works without unpacking anything,
  without a shaded/uber-jar merge pass, and without classpath management.
* **`tanukisoft`** — wraps the project as a proper OS-managed service
  using https://wrapper.tanukisoftware.com/[Tanuki Software's Java Service
  Wrapper (JSW)]. Produces a ready-to-run tree with `bin/`, `lib/`, and
  `conf/`, the correct platform-specific wrapper executable, rendered
  launch scripts (shell / batch / Windows service install-uninstall),
  and a populated `wrapper.conf`. Understands Upstart, systemd, and
  Windows services.

Both are delivered as Maven extension plugins that install new *packaging
types* — declare `<packaging>singularity</packaging>` or
`<packaging>tanukisoft</packaging>` on a project, and the plugin's
lifecycle binding handles everything `package`, `install`, and `deploy`
normally would.

The module is organized as two independently usable subprojects, each in
turn split into an API/runtime half and a Maven-plugin half:

[cols="1,2,3",options="header"]
|===
| Subproject
| Runtime artifact
| Maven plugin

| `spark/singularity`
| `spark-singularity-boot` — the boot classes that end up inside every
  Singularity jar (entry point, `ClassLoader`, URL handler, index model).
| `spark-singularity-maven-plugin` — the `generate-singularity`,
  `install-singularity`, and `deploy-singularity` goals plus the
  `singularity` packaging type.

| `spark/tanukisoft`
| `spark-tanukisoft-integration` — `AbstractWrapperListener` and its
  Spring specializations, the classes a running wrapped application
  links against to bridge the wrapper's `start` / `stop` callbacks to
  application code.
| `spark-tanukisoft-maven-plugin` — the `generate-wrapper`,
  `install-wrapper`, and `deploy-wrapper` goals plus the `tanukisoft`
  packaging type. Carries every Tanuki wrapper binary as a bundled
  resource.
|===

=== Why This Package Exists

Every Java service eventually meets three problems:

1. **Shipping**. How does the artifact reach a deployed host? A plain
   jar is useless without its classpath. An uber-jar works, but most
   shading pipelines have to rewrite bytecode and resources, break
   signed jars, merge `META-INF/services/` files, and strip manifest
   attributes. Services that need reproducible, untouched dependency
   jars (signed crypto providers, OSGi fragments, JNI library jars)
   cannot use a shaded jar.

2. **Running**. Once the jar arrives on the host, how does it start?
   Platform-native service wrappers exist on every OS — Windows
   services, launchd, systemd, SysV init, Upstart — but every one
   expects its own configuration format and lifecycle conventions. A
   shell script that works on one Linux distro rarely works on another.
   A Windows service needs an executable wrapper, a registry entry, an
   install step, an uninstall step, and a command-line surface.

3. **Behaving**. Once started, how does the service respond to being
   told to stop? A Java process that blocks a graceful shutdown behind
   an unfinished transaction is worse than one that crashes; an ops
   team needs a watchdog that will eventually *make* the process stop,
   and a signal path that lets the application flush and close in the
   meantime.

`smallmind-spark` solves (1) with `singularity` and (2) and (3) with
`tanukisoft`. The two can be used independently or together — the most
common production pattern is a Singularity jar running under a
Tanukisoft wrapper — and each is a Maven packaging type so adopting
either is a one-line POM change.

=== What It Does, Concretely

* **Packages a runnable jar that contains every nested dependency
  without rewriting them** (`spark-singularity-maven-plugin`). Boot
  classes laid down in the jar root, dependency jars stored *as-is*
  under `META-INF/singularity/lib/`, project classes at the root,
  serialized `SingularityIndex` recording the provenance of every
  class and resource.

* **Provides a custom `ClassLoader` that resolves classes from nested
  jars in place** (`SingularityClassLoader`), backed by a custom
  `singularity:` URL protocol that can stream bytes from any entry in
  any nested library without extracting anything to disk. Nested jars
  are cached in memory via soft references so repeated lookups do not
  reopen the outer archive.

* **Turns the jar into an executable bootstrap** via
  `SingularityEntryPoint`: `Main-Class` of the outer jar, reflectively
  invokes whatever fully qualified class name was configured as
  `mainClass` at build time.

* **Assembles a ready-to-run Tanuki wrapper distribution**
  (`spark-tanukisoft-maven-plugin`) with `bin/`, `lib/`, `conf/`, the
  correct platform-specific wrapper executable and native library
  (19 supported platform triples), runtime dependencies, the project's
  own jar, the Tanuki license, and any additional configuration files.

* **Renders launch scripts and `wrapper.conf` from Freemarker
  templates** — `<app>.sh` for Unix, `<app>.bat` /
  `Install<app>-NT.bat` / `Uninstall<app>-NT.bat` for Windows, and
  `wrapper.conf` for every platform. The Freemarker model includes
  heap sizing, JVM arguments, environment variables, NT service
  dependencies, Upstart / systemd toggles, run-as user, and classpath.

* **Provides an `AbstractWrapperListener`** that subclasses can extend
  to hook the wrapper's start / stop callbacks to their own
  application lifecycle — with a built-in keep-alive thread so slow
  startups do not trip the wrapper's watchdog.

* **Provides `SpringContextWrapperListener` and
  `ClassPathXmlSpringContextWrapperListener`** so Spring-based
  applications can be wrapped by declaring the XML locations in the
  plugin configuration — the most common integration path, requiring
  *no* application-specific wrapper-listener class at all.

* **Registers new Maven packaging types** (`singularity` and
  `tanukisoft`) that rebind `package`, `install`, and `deploy` phases
  to the module's own goals via Plexus `components.xml`.

=== What It Intentionally Does Not Do

* It does not rewrite dependency bytecode or resources. The whole
  *point* of Singularity is that the jars in `META-INF/singularity/lib/`
  are the exact jars Maven resolved — signatures intact, service
  descriptors intact, `META-INF/` intact. If you need shading, use
  `maven-shade-plugin`; Singularity is the *other* choice.

* It does not hide Tanuki's Java Service Wrapper. `wrapper.conf` is
  still `wrapper.conf`, in exactly the form Tanuki documents. The
  plugin generates it from a Freemarker template whose inputs are
  Maven-configurable; everything else is Tanuki's.

* It does not ship Tanuki's wrapper license. The Tanuki wrapper binaries
  themselves are bundled (as `wrapper-<os>-<arch>(.exe)` resources in
  the plugin jar) for convenience, but a production deployment needs
  a real Tanuki license file and is expected to point `licenseFile` at
  it.

* It does not replace Spring Boot, Quarkus, Micronaut, or any other
  "framework that packages its own runtime." The target audience is a
  plain Java service — typically a Spring application — that wants a
  jar it can hand to ops *and* an OS service wrapping story, without
  adopting a heavier framework.

* It does not provide hot redeploy, health probes, config reload, or
  any other long-running-service amenity. The Tanuki wrapper has its
  own watchdog; anything beyond that is the application's concern.

[[installation]]
== Installation

`smallmind-spark` ships four independently resolvable artifacts. Pick
the smallest set that matches your integration surface.

=== `spark-singularity-boot`

You never depend on this directly. The Singularity Maven plugin has it
as its own `compile`-scope dependency and extracts its classes into the
outer jar at package time, which is why the boot classes appear under
`org/smallmind/spark/singularity/boot/` at the root of every Singularity
jar. If you are consuming a Singularity jar (rather than building one)
you can ignore this artifact entirely.

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>spark-singularity-boot</artifactId>
  <version>${project.version}</version>
</dependency>
----

=== `spark-singularity-maven-plugin`

Declare this as a build plugin with `<extensions>true</extensions>`. The
`extensions` attribute is what registers the `singularity` packaging
type and its lifecycle mapping — without it, `<packaging>singularity</packaging>`
will not be recognized.

[source,xml]
----
<packaging>singularity</packaging>

<build>
  <plugins>
    <plugin>
      <groupId>org.smallmind</groupId>
      <artifactId>spark-singularity-maven-plugin</artifactId>
      <version>${project.version}</version>
      <extensions>true</extensions>
      <configuration>
        <mainClass>com.example.service.Main</mainClass>
      </configuration>
    </plugin>
  </plugins>
</build>
----

=== `spark-tanukisoft-integration`

Depend on this from application code that extends
`AbstractWrapperListener` or uses the shipped Spring specializations.
Brings in the Tanuki `wrapper` runtime jar (`tanukisoft:wrapper`),
Spring's `spring-context`, and `smallmind-nutsnbolts`.

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>spark-tanukisoft-integration</artifactId>
  <version>${project.version}</version>
</dependency>
----

If you are using the shipped
`ClassPathXmlSpringContextWrapperListener` and you do not need to
compile any code against the wrapper API, a project with
`<packaging>tanukisoft</packaging>` will pull this in through the
example POM's dependency on `spark-tanukisoft-integration`.

=== `spark-tanukisoft-maven-plugin`

Declare this as a build plugin with `<extensions>true</extensions>` so
the `tanukisoft` packaging type is registered.

[source,xml]
----
<packaging>tanukisoft</packaging>

<build>
  <plugins>
    <plugin>
      <groupId>org.smallmind</groupId>
      <artifactId>spark-tanukisoft-maven-plugin</artifactId>
      <version>${project.version}</version>
      <extensions>true</extensions>
      <configuration>
        <operatingSystem>${tanukisoft.wrapper.os}</operatingSystem>
        <wrapperListener>org.smallmind.spark.tanukisoft.integration.spring.ClassPathXmlSpringContextWrapperListener</wrapperListener>
        <appParameters>
          <appParameter>classpath:spring/application-context.xml</appParameter>
        </appParameters>
      </configuration>
    </plugin>
  </plugins>
</build>
----

[IMPORTANT]
====
The Tanuki wrapper binaries are distributed inside the plugin jar under
the `org/smallmind/spark/tanukisoft/maven/bin/` and `.../lib/`
resource roots. They are copied verbatim into the generated distribution.
If a platform you need is not in `OSType`, the plugin will reject the
build with "Unknown operating system type(...)". Adding a platform
requires placing the correct wrapper binaries under those resource
paths *and* adding the corresponding `OSType` enum constant.
====

[[conceptual-model]]
== Conceptual Model

The module rests on two orthogonal ideas, one per subproject. Holding
them distinct in your head is the shortest path to correct use.

=== Singularity's Central Idea

A Singularity bundle is a normal `.jar` file whose contents are
partitioned three ways:

. **Boot classes** at the jar root under
  `org/smallmind/spark/singularity/boot/`. These are copied into the
  outer jar at package time from the `spark-singularity-boot` plugin
  dependency. They are reachable by a fresh JVM's system class loader.

. **Application classes** at the jar root, in their natural package
  directories. Same position as a normal jar's contents; same
  resolution story.

. **Library jars** under `META-INF/singularity/lib/`, each one kept
  verbatim as a `.jar` file. *Nothing* unpacks them. A normal system
  class loader cannot reach classes inside these nested jars, so a
  custom `ClassLoader` is necessary.

Glue holds the three apart:

Index::
    A serialized `SingularityIndex` at
    `META-INF/singularity/index/singularity.idx` records, for every
    class and resource, which of the three populations it belongs to.
    For library entries, it also records which specific library jar
    inside `lib/` supplied it.

Custom URL protocol::
    The `singularity:` protocol has the form
    `singularity:<outer-jar-url>@/META-INF/singularity/lib/<lib>!/<entry>`.
    `SingularityJarURLStreamHandler` claims it JVM-wide from
    `SingularityClassLoader`'s static initializer, so any component
    that resolves an entry's URL (including `javax.crypto.JarVerifier`,
    `ServiceLoader`, Spring's `PathMatchingResourcePatternResolver`,
    and URL-based classloaders layered on top) can reach the bytes.

Custom ClassLoader::
    `SingularityClassLoader` reads the index at boot, maps every
    entry name to the appropriate URL form (`jar:` for jar-root entries,
    `singularity:` for library entries), and serves `findClass` /
    `findResource` from that map. It *refuses* a small set of
    JDK-shadowed namespaces so the platform's own copies win.

Entry point::
    `SingularityEntryPoint` is the jar's `Main-Class`. On `main`, it
    opens the enclosing jar via `CodeSource`, constructs a
    `SingularityClassLoader`, installs it as the context class loader,
    and reflectively invokes the application's *real* `main` — whose
    class name is stored in the manifest attribute `Singularity-Class`.

=== Tanukisoft's Central Idea

Tanukisoft's Java Service Wrapper is a native executable that launches
a JVM as a child process, watches that child for health, restarts it
on crash, and translates OS-level service / daemon conventions
(Windows services, systemd units, SysV init, launchd, Upstart) into a
consistent API. The application code inside that JVM is a normal Java
program that implements one interface (`WrapperListener`) so the
wrapper can deliver `start` and `stop` callbacks across the JNI
boundary.

`smallmind-spark` gives that a three-layer shape:

Bridge::
    `AbstractWrapperListener` implements `WrapperListener` and bolts on
    the one-time setup each wrapped application needs: a CLI bootstrap
    that reads the configured listener class name from the first
    argument and reflectively creates it; a Property-Expander pass
    over the remaining arguments; a daemon thread that calls
    `WrapperManager.signalStarting(int)` on a loop so a slow `startup`
    is not killed by the watchdog.

Spring specialization::
    `SpringContextWrapperListener` extends `AbstractWrapperListener` and
    maps the abstract `startup` / `shutdown` hooks onto the life of a
    Spring `ConfigurableApplicationContext`. Subclasses need only
    implement one factory method. This is the layer most applications
    should use, directly or through `ClassPathXmlSpringContextWrapperListener`.

Build-time assembly::
    `spark-tanukisoft-maven-plugin` stages a `bin/` + `lib/` + `conf/`
    tree, copies in the platform-specific wrapper binary plus native
    library, copies in all dependency jars, renders the Freemarker
    templates into `<app>.sh` (or the Windows batch equivalents) and
    `wrapper.conf`, and optionally compresses the result into an
    archive that can be dropped on a host.

Nothing about the runtime layer assumes a Singularity jar; the two
subprojects are independent. Their natural composition is: build the
application as `<packaging>singularity</packaging>`, then reference the
resulting single jar as the only entry in the wrapper's classpath — but
this is a deployment choice, not a code coupling.

[[singularity-reference]]
== Singularity Reference

=== Maven Goals

`generate-singularity` (phase: `package`)::
    Assembles the bundle. Creates
    `${project.build.directory}/${singularityBuildDir}/` with its
    `META-INF/singularity/{lib,index}` subtree, extracts the boot
    classes from the plugin's own `spark-singularity-boot` dependency
    into the tree, walks `project.getRuntimeArtifacts()` and copies
    each non-excluded dependency jar verbatim into `lib/` (while
    recording every one of its entries in the `SingularityIndex`),
    walks `target/classes/` and copies every file into the tree (while
    recording each in the index), serializes the index to
    `META-INF/singularity/index/singularity.idx`, composes a manifest
    with `Main-Class: org.smallmind.spark.singularity.boot.SingularityEntryPoint`
    and `Singularity-Class: <mainClass>`, compresses the tree into
    `${project.build.directory}/<artifactId>-<version>[-<classifier>].jar`,
    and attaches the result as a project artifact.

`install-singularity` (phase: `install`)::
    Locates the output jar in `target/` using the same naming rule as
    `generate-singularity`, looks for a sibling `.asc` signature file,
    attaches it via `AscArtifactMetadata` if present, and installs
    the artifact into the local repository via
    `ArtifactInstaller.install`.

`deploy-singularity` (phase: `deploy`)::
    Same as `install-singularity`, except it pushes to the project's
    configured `distributionManagement` repository through
    `ArtifactDeployer.deploy`.

=== Configuration Parameters (`generate-singularity`)

`mainClass` (required)::
    Fully qualified name of the *application's* main class — the one
    the user would have invoked had they had a classpath. Stored in
    the jar manifest as `Singularity-Class`. `SingularityEntryPoint`
    reads it and reflectively calls its `main(String[])`.

`singularityBuildDir` (default: `singularity`)::
    Name of the subdirectory under `${project.build.directory}` into
    which the bundle tree is staged before compression. Rarely needs
    changing.

`exclusions`::
    List of `<exclusion>` entries, each with `<groupId>` and
    `<artifactId>`, that should be dropped from the bundle. Matched
    exactly on both coordinates.

`verbose` (default: `false`)::
    Enables per-file progress logging.

`skip` (default: `false`)::
    Skips execution. Combine with a Maven profile to build a
    "development" jar that intentionally omits the Singularity step.

=== Runtime Classes (`spark-singularity-boot`)

`SingularityEntryPoint`::
    `Main-Class` of every generated jar. Its `main(String...)` opens
    the enclosing jar through its own `CodeSource`, reads the
    manifest, constructs a `SingularityClassLoader` bound to that
    jar, installs the loader as the context class loader, and
    reflectively invokes the `Singularity-Class`'s `main` with the
    original argument array. Silently ignores the degenerate case
    where `Singularity-Class` equals itself (so the boot does not
    recurse).

`SingularityClassLoader`::
    The runtime's center. Reads the serialized `SingularityIndex` off
    the jar, maps every entry name to either a `jar:`-protocol URL
    (for entries sitting at the jar root) or a `singularity:`-protocol
    URL (for entries inside a nested library jar), and serves
    `findClass` / `findResource` / `findResources` from that map.
+
Its static initializer installs `SingularityJarURLStreamHandlerFactory`
*JVM-wide*, which is a deliberate singleton: the JVM permits exactly
one `URLStreamHandlerFactory` to be registered, and
`SingularityClassLoader` must be the one that registers it.

`SingularityIndex`::
    `Serializable` POJO with two internal collections: `fileNameSet`
    for entries stored directly in the outer jar and `inverseEntryMap`
    for entries inside a nested library (mapping entry-name → jar-name).
    Exposes two `Iterable<URLEntry>` views used by the loader to
    materialize its resolution table. The `URLEntry` nested record
    pairs a logical entry name with the URL at which its bytes can
    be read.

`CachedJarFile`::
    In-memory snapshot of a nested jar. Its constructor drains a
    `JarInputStream` once, storing every entry's compressed bytes in
    a map keyed by entry name. Subsequent calls to
    `getInputStream(name)` return a fresh `InflaterInputStream` over
    those bytes. Used by `SingularityJarURLConnection` behind a
    soft-reference map so the JVM can reclaim cached jars under
    memory pressure.

`SingularityJarURLConnection`::
    `URLConnection` for the `singularity:` protocol. Parses the URL
    form `singularity:<outer>@/[<inner-jar>!/]<entry>` and returns an
    `InputStream` over the addressed entry, reusing the cached
    nested-jar form when possible.

`SingularityJarURLStreamHandler`::
    Standard `URLStreamHandler` that returns a fresh
    `SingularityJarURLConnection` for every URL.

`SingularityJarURLStreamHandlerFactory`::
    `URLStreamHandlerFactory` registered JVM-wide. Claims the
    `singularity` protocol; returns `null` for everything else so the
    platform's default handler selection stays in effect.

=== Plugin-Support Classes (`spark-singularity-maven-plugin`)

`Exclusion`::
    Configuration POJO with `groupId` and `artifactId` fields and a
    single `matchesArtifact(Artifact)` predicate. Both coordinates
    must match exactly; version is ignored.

`AscArtifactMetadata`::
    Maven `ArtifactMetadata` implementation that carries a
    detached-GPG `.asc` file alongside an artifact through install
    and deploy. The local filename is
    `<artifactId>-<version>[-<classifier>].jar.asc`. Lives in the
    artifact's version directory. Used only if an `.asc` is found
    next to the generated jar.

`CopyFileVisitor`::
    `SimpleFileVisitor<Path>` used by `generate-singularity` to walk
    `target/classes/`. On each file, copies it to the matching
    location under the singularity build tree and registers the
    relative resource path (with `/` separators) in the
    `SingularityIndex` via `addFileName`.

[[class-loading-rules]]
=== Class Loading Rules

`SingularityClassLoader.loadClass(String, boolean)` follows a
self-before-parent strategy:

. Check `findLoadedClass` — return an already-defined class directly.
. Otherwise call `findClass` — the *bundled* definition is preferred.
. If `findClass` throws `ClassNotFoundException`, delegate to the
  parent (or to `findSystemClass` if no parent is configured).

`findClass` itself:

. Rejects names in reserved namespaces (see below) so the platform's
  own copies resolve through the parent.
. Looks up `<binary>.class` in the index; if absent, throws
  `ClassNotFoundException`.
. Determines a `CodeSource` URL based on the entry's protocol:
  * `jar:<outer>!/…` → `CodeSource` is `<outer>!/`.
  * `singularity:<outer>@/<inner>!/…` → `CodeSource` is
    `singularity:<outer>@/<inner>` (no trailing `!/`). This form
    compensates for a quirk in `javax.crypto.JarVerifier` that
    would otherwise mangle the URL.
. Reads the bytes via `URL.openStream()`.
. Lazily defines the enclosing package on first touch, using the
  outer manifest's specification/implementation titles and the
  optional seal base.
. Calls `defineClass(name, bytes, 0, bytes.length, protectionDomain)`
  where the protection domain has `AllPermission`.

[[reserved-jdk-namespaces]]
=== Reserved JDK Namespaces

`SingularityClassLoader` deliberately refuses to define classes whose
names begin with:

* `jakarta.xml.` — except `jakarta.xml.bind.` which *is* admitted.
* `org.xml.`
* `org.w3c.`

Bundled dependencies sometimes ship these packages (older SAX stubs,
stale JAXB API jars, or test fixtures that repackage XML APIs). The
JDK requires its *own* copies of these classes for platform code to
function; allowing the bundle to provide conflicting versions causes
`LinkageError`s when the JDK's own parser code hits them. The allow
list for `jakarta.xml.bind` exists because production code that wants
JAXB needs a real implementation, and a Singularity bundle is the
correct place to ship one.

[[excluding-dependencies]]
=== Excluding Dependencies

A host JVM that already exposes a library (a container-provided JDBC
driver, say, or a shared agent) should not have it duplicated inside
the Singularity jar — same class, two definitions, different
`Class<?>` objects, mystery `ClassCastException` at the seam.

[source,xml]
----
<plugin>
  <groupId>org.smallmind</groupId>
  <artifactId>spark-singularity-maven-plugin</artifactId>
  <version>${project.version}</version>
  <extensions>true</extensions>
  <configuration>
    <mainClass>com.example.service.Main</mainClass>
    <exclusions>
      <exclusion>
        <groupId>com.oracle.database.jdbc</groupId>            <!--1-->
        <artifactId>ojdbc11</artifactId>
      </exclusion>
      <exclusion>
        <groupId>org.slf4j</groupId>                           <!--2-->
        <artifactId>slf4j-api</artifactId>
      </exclusion>
    </exclusions>
  </configuration>
</plugin>
----
<1> Oracle JDBC driver provided by the host.
<2> SLF4J API already loaded by the host's logging framework.

Exclusions match exactly on both coordinates. There is no
glob / version / scope filtering — add an entry per artifact id.

[[tanukisoft-reference]]
== Tanukisoft Reference

=== Maven Goals

`generate-wrapper` (phase: `package`)::
    Assembles the wrapper distribution under
    `${project.build.directory}/${applicationDir}/<app-name>` with the
    standard `bin/`, `lib/`, `conf/` layout. Validates the
    `operatingSystem` and `compression` values against the `OSType`
    and `CompressionType` enums; copies the license file, user
    configurations, runtime dependencies, any `<dependencies>` matches,
    and the project jar (creating one if the project's packaging is
    not `jar`); copies the platform-specific Tanuki executable to
    `bin/` and native library to `lib/`; renders `freemarker.App.sh.in`
    to `<app-name>.sh` on Unix or copies `AppCommand.bat.in` /
    `InstallApp-NT.bat.in` / `UninstallApp-NT.bat.in` on Windows;
    renders `freemarker.wrapper.conf.in` to `wrapper.conf` for all
    platforms; optionally compresses the tree into
    `${project.build.directory}/<app-name>-<version>[-<classifier>]-app.<ext>`.

`install-wrapper` (phase: `install`)::
    Installs the aggregate archive into the local Maven repository
    with classifier `app` (or `<classifier>-app` if a `classifier` is
    also configured).

`deploy-wrapper` (phase: `deploy`)::
    Deploys the aggregate archive to the project's configured
    `distributionManagement` repository with the same classifier.

=== Configuration Parameters (`generate-wrapper`)

The full set of parameters, in the order a user typically cares about:

`operatingSystem` (required)::
    An `OSType` enum value, supplied as its string name (hyphens
    accepted in place of underscores, case-insensitive — e.g.
    `LINUX_X86_64`, `linux-x86-64`, and `Linux_X86_64` all work). See
    <<ostype-reference>>.

`wrapperListener` (required)::
    Fully qualified name of the `WrapperListener` subclass that the
    wrapper should instantiate. This is the class whose `startup` and
    `shutdown` hooks get called. Typical values:
+
* `org.smallmind.spark.tanukisoft.integration.spring.ClassPathXmlSpringContextWrapperListener`
  — pass Spring XML locations as `appParameters`.
* Your own subclass of `AbstractWrapperListener` or
  `SpringContextWrapperListener`.

`applicationName` (default: `${project.artifactId}`)::
    Short name used for the staging directory, the shell script
    (`<name>.sh`), the Windows batch files (`<name>.bat`,
    `Install<name>-NT.bat`, `Uninstall<name>-NT.bat`), and the
    wrapper's own `wrapper.name` property.

`applicationLongName` (default: `${project.name}`)::
    Long name, used in the wrapper's `wrapper.displayname` and in
    Windows service registration.

`applicationDescription` (default: `"<class> generated project"`)::
    Description, used in `wrapper.description` and in Windows
    service registration. Override for an informative service
    description rather than the placeholder default.

`applicationDir` (default: `application`)::
    Name of the subdirectory of `${project.build.directory}` into
    which the distribution is staged.

`licenseFile`::
    Filesystem path (not classpath) to a Tanuki wrapper license file.
    Copied into `conf/`. Production builds always need one.

`configurations`::
    List of filesystem paths. Each file is copied into `conf/`. Use
    this for any additional configuration files the running
    application needs (Log4j configuration, Spring property files,
    Terracotta config, etc.).

`dependencies`::
    List of `<dependency>` entries (`groupId` + `artifactId`) that
    should be bundled in `lib/` even when their Maven scope is
    `compile`, `provided`, or `system` (i.e. not normally part of
    the runtime artifact set). Use for driver jars shipped at
    `provided` scope to avoid running tests against them.

`jvmArgs`::
    List of raw JVM arguments for the wrapper to pass on the child
    `java` command. Each element becomes a
    `wrapper.java.additional.N=<arg>` entry in `wrapper.conf`.

`jvmInitMemoryMB` / `jvmMaxMemoryMB`::
    Strings setting the wrapper's `wrapper.java.initmemory` and
    `wrapper.java.maxmemory` properties (both in megabytes). Omit
    to let Tanuki use its own defaults.

`appParameters`::
    Ordered list of arguments passed to the wrapper listener's
    `main`. The plugin *prepends* `wrapperListener` to this list
    (because `AbstractWrapperListener.main` expects the concrete
    listener class as its first argument). Subsequent entries are
    the application's own arguments — typically Spring XML
    resource locations when using `ClassPathXmlSpringContextWrapperListener`.

`envArgs`::
    List of `<envArg>` entries (`name` / `value`). Each one becomes
    a `set.<name>=<value>` in `wrapper.conf`, available to the
    wrapper's own substitution and to the launched JVM's environment.

`serviceDependencies`::
    Ordered list of Windows NT service names this service depends on.
    Each becomes `wrapper.ntservice.dependency.N=<name>` in
    `wrapper.conf`.

`javaCommand` (default: `java`)::
    Name of the `java` executable to invoke. A fully qualified path
    avoids relying on `PATH`.

`umask` (default: `0022`)::
    Process umask on Unix. Rendered as `wrapper.umask` in
    `wrapper.conf`.

`runAs`::
    Unix user to run the service as. On Windows, renders
    permissions entries for that account. Left unset, the service
    runs as the user that launched the wrapper.

`waitAfterStartup` (default: `0`)::
    Milliseconds to delay between the child JVM becoming ready and
    the wrapper reporting "started" to the OS. Useful when a service
    binds a port or reaches a health threshold slightly after
    `startup` returns.

`useUpstart` (default: `false`)::
    Generate Upstart-style scripts (Ubuntu). Mutually exclusive with
    `useSystemD` for any single build.

`useSystemD` (default: `false`)::
    Generate systemd unit files.

`compression` (default: `zip`)::
    Compression type for the aggregate archive. Must be a value of
    `org.smallmind.nutsnbolts.zip.CompressionType` (`ZIP`, `JAR`,
    `TGZ`, `BZ2`, etc.).

`createArtifact` (default: `true`)::
    If `true`, compress the entire staged tree into a single archive
    and attach it. If `false`, leave the staged tree in place — use
    this if a later build step assembles the distribution its own
    way.

`includeVersion` (default: `true`)::
    If `true`, the staging directory and archive name include
    `-<version>`. If `false`, the stable `<applicationName>` is used.

`compactClasspath` (default: `true`)::
    If `true`, the wrapper's classpath is a single `*` wildcard that
    matches every file in `lib/`. If `false`, each jar is listed
    individually as `wrapper.java.classpath.N`. `true` is simpler and
    is the only sensible choice when dependencies change between
    builds.

`classifier`::
    Optional Maven-style classifier for the aggregate archive
    filename.

`verbose` (default: `false`)::
    Per-file progress logging.

`skip` (default: `false`)::
    Skip execution.

=== Runtime Classes (`spark-tanukisoft-integration`)

`AbstractWrapperListener`::
    The bridge class every wrapper integration flows through. Supplies
    a static `main` that Tanuki's wrapper invokes; that `main` reflects
    the listener class named in `args[0]`, pre-expands the remaining
    arguments through `PropertyExpander` (so Ant-style `${property}`
    placeholders in `wrapper.conf` resolve at runtime), and hands the
    listener to `WrapperManager.start`.
+
On `start(String[])`, spawns a daemon `SignalWorker` thread that
loops for up to `wrapper.startup.timeout` seconds (default 30,
minimum 10, fail-safe ceiling 180) calling
`WrapperManager.signalStarting(timeout)` every `timeout - 2`
seconds. That keeps the wrapper's watchdog from killing the JVM
during slow context refreshes.
+
Converts uncaught exceptions from `startup` or `shutdown` into exit
code `2` (`STACK_TRACE_ERROR_CODE`). Implements `controlEvent` to
translate Ctrl-C / Close / Shutdown into `WrapperManager.stop(0)`
*only when the process is not under the native wrapper* — when the
native wrapper is in charge, it delivers these events itself.

`SpringContextWrapperListener`::
    Abstract specialization that owns a
    `ConfigurableApplicationContext` field and maps
    `startup(String[])` / `shutdown()` onto the context:
+
* `startup` creates the context on first call by invoking
  `loadApplicationContext(args)`, then refreshes it if it is not
  already active.
* `shutdown` closes the context if one was created and is still
  active.
+
Subclasses implement exactly one method:
`loadApplicationContext(String[] args)`.

`ClassPathXmlSpringContextWrapperListener`::
    The zero-code listener. Its `loadApplicationContext(String[] args)`
    returns `new ClassPathXmlApplicationContext(args)`, so every
    argument passed through the plugin's `appParameters` becomes a
    classpath-relative Spring XML location. In most deployments this
    is the only listener you ever reference.

=== Plugin-Support Classes (`spark-tanukisoft-maven-plugin`)

[[ostype-reference]]
`OSType`::
    Enum of every supported platform triple. Each constant carries
    an `OSStyle` (Windows vs Unix, which decides launch-script
    selection) plus the platform-specific wrapper executable filename
    and native library filename. The 19 constants cover AIX (ppc-32,
    ppc-64), HP-UX (ia-32, ia-64, parisc-32, parisc-64), Linux (ia-64,
    ppc-32, ppc-64, x86-32, x86-64), macOS (universal-32,
    universal-64), Solaris (sparc-32, sparc-64, x86-32, x86-64), and
    Windows (x86-32, x86-64). The plugin's `operatingSystem`
    parameter accepts the enum name with underscores *or* hyphens.

`OSStyle`::
    `WINDOWS` or `UNIX`. Selects the launch-script template and the
    generic native library name (`wrapper.dll` or `libwrapper.so`)
    that the wrapper looks for in `lib/`. Also present as the
    platform-specific name so the wrapper can locate it either way.

`Dependency`::
    Configuration POJO for the `<dependencies>` list. Fields:
    `groupId`, `artifactId`. Single method `matchesArtifact(Artifact)`.

`EnvironmentArgument`::
    Configuration POJO for the `<envArgs>` list. Fields: `name`,
    `value`.

=== Freemarker Templates

All Freemarker templates ship as resources inside
`spark-tanukisoft-maven-plugin` and use SQUARE_BRACKET tag syntax
(`[#if …]`, `[#list … as …]`) so the templates remain legal inside
POM files and shell scripts.

`bin/freemarker.App.sh.in`::
    Unix launch script. Rendered to `<applicationName>.sh` when the
    selected `OSType` has `OSStyle.UNIX`.

`bin/AppCommand.bat.in` / `bin/InstallApp-NT.bat.in` / `bin/UninstallApp-NT.bat.in`::
    Windows scripts. Copied — not Freemarker-rendered — to
    `<applicationName>.bat`, `Install<applicationName>-NT.bat`, and
    `Uninstall<applicationName>-NT.bat` respectively when the
    selected `OSType` has `OSStyle.WINDOWS`.

`conf/freemarker.wrapper.conf.in`::
    The wrapper configuration template. Rendered to `wrapper.conf`
    regardless of `OSStyle`. Its Freemarker model is populated with
    every plugin parameter that shapes wrapper behavior — the
    application name / long name / description, the Java command,
    each JVM argument, heap sizes, umask, run-as user, Upstart /
    systemd flags, wait-after-startup, appParameters, envArgs,
    service dependencies, and the classpath element list.

[[maven-packaging-types]]
== Maven Packaging Types

Each subproject registers a new Maven *packaging type* via the Plexus
`components.xml` bundled in its plugin. The `<extensions>true</extensions>`
attribute on the plugin declaration is what activates the mapping.

=== `<packaging>singularity</packaging>`

Registered by `spark-singularity-maven-plugin`. Binds:

[cols="1,3",options="header"]
|===
| Phase
| Bound goal

| `process-resources` | `maven-resources-plugin:resources`
| `compile`           | `maven-compiler-plugin:compile`
| `process-test-resources` | `maven-resources-plugin:testResources`
| `test-compile`      | `maven-compiler-plugin:testCompile`
| `test`              | `maven-surefire-plugin:test`
| `package`           | `spark-singularity-maven-plugin:generate-singularity`
| `install`           | `spark-singularity-maven-plugin:install-singularity`
| `deploy`            | `spark-singularity-maven-plugin:deploy-singularity`
|===

The `singularity` artifact handler reports `extension=jar` and
`addedToClasspath=false`. A downstream project that adds a
`singularity`-packaged module as a dependency gets the jar file but
not classpath inclusion — which is correct, because a Singularity
jar is an *executable artifact*, not a library.

=== `<packaging>tanukisoft</packaging>`

Registered by `spark-tanukisoft-maven-plugin`. Binds:

[cols="1,3",options="header"]
|===
| Phase
| Bound goal

| `process-resources` | `maven-resources-plugin:resources`
| `compile`           | `maven-compiler-plugin:compile`
| `process-test-resources` | `maven-resources-plugin:testResources`
| `test-compile`      | `maven-compiler-plugin:testCompile`
| `test`              | `maven-surefire-plugin:test`
| `package`           | `maven-jar-plugin:jar`, then `spark-tanukisoft-maven-plugin:generate-wrapper`
| `install`           | `spark-tanukisoft-maven-plugin:install-wrapper`
| `deploy`            | `spark-tanukisoft-maven-plugin:deploy-wrapper`
|===

The double bind on `package` is deliberate: `maven-jar-plugin:jar`
produces the normal project jar first, and `generate-wrapper` picks
it up as one of the files staged into `lib/`. If the normal jar step
were skipped, `generate-wrapper` would synthesize one from
`target/classes/` on its own, but letting `maven-jar-plugin` run
keeps ordinary test-jar and classifier logic intact.

[[singularity-quick-start]]
=== Singularity Quick Start

A minimal POM for a service that should ship as a single
executable jar:

[source,xml]
----
<project>
  <modelVersion>4.0.0</modelVersion>
  <groupId>com.example</groupId>
  <artifactId>my-service</artifactId>
  <version>1.0.0</version>
  <packaging>singularity</packaging>                                  <!--1-->

  <dependencies>
    <!-- ordinary runtime dependencies -->
    <dependency>
      <groupId>com.fasterxml.jackson.core</groupId>
      <artifactId>jackson-databind</artifactId>
      <version>2.17.0</version>
    </dependency>
  </dependencies>

  <build>
    <plugins>
      <plugin>
        <groupId>org.smallmind</groupId>
        <artifactId>spark-singularity-maven-plugin</artifactId>
        <version>${smallmind.version}</version>
        <extensions>true</extensions>                                  <!--2-->
        <configuration>
          <mainClass>com.example.service.Main</mainClass>              <!--3-->
        </configuration>
      </plugin>
    </plugins>
  </build>
</project>
----
<1> Declaring `singularity` as the packaging triggers the plugin's
    lifecycle mapping. Requires the plugin extension to be enabled.
<2> Registers the plugin as a build extension so its packaging type
    and lifecycle mapping are loaded.
<3> Fully qualified name of the application's main class.

Run `mvn package`. The output is
`target/my-service-1.0.0.jar`, containing:

[source,text]
----
my-service-1.0.0.jar
  META-INF/
    MANIFEST.MF                              Main-Class: SingularityEntryPoint

      lib/
        jackson-databind-2.17.0.jar          Dependency jar, untouched

  org/
    smallmind/spark/singularity/boot/        Boot classes at the jar root
      SingularityEntryPoint.class

      ...
  com/
    example/service/                         Project classes at the jar root
      Main.class
      ...

[[spring-wrapper-listener-pattern]]
=== Spring Wrapper-Listener Pattern

The recommended integration path for Spring applications is to *not*
write your own wrapper listener — use the shipped
`ClassPathXmlSpringContextWrapperListener` and pass Spring XML
locations as `appParameters`.

If you need programmatic context construction (not XML, or XML plus
Java config, or a parent-child hierarchy), subclass
`SpringContextWrapperListener`:

[source,java]
----
package com.example.service.wrapper;

import org.smallmind.spark.tanukisoft.integration.spring.SpringContextWrapperListener;
import org.springframework.context.ConfigurableApplicationContext;
import org.springframework.context.annotation.AnnotationConfigApplicationContext;

public class AnnotationConfigWrapperListener extends SpringContextWrapperListener {

  @Override
  public ConfigurableApplicationContext loadApplicationContext (String[] args) {   <1>

    AnnotationConfigApplicationContext context = new AnnotationConfigApplicationContext();

    for (String arg : args) {                                                        <2>
      try {
        context.register(Class.forName(arg));
      } catch (ClassNotFoundException classNotFoundException) {
        throw new RuntimeException(classNotFoundException);
      }
    }

    return context;                                                                  <3>
  }
}
----
<1> The one abstract hook. Receives the `appParameters` list (minus
    the wrapper-framing first element).
<2> Interpret each argument however makes sense for your app. Here
    they are `@Configuration` class names.
<3> Return an *unrefreshed* context — `SpringContextWrapperListener`
    calls `refresh()` itself. Returning a pre-refreshed context is
    harmless (it is detected via `isActive()`) but redundant.

Point the plugin at your class:

[source,xml]
----
<wrapperListener>com.example.service.wrapper.AnnotationConfigWrapperListener</wrapperListener>
<appParameters>
  <appParameter>com.example.service.config.AppConfig</appParameter>
  <appParameter>com.example.service.config.PersistenceConfig</appParameter>
</appParameters>
----

For non-Spring applications, extend `AbstractWrapperListener` directly:

[source,java]
----
package com.example.service.wrapper;

import org.smallmind.spark.tanukisoft.integration.AbstractWrapperListener;

public class PlainWrapperListener extends AbstractWrapperListener {

  private MyService service;

  @Override
  public void startup (String[] args)                                                <1>
    throws Exception {

    service = new MyService();
    service.start();
  }

  @Override
  public void shutdown ()                                                            <2>
    throws Exception {

    if (service != null) {
      service.stop();
    }
  }
}
----
<1> Called on the wrapper's start thread. The keep-alive daemon is
    already running — take as long as you need, up to 180 seconds
    (`FAIL_SAFE_TIMEOUT_SECONDS`).
<2> Called on the wrapper's stop thread when the OS or an operator
    asks for shutdown.

=== Composing Singularity And Tanukisoft

The two subprojects can be stacked in one of two shapes.

*Stack A — one Maven module per shape.* Build `my-service-jar` with
`<packaging>singularity</packaging>`, then a sibling `my-service-app`
with `<packaging>tanukisoft</packaging>` that takes `my-service-jar`
as its only runtime dependency. The wrapper's `lib/` contains exactly
one jar — the Singularity jar — and the `mainClass` /
`wrapperListener` are both inside it.

*Stack B — plain Tanuki, no Singularity.* For the common case where
the application has ordinary dependency management, the plain
`tanukisoft` packaging already gives you a complete runnable
distribution. No need to add Singularity on top.

Stack A is worth the extra module only if the Singularity jar is
itself a distribution surface — for example, the same jar is also
handed to developers for local `java -jar` use, or the jar is signed
and that signature must survive end-to-end.

[[behavioral-reference]]
== Behavioral Reference

=== `SingularityEntryPoint` Bootstrap Order

`SingularityEntryPoint.main`:

. Reads the enclosing jar through
  `SingularityEntryPoint.class.getProtectionDomain().getCodeSource().getLocation()`.
. Opens that URL as a `JarInputStream`, advances to the manifest, and
  reads the manifest in the same pass.
. *Constructs* `SingularityClassLoader(null, manifest, codeSource,
  jarInputStream)` — which (a) triggers the static registration of
  `SingularityJarURLStreamHandlerFactory` and (b) advances the stream
  further until it hits
  `META-INF/singularity/index/singularity.idx`, deserializes the
  index, and closes the stream.
. Sets the new loader as `Thread.currentThread().getContextClassLoader()`.
  The *system* loader remains the loader defined by the JVM on
  startup — only the context loader is replaced.
. Reads `Singularity-Class` from the manifest.
. If the value equals `SingularityEntryPoint.class.getName()`, does
  nothing (the recursive case). Otherwise, loads the class *through
  the context loader* and reflectively invokes its
  `main(String[])`.

Side effects:

* `URL.setURLStreamHandlerFactory` is called once, from the static
  initializer of `SingularityClassLoader`. The JVM permits this
  exactly once per process; if another library has already
  registered a factory, bundle construction throws an `Error`.
* `ClassLoader.registerAsParallelCapable()` is called at the same
  time so `loadClass` can run in parallel without a global lock.

=== Class Loading Parent Delegation

`loadClass(name, resolve)` in `SingularityClassLoader` is:

[source,text]
----
if (already loaded)
  return cached
try findClass            // bundled first
except ClassNotFoundException:
  if parent != null:
    return parent.loadClass(name)
  else:
    return findSystemClass(name)
----

The conventional parent-first order is deliberately inverted: a
Singularity bundle may contain classes in packages that also exist on
the system classpath (because the JVM was launched with
`-classpath …:the-bundle.jar` and some transitively reachable
classpath entry duplicates a bundled name). Checking the bundle first
avoids silently preferring a system copy when the bundle has a newer
definition.

Classes in the reserved JDK namespaces (`jakarta.xml.`, `org.xml.`,
`org.w3c.`) bypass this loop and go straight to the parent, so the
JVM's own XML stack is always used even if a bundled library
accidentally ships those packages.

=== `singularity:` URL Form

[source,text]
----
singularity:<outer-jar-url>@/[<inner-jar-path>!/]<entry-name>
----

Parts:

* `<outer-jar-url>` — external form of the outer jar's URL, typically
  `file:/path/to/my-service-1.0.0.jar`.
* `@/` — separator that replaces the `!/` of the standard `jar:`
  protocol to avoid a three-level `jar:jar:` nesting.
* `<inner-jar-path>` (optional) — path of a nested jar under
  `META-INF/singularity/lib/`. Absent when the entry lives directly
  in the outer jar.
* `!/` — standard separator between jar location and intra-jar entry.
* `<entry-name>` — the entry's path inside whichever jar contains it.

=== Cached Nested Jars

`SingularityJarURLConnection` maintains a `Map<String, SoftReference<CachedJarFile>>`
keyed by inner-jar entry name. On every `getInputStream`:

. If the map has a live reference, that `CachedJarFile` is reused.
. Otherwise, the inner jar is opened through a `jar:` URL that points
  at it inside the outer jar, drained into a new `CachedJarFile`
  (which holds every entry's compressed bytes in memory), and stored
  back into the map.

Soft references mean the JVM can reclaim a cached jar under memory
pressure; the next access simply re-drains it. There is no explicit
eviction policy and no size cap.

[[startup-timeout-and-keep-alive]]
=== Startup Timeout And Keep-Alive

`AbstractWrapperListener.start(args)`:

. Reads `args[0]` as the wrapper's startup timeout in seconds (the
  value was prepended by `main` after consulting
  `WrapperManager.getProperties().getProperty("wrapper.startup.timeout")`
  and defaulting to 30).
. Creates a `CountDownLatch` with count 1 and a `SignalWorker`
  configured with that timeout.
. Runs `SignalWorker` as a daemon thread, then invokes
  `startup(trimmedArgs)` on the current thread.
. In a `finally`, counts the latch down so `SignalWorker` can exit.

`SignalWorker.run`:

[source,text]
----
startMillis = now
while (now - startMillis < 180_000
    && !completedLatch.await(startupTimeoutSeconds - 2, SECONDS)):
  WrapperManager.signalStarting(startupTimeoutSeconds)
----

The invariant: the wrapper's watchdog grants `startupTimeoutSeconds`
of patience on each `signalStarting` call, so as long as the
application signals at least once per `startupTimeoutSeconds - 2`
seconds, the wrapper does not kill the JVM. `SignalWorker` therefore
races `startup` completion against the wake-up interval, keeping the
wrapper patient until startup returns.

Tuning hints:

* Set `wrapper.startup.timeout` in `wrapper.conf` (or pass it as a
  `WrapperManager` property from elsewhere) when your application
  routinely takes more than 30 seconds to reach "ready". Values
  below 10 are rejected.
* The 180-second fail-safe ceiling (`FAIL_SAFE_TIMEOUT_SECONDS`) is
  a hard stop — `SignalWorker` exits even if `startup` is still
  running — so very-long-startup applications should invest in
  eager module loading or bean warmup before `startup` returns.

=== Exit Codes

`AbstractWrapperListener` converts exceptions to exit codes as
follows:

`null` (from `start`)::
    Startup succeeded; the wrapper keeps running.

`2` (from `start` or `stop`)::
    `STACK_TRACE_ERROR_CODE`. `startup` or `shutdown` threw; the
    exception's stack trace was printed. The wrapper interprets a
    non-`null` return from `start` as "startup failed" and stops
    the JVM.

`exitCode` (from `stop`, when `shutdown` returned normally)::
    The wrapper's requested exit code is passed through unchanged.

=== Control-Event Handling Without The Native Wrapper

`WrapperManager.isControlledByNativeWrapper()` reports `false` when
the application is launched without the `wrapper` executable (for
example, by running `java … AbstractWrapperListener
com.example.Listener` directly during development). In that mode:

* `controlEvent(WrapperManager.WRAPPER_CTRL_C_EVENT)` → `WrapperManager.stop(0)`
* `controlEvent(WrapperManager.WRAPPER_CTRL_CLOSE_EVENT)` → `WrapperManager.stop(0)`
* `controlEvent(WrapperManager.WRAPPER_CTRL_SHUTDOWN_EVENT)` → `WrapperManager.stop(0)`

When the native wrapper *is* in charge, these events are intercepted
by the wrapper itself; the Java layer does nothing.

=== Maven Artifact Attachment

`generate-singularity` calls
`project.addAttachedArtifact(applicationArtifact)` with a freshly
created artifact of packaging type `jar`. The project's *primary*
artifact is therefore the generated Singularity jar — the
`singularity` packaging handler reports `extension=jar`, so there is
no type mismatch.

`generate-wrapper` does *not* attach the aggregate archive itself —
`install-wrapper` and `deploy-wrapper` locate it by filename in
`target/` and install / deploy it with classifier `app`. The
classifier ensures the aggregate does not collide with the primary
`.jar` artifact from `maven-jar-plugin:jar` that ran earlier in the
lifecycle.

=== OS Style And Native Library Naming

`generate-wrapper` copies the platform-specific native library to
`lib/` twice:

. Under its platform-specific name (`libwrapper-linux-x86-64.so`,
  `wrapper-windows-x86-64.dll`, etc.) so maintenance scripts and
  `file` command inspection can identify it.
. Under the *generic* `OSStyle` name (`libwrapper.so` or
  `wrapper.dll`) so the wrapper's own default lookup finds it
  without platform-aware configuration.

Both copies point at the same bundled resource. The duplication
costs a few hundred KB on disk and simplifies `wrapper.conf`.

[[operational-notes]]
== Operational Notes

=== Dependencies Pulled In

`spark-singularity-boot`::
    `smallmind-nutsnbolts` only.

`spark-singularity-maven-plugin`::
    `smallmind-nutsnbolts`, `spark-singularity-boot`,
    `maven-plugin-api`, `maven-core`, `maven-model`,
    `maven-artifact`, `maven-compat`, and
    `maven-plugin-annotations`. The Maven artifacts are `provided`
    or `compile` — nothing unusual for a Maven plugin.

`spark-tanukisoft-integration`::
    `smallmind-nutsnbolts`, `spring-context`, and
    `tanukisoft:wrapper` (the Java Service Wrapper's own Java API
    jar). `spring-context` is a hard dependency because
    `SpringContextWrapperListener` is part of the artifact; a pure
    non-Spring user still needs it on the classpath, but only the
    `AbstractWrapperListener` class is referenced.

`spark-tanukisoft-maven-plugin`::
    `freemarker` (for `wrapper.conf` rendering), plus the same Maven
    provided/compile matrix.

=== Build-Time Failure Signatures

"Unable to locate the boot class dependencies"::
    `generate-singularity` walks `pluginArtifacts` looking for
    `org.smallmind:spark-singularity-boot`. The plugin's POM has
    it as a compile-scope dependency, so under normal circumstances
    it is always present. This error means the plugin was installed
    with a modified POM, or Maven's dependency resolution stripped
    the boot artifact.

"Missing singularity index"::
    `SingularityClassLoader`'s constructor advanced through every
    entry of the outer jar and did not find
    `META-INF/singularity/index/singularity.idx`. Either the jar is
    not actually a Singularity bundle (wrong `Main-Class`, wrong
    layout) or the index-writing step in `generate-singularity`
    was interrupted.

"Unknown operating system type(…)"::
    `generate-wrapper` could not parse the `operatingSystem`
    parameter as an `OSType` constant. Valid names are listed in
    `OSType.values()` — underscores and hyphens are interchangeable,
    case-insensitive.

"Unknown compression type(…)"::
    Same as above for `compression` vs `CompressionType`.

"Unknown os style(…)"::
    The plugin derived an unrecognized `OSStyle` from a validated
    `OSType`. This is an internal consistency check; an `OSType`
    constant whose `OSStyle` is neither `WINDOWS` nor `UNIX` would
    trigger it, but no such constant exists, so seeing this error
    means the enum has been modified.

"Unable to load template(…)"::
    `processFreemarkerTemplate` could not resolve a bundled
    Freemarker resource. The template resource path is built from
    `GenerateWrapperMojo.class.getPackage().getName()` plus `bin/`
    or `conf/`; a plugin jar that has been repackaged without the
    `installation/` resource root will fail here.

=== Runtime Failure Signatures

`NoClassDefFoundError` on a class that *is* inside a bundled library::
    Almost always one of two things:
+
* The class's package is in the reserved JDK namespaces
  (`jakarta.xml.`, `org.xml.`, `org.w3c.`) and the loader refused to
  define it. Add it to an allow-list if genuinely necessary (requires
  modifying `SingularityClassLoader.OPERABLE_NAMESPACES`), or prefer
  the JDK's own implementation.
* The class is loaded by a framework that captured the system class
  loader before `SingularityEntryPoint.main` replaced the context
  class loader. JNI libraries and some JDBC drivers cache loaders
  early; make sure the offending framework is initialized from
  within the application's `main`, not from a static initializer
  that runs before it.

`StackOverflowError` during static initialization of
`SingularityClassLoader`::
    Another library already installed a `URLStreamHandlerFactory`.
    The JVM blocks a second installation, and if the first one
    delegates to the platform default for unknown protocols, a
    Singularity-built jar running on top of an app that also wants
    to own the URL protocol must be reconciled manually — register
    the Singularity handler into the pre-existing factory.

`WrapperManager` errors ("WrapperStartException", "not initialized"):
    `AbstractWrapperListener.main` was invoked outside a wrapper
    process, or the wrapper process was started without
    `wrapper.conf` present. During development, run
    `./bin/<app>.sh console` to exercise the full wrapper path;
    running the jar directly with `java -jar …` puts the listener
    into "unwrapped" mode where Ctrl-C triggers
    `WrapperManager.stop(0)` but most wrapper features are inert.

=== Tanuki License Management

The Tanuki wrapper requires a license for production use. The plugin
has no opinion about where the license comes from — it just copies
whatever path `licenseFile` names into `conf/`. Common patterns:

* Keep the license under `~/.m2/wrapper-license.conf` on every
  build host; pass `<licenseFile>${user.home}/.m2/wrapper-license.conf</licenseFile>`.
* Keep the license in a `secret-resources` submodule that the build
  system unpacks on the CI runner; pass a repo-relative path.
* Leave `licenseFile` unset during development builds (the wrapper
  still runs in evaluation mode) and set it only in the release
  profile.

=== Running Unwrapped

`AbstractWrapperListener.main` can be invoked with the wrapper binary
not in the loop:

[source,shell]
----
java -cp … org.smallmind.spark.tanukisoft.integration.AbstractWrapperListener \
    com.example.MyListener arg1 arg2
----

In that mode:

* Property expansion (`${…}`) on `args` still runs.
* The keep-alive loop still runs — `WrapperManager.signalStarting`
  just becomes a no-op when `isControlledByNativeWrapper()` is
  `false`.
* Ctrl-C reaches `controlEvent` and triggers
  `WrapperManager.stop(0)`, which unwinds through the listener's
  `stop` callback and calls `shutdown`.

Use this mode during development; use the real wrapper binary for
anything that looks like production.

[[limitations-and-known-issues]]
== Limitations And Known Issues

One `URLStreamHandlerFactory` per JVM::
    `SingularityClassLoader`'s static initializer installs the
    `singularity:` handler by calling
    `URL.setURLStreamHandlerFactory`. The JVM permits exactly one
    such installation; any subsequent call — including, recursively,
    a second Singularity jar loaded into the same JVM — throws an
    `Error`. This is rarely a problem because a Singularity jar is
    typically the *only* launch artifact, but embedding Singularity
    inside a host that already owns URL protocols needs care.

Reserved-namespace list is fixed::
    The hard-coded `INOPERABLE_NAMESPACES` /
    `OPERABLE_NAMESPACES` arrays in `SingularityClassLoader` cannot
    be configured at build or boot time. A bundled library whose
    platform-shadowed class *must* come from the bundle (a
    newer `org.xml.sax`, say) cannot be accommodated without a
    code change.

No module-path support::
    `SingularityClassLoader` is a classic flat `ClassLoader` — it
    does not understand Java modules. The TODO comment in
    `findClass(String moduleName, String name)` records this. Jars
    that rely on automatic-module names still work (modules are a
    runtime concern layered on top of classpath loading), but a
    module-path-oriented application using explicit `requires`
    declarations inside a Singularity bundle is not supported.

Exclusions and bundled-dependencies match exactly::
    `Exclusion` (singularity) and `Dependency` (tanukisoft) both
    compare `groupId` and `artifactId` as exact strings. There is
    no glob, no version filter, and no scope filter. For most
    projects this is sufficient; large multi-module builds that
    want to exclude whole dependency groups must enumerate each
    artifact.

Single `URLStreamHandler` chain::
    A library that installs its own
    `URLStreamHandler` for a non-`singularity` protocol *after*
    `SingularityClassLoader`'s static initializer has run will be
    dispatched correctly, but any library that registers *before*
    the class loader is first touched must be compatible with the
    JVM default-handler pattern, because Singularity's factory
    returns `null` for non-`singularity` protocols and the JVM
    falls back to its built-in search.

No in-flight reloading::
    `SingularityClassLoader` reads the index once in its
    constructor and never re-reads it. Swapping the bundle file
    while the JVM is running has no effect until the process is
    restarted.

Wrapper generation is single-platform per build::
    `generate-wrapper` takes exactly one `operatingSystem` value per
    execution. Producing a multi-platform distribution requires
    multiple plugin executions (or multiple Maven modules). The
    example POM uses `<os>` activation to pick the right value for
    whichever host runs the build — which is the normal pattern,
    but means a single build host cannot produce every platform's
    wrapper.

Upstart and systemd are mutually exclusive::
    Both `useUpstart` and `useSystemD` feed the same Freemarker
    template. Setting both `true` in one build is not a supported
    configuration; pick one per distribution.

Tanuki wrapper binaries are bundled, not downloaded::
    Every supported platform's wrapper binary lives inside
    `spark-tanukisoft-maven-plugin` as a resource. The plugin jar is
    therefore large (~25 MB). Adding a new platform means adding a
    binary *and* an `OSType` constant, which requires rebuilding
    the plugin — there is no runtime extension point.

Build directory reuse is not self-cleaning::
    `generate-singularity` and `generate-wrapper` both rely on
    Maven's `clean` to remove stale contents. Incremental builds
    that do *not* clean can carry stale library jars or index
    entries forward, producing a bundle with ghost entries. Prefer
    `mvn clean package` over `mvn package` for release builds.

`AbstractWrapperListener.main` is a static entry point, not a bean::
    The listener is reflectively constructed via
    `Class.forName(...).getConstructor().newInstance()`, so every
    concrete listener class must expose a public no-arg constructor.
    Dependencies have to be acquired inside `startup` (e.g. through
    the Spring context constructed there), not through constructor
    injection.

Singularity jars are not hermetic::
    A Singularity jar includes its bundled dependencies, but the
    JVM still resolves platform classes from the JDK. A bundle
    compiled against JDK 21 will fail at runtime on JDK 17 the same
    way a normal jar would. Plan JDK version management at the
    deployment layer, not at the bundle layer.
