== The Problem This Solves

Most Java services end up with three layers that look almost, but not quite,
the same:

. a domain entity ã the shape the database or the business logic uses;
. one or more inbound request DTOs ã the shapes a client is allowed to send
  for *create*, *update*, *patch*, or *admin* operations;
. one or more outbound response DTOs ã the shapes the server is willing to
  reveal for *public*, *internal*, or *admin* callers.

Each pair of (direction, purpose) typically wants a different set of
properties, different `@NotNull`/`@Size`/`@Email` constraints, and sometimes a
different JSON shape. Writing these DTOs by hand is tedious and error-prone;
writing Jackson `@JsonView` classes is better but still requires hand-rolled
types, and it filters at runtime instead of compile time.

**Doppelganger** generates these DTOs. You annotate a single domain class;
the annotation processor emits one class per (direction, purpose) combination
you asked for, complete with JAXB/Jackson annotations, Bean Validation
constraints, optional polymorphic adapters, and optional `Serializable`
implementation. The generated classes are plain POJOs ã the `@Doppelganger`
annotations themselves are not present on them, so there is no hidden
runtime machinery to diagnose.

The module is the annotation-processor/code-generator counterpart to the rest
of `smallmind-web`: `web-jersey` serialises the generated views via
`JsonCodec` from `web-json-scaffold`; `web-grizzly` or `web-jetty` host the
endpoints that consume and emit them. See xref:WEB.adoc[WEB.adoc] for the
containers and the Jersey layer.

A "doppelganger" (German, "double-goer") is a lookalike of a person. Each
generated view is structurally similar to the entity it was cut from, but
shows only the properties appropriate to its direction and purpose.

=== Why not Lombok, MapStruct, `@JsonView`, Immutables?

`@JsonView`::
    Filters fields at runtime on a single class. It cannot enforce constraints
    that vary by view, cannot carry different `@XmlElement` names per view,
    and offers no compile-time signal that you handed an admin DTO back from
    a public endpoint.

Lombok::
    Generates getters/setters/builders/equals/hashCode on one class at a
    time. It has no concept of direction or purpose ã everything is on the
    one annotated type.

MapStruct::
    Maps between two types you have already defined. It cannot define the
    target type, and it knows nothing about which fields are appropriate for
    which purpose.

Immutables::
    Generates a matching immutable class next to an abstract type. Excellent
    for value objects, but not direction-aware and not purpose-aware.

Doppelganger owns the target classes. That is the whole point ã a
create-request DTO should not have an `id`; an admin-response DTO should
have `roles`; neither of those should be a runtime assertion.

== At A Glance

[cols="1,3",options="header"]
|===
| Concept | Meaning

| *Direction*
| `IN` (deserialise from client) or `OUT` (serialise to client). Generated
  class names are suffixed with `In` or `Out`.

| *Purpose*
| A caller-assigned label such as `"create"`, `"update"`, `"admin"`, or the
  default (unnamed) purpose. Generated class names carry the capitalised
  purpose between the base name and the direction suffix.

| *Idiom*
| A tuple of `(purposes, visibility, constraints, required)` that describes
  how a property participates in one or more purposes. A property can
  declare multiple idioms.

| *Visibility*
| `IN`, `OUT`, or `BOTH`. Narrows an idiom to one direction.

| *Real*
| A view property backed by a field on the annotated entity. Declared via
  `@Real` inside `@Doppelganger`.

| *Virtual*
| A view property with no backing entity field ã purely synthetic.

| *Pledge*
| A promise that a view for a given `(purposes, visibility)` must be
  generated even when no property currently forces it to exist. Makes empty
  views addressable by name.

| *Polymorphic hierarchy*
| A base class whose subclasses are tagged with a JSON/XML discriminator.
  Handled via `@Polymorphic`.

| *Plain hierarchy*
| A base class whose subclasses are exposed through generic type parameters
  on the generated view rather than through a discriminator adapter.
  Handled via `@Hierarchy`.
|===

== Installation

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>web-json-doppelganger</artifactId>
  <version>7.0.0-SNAPSHOT</version>
</dependency>
----

The processor is registered as a Java SPI service (via Google Auto Service)
at `META-INF/services/javax.annotation.processing.Processor`, so `javac`
discovers it automatically as soon as the artifact is on the annotation
processor path. For Maven, that means adding the dependency to the module
that contains the annotated domain classes; no
`maven-compiler-plugin` `<annotationProcessorPaths>` entry is required if the
artifact is already on the compile classpath.

Generated sources land under `target/generated-sources/annotations/` in the
same package as the annotated entity. The generated classes depend at
runtime on:

* `org.smallmind:web-json-scaffold` ã supplies `JsonCodec`, `@Comment`,
  `@NullifiedBy`, `Fault`, polymorphic adapter base classes, `Page`;
* `jakarta.xml.bind:jakarta.xml.bind-api` ã JAXB annotations;
* `jakarta.validation:jakarta.validation-api` ã the constraint annotations
  you chose to apply (for example, `@NotNull`, `@Size`, `@Email`);
* `tools.jackson.core:jackson-databind` ã Jackson serialisation.

The Doppelganger module itself does not need to be on the runtime classpath.
Declaring it with `<scope>provided</scope>` is acceptable when you want to
keep compile-time tooling out of deployed artefacts.

== Quick Start

=== Annotate a domain class

[source,java]
----
package com.example.domain;

import org.smallmind.web.json.doppelganger.*;
import jakarta.validation.constraints.NotNull;
import jakarta.validation.constraints.Email;
import jakarta.validation.constraints.Size;

@Doppelganger(
  real = {
    @Real(field = "id", type = @Type(Long.class)),                       // <1>
    @Real(field = "username", type = @Type(String.class),
          idioms = {
            @Idiom(purposes = "create",
                   constraints = {
                     @Constraint(value = NotNull.class),
                     @Constraint(value = Size.class,
                                 arguments = "min = 1, max = 64")
                   },
                   required = true),
            @Idiom(purposes = {"create", "admin"}, visibility = Visibility.OUT)
          }),
    @Real(field = "password", type = @Type(String.class),                // <2>
          idioms = @Idiom(purposes = "create",
                          visibility = Visibility.IN,
                          constraints = @Constraint(value = NotNull.class),
                          required = true)),
    @Real(field = "email", type = @Type(String.class),
          idioms = @Idiom(purposes = "create",
                          constraints = @Constraint(value = Email.class)))
  },
  serializable = true
)
public class User {
  private Long id;
  private String username;
  private String password;
  private String email;
  // getters/setters
}
----
<1> `id` participates only in the default purpose; no idioms means it is
    visible both inbound and outbound in views without a named purpose.
<2> `password` is only ever accepted inbound on `create`; it never appears
    outbound in any purpose.

=== Compile

`mvn compile` runs `javac` with the Doppelganger processor on the
annotation-processor path. The processor emits:

[source,text]
----
target/generated-sources/annotations/com/example/domain/UserIn.java
target/generated-sources/annotations/com/example/domain/UserOut.java
target/generated-sources/annotations/com/example/domain/UserCreateIn.java
target/generated-sources/annotations/com/example/domain/UserCreateOut.java
target/generated-sources/annotations/com/example/domain/UserAdminOut.java
----

=== Use the generated views

[source,java]
----
@Path("/users")
public class UserResource {

  @POST
  public UserCreateOut create(@Valid UserCreateIn in) {
    User user = userService.create(in.getUsername(),
                                   in.getPassword(),
                                   in.getEmail());
    return new UserCreateOut(user.getId(), user.getUsername());
  }

  @GET
  @Path("/{id}")
  public UserOut read(@PathParam("id") Long id) {
    return toOut(userService.find(id));
  }

  @GET
  @Path("/{id}/admin")
  public UserAdminOut readAdmin(@PathParam("id") Long id) {
    return toAdminOut(userService.find(id));
  }
}
----

The generated classes are plain JAXB/Jackson POJOs. They serialise with
`JsonCodec.writeAsBytes(...)` and deserialise with
`JsonCodec.readAsObject(..., UserCreateIn.class)` without further wiring.

== Conceptual Model

=== Directions, purposes, idioms

The three axes Doppelganger reasons about are orthogonal.

Direction::
    Always exactly `IN` or `OUT`. A class generated in direction `IN` is for
    deserialising client input; a class generated in direction `OUT` is for
    serialising server output.

Purpose::
    A caller-chosen string identifier. The *default purpose* is the one you
    get when no `purposes` are listed on an `@Idiom`. Named purposes
    (`"create"`, `"update"`, `"admin"`, ãÎ) select a distinct generated
    class per direction.

Idiom::
    An `@Idiom` attaches a set of *constraints* and a *required* flag to a
    property for a given `(purposes, visibility)` tuple. A single property
    can declare any number of idioms. If a property declares no idioms at
    all, it participates in the default idiom with visibility `BOTH`.

For every `(direction, purpose)` tuple the processor can reach through the
annotated class, it generates one view class. The class contains the
union of the properties whose idioms cover that tuple, with the constraint
annotations that the matching idioms emit.

=== Visibility composition

Two visibilities compose by rule:

* if they are equal, the result is the common value;
* if they differ and neither is `null`, the result is `BOTH`.

`Visibility.matches(Direction)` is what the processor uses to decide whether
a property appears in a view for a given direction. A field-level `@View`
without idioms is, for visibility purposes, `BOTH` ã unless a
class-level `constrainingIdioms` entry on `@Doppelganger` narrows it.

=== Real vs. virtual vs. `@View`

Three equivalent ways to declare a view property exist:

Field-level `@View`::
    Placed on a field or accessor method inside the domain class. Useful when
    the domain class itself is what you want to annotate. Retention is
    `RetentionPolicy.CLASS` specifically so annotations on fields in
    other modules are still visible to the processor.

Class-level `@Real`::
    Placed inside `@Doppelganger.real`. References a real field of the
    entity by name and type. Preferred when you want the view configuration
    to live with the view declaration rather than sprinkled through the
    entity.

Class-level `@Virtual`::
    Placed inside `@Doppelganger.virtual`. Declares a property that has no
    backing entity field. The generated view holds the field and
    accessors, but there is no translation to and from the entity for this
    property ã the caller is expected to populate it explicitly.

=== Pledges

By default, the processor only generates a view class for a
`(direction, purpose)` tuple if at least one property participates. A
`@Pledge` forces generation even when no properties would otherwise exist.
Pledges are the way to publish an intentionally-empty DTO ã useful for
API versioning or for endpoints whose only request body is "do it".

=== Polymorphic vs. plain hierarchies

`@Polymorphic`::
    The generated views carry JAXB/Jackson type information so a polymorphic
    payload can be deserialised to the right concrete view. `subClasses`
    lists the leaf subtypes; `useAttribute` controls whether the
    discriminator is an XML attribute or an element (JSON sees it as a
    property either way).

`@Hierarchy`::
    The generated views use generic type parameters instead of a
    discriminator adapter. Appropriate when a parent view should expose a
    subtype without the payload needing to identify itself.

== API Reference

=== `@Doppelganger` (class-level, runtime retention)

[cols="1,1,1,3",options="header"]
|===
| Element | Type | Default | Role

| `virtual`             | `Virtual[]`       | `{}`              | Synthetic properties present on every generated view.
| `real`                | `Real[]`          | `{}`              | Properties backed by entity fields.
| `pledges`             | `Pledge[]`        | `{}`              | Guarantee generation of specific `(purposes, visibility)` views.
| `polymorphic`         | `Polymorphic`     | `@Polymorphic()`  | Polymorphic hierarchy configuration.
| `hierarchy`           | `Hierarchy`       | `@Hierarchy()`    | Plain hierarchy configuration.
| `constrainingIdioms`  | `Idiom[]`         | `{}`              | Class-level idioms applied to generated views for matching purposes/directions.
| `implementations`     | `Implementation[]`| `{}`              | Additional interfaces on every generated view.
| `imports`             | `Import[]`        | `{}`              | Additional `import` statements in the generated source.
| `name`                | `String`          | `""`              | XML root-element name; defaults to the decapitalised class name.
| `namespace`           | `String`          | `"http://org.smallmind/web/json/doppelganger"` | XML namespace.
| `serializable`        | `boolean`         | `false`           | If `true`, generated views implement `java.io.Serializable`.
| `comment`             | `String`          | `""`              | Text embedded in `@Comment` on every view.
|===

=== `@View` (field or method, CLASS retention)

[cols="1,1,1,3",options="header"]
|===
| Element | Type | Default | Role

| `idioms`   | `Idiom[]`                    | `{}`              | Idiom overrides for this property; empty means the default idiom only.
| `adapter`  | `Class<? extends XmlAdapter>`| `NullXmlAdapter.class` | JAXB adapter for the property.
| `as`       | `Class<?>`                   | `Void.class`      | Type hint for tooling; becomes an `@As` annotation on the generated field.
| `name`     | `String`                     | `""`              | XML element name; defaults to the field name.
| `required` | `boolean`                    | `false`           | Default-idiom required flag; overridable per idiom.
| `comment`  | `String`                     | `""`              | Text embedded in `@Comment` on the generated property.
|===

`@View`'s retention is deliberately `CLASS`, not `SOURCE`, so annotations on
fields of a class declared in a different Maven module remain visible to the
processor when that class is referenced.

=== `@Real` (embedded in `@Doppelganger.real`)

Same shape as `@View` plus:

[cols="1,1,1,3",options="header"]
|===
| Element | Type | Default | Role

| `field` | `String` | *required* | The name of the entity field this view property reflects.
| `type`  | `@Type`  | *required* | The field's raw class and (optional) generic parameters.
|===

=== `@Virtual` (embedded in `@Doppelganger.virtual`)

Same shape as `@Real` but the field does not have to exist on the entity.
`field` names the logical property on the generated view; `type` supplies
the declared type.

=== `@Idiom` (embedded)

[cols="1,1,1,3",options="header"]
|===
| Element | Type | Default | Role

| `constraints`| `Constraint[]`| `{}`            | Constraints to apply within this idiom.
| `visibility` | `Visibility`  | `Visibility.BOTH`| Which directions this idiom covers.
| `purposes`   | `String[]`    | `{}`            | Purpose names this idiom covers; empty means the default purpose.
| `required`   | `boolean`     | `false`         | Whether the property is required within this idiom.
|===

=== `@Constraint` (embedded)

[cols="1,1,1,3",options="header"]
|===
| Element | Type | Default | Role

| `value`     | `Class<?>` | *required* | The constraint annotation class (for example, `NotNull.class`, `Size.class`).
| `arguments` | `String`   | `""`       | Raw text placed inside the annotation's parentheses.
|===

The processor embeds the constraint as-is:

[source,java]
----
@Constraint(value = Size.class, arguments = "min = 1, max = 255")
// ã  @Size(min = 1, max = 255)
----

=== `@Pledge` (embedded)

[cols="1,1,1,3",options="header"]
|===
| Element | Type | Default | Role

| `visibility`| `Visibility` | `Visibility.BOTH`| Directions to guarantee generation for.
| `purposes`  | `String[]`   | `{}`             | Purposes to guarantee generation for.
|===

=== `@Polymorphic` (embedded under `@Doppelganger.polymorphic`)

[cols="1,1,1,3",options="header"]
|===
| Element | Type | Default | Role

| `subClasses`   | `Class<?>[]` | `{}`   | Leaf subclasses for which to emit discriminator adapters.
| `useAttribute` | `boolean`    | `false`| When `true`, the XML discriminator is an attribute; when `false`, an element.
|===

=== `@Hierarchy` (embedded under `@Doppelganger.hierarchy`)

[cols="1,1,1,3",options="header"]
|===
| Element | Type | Default | Role

| `subClasses` | `Class<?>[]` | `{}` | Leaf subclasses exposed through generic type parameters on the generated view.
|===

=== `@Type` / `@Implementation` / `@Import`

`@Type` carries a raw class plus optional generic `parameters` for the
property's declared type. `@Implementation` adds an interface to every
generated view. `@Import` adds an `import` line to the generated source ã
useful when a constraint or adapter references a class whose fully-qualified
name would otherwise collide or clutter the generated code.

=== Direction and Visibility enums

`Direction`::
    `IN` (code `"In"`, visibility `IN`), `OUT` (code `"Out"`, visibility
    `OUT`). `getCode()` is the suffix appended to generated class names.

`Visibility`::
    `IN`, `OUT`, `BOTH`. `compose(other)` returns the common value when
    equal, otherwise `BOTH`; `matches(direction)` reports whether a given
    direction is covered.

== Generated Class Shape

For the `User` quick-start example above, `UserCreateIn` is a
JAXB/Jackson-annotated POJO whose fields are the union of properties whose
idioms include the `"create"` purpose with `IN` or `BOTH` visibility:

[source,java]
----
package com.example.domain;

import java.io.Serializable;
import jakarta.xml.bind.annotation.XmlElement;
import jakarta.xml.bind.annotation.XmlRootElement;
import jakarta.validation.constraints.NotNull;
import jakarta.validation.constraints.Size;
import jakarta.validation.constraints.Email;

@XmlRootElement(name = "user",
                namespace = "http://org.smallmind/web/json/doppelganger")
public final class UserCreateIn implements Serializable {

  @XmlElement(required = true)
  @NotNull
  @Size(min = 1, max = 64)
  private String username;

  @XmlElement(required = true)
  @NotNull
  private String password;

  @XmlElement
  @Email
  private String email;

  public UserCreateIn () { }
  public UserCreateIn (String username, String password, String email) {
    this.username = username;
    this.password = password;
    this.email = email;
  }

  public String getUsername () { return username; }
  public void setUsername (String username) { this.username = username; }

  public String getPassword () { return password; }
  public void setPassword (String password) { this.password = password; }

  public String getEmail () { return email; }
  public void setEmail (String email) { this.email = email; }
}
----

`UserCreateOut` is generated the same way but contains only the properties
whose `"create"` idiom includes `OUT` or `BOTH` visibility ã in this case
`username` (via its second idiom) and `id` (via the default idiom, which is
also emitted on the `create`-purpose outbound view because the `id`'s
idiom-free `@Real` declaration participates in every purpose).

`UserAdminOut` contains the properties whose `"admin"` idiom includes `OUT`
or `BOTH`. In this case that is only `username`.

The class name composition rule is:

[source,text]
----
{ClassName} + {CapitalisedPurpose?} + {In|Out}
----

Default purpose omits the purpose segment; named purposes produce distinct
class names per direction.

== Processing Pipeline

The following is what happens when `javac` encounters a
`@Doppelganger`-annotated class with the processor on the annotation
processor path.

. *Discovery.* `DoppelgangerAnnotationProcessor.process(...)` is invoked
  per compilation round with the annotated elements. Each annotated
  `TypeElement` is handed to `generate(...)`. A `ClassTracker` records
  already-processed classes to avoid duplicate work.
. *Validation.* The element must be a top-level, public class. Inner
  classes, interfaces, and enums are rejected with a `DefinitionException`.
. *Metadata extraction.* `DoppelgangerInformation` parses the annotation.
  Properties from `@Real`, `@Virtual`, and field-level `@View` are
  normalised into `PropertyInformation` plus `PropertyBox` entries. Each
  idiom is expanded into one `(purpose, visibility)` pair per purpose
  string.
. *Visibility tracking.* `VisibilityTracker` records every
  `(purpose, visibility)` combination that must produce a generated class,
  including those contributed by `@Pledge` entries.
. *Hierarchy resolution.* If the class is annotated with `@Polymorphic` or
  `@Hierarchy`, `ClassTracker` marks the subclasses so their own
  `@Doppelganger` processing takes the hierarchy into account.
. *Recursion.* If a property type is itself a `@Doppelganger`-annotated
  class, the processor recurses so the referenced views exist before the
  current class tries to reference them.
. *Emission.* For every `(direction, purpose)` combination the tracker
  recorded, `generate(...)` writes a Java source file through
  `processingEnv.getFiler().createSourceFile(...)`:
+
* class header with `@XmlRootElement`, Jackson `@JsonInclude` if warranted,
  and any `@Implementation` interfaces;
* a private field per property, with `@XmlElement(required=...)`,
  `@XmlJavaTypeAdapter` when an adapter is declared, `@As` when a type hint
  was supplied, and each constraint annotation's formatted text;
* no-arg and all-args constructors;
* getters and setters for every property.
. *Translators.* Where a property references another `@Doppelganger` type,
  `TranslatorFactory` chooses a `Translator` ã `NoopTranslator`,
  `ClassTranslator`, `ListTranslator`, or `ArrayTranslator` ã which emits
  the entityãview conversion snippets.
. *Polymorphic adapters.* For a `@Polymorphic` base class, a companion
  `XmlAdapter` is generated that routes discriminator values to the
  appropriate generated subclass.

Source emission is done with direct `BufferedWriter` output, not a
templating engine. The generated files are ordinary `.java` files that
`javac` compiles in the same invocation.

== Integration With JSON and XML

Generated views play by existing Jackson and JAXB rules.

* `JsonCodec` from `web-json-scaffold` is the project's configured
  `ObjectMapper` facade. It already registers
  `JakartaXmlBindAnnotationModule`, so `@XmlElement(required=...)` and
  related JAXB metadata are honoured during JSON (de)serialisation.
* Jackson sees each private field's JAXB annotations plus any
  `@JsonProperty` and friends you add through `@Implementation` or
  `@Import`.
* Bean Validation constraints (`@NotNull`, `@Size`, `@Email`, ãÎ) are
  enforced the same way they would be on any JAX-RS parameter annotated
  with `@Valid` ã the processor does not validate at generation time; it
  only places the annotations.

== Typical Usage Patterns

=== Splitting create and update

[source,java]
----
@Doppelganger(
  real = {
    @Real(field = "id", type = @Type(Long.class),
          idioms = @Idiom(purposes = "update",
                          visibility = Visibility.IN,
                          required = true)),
    @Real(field = "name", type = @Type(String.class),
          idioms = {
            @Idiom(purposes = {"create", "update"},
                   visibility = Visibility.IN,
                   constraints = @Constraint(value = NotNull.class),
                   required = true),
            @Idiom(purposes = {"create", "update"},
                   visibility = Visibility.OUT)
          }),
    @Real(field = "createdAt", type = @Type(Instant.class),
          idioms = @Idiom(purposes = {"create", "update"},
                          visibility = Visibility.OUT))
  }
)
public class Widget { /* ãÎ */ }
----

Yields `WidgetCreateIn`, `WidgetCreateOut`, `WidgetUpdateIn`, and
`WidgetUpdateOut` ã `id` present on `UpdateIn` but never on `CreateIn`;
`createdAt` present on both `Out` views but never accepted inbound.

=== A pledge for an empty "delete confirmation"

[source,java]
----
@Doppelganger(
  pledges = @Pledge(purposes = "deleteAck", visibility = Visibility.OUT)
)
public class Widget { /* ãÎ */ }
----

Generates `WidgetDeleteAckOut` even though no property participates. The
class is a named, deserialisable return type without having to invent one
by hand.

=== Polymorphic response

[source,java]
----
@Doppelganger(
  polymorphic = @Polymorphic(subClasses = {Cat.class, Dog.class},
                             useAttribute = false),
  real = @Real(field = "name", type = @Type(String.class))
)
public class Animal { /* ãÎ */ }

@Doppelganger(
  real = @Real(field = "purrHz", type = @Type(double.class))
)
public class Cat extends Animal { /* ãÎ */ }
----

`AnimalOut` emits a Jackson/JAXB discriminator-aware type, and an
`AnimalXmlAdapter` is generated so JAXB knows how to marshal between an
incoming discriminator value and the appropriate generated subclass view.

== Behavioral Reference

Directional defaults::
    A property with no `@Idiom` declarations participates in the default
    purpose with `Visibility.BOTH`. Both directions generate a class that
    includes it.

Purpose semantics::
    Purposes are entirely caller-chosen strings. The processor does not
    interpret `"create"` differently from `"fribble"`; it just uses the
    string for class naming and idiom matching.

Required merges::
    The `required` flag on `@View` and `@Real` applies in the default
    idiom. An explicit `@Idiom` can flip it on or off for a specific
    purpose/visibility.

Idiom expansion::
    When an idiom lists multiple purposes, the processor treats that as
    separate idioms with one purpose each. Constraints and the required
    flag apply in every expanded copy.

Recursive properties::
    If property `Foo.bar` has type `Baz` and `Baz` is also
    `@Doppelganger`-annotated, the generated `FooOut.bar` field will be
    typed as `BazOut` (and `FooCreateOut.bar` as `BazCreateOut` when a
    matching purpose exists). `ClassTracker` prevents infinite recursion on
    self- or mutually-referential types.

Generated modifiers::
    Generated view classes are `public final` unless a hierarchy is in
    play. Hierarchy-participating generated views are non-final so
    subclass views can extend them.

Constructors::
    Every generated view has a no-arg constructor (for JSON/JAXB
    deserialisation) and an all-args constructor in declared property
    order.

Name overrides::
    `@View.name` and `@Real.name` control the XML element name; the root
    element name for the class is `@Doppelganger.name` (defaulting to the
    decapitalised class name). Jackson respects the JAXB names through the
    configured `ObjectMapper`.

Comments::
    A non-empty `comment` on `@Doppelganger`, `@View`, or `@Real` becomes
    an `@Comment("...")` annotation on the generated class or field.
    `@Comment` lives in `web-json-scaffold` and carries no runtime
    behaviour; it is metadata for documentation tools.

== Error Handling

The processor reports issues through `processingEnv.getMessager()` at
`Diagnostic.Kind.ERROR` so `javac` fails the build. Common diagnostics:

"Class must be a root implementation of type 'class'"::
    `@Doppelganger` placed on an inner class, interface, or enum. Move it
    to a top-level class.

"The purpose(X) must be a legal identifier fragment"::
    Purpose strings become class-name segments; they must be valid Java
    identifier fragments (letters, digits, `_`; must not start with a
    digit).

"The field(name=X, purpose=Y, direction=Z) has already been processed"::
    Multiple `@Real`, `@Virtual`, or `@View` declarations target the same
    property under the same `(purpose, direction)`. Consolidate them.

`DefinitionException` at compile time::
    General structural error from the processor. The message names the
    offending element.

== Limitations and Known Issues

Generics on the annotated class::
    `@Doppelganger` is not supported on a generic type declaration such as
    `class Container<T>`. Generics on *properties* via `@Type.parameters`
    are supported.

No `equals`, `hashCode`, or `toString` generation::
    Generated views are mutable POJOs with only constructors and
    accessors. Equality by value is out of scope; if you need it, add an
    interface through `@Implementation` and implement the methods on your
    own.

No copy constructor or builder::
    Only no-arg and all-args constructors are emitted.

`serialVersionUID`::
    When `serializable = true`, the generated class implements
    `Serializable` but no explicit `serialVersionUID` is emitted.

Dynamic purposes::
    All purposes are compile-time strings baked into generated class
    names. A purpose cannot be decided at runtime.

Implicit hierarchy merging::
    If a superclass is `@Doppelganger`-annotated, the subclass inherits
    view properties. Overriding a superclass property in the subclass's
    view declaration is not a supported operation; redeclaring it with the
    same name and type is.

== Minimal Checklist

. `web-json-doppelganger` is on the compile classpath (or annotation
  processor path) of the module containing the domain classes.
. Every annotated class is `public` and top-level.
. Every `@Real.field` names an actual entity field; every `@Real.type`
  matches the field's declared type.
. Idiom purposes use valid identifier fragments.
. The runtime classpath has `web-json-scaffold`,
  `jakarta.xml.bind-api`, `jakarta.validation-api`, and the Jackson
  modules your views reference.
. The application's `ObjectMapper` is obtained through `JsonCodec`
  (or a Jackson mapper with the same JAXB and polymorphic module
  configuration).
