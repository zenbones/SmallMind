[[mongodb, MongoDB]]
= MongoDB

[partintro]
Smallmind MongoDB is split into two narrow modules:

* `mongodb-throng`: explicit entity mapping, lifecycle callbacks, index declarations, and small query/update builders over the MongoDB sync driver
* `mongodb-utility`: thin Spring factory beans and small helper types around `MongoClient` construction

The modules stay close to the driver and package the repetitive mapping, query, and client-construction seams into explicit APIs.

Audience: developers wiring MongoDB into applications or reusable libraries.
Versioning: examples use `${smallmind.version}` for Smallmind artifacts; keep related Smallmind modules on the same version unless you have a specific reason to split them.

[[mongodb-at-a-glance, At A Glance]]
== At A Glance

=== Integration Model In 60 Seconds

1. Use `mongodb-throng` when you want explicit object mapping plus a small CRUD/query layer over the sync driver.
2. Use `mongodb-utility` when you want optional Spring bean materialization around driver objects.
3. Mark every persisted field explicitly with `@Property` and mark exactly one field per entity with `@Id`.
4. Give every decoded `@Entity` and `@Embedded` type a public no-arg constructor.
5. Register every participating mapped class in the `ThrongClient` constructor. There is no classpath scanning.
6. Build filters, sorts, projections, updates, and compound indexes with stored MongoDB field names, not Java field names.
7. Remember that `update(...)` and `delete(...)` are `updateMany(...)` and `deleteMany(...)` wrappers.
8. Keep `MongoClient` lifecycle explicit unless Spring owns it through `MongoClientFactoryBean`.

[[mongodb-why-use, Why Use Smallmind MongoDB]]
== Why Use Smallmind MongoDB

Use Smallmind MongoDB when you want MongoDB integration to stay explicit and library-safe.

What these modules do well:

* keep mapping and query behavior close to the MongoDB sync driver
* make entity registration visible in `ThrongClient` construction instead of hiding it behind scanning
* keep Spring support limited to object construction and lifecycle handoff
* expose raw MongoDB field names and driver options instead of wrapping them in a larger framework vocabulary
* work well inside shared libraries that cannot assume one container or persistence stack

These modules are especially useful when you want driver-level visibility with repeatable mapping and optional Spring construction helpers.

[[mongodb-throng-vs-morphia, Throng Vs Higher-Level ODMs]]
== Throng Vs Higher-Level ODMs

Morphia is the obvious comparison point, and the same tradeoff applies to similar ODMs.
Throng keeps mapping, query building, and lifecycle closer to the MongoDB driver.

Choose Throng when you want:

* explicit mapped-type registration in constructor code
* field-level annotations without a broader repository abstraction
* query, sort, projection, and update builders that still look like MongoDB
* caller-owned runtime policy around connection lifecycle, startup, and configuration

Throng is especially strong in codebases that prefer explicit mapped types, visible query construction, and driver-level lifecycle control.

[cols="2,2,4",options="header"]
|===
|Need |Use |Notes

|Build a mapped client over the sync driver

[[mongodb-quick-start, Quick Start]]
== Quick Start

=== 1) Define Mapped Types

[source,java]
----
import java.util.Date;
import org.bson.types.ObjectId;
import org.smallmind.mongodb.throng.index.IndexType;
import org.smallmind.mongodb.throng.index.annotation.Index;
import org.smallmind.mongodb.throng.index.annotation.IndexOptions;
import org.smallmind.mongodb.throng.index.annotation.Indexed;
import org.smallmind.mongodb.throng.index.annotation.Indexes;
import org.smallmind.mongodb.throng.mapping.annotation.Embedded;
import org.smallmind.mongodb.throng.mapping.annotation.Entity;
import org.smallmind.mongodb.throng.mapping.annotation.Id;
import org.smallmind.mongodb.throng.mapping.annotation.Property;

@Embedded
public class Profile {

  @Property
  private String displayName;

  @Property
  private String timezone;

  public Profile () {
  }

  // getters and setters omitted for brevity
}

@Entity("users")
@Indexes(
    value = {
        @Index("user_name"),
        @Index(value = "createdAt", type = IndexType.DESCENDING)
    },
    options = @IndexOptions(name = "users_lookup")
)
public class User {

  @Id
  private ObjectId id;

  @Property("user_name")
  @Indexed(options = @IndexOptions(unique = true, name = "users_username"))
  private String username;

  @Property
  private String email;

  @Property
  private Profile profile;

  @Property
  private String[] roles;

  @Property
  private Date createdAt;

  public User () {
  }

  // getters and setters omitted for brevity
}
----

[IMPORTANT]
====
Only `@Property` fields and one `@Id` field are mapped. Private fields are fine. Public bean getters and setters are used when present, otherwise Throng reads and writes the fields directly.
====

=== 2) Build `ThrongClient`

[source,java]
----
import com.mongodb.client.MongoClient;
import com.mongodb.client.MongoClients;
import org.smallmind.mongodb.throng.ThrongClient;
import org.smallmind.mongodb.throng.ThrongOptions;

MongoClient mongoClient = MongoClients.create("mongodb://127.0.0.1:27017");

ThrongClient throngClient = new ThrongClient(
    mongoClient,
    "application",
    new ThrongOptions(false, true, true),
    User.class,
    Profile.class
);

// later, during application shutdown
mongoClient.close();
----

[IMPORTANT]
====
Register every participating mapped class in the constructor, not just the top-level entity types. If a mapped field refers to an `@Embedded` class, register that embedded class too.
====

=== 3) Insert, Query, Update, Count, And Delete

[source,java]
----
import java.util.Date;
import com.mongodb.client.model.DeleteOptions;
import com.mongodb.client.model.InsertOneOptions;
import com.mongodb.client.model.UpdateOptions;
import com.mongodb.client.result.InsertOneResult;
import org.smallmind.mongodb.throng.UpdateResult;
import org.smallmind.mongodb.throng.query.Filter;
import org.smallmind.mongodb.throng.query.Query;
import org.smallmind.mongodb.throng.query.Updates;

User user = new User();
user.setUsername("ada");
user.setEmail("ada@example.org");
Profile profile = new Profile();
profile.setDisplayName("Ada Lovelace");
profile.setTimezone("America/Los_Angeles");
user.setProfile(profile);
user.setRoles(new String[] {"admin", "author"});
user.setCreatedAt(new Date());

InsertOneResult insertOneResult = throngClient.insert(user, new InsertOneOptions());
if (user.getId() == null && insertOneResult.getInsertedId() != null) {
  user.setId(insertOneResult.getInsertedId().asObjectId().getValue());
}

User loadedUser = throngClient.findOne(
    User.class,
    Query.with().filter(Filter.where("user_name").eq("ada"))
);

UpdateResult updateResult = throngClient.update(
    User.class,
    Filter.where("user_name").eq("ada"),
    Updates.of()
        .set("profile.timezone", "UTC")
        .addToSet("roles", "reviewer"),
    new UpdateOptions()
);

long totalUsers = throngClient.count(User.class, Filter.empty());

long deletedUsers = throngClient.delete(
    User.class,
    Filter.where("user_name").eq("ada"),
    new DeleteOptions()
).getDeletedCount();
----

[IMPORTANT]
====
`Filter`, `Sort`, `Projections`, `Updates`, and compound `@Index` declarations all use stored MongoDB field names. In the example above, the correct field name is `user_name`, not `username`.
====

[[mongodb-throng-overview, Throng Overview]]
== Throng Overview

`ThrongClient` is a startup-time assembler plus a small runtime facade.
When you construct it, it:

* reads entity and embedded metadata from annotations
* resolves field codecs
* rejects invalid mappings early with `ThrongMappingException`
* optionally creates indexes for registered entity classes
* caches the codecs needed for `count`, `find`, `findOne`, `insert`, `update`, and `delete`

[cols="2,2,4",options="header"]
|===
|Method |Driver behavior |Notes

|`count(Class<T>, Filter)`
|`countDocuments(...)`
|Use `Filter.empty()` to count all documents

|`find(Class<T>, Query)`
|`find(...)`
|Returns `ThrongIterable<T>` for lazy iteration or `asList()` materialization

|`findOne(Class<T>, Query)`
|`find(...).limit(1).first()`
|Returns the first match or `null`

|`insert(T, InsertOneOptions)`
|`insertOne(...)`
|Current implementation ignores the `InsertOneOptions` argument and does not back-fill generated ids into the entity

|`update(Class<T>, Filter, Updates, UpdateOptions)`
|`updateMany(...)`
|Returns Smallmind `UpdateResult`

|`delete(Class<T>, Filter, DeleteOptions)`
|`deleteMany(...)`
|Returns the driver's `DeleteResult`
|===

[NOTE]
====
There is no `save(...)`, `replaceOne(...)`, or repository abstraction here. If you need driver behavior outside these focused helpers, use the MongoDB driver directly alongside Throng.
====

[[mongodb-mapping-annotations, Mapping Annotations]]
== Mapping Annotations

=== Entity And Embedded Types

[cols="2,2,4",options="header"]
|===
|Annotation |Target |Meaning

|`@Entity("collection")`
|type
|Marks a top-level mapped type and chooses its MongoDB collection

|`@Embedded`
|type
|Marks a type that can be embedded inside entities or other embedded types

|`@Id`
|field
|Marks the entity id field; the stored name defaults to `_id`

|`@Property`
|field
|Marks a persisted field and optionally overrides its stored name

|`@Codec`
|field
|Forces a specific BSON codec for that property

|`@Indexed`
|field
|Declares a single-field index for a mapped property

|`@Indexes`
|type
|Declares a compound index using one or more nested `@Index` elements
|===

Mapping rules that matter in practice:

* every `@Entity` must define exactly one `@Id`
* decoded `@Entity` and `@Embedded` types must expose a public no-arg constructor
* only `@Property` fields participate in persistence beyond the single `@Id` field
* inherited non-static, non-transient fields participate in mapping
* persisted property names must be unique across the mapped type
* `@Indexed` and `@Codec` are only collected from fields already marked with `@Property`

=== Field Mapping

Throng is field-driven, not getter-driven.
Public getters and setters are used when present, but they are optional.
A private field with `@Property` is enough for persistence.

Stored names are the names that matter outside the Java class.
If you rename a property in MongoDB with `@Property("user_name")`, every query and update string must use `user_name`.

[source,java]
----
@Entity("users")
public class User {

  @Id
  private ObjectId id;

  @Property("user_name")
  private String username;

  public User () {
  }
}

Filter.where("user_name").eq("ada");
Updates.of().set("user_name", "grace");
Sort.on().asc("user_name");
----

=== Custom Codecs

Use `@Codec` when the driver registry does not already know how to handle a field type.
The codec class must expose a public no-arg constructor because Throng instantiates it reflectively.

[source,java]
----
import org.bson.BsonReader;
import org.bson.BsonWriter;
import org.bson.codecs.Codec;
import org.bson.codecs.DecoderContext;
import org.bson.codecs.EncoderContext;
import org.bson.types.ObjectId;
import org.smallmind.mongodb.throng.mapping.annotation.Entity;
import org.smallmind.mongodb.throng.mapping.annotation.Id;
import org.smallmind.mongodb.throng.mapping.annotation.Property;

public class EmailAddress {

  private final String value;

  public EmailAddress (String value) {
    this.value = value;
  }

  @Override
  public String toString () {
    return value;
  }
}

public class EmailAddressCodec implements Codec<EmailAddress> {

  @Override
  public Class<EmailAddress> getEncoderClass () {
    return EmailAddress.class;
  }

  @Override
  public EmailAddress decode (BsonReader reader, DecoderContext decoderContext) {
    return new EmailAddress(reader.readString());
  }

  @Override
  public void encode (BsonWriter writer, EmailAddress value, EncoderContext encoderContext) {
    writer.writeString(value.toString());
  }
}

@Entity("subscribers")
public class Subscriber {

  @Id
  private ObjectId id;

  @Property
  @org.smallmind.mongodb.throng.mapping.annotation.Codec(EmailAddressCodec.class)
  private EmailAddress email;

  public Subscriber () {
  }
}
----

[IMPORTANT]
====
Arrays of embedded values and arrays of codec-backed values are handled automatically.
Generic containers such as `List<Profile>` or `List<EmailAddress>` are not fully automated in the current implementation.
If you need those shapes, use a driver codec for the concrete container type itself or model the container as its own concrete mapped type.
====

=== Polymorphic Embedded Types

Throng has embedded polymorphism support, but it is intentionally narrow.
It applies to `@Embedded` types, not top-level entities.

[source,java]
----
import org.smallmind.mongodb.throng.mapping.annotation.Embedded;
import org.smallmind.mongodb.throng.mapping.annotation.Polymorphic;

@Embedded(polymorphic = @Polymorphic(
    key = "kind",
    value = {CardPayment.class, WirePayment.class}
))
public abstract class PaymentMethod {

  public PaymentMethod () {
  }
}
----

What to remember:

* the discriminator key defaults to `java/object` when `key` is empty
* the discriminator value is the fully qualified Java class name
* lifecycle annotations are still forbidden on embedded types

=== Annotation Scope Matrix (`@Index`, `@Indexes`, `@Property`, `@Codec`)

[cols="2,1,1,4",options="header"]
|===

|`@Indexes`

|`@Property`

[[mongodb-lifecycle, Lifecycle Callbacks]]
== Lifecycle Callbacks

Lifecycle annotations are only supported on `@Entity` classes.
The callback methods must be public because discovery uses `Class.getMethods()`.

[cols="2,3,3",options="header"]
|===
|Annotation |Required signature |When it runs

|`@PreLoad`
|`public static void method (BsonDocument bsonDocument)`
|Before a BSON document is decoded into an entity

|`@PostLoad`
|`public void method ()`
|After the entity has been decoded

|`@PrePersist`
|`public void method ()`
|Before the entity is encoded into BSON

|`@PostPersist`
|`public void method (BsonDocument bsonDocument)`
|After the entity has been encoded into BSON
|===

[source,java]
----
import java.util.Date;
import org.bson.BsonDocument;
import org.smallmind.mongodb.throng.lifecycle.annotation.PostLoad;
import org.smallmind.mongodb.throng.lifecycle.annotation.PrePersist;

@Entity("users")
public class User {

  @Property
  private Date createdAt;

  public User () {
  }

  @PrePersist
  public void ensureCreatedAt () {
    if (createdAt == null) {
      createdAt = new Date();
    }
  }

  @PostLoad
  public void normalizeAfterLoad () {
    if (createdAt == null) {
      createdAt = new Date(0L);
    }
  }
}
----

[IMPORTANT]
====
Embedded types cannot declare lifecycle annotations.
`ThrongClient` construction fails with `ThrongMappingException` if lifecycle annotations are present on an `@Embedded` class.
====

[NOTE]
====
Static versus instance usage is validated at startup, but parameter mismatches are not exhaustively validated up front.
A wrongly shaped callback can still fail later as a `ThrongRuntimeException` during encode or decode.
====

[[mongodb-indexing, Indexing]]
== Indexing

=== Index Annotations

Use `@Indexed` for single-field indexes and `@Indexes(...)` for compound indexes.
The nested `@Index(...)` values are field names as stored in MongoDB documents.

Supported `IndexType` values:

* `ASCENDING`
* `DESCENDING`
* `HASHED`
* `TEXT`
* `GEO2D`
* `GEO2DSPHERE`

[source,java]
----
import com.mongodb.client.model.CollationStrength;
import java.util.Date;
import org.bson.types.ObjectId;
import org.smallmind.mongodb.throng.index.IndexType;
import org.smallmind.mongodb.throng.index.annotation.Collation;
import org.smallmind.mongodb.throng.index.annotation.Index;
import org.smallmind.mongodb.throng.index.annotation.IndexOptions;
import org.smallmind.mongodb.throng.index.annotation.Indexed;
import org.smallmind.mongodb.throng.index.annotation.Indexes;

@Entity("sessions")
@Indexes(
    value = {
        @Index("user_name"),
        @Index(value = "createdAt", type = IndexType.DESCENDING)
    },
    options = @IndexOptions(
        name = "sessions_user_created",
        partialFilterExpression = "{ archived : false }"
    )
)
public class SessionRecord {

  @Id
  private ObjectId id;

  @Property("user_name")
  private String username;

  @Property
  @Indexed(options = @IndexOptions(
      unique = true,
      name = "sessions_token",
      collation = @Collation(locale = "en", strength = CollationStrength.SECONDARY)
  ))
  private String sessionToken;

  @Property
  private boolean archived;

  @Property
  private Date createdAt;

  public SessionRecord () {
  }
}
----

=== Automatic Creation

Annotated indexes are created when `ThrongClient` is constructed and `ThrongOptions.createIndexes` is `true`.
That means index creation is startup behavior, not a background feature that happens later.

`ThrongOptions.includeCollation` controls whether `@Collation` values are applied when those indexes are created.
If `includeCollation` is `false`, the index is still created but its collation settings are omitted.

=== Index Option Notes

A few details are easy to miss:

* `partialFilterExpression`, `weights`, `storageEngine`, and `wildcardProjection` are JSON strings parsed with `BasicDBObject.parse(...)`
* `expireAfterSeconds` is only applied when it is greater than zero
* `bits`, `version`, `textVersion`, and `sphereVersion` are only applied when they are greater than zero
* `min` is only applied when it is at least `-180`
* `max` is only applied when it is at most `180`

[[mongodb-crud, CRUD API]]
== CRUD API

=== Counting

Use `count(...)` with an explicit `Filter`.
For a collection-wide count, pass `Filter.empty()`.

[source,java]
----
long userCount = throngClient.count(User.class, Filter.empty());
long adminCount = throngClient.count(User.class, Filter.where("roles").in("admin"));
----

=== Querying

`find(...)` returns `ThrongIterable<T>`.
Iterate lazily when you want streaming behavior, or call `asList()` when materializing the full result is acceptable.

[source,java]
----
ThrongIterable<User> iterable = throngClient.find(
    User.class,
    Query.with()
        .filter(Filter.where("user_name").regex(java.util.regex.Pattern.compile("^a")))
        .limit(25)
);

for (User user : iterable) {
  process(user);
}

List<User> users = iterable.asList();
----

[NOTE]
====
`findOne(...)` is just the first result from a query. It applies `limit(1)` and returns `null` when nothing matches.
====

=== Insert

`insert(...)` returns the driver's `InsertOneResult`.
There are two current implementation details you should account for:

* the `InsertOneOptions` argument is accepted by the signature but is not passed to the driver
* if your entity id field is `null`, MongoDB may generate an `_id`, but Throng does not write that generated value back into the Java object

[source,java]
----
InsertOneResult insertOneResult = throngClient.insert(user, new InsertOneOptions());

if (user.getId() == null && insertOneResult.getInsertedId() != null) {
  user.setId(insertOneResult.getInsertedId().asObjectId().getValue());
}
----

=== Update

`update(...)` is a thin wrapper around `updateMany(...)`.
It affects every document matched by the `Filter`.
The returned Smallmind `UpdateResult` exposes:

* `wasAcknowledged()`
* `getMatchedCount()`
* `getModifiedCount()`
* `getUpsertedId()`
* `getModifiedPlusInsertedCount()`

`UpdateOptions` is passed straight through to the driver, so driver features such as upsert, collation, hints, comments, array filters, and `let(...)` behave according to the driver and server.

[source,java]
----
UpdateResult updateResult = throngClient.update(
    User.class,
    Filter.where("user_name").eq("ada"),
    Updates.of()
        .set("email", "ada@new.example.org")
        .setOnInsert("createdAt", new Date())
        .addToSet("roles", "reviewer"),
    new UpdateOptions().upsert(true)
);
----

=== Delete

`delete(...)` is a thin wrapper around `deleteMany(...)`.
It returns the driver's `DeleteResult`, so you still inspect `wasAcknowledged()` and `getDeletedCount()` on the driver type.

`DeleteOptions` is also passed straight through to the driver.
If you need collation, hints, comments, or `let(...)`, configure them on `DeleteOptions` exactly as you would with direct driver usage.

[source,java]
----
long deleted = throngClient.delete(
    User.class,
    Filter.where("user_name").eq("ada"),
    new DeleteOptions()
).getDeletedCount();
----

=== Duplicate Key Detection

If you want a small helper for duplicate `_id` or unique-index failures, `DuplicateKeyUtility` checks MongoDB error code `11000`.

[source,java]
----
import com.mongodb.MongoWriteException;
import org.smallmind.mongodb.throng.DuplicateKeyUtility;

try {
  throngClient.insert(user, new InsertOneOptions());
} catch (MongoWriteException writeException) {
  if (DuplicateKeyUtility.idDuplicateKeyException(writeException)) {
    // handle duplicate key case
  }

  throw writeException;
}
----

[[mongodb-query-api, Query API]]
== Query API

=== `Filter`

`Filter` is intentionally small.
Each `Filter.where("field")` builds one predicate.
To combine predicates, create multiple filters and join them with `Filter.and(...)` or `Filter.or(...)`.

[source,java]
----
import java.util.regex.Pattern;
import org.smallmind.mongodb.throng.query.Filter;

Filter activeAuthorFilter = Filter.and(
    Filter.where("status").eq("ACTIVE"),
    Filter.where("roles").in("author", "reviewer"),
    Filter.where("user_name").regex(Pattern.compile("^a"))
);
----

Rules worth remembering:

* one field-level builder gets one terminal operator such as `eq(...)`, `lt(...)`, or `regex(...)`
* call `not()` only after the predicate exists
* incomplete or already-closed filters throw `IllegalFilterStateException`
* `Filter.empty()` matches everything and is useful for `count(...)`

=== `Query`

`Query` adds filter, sort, projection, skip, limit, and batch size to a `find(...)` call.
It does not try to mirror every `FindIterable` feature.
If you need driver features such as query collation, comments, hints, no-cursor-timeout flags, or specialized cursor controls, use the MongoDB driver directly.

[source,java]
----
import org.smallmind.mongodb.throng.ThrongIterable;
import org.smallmind.mongodb.throng.query.Projections;
import org.smallmind.mongodb.throng.query.Query;
import org.smallmind.mongodb.throng.query.Sort;

ThrongIterable<User> page = throngClient.find(
    User.class,
    Query.with()
        .filter(activeAuthorFilter)
        .sort(Sort.on().asc("user_name").desc("createdAt"))
        .projection(Projections.with().include("user_name", "email").exclude("_id"))
        .skip(20)
        .limit(10)
        .batchSize(10)
);
----

[NOTE]
====
`skip` is a `long` in the builder but is narrowed to `int` before it reaches the driver.
Non-positive values for `skip`, `limit`, and `batchSize` are effectively ignored.
====

=== `Sort`, `Projections`, `Updates`

`Sort` is a chain of `asc(...)` and `desc(...)` calls.
`Projections` combines `include(...)` and `exclude(...)` calls.
`Updates` combines the following operations:

* `set(String fieldName, Object value)`
* `unset(String fieldName)`
* `setOnInsert(String fieldName, Object value)`
* `inc(String fieldName, Number number)`
* `mul(String fieldName, Number number)`
* `max(String fieldName, Object value)`
* `min(String fieldName, Object value)`
* `push(String fieldName, Object value)`
* `addToSet(String fieldName, Object value)`
* `pull(String fieldName, Object value)`
* `popFirst(String fieldName)`
* `popLast(String fieldName)`
* `rename(String fieldName, String updatedFieldName)`

Example:

[source,java]
----
Updates updates = Updates.of()
    .set("profile.timezone", "UTC")
    .inc("loginCount", 1)
    .addToSet("roles", "reviewer")
    .rename("legacyField", "modernField");
----

[[mongodb-json-bson, JSON/BSON Utilities]]
== JSON/BSON Utilities

=== `BsonUtility`

`BsonUtility` converts between streaming `BsonReader` and `BsonWriter` instances and in-memory `BsonValue` trees.
It supports a useful subset of BSON values rather than the entire BSON type system.

Supported BSON types in `BsonUtility`:

* `OBJECT_ID`
* `TIMESTAMP`
* `DATE_TIME`
* `DOCUMENT`
* `ARRAY`
* `NULL`
* `STRING`
* `BOOLEAN`
* `DOUBLE`
* `INT32`
* `INT64`

=== `JsonUtility`

`JsonUtility` converts between Jackson `JsonNode` values and BSON.
It covers common document, array, string, boolean, and number cases.

[source,java]
----
import org.bson.BsonValue;
import tools.jackson.databind.JsonNode;
import tools.jackson.databind.node.JsonNodeFactory;
import tools.jackson.databind.node.ObjectNode;
import org.smallmind.mongodb.throng.json.JsonUtility;

ObjectNode payload = JsonNodeFactory.instance.objectNode();
payload.put("user_name", "ada");
payload.put("loginCount", 3);

BsonValue bsonValue = JsonUtility.fromJson(payload);
JsonNode roundTrip = JsonUtility.toJson(bsonValue);
----

`JsonUtility` currently supports:

* `JsonNode` object, array, null, string, boolean, integer, long, float, and double values when converting from JSON to BSON
* `BsonValue` document, array, null, undefined, string, boolean, `INT32`, `INT64`, `DOUBLE`, and `DECIMAL128` values when converting from BSON to JSON
* the same common scalar and container types when converting through `BsonReader` and `BsonWriter`

[IMPORTANT]
====
Neither utility is a complete BSON translation layer.
If your documents rely on additional BSON types such as binary values, regular expressions, code values, or other specialized nodes, use the MongoDB driver's own document APIs instead.
====

=== `DuplicateKeyUtility`

`DuplicateKeyUtility.idDuplicateKeyException(MongoWriteException)` is the small helper for duplicate key detection described earlier in <<mongodb-crud,CRUD API>>.
It does one thing: checks whether the write error code is `11000`.

== Utility Module

`mongodb-utility` is intentionally small.
It contains a pair of general helper types plus the Spring factory beans described in the next section.

[cols="2,4",options="header"]
|===
|Type |Role

|`MongoServer`
|Simple host/port value object. Default port is `27017`.

|`MongoAcknowledgment`
|Enumeration of write concern levels: `ZERO`, `ONE`, `TWO`, `THREE`, and `MAJORITY`.
|===

[NOTE]
====
If you are not using Spring, most of your direct Java wiring will still be plain MongoDB driver code. `mongodb-utility` does not try to replace the driver's own settings builders.
====

[[mongodb-spring, Spring Wiring]]
== Spring Wiring

The Spring support in `mongodb-utility` is deliberately narrow.
It materializes driver objects; it does not auto-discover entities or own broader runtime policy.

[source,xml]
----
<bean id="mongoServers" class="org.smallmind.mongodb.utility.spring.MongoServerFactoryBean">
  <property name="serverPattern" value="mongo-#.example.org:27017"/>
  <property name="serverSpread" value="1-3"/>
</bean>

<bean id="mongoCredential" class="org.smallmind.mongodb.utility.spring.MongoCredentialFactoryBean">
  <property name="user" value="${mongo.user}"/>
  <property name="password" value="${mongo.password}"/>
  <property name="source" value="admin"/>
</bean>

<bean id="writeConcern" class="org.smallmind.mongodb.utility.spring.MongoWriteConcern">
  <property name="acknowledgment" value="MAJORITY"/>
</bean>

<bean id="mongoClientSettings" class="org.smallmind.mongodb.utility.spring.MongoClientSettingsFactoryBean">
  <property name="serverAddresses" ref="mongoServers"/>
  <property name="mongoCredential" ref="mongoCredential"/>
  <property name="writeConcern" ref="writeConcern"/>
  <property name="retryReads" value="true"/>
  <property name="retryWrites" value="true"/>
  <property name="sslEnabled" value="false"/>
</bean>

<bean id="mongoClient" class="org.smallmind.mongodb.utility.spring.MongoClientFactoryBean">
  <property name="clientSettings" ref="mongoClientSettings"/>
</bean>
----

If you want Spring to construct the driver objects but still use Throng directly, add an ordinary bean around the `ThrongClient` constructor:

[source,java]
----
import com.mongodb.client.MongoClient;
import org.smallmind.mongodb.throng.ThrongClient;
import org.smallmind.mongodb.throng.ThrongOptions;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class MongoConfiguration {

  @Bean
  public ThrongClient throngClient (MongoClient mongoClient)
    throws Exception {

    return new ThrongClient(
        mongoClient,
        "application",
        new ThrongOptions(false, true, true),
        User.class,
        Profile.class
    );
  }
}
----

=== `MongoServerFactoryBean`

`MongoServerFactoryBean` creates `ServerAddress[]` from a `serverPattern` and optional `serverSpread`.

Pattern behavior:

* if there is no `#` placeholder, it creates one `ServerAddress`
* if there is a `#` placeholder, it expands that placeholder with `Spread.calculate(serverSpread)`
* if the pattern omits `:port`, the default port is `27017`

Example: `mongo-#.example.org:27017` with `serverSpread` set to `1-3` produces `mongo-1.example.org`, `mongo-2.example.org`, and `mongo-3.example.org`.

=== `MongoCredentialFactoryBean`

`MongoCredentialFactoryBean` produces a driver `MongoCredential` from `user`, `password`, and `source`.
It always builds a SCRAM-SHA-1 credential.
If you need a different authentication mechanism, construct `MongoCredential` yourself and inject it directly.

=== `MongoClientSettingsFactoryBean`

`MongoClientSettingsFactoryBean` exposes the most commonly needed driver settings as Spring bean properties.
It supports:

* cluster hosts through `serverAddresses`
* optional credential assignment through `mongoCredential`
* optional `codecRegistry` merging with the driver's default registry
* `readPreference`, `readConcern`, `writeConcern`, `writeConcernEnabled`, `retryReads`, and `retryWrites`
* TLS configuration through `certResource`, `sslEnabled`, and `allowInvalidHostNames`
* timeouts and pool settings such as `socketConnectTimeoutMilliseconds`, `serverSelectionTimeoutMilliseconds`, `connectionPoolMinSize`, `connectionPoolMaxSize`, `connectionPoolMaxConnecting`, `connectionPoolMaxWaitTimeMilliseconds`, `connectionPoolMaxConnectionLifeTimeSeconds`, and `connectionPoolMaxConnectionIdleTimeSeconds`

A few defaults and constraints matter:

* `serverAddresses` is required; the factory always applies hosts from that property
* `uuidRepresentation` is forced to `UuidRepresentation.STANDARD`
* `certResource` alone does not enable TLS; set `sslEnabled` to `true` as well
* if you need SSL, auth, or cluster behavior beyond these setters, build `MongoClientSettings` directly with the driver

=== `MongoClientFactoryBean`

`MongoClientFactoryBean` builds a singleton `MongoClient` from `MongoClientSettings` and closes it in `destroy()`.
Use it when you want Spring to own client shutdown.

=== `MongoWriteConcern`

`MongoWriteConcern` builds a `WriteConcern` from `MongoAcknowledgment` and an optional `journaled` flag.

[IMPORTANT]
====
In the current implementation, `setJournaled(true)` does not change the returned `WriteConcern` because the result of `withJournal(...)` is not reassigned.
If journaling matters, construct the driver `WriteConcern` yourself and inject it directly.
====

[[mongodb-lifecycle-ownership, Lifecycle And Resource Ownership]]
== Lifecycle And Resource Ownership

Ownership is explicit:

* `ThrongClient` does not close the `MongoClient` passed to it
* `MongoClientFactoryBean` does close the client when the Spring bean is destroyed
* `ThrongClient` construction can create indexes if `createIndexes` is enabled, so startup may perform writes
* `ThrongIterable` does not expose an explicit cursor-close API; for cursor-level control, use the driver directly

Operational guidance:

* prefer long-lived `MongoClient` instances rather than one client per operation
* treat `ThrongClient` as reusable infrastructure after it has validated mappings at startup
* if you disable automatic index creation, move index management into your own deployment or migration path

[[mongodb-edge-cases, Behavioral Edge Cases]]
== Behavioral Edge Cases

[cols="3,5",options="header"]
|===
|Behavior |Why it matters

|Query, update, sort, projection, and compound-index field names use stored MongoDB names
|If you rename fields with `@Property("...")`, all string-based query and index definitions must use the stored name

|Entity decode expects the id field first in the document
|Documents produced outside Throng can fail to decode with `ThrongRuntimeException` if the first field is not the entity id

|`insert(T, InsertOneOptions)` ignores `InsertOneOptions`
|Do not expect insert options such as bypass flags to take effect through Throng's insert wrapper

|Generated ids are not written back into the entity after insert
|If the id field starts as `null`, inspect `InsertOneResult.getInsertedId()` and assign it yourself if needed

|`update(...)` and `delete(...)` are many-document wrappers
|If you expected single-document semantics, you need direct driver calls instead

|`Query.skip(long)` is narrowed to `int`
|Large skip values can overflow the driver call boundary

|Non-positive `skip`, `limit`, and `batchSize` values are ignored
|The builder is permissive and does not fail fast on those values

|Generic containers of `@Embedded` or `@Codec` types are not fully automated
|Arrays work, but generic containers often need their own codec or concrete wrapper type

|Lifecycle annotations are forbidden on embedded types
|Startup fails with `ThrongMappingException` if embedded classes declare lifecycle callbacks

|`@Entity(polymorphic = ...)` is not used by current mapping logic
|Entity polymorphism is not a feature you should rely on in the current implementation

|Embedded polymorphic subtype validation is strict
|Ordinary subtype lists can be rejected; test that path directly before adopting it

|`MongoWriteConcern.setJournaled(...)` is currently ineffective
|Supply a driver-built `WriteConcern` when journaling matters

|`MongoClientSettingsFactoryBean` requires `serverAddresses`, and `certResource` does not imply `sslEnabled=true`
|Missing hosts or incomplete TLS setup causes Spring bean initialization failures

|`BsonUtility` and `JsonUtility` only cover subsets of BSON and JSON shapes
|Use the driver document APIs for specialized BSON types
|===

[[mongodb-troubleshooting, Troubleshooting]]
== Troubleshooting

=== `ThrongMappingException` During Construction

Common causes:

* missing `@Id` on an entity
* multiple `@Id` fields on one entity
* duplicate stored property names
* missing public no-arg constructor on a decoded type
* unknown codec for a field type
* lifecycle annotations on an embedded type
* forgetting to register an `@Embedded` class in the `ThrongClient` constructor

Checks:

* verify that every persisted field is marked with `@Property`
* verify that custom codec classes expose public no-arg constructors
* verify that all mapped entities and embedded types are passed to `ThrongClient`
* verify that renamed properties are unique after applying `@Property("...")`

=== Queries Or Updates Match Nothing

The most common cause is using Java field names after you renamed a stored property.

If the entity uses `@Property("user_name")`, these are the correct strings:

* `Filter.where("user_name")`
* `Sort.on().asc("user_name")`
* `Updates.of().set("user_name", "...")`
* `@Index("user_name")`

=== `ThrongRuntimeException` On Read

Common causes:

* the first document field is not the mapped id field
* a lifecycle callback threw an exception
* a non-public callback was assumed to be discovered but was not
* a stored polymorphic discriminator does not map to a registered class
* a document contains a shape your custom codec does not understand

Checks:

* inspect the raw stored document shape in MongoDB
* verify callback visibility and signatures
* verify polymorphic discriminator names if you are using embedded polymorphism
* verify that documents written outside Throng still match Throng's decode expectations

=== Filter Builder Exceptions

`IllegalFilterStateException` usually means one of three things:

* you called more than one terminal operator on one `Filter.where("...")`
* you called `not()` before defining the predicate
* you combined an incomplete filter inside `Filter.and(...)` or `Filter.or(...)`

=== JSON/BSON Conversion Errors

`DocumentParsingException` and `BSONParsingException` usually mean the utility helpers saw a BSON or JSON type outside their supported subset.
Switch to the MongoDB driver's document APIs when your documents include specialized BSON nodes.

=== Spring Bean Init Failures

Common causes:

* `serverAddresses` was never supplied to `MongoClientSettingsFactoryBean`
* `MongoCredentialFactoryBean` was given missing `user`, `password`, or `source`
* `certResource` was supplied but `sslEnabled` was left `false`
* `MongoWriteConcern` was expected to apply journaling

Checks:

* verify `serverPattern` and `serverSpread` values for `MongoServerFactoryBean`
* verify that TLS and trust material are both configured when SSL is required
* inject a driver-built `WriteConcern` directly when you need exact journaling behavior

[[mongodb-minimal-checklist, Minimal Checklist]]
== Minimal Checklist

1. Add `mongodb-throng`, `mongodb-utility`, or both.
2. Give every decoded `@Entity` and `@Embedded` type a public no-arg constructor.
3. Mark one entity field with `@Id` and every persisted field with `@Property`.
4. Register every entity and embedded class explicitly in `ThrongClient`.
5. Use stored MongoDB field names in queries, updates, and compound indexes.
6. Decide on `storeNulls`, `createIndexes`, and `includeCollation` explicitly.
7. Close `MongoClient` yourself unless Spring owns it through `MongoClientFactoryBean`.

[[mongodb-verification, Verification Checklist]]
== Verification Checklist

* construct `ThrongClient` in a focused test so mapping failures surface immediately
* insert, query, update, count, and delete at least one representative entity shape against a real MongoDB instance or test fixture
* verify renamed property queries and compound index field names with the stored document shape
* if you enable automatic index creation, confirm the expected indexes exist after startup
* if you use custom codecs, cover both encode and decode paths in tests
* if you use Spring factory beans, start and stop the application context and confirm `MongoClient` shutdown behavior
