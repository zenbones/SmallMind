[[quorum, Quorum]]
= Quorum

[partintro]
Quorum is Smallmind's library module for local coordination concerns that usually sit at awkward integration boundaries: rate limiting, provider failover, pooling, and `java:` namespace translation over JNDI.

It exposes five distinct surfaces:

* hierarchical token buckets in `org.smallmind.quorum.bucket`
* provider failover and blacklisting in `org.smallmind.quorum.juggler`
* a minimal synchronized object pool in `org.smallmind.quorum.pool.simple`
* a lifecycle-aware, metrics-aware pool in `org.smallmind.quorum.pool.complex`
* a `java:` namespace facade over backing directory stores, with LDAP support, in `org.smallmind.quorum.namespace`

Quorum follows the normal Smallmind rules:

* construction is explicit
* lifecycle is caller-owned
* integration points are narrow and visible
* runtime policy stays in application code instead of hidden framework behavior

Audience: application and library developers who need explicit coordination primitives without bringing in a larger framework or a remote control plane.
Versioning: examples use `${smallmind.version}` for Smallmind artifacts. Keep related Smallmind modules on the same version unless you have a specific reason not to.

[[quorum-at-a-glance, At A Glance]]
== At A Glance

[cols="2,2,4",options="header"]
|===
|Problem |Start with |Important obligation

|Local admission control inside one JVM
|`TokenBucket<T>`
|You decide token cost, refill rate, and any child-bucket partitioning. Child buckets must be registered explicitly.

|Several interchangeable providers where one may fail independently
|`Juggler<P, R>`
|You own the lifecycle: `initialize()`, optional `startup()`, then `shutdown()` and `deconstruct()`. Recovery behavior comes entirely from your `JugglingPin` implementation.

|Simple reusable objects with deterministic teardown
|`org.smallmind.quorum.pool.simple.ComponentPool<T>`
|The caller must return each borrowed component with `returnComponent(...)` and eventually call `close()` on the pool.

|Reusable resources that need validation, error reporting, lifecycle hooks, or retirement rules
|`org.smallmind.quorum.pool.complex.ComponentPool<C>`
|The served component is not returned automatically. Your `ComponentInstance<C>` or the served object must route close and error paths back into `returnInstance(...)` or `terminateInstance(...)`.

|A `java:` naming facade over LDAP
|`JavaContext` plus `javaURLContextFactory`
|Root lookups must start with `java:` and you must provide JNDI environment entries describing the backing store.

|Pooled LDAP-backed naming contexts
|`PooledJavaContextComponentInstanceFactory` plus the complex pool
|Calling `close()` on the borrowed `PooledJavaContext` returns it to the pool. `close(true)` is the destructive close used by pool teardown.
|===

Quorum focuses on local JVM coordination primitives that stay explicit at construction time and easy to compose into application code.

[[quorum-why-use, Why Use Quorum]]
== Why Use Quorum

Use Quorum when the hard part is coordinating integration edges inside the application boundary.

Typical reasons to choose it:

* you need rate limiting, failover, or pooling in library code without taking over the application's runtime model
* you want behavior that stays visible in constructors and method calls
* you need thin adapters over JNDI, metrics, and lifecycle rather than magic registration
* you want Smallmind-style glue code that composes with other modules instead of displacing them

It works best when the coordination problem lives inside one JVM or one application's integration boundary.

[[quorum-choose, Choosing The Right Surface]]
== Choosing The Right Surface

[cols="2,2,4",options="header"]
|===
|If you need... |Use... |Why

|A synchronous, in-memory token budget
|`TokenBucket<T>`
|It is local, deterministic, thread-safe, and has no background worker.

|Randomized selection across interchangeable providers with quarantine and optional recovery
|`Juggler<P, R>`
|It rotates among pins, blacklists failures, and can re-enable them after `recover()` succeeds.

|A small borrow/return pool where `terminate()` is enough cleanup
|`pool.simple.ComponentPool<T>`
|It is intentionally unsophisticated and easy to reason about.

|Validation-on-create or validation-on-borrow, lifecycle hooks, time-based retirement, metrics, or JMX exposure
|`pool.complex.ComponentPool<C>`
|It models resource lifecycle explicitly through `ComponentInstanceFactory<C>` and `ComponentInstance<C>`.

|Code written against `DirContext`, but with `java:` names instead of raw LDAP distinguished names
|`JavaContext`
|It translates between internal `java:` paths and backing-store names.

|A ready-made complex-pool integration for LDAP contexts
|`PooledJavaContextComponentInstanceFactory`
|It turns pooled directory contexts into complex-pool components with close and abort callbacks already wired.
|===

[IMPORTANT]
====
If you are choosing between the simple and complex pools, the deciding question is not "do I want more features?" It is "who is responsible for returning or destroying a borrowed component?"

The simple pool is explicit caller-managed borrow and return.
The complex pool expects the component wrapper or component events to drive that lifecycle.
====

[[quorum-lifecycle, Lifecycle Expectations]]
== Lifecycle Expectations

Each Quorum surface has a different lifecycle model.

[cols="2,4",options="header"]
|===
|Surface |Lifecycle model

|`TokenBucket`
|No startup or shutdown. Construction fully defines behavior.

|`Juggler`
|Call `initialize()` before use. Call `startup()` if pins need start logic or blacklist recovery. Later call `shutdown()` and then `deconstruct()`.

|Simple pool
|Borrow with `getComponent()`, return with `returnComponent(...)`, finish with `close()`.

|Complex pool
|Call `startup()` before borrowing. The pool creates, validates, retires, and destroys `ComponentInstance<C>` wrappers. Call `shutdown()` to tear the pool down.

|`JavaContext`
|Behaves like a translated `DirContext`. Close it when you are finished.

|`PooledJavaContext`
|`close()` means "return to the pool". `close(true)` means "really close the underlying directory context".
|===

[[quorum-quick-start, Quick Start]]
== Quick Start

=== Global Plus Per-Tenant Token Bucket

[source,java]
----
import java.util.concurrent.TimeUnit;
import org.smallmind.nutsnbolts.time.Stint;
import org.smallmind.quorum.bucket.BucketKey;
import org.smallmind.quorum.bucket.TokenBucket;

final class Request {

  private final String tenantId;

  Request (String tenantId) {
    this.tenantId = tenantId;
  }

  public String getTenantId () {
    return tenantId;
  }
}

enum TenantBucketKey implements BucketKey<Request> {
  PREMIUM, STANDARD
}

TokenBucket<Request> requestBucket = new TokenBucket<>(
    request -> 1.0D,
    request -> "premium".equals(request.getTenantId()) ? TenantBucketKey.PREMIUM : TenantBucketKey.STANDARD,
    200.0D,
    200.0D,
    Stint.of(1, TimeUnit.SECONDS)
);

requestBucket.add(
    TenantBucketKey.PREMIUM,
    () -> new TokenBucket<>(
        request -> 1.0D,
        request -> TenantBucketKey.PREMIUM,
        50.0D,
        50.0D,
        Stint.of(1, TimeUnit.SECONDS)
    )
);

boolean allowed = requestBucket.allowed(new Request("premium"));
----

=== Provider Failover With Recovery

[source,java]
----
import org.smallmind.quorum.juggler.AbstractJugglingPin;
import org.smallmind.quorum.juggler.Juggler;
import org.smallmind.quorum.juggler.JugglerResourceException;

final class Endpoint {

  ServiceHandle open () throws Exception {
    return new ServiceHandle();
  }

  boolean ping () {
    return true;
  }
}

final class ServiceHandle {
}

final class EndpointPin extends AbstractJugglingPin<ServiceHandle> {

  private final Endpoint endpoint;

  EndpointPin (Endpoint endpoint) {
    this.endpoint = endpoint;
  }

  @Override
  public ServiceHandle obtain ()
    throws JugglerResourceException {

    try {
      return endpoint.open();
    } catch (Exception exception) {
      throw new JugglerResourceException(exception);
    }
  }

  @Override
  public boolean recover () {
    return endpoint.ping();
  }

  @Override
  public String describe () {
    return endpoint.toString();
  }
}

Endpoint primary = new Endpoint();
Endpoint secondary = new Endpoint();

Juggler<Endpoint, ServiceHandle> juggler = new Juggler<>(
    Endpoint.class,
    ServiceHandle.class,
    30,
    (endpoint, resourceClass) -> new EndpointPin(endpoint),
    primary,
    secondary
);

juggler.initialize();
juggler.startup();

try {
  ServiceHandle handle = juggler.pickResource();
  // use the returned resource
} finally {
  juggler.shutdown();
  juggler.deconstruct();
}
----

[[quorum-bucket, Token Buckets]]
== Token Buckets

=== What `TokenBucket<T>` Does

`TokenBucket<T>` is a hierarchical in-memory token bucket.

For each `allowed(T input)` call it:

1. refills capacity according to elapsed nanoseconds
2. asks `BucketQuantifier<T>` how many tokens the input costs
3. rejects immediately if this bucket lacks capacity
4. optionally checks a selected child bucket
5. deducts tokens from the current bucket only after the child path succeeds

This gives two common shapes:

* a flat bucket for global throttling
* a parent bucket plus one or more child buckets for "global plus partition" throttling

=== Core Types

[cols="2,4",options="header"]
|===
|Type |Role

|`BucketQuantifier<T>`
|Maps an input to a token cost.

|`BucketSelector<T>`
|Maps an input to a child-bucket key.

|`BucketKey<T>`
|Marker interface for child-bucket identifiers.

|`BucketFactory<T>`
|Creates a child bucket when you register one with `add(...)`.

|`TokenBucket<T>`
|Tracks capacity, refill rate, and optional child buckets.
|===

=== Construction Rules

The constructor takes five things:

* a quantifier
* a selector
* a capacity limit
* a refill quantity
* a refill period as `Stint`

Internally it computes:

* `refillPerNanosecond = refillQuantity / refillRateInNanoseconds`

Behavior that matters in practice:

* capacity starts full
* refill uses `double`, so fractional token rates are supported
* refill is time-based, not call-count-based
* capacity never grows above the configured limit

=== Child Buckets Are Explicit

Child buckets are registered with `add(BucketKey<T>, BucketFactory<T>)`.

Important details:

* child buckets are not created automatically during `allowed(...)`
* if the selector returns a key with no registered child, the parent bucket alone decides the call
* child buckets can themselves contain children because each child is another `TokenBucket<T>`

=== Concurrency Model

Both `add(...)` and `allowed(...)` are synchronized.

That means:

* one shared bucket instance is safe to use across threads
* refill and deduction are atomic with respect to other callers
* a child bucket can be shared safely because the private recursive `allowed(long, T)` path is also synchronized

=== Design Guidance

Use `TokenBucket` when:

* the limit is local to one JVM
* you want no background thread and no remote dependency
* you can describe the decision as capacity plus refill math

Do not use it when:

* multiple JVMs must coordinate a shared budget
* limit state must survive restart
* you need adaptive or probabilistic admission rather than a token bucket

[NOTE]
====
The constructor currently prints the computed refill rate to standard out through `System.out.println(...)`.
Treat that as a known implementation quirk when deciding where bucket construction should happen.
====

[[quorum-juggler, Juggler]]
== Juggler

=== What `Juggler<P, R>` Does

`Juggler<P, R>` manages a set of interchangeable providers by wrapping each one in a `JugglingPin<R>`.

Its job is narrow:

* build pins from providers
* rotate among them in randomized cycles
* blacklist pins whose `start(...)` or `obtain()` path fails
* optionally restore blacklisted pins after `recover()` succeeds

It is a failover primitive, not a full load balancer.
It has no weights, topology model, or cross-node state.

=== Core Types

[cols="2,4",options="header"]
|===
|Type |Role

|`Juggler<P, R>`
|Coordinates pin lifecycle, selection, blacklisting, and recovery.

|`JugglingPin<R>`
|Contract for start, stop, close, obtain, recover, and description behavior.

|`AbstractJugglingPin<R>`
|Convenience base class with no-op lifecycle methods.

|`JugglingPinFactory<P, R>`
|Creates pins from providers.

|`BlacklistEntry<R>`
|Pairs a failed pin with the throwable that caused blacklisting.

|`BlackList<R>`
|Callback surface for adding pins to a blacklist.

|`NoAvailableJugglerResourceException`
|Thrown when no pin can supply a resource.
|===

=== Lifecycle

The intended sequence is explicit:

1. `initialize()` builds pin collections and randomizes the first circulation order
2. `startup()` calls `start(...)` on each pin and optionally starts the recovery worker
3. `pickResource()` obtains resources from healthy pins
4. `shutdown()` calls `stop(...)` on each remaining pin and stops recovery
5. `deconstruct()` calls `close(...)` on pins after shutdown

Technical details worth knowing:

* `pickResource()` is allowed after `initialize()` even if `startup()` was never called
* `shutdown()` only does work from the `STARTED` state
* `deconstruct()` only does work from the `STOPPED` state

=== Blacklisting And Recovery

Failures are handled differently depending on where they occur.

When `startup()` calls `pin.start(...)`:

* pins that throw `JugglerResourceException` are logged
* those pins are removed from active circulation
* a `BlacklistEntry<R>` is stored with the current time

When `pickResource()` calls `pin.obtain()`:

* any thrown exception blacklists the pin immediately
* selection continues until a healthy pin responds or no pins remain
* if no pins remain, `NoAvailableJugglerResourceException` is thrown with suppressed causes from prior failures

Recovery is optional:

* `recoveryCheckSeconds == 0` disables recovery entirely
* otherwise a daemon thread polls every 3 seconds
* a blacklisted pin becomes eligible for recheck after `recoveryCheckSeconds`
* if `recover()` returns `true`, the pin re-enters service

=== Design Guidance

Use `Juggler` when:

* providers are equivalent enough that any healthy one can serve the request
* provider failure should reduce capacity rather than fail the whole subsystem immediately
* the provider itself knows how to recover or be probed for health

Do not use it when:

* you need weighted routing or request sharding
* you need per-request balancing policy
* you need load-balancer features such as circuit-breaking windows, hedging, or distributed coordination

[IMPORTANT]
====
The single-provider constructor does not clone the provider. It fills the backing array with the same provider object reference.
Use it only when that is exactly what you want.
====

[[quorum-pool-base, Shared Pool Base]]
== Shared Pool Base

Both pool implementations build on `PoolConfig<P>` and `Pool`.

Shared configuration defaults:

* `maxPoolSize = 10`
* `acquireWaitTimeMillis = 0`

Shared utility surfaces:

* `Pool.register()` stores the pool in `PoolManager`
* `PoolManager` exposes only one pool per application context key

The two pool implementations interpret `acquireWaitTimeMillis == 0` differently, so read the specific section before using the shared default blindly.

[[quorum-simple-pool, Simple Pool]]
== Simple Pool

=== What It Is

`org.smallmind.quorum.pool.simple.ComponentPool<T>` is a synchronized linked-list pool for `T extends PooledComponent`.

Its model is intentionally direct:

* borrow with `getComponent()`
* return with `returnComponent(T)`
* destroy with `PooledComponent.terminate()`
* finish with `close()`

There is no validation hook, no background retirement, no metrics API, and no event model.

=== Core Types

[cols="2,4",options="header"]
|===
|Type |Role

|`PooledComponent`
|Component contract with a single `terminate()` method.

|`ComponentFactory<T>`
|Creates new components when the pool can grow.

|`SimplePoolConfig`
|Concrete configuration class over `PoolConfig<P>`.

|`ComponentPool<T>`
|Synchronized borrow/return pool.
|===

=== Borrow And Return Semantics

`getComponent()` behaves as follows:

* if a free component exists, return it
* otherwise, if growth is allowed, create a new component
* otherwise, wait according to `acquireWaitTimeMillis`

`returnComponent(T)` behaves as follows:

* remove the component from the used list
* if the retained size is still below `maxPoolSize`, put it back on the free list
* otherwise, call `terminate()` immediately

`close()` behaves as follows:

* mark the pool closed
* wake waiting borrowers
* wait for all borrowed components to be returned
* terminate all free components

=== Wait Behavior

The simple pool treats `acquireWaitTimeMillis` this way:

* `0` means wait indefinitely when the pool is full
* `> 0` means wait up to that duration and then throw `ComponentPoolException`

This is easy to overlook because the complex pool does not use the same convention.

=== Design Guidance

Use the simple pool when:

* the resource has cheap, deterministic validation or does not need validation at all
* a single `terminate()` hook is enough teardown
* the caller can be trusted to always return what it borrows

Do not use it when:

* resources need health checks on create or acquire
* you need time-based retirement
* you need lease diagnostics or JMX visibility
* the served component must terminate itself on error

[NOTE]
====
`maxPoolSize == 0` creates an unusual mode: the pool can create without a growth limit, but returned components are never retained because the free-list retention check can never succeed. In practice that means "create on demand, terminate on return," not a reusable cache.
====

[[quorum-complex-pool, Complex Pool]]
== Complex Pool

=== What It Is

`org.smallmind.quorum.pool.complex.ComponentPool<C>` is a lifecycle-aware pool built around `ComponentInstance<C>` wrappers.

The pool itself does not know how to validate, serve, or destroy `C`.
That logic lives in:

* `ComponentInstanceFactory<C>` for creation and factory lifecycle
* `ComponentInstance<C>` for validation, serving, existential tracking, and destruction

This is the main conceptual difference from the simple pool.

=== Core Types

[cols="2,4",options="header"]
|===
|Type |Role

|`ComponentInstanceFactory<C>`
|Creates pooled instances and exposes optional initialize/startup/shutdown/deconstruct hooks.

|`AbstractComponentInstanceFactory<C>`
|Convenience base class with no-op lifecycle methods.

|`ComponentInstance<C>`
|Per-instance contract for `validate()`, `serve()`, and `close()`.

|`ComplexPoolConfig`
|Configuration for sizing, validation, timeouts, diagnostics, and retirement rules.

|`ComponentPool<C>`
|User-facing pool abstraction.

|`ComponentPoolSurface`
|Management surface mirrored by the JMX monitor.

|`ComponentPoolEventListener`
|Listener for error and lease-time events.
|===

=== Lifecycle And Ownership

`ComponentPool.startup()` does three things in order:

1. `componentInstanceFactory.initialize()`
2. create `max(initialPoolSize, minPoolSize)` instances and queue them as free
3. `componentInstanceFactory.startup()`

`ComponentPool.shutdown()` does this in order:

1. `componentInstanceFactory.shutdown()`
2. terminate every managed component, including currently borrowed ones
3. `componentInstanceFactory.deconstruct()`

That last point matters: unlike the simple pool, complex-pool shutdown is not "wait politely for borrowers." It is pool teardown.

=== Borrowing Model

`getComponent()` returns the value from `ComponentInstance<C>.serve()`.

That means the pool does not automatically know how the caller will finish with the borrowed component.
One of these must be true:

* the served component's `close()` or similar success path eventually calls `componentPool.returnInstance(thisComponentInstance)`
* the served component's error path eventually calls `componentPool.terminateInstance(thisComponentInstance)`
* some listener or callback around the served component performs those operations

If none of that is wired, borrowed instances never re-enter the pool.

[IMPORTANT]
====
If your first instinct is to call `ComponentPool.getComponent()` and later hand the same `C` back to the pool, you are using the wrong abstraction.

The complex pool is instance-centric, not raw-value borrow/return centric.
====

=== Capacity And Acquire Semantics

On each borrow attempt the pool:

1. tries an already-free component
2. validates it on acquire if `testOnAcquire` is enabled
3. creates a new component on demand if capacity allows
4. otherwise waits up to `acquireWaitTimeMillis`
5. throws `ComponentPoolException` if the wait expires

Important details:

* `maxPoolSize == 0` means unbounded growth
* `acquireWaitTimeMillis == 0` means do not wait at all when the pool is saturated; fail immediately
* startup only pre-creates `max(initialPoolSize, minPoolSize)`
* later growth happens lazily on borrow

=== Validation, Creation Timeout, And Replacement

`ComplexPoolConfig` gives you three important validation controls:

* `testOnCreate`: validate immediately after creation
* `testOnAcquire`: validate before serving a free instance
* `creationTimeoutMillis`: bound instance creation by delegating to a worker thread

Replacement behavior is intentionally conservative:

* terminating an instance does not automatically restore the previous pool size
* replacement is guaranteed only as needed to satisfy `minPoolSize`
* otherwise the pool grows back on a later borrow if demand requires it

=== Deconstruction Fuses

When one or more fuse settings are enabled, each `ComponentPin<C>` gets a `DeconstructionCoordinator` with one or more fuses.

[cols="2,4",options="header"]
|===
|Setting |Behavior

|`maxLeaseTimeSeconds`
|Absolute lifetime from instance creation. The timer does not reset on return.

|`maxIdleTimeSeconds`
|Runs only while the component is free. Borrowing cancels the idle fuse and returning reschedules it.

|`maxProcessingTimeSeconds`
|Runs only while the component is borrowed. Ignition is treated as prejudicial and can log the existential stack trace when enabled.
|===

=== Existential Tracking And Diagnostics

If `existentiallyAware` is enabled, the pool expects `ComponentInstance<C>.serve()` to record `Thread.currentThread().getStackTrace()` or equivalent state.

The built-in namespace integration and the persistence module both follow that pattern.
`ComponentPool.getExistentialStackTraces()` exposes stack traces for currently borrowed instances.

=== A Minimal Custom Pattern

The built-in namespace integration is the best concrete example because it already wires close and abort callbacks correctly.
If you are building your own `ComponentInstance<C>`, the pattern usually looks like this:

[source,java]
----
final class LeaseHandle implements AutoCloseable {

  private final Runnable returnAction;
  private final Runnable abortAction;

  LeaseHandle (Runnable returnAction, Runnable abortAction) {
    this.returnAction = returnAction;
    this.abortAction = abortAction;
  }

  public void markBroken () {
    abortAction.run();
  }

  @Override
  public void close () {
    returnAction.run();
  }
}
----

The matching `ComponentInstance<LeaseHandle>` supplies those callbacks so success returns the instance to the pool and failure terminates it.
That is the design center of the complex pool.

[[quorum-namespace, Java Namespace]]
== Java Namespace

=== What `JavaContext` Does

`JavaContext` is a `DirContext` facade that translates internal `java:` paths into backing-store names.

With LDAP, this lets you write code against names such as:

* `java:people`
* `java:people/admin`
* `engineering/backend` after you already hold a child context

instead of constructing distinguished names manually throughout the calling code.

=== Core Types

[cols="2,4",options="header"]
|===
|Type |Role

|`JavaContext`
|Translated `DirContext` facade.

|`PooledJavaContext`
|`JavaContext` variant that emits close and abort events for pooling.

|`javaURLContextFactory`
|JNDI object factory used when resolving `java:` names.

|`NamingConnectionDetails`
|Connection settings for the backing store.

|`StorageType`
|Built-in backing-store enum. Today that means `LDAP` only.

|`NameTranslator`
|Translates between internal names and backing-store names.

|`LdapContextCreator`, `LdapNameTranslator`
|Built-in LDAP implementations.
|===

=== Required Environment Entries

To create a top-level `JavaContext` through JNDI you normally provide:

* `Context.URL_PKG_PREFIXES = org.smallmind.quorum.namespace`
* `JavaContext.CONNECTION_DETAILS = new NamingConnectionDetails(...)`
* `JavaContext.CONTEXT_STORE = StorageType.LDAP.getBackingStore()`
* optionally `JavaContext.CONTEXT_MODIFIABLE = "true"`
* optionally `JavaContext.POOLED_CONNECTION = "true"`

Using `StorageType.LDAP.getBackingStore()` is safer than spelling the backing-store string yourself, because `javaURLContextFactory` reflects over that value.

=== Name Translation Rules For LDAP

For `StorageType.LDAP` the built-in translation model is straightforward:

* internal path segments become `cn=<segment>` components
* the root `java:` segment is required for top-level lookup
* absolute LDAP names are translated back into `java:` paths by removing the configured root namespace

Example with root namespace `dc=example,dc=net`:

[cols="2,4",options="header"]
|===
|Internal form |LDAP form

|`java:people/admin`
|relative lookup under the root namespace as `cn=admin,cn=people`

|`cn=admin,cn=people,dc=example,dc=net`
|reported back by `getNameInNamespace()` as `java:people/admin`
|===

=== Read-Only And Modifiable Modes

`JavaContext` checks the string value of `JavaContext.CONTEXT_MODIFIABLE`.
Only the exact string `"true"` enables mutations.

If modifiability is not enabled, these operations throw `OperationNotSupportedException`:

* `bind`
* `rebind`
* `unbind`
* `rename`
* `destroySubcontext`
* `createSubcontext`
* `modifyAttributes`
* `addToEnvironment`
* `removeFromEnvironment`

Read operations remain available.

=== Practical Use

Use `JavaContext` when:

* you already have JNDI-oriented code or APIs
* you want a `java:` namespace boundary over LDAP
* you need namespace translation without taking on a broader directory abstraction

Avoid it when:

* raw LDAP distinguished names are already your true API
* you do not want JNDI in the calling code at all

[[quorum-namespace-pool, Pooled JavaContext Integration]]
== Pooled JavaContext Integration

=== What It Adds

`PooledJavaContext` extends `JavaContext` with an event model that distinguishes:

* normal close, which means "return this instance to the pool"
* communication failure, which means "terminate this instance and report the error"

That makes it a natural fit for the complex pool.

=== `PooledJavaContext`

`PooledJavaContext` wraps most public `JavaContext` operations and catches `CommunicationException`.
When one occurs it fires `contextAborted(...)` before rethrowing the exception.

Its close behavior is intentionally different from ordinary JNDI:

* `close()` delegates to `close(false)`
* `close(false)` fires a `contextClosed(...)` event
* `close(true)` actually closes the underlying `DirContext`

That distinction is what lets the complex pool reuse the same underlying directory context across borrows.

=== `JavaContextComponentInstance`

`JavaContextComponentInstance` is the built-in `ComponentInstance<PooledJavaContext>` bridge.
It does four important things:

* validates with `lookup("")`
* captures a borrower stack trace when existential tracking is enabled
* returns the instance to the pool on `contextClosed(...)`
* terminates the instance and reports an error on `contextAborted(...)`

This class is the clearest example of how the complex pool expects a served component to manage its own return and failure paths.

=== `PooledJavaContextComponentInstanceFactory`

This factory packages the namespace and complex-pool pieces together.
It builds a JNDI environment with:

* `Context.URL_PKG_PREFIXES = org.smallmind.quorum.namespace`
* `JavaContext.CONNECTION_DETAILS = new NamingConnectionDetails(...)`
* `JavaContext.CONTEXT_STORE = storageType.getBackingStore()`
* `JavaContext.CONTEXT_MODIFIABLE = "true"`
* `JavaContext.POOLED_CONNECTION = "true"`

Then it:

1. creates an `InitialContext`
2. looks up the requested `contextPath`
3. casts the result to `PooledJavaContext`
4. wraps it in `JavaContextComponentInstance`

[NOTE]
====
The built-in pooled factory always enables `JavaContext.CONTEXT_MODIFIABLE = "true"`.
If you need pooled but read-only contexts, write your own `ComponentInstanceFactory<PooledJavaContext>`.
====

=== Shallow Pooling Behavior

Pooling does not propagate arbitrarily deep through the namespace API.
A pooled wrapper is shallow.

Important consequences from the current implementation:

* the top-level lookup that created the pooled context can return `PooledJavaContext`
* nested `lookup(...)` calls from a `PooledJavaContext` return plain `JavaContext` wrappers when they resolve subcontexts
* `lookupLink(...)`, enumeration conversion, and `createSubcontext(...)` also return plain `JavaContext` wrappers

That is usually fine for directory navigation, but it matters if you expected every nested context to carry pool-return semantics.

[[quorum-events-metrics, Events, Metrics, And JMX]]
== Events, Metrics, And JMX

=== Complex-Pool Events

`ComponentPool<C>` can publish two event types through `ComponentPoolEventListener`:

* `reportErrorOccurred(ErrorReportingComponentPoolEvent<?>)`
* `reportLeaseTime(LeaseTimeReportingComponentPoolEvent<?>)`

Those events come from explicit pool behavior:

* error reporting is triggered by integrations such as `JavaContextComponentInstance`
* lease-time reporting is triggered when `reportLeaseTimeNanos` is enabled and a component is returned

=== Claxon Metrics

The complex pool emits several Claxon metrics.
The exact meter wiring lives in the pool classes, but the user-visible tags are stable enough to plan around.

[cols="2,4",options="header"]
|===
|Tag/value shape |Meaning

|`pool=<poolName>, size=Free`
|Current free-count update.

|`pool=<poolName>, size=Processing`
|Current borrowed-count update.

|`pool=<poolName>, event=Timeout`
|Acquire timed out.

|`pool=<poolName>, event=Released`
|A component was returned; the recorded value is lease duration.

|`pool=<poolName>, event=Waited`
|Acquisition path instrumentation around `getComponent()`.
|===

=== JMX Monitor

`ComponentPoolMonitor` is both:

* a `ComponentPoolSurface` implementation
* an MXBean through `ComponentPoolMonitorMXBean`

It exposes the main pool configuration and current counts, and it emits JMX notifications for:

* `CreationErrorOccurredNotification.TYPE = ERROR_OCCURRED`
* `ComponentLeaseTimeNotification.TYPE = LEASE_TIME`

Useful management operations include:

* `startup()`
* `shutdown()`
* getters and setters for validation, pool sizing, and timeout fields

[NOTE]
====
The public setter name for maximum processing time is `setMaxProcessingTimeTimeSeconds(...)`.
The duplicated word is part of the current public surface.
====

=== Logging

Operational logging in Quorum goes through Scribe.
The most relevant sources are:

* juggler startup, pick, blacklist, and recovery paths
* complex-pool creation and retirement failures
* max-processing-time warnings that include existential stack traces when enabled
* pooled namespace abort handling

[[quorum-patterns, Common Integration Patterns]]
== Common Integration Patterns

=== Use Token Buckets At Clear Boundaries

`TokenBucket` is easiest to reason about when it sits at a visible boundary:

* HTTP admission
* message consumption
* calls into an expensive downstream system

If you build multi-level bucket trees, keep the partition meaning obvious and register child buckets explicitly at startup.

=== Use Juggler For Provider Health, Not Request Policy

A good `Juggler` use case is "several equivalent providers, any of which may fail independently." If the real requirement is request-aware routing, put that policy above the juggler or use a different abstraction.

=== Keep Complex-Pool Return Logic Inside The Served Component

The pool should not force application code to hold a side reference to `ComponentInstance<C>`.
Prefer one of these designs:

* `close()` on the served component returns it to the pool
* an error callback on the served component terminates it in the pool
* an event listener attached by the `ComponentInstance<C>` does both

That is the pattern used by Quorum's pooled namespace integration and by the persistence module's pooled SQL connections.

=== Combine Juggler And Complex Pool Deliberately

A useful composite design is:

1. `Juggler` chooses a healthy provider
2. a complex-pool factory creates instances against that provider
3. the served component returns or terminates itself through pool callbacks

This is a good fit when provider availability and component health are related but not identical.

=== Use A Custom Factory When Namespace Defaults Are Too Opinionated

The built-in pooled namespace factory is convenient, but it makes two policy decisions for you:

* pooled contexts are modifiable
* lifecycle is driven by `PooledJavaContext`

If either choice is wrong for your environment, build your own `ComponentInstanceFactory<PooledJavaContext>` or `ComponentInstanceFactory<JavaContext>`.

[[quorum-troubleshooting, Troubleshooting]]
== Troubleshooting

[cols="2,4",options="header"]
|===
|Symptom |Likely cause and response

|`NamingException: No starting context from which to resolve (...)`
|The root lookup did not start with `java:`. Create the root through `InitialContext` and begin with a `java:` path such as `java:people`.

|`OperationNotSupportedException: This backing store is not modifiable`
|`JavaContext.CONTEXT_MODIFIABLE` was not set to the exact string `"true"`. Enable it only if the backing store should accept writes.

|`ComponentPoolException: Exceeded the maximum acquire wait time(...)`
|The complex pool was saturated and no valid instance became available in time. Verify return/terminate wiring first, then review `maxPoolSize` and `acquireWaitTimeMillis`.

|`ComponentPool(<factory>) is completely booked`
|The simple pool hit its timed wait limit while full. Increase capacity, shorten hold time, or return components more reliably.

|`NoAvailableJugglerResourceException`
|All juggler pins failed or were blacklisted. Inspect suppressed causes and confirm `recover()` behavior if recovery is enabled.

|`IllegalStateException: Juggler must be in the initialized or started state`
|`pickResource()` was called before `initialize()`.

|Borrowed components never seem to return to the complex pool
|The served component or its wrapper is not calling `returnInstance(...)` or `terminateInstance(...)` through the owning `ComponentInstance<C>`.

|Repeated `CommunicationException` from `PooledJavaContext`
|The LDAP-backed context is broken. The pooled wrapper should abort and terminate itself. Check credentials, network reachability, TLS settings, and root namespace.

|Unexpected output on standard out during bucket construction
|This is current `TokenBucket` behavior. Construct buckets in a place where that output is acceptable or treat it as a known module quirk.

|A pooled namespace child context does not return itself to the pool on `close()`
|You are probably holding a nested `JavaContext`, not the original `PooledJavaContext`. Pool-return semantics are shallow.
|===

[[quorum-minimal-checklist, Minimal Checklist]]
== Minimal Checklist

* choose the smallest Quorum surface that actually solves the problem
* document lifecycle ownership in the calling code
* for the simple pool, verify whether `acquireWaitTimeMillis == 0` is acceptable
* for the complex pool, write down exactly how a served component returns or terminates itself
* if you enable complex-pool fuses, document whether you want absolute lifetime, idle retirement, processing timeout, or some combination
* for namespace usage, decide deliberately whether mutation should be enabled
* if you rely on diagnostics, decide how Claxon metrics, Scribe logs, and optional JMX notifications will be consumed

[[quorum-verification, Verification Checklist]]
== Verification Checklist

There is currently no `quorum/src/test` tree in this repository, so focused verification matters.

Recommended checks for a new integration:

1. Token bucket: verify both allowed and denied requests around the configured limit.
2. Token bucket: verify refill after elapsed time.
3. Token bucket: verify parent-plus-child behavior for one registered and one unregistered child key.
4. Juggler: force one pin to fail on `obtain()` and confirm blacklisting.
5. Juggler: if recovery is enabled, confirm that `recover()` returning `true` brings the pin back into service.
6. Simple pool: borrow to capacity and confirm wait behavior when full.
7. Simple pool: confirm that `close()` blocks until borrowed components are returned.
8. Complex pool: confirm `startup()` pre-creates `max(initialPoolSize, minPoolSize)` instances.
9. Complex pool: exercise `testOnCreate` and `testOnAcquire` paths if enabled.
10. Complex pool: verify acquire timeout or immediate-failure behavior at saturation.
11. Complex pool: verify that normal completion returns the instance and error completion terminates it.
12. Complex pool: if fuses are enabled, force one to ignite and confirm the intended retirement behavior.
13. Namespace: verify `java:` name translation against a real LDAP hierarchy.
14. Namespace: verify read-only and modifiable modes separately if both matter.
15. Pooled namespace: confirm that the top-level borrowed context returns to the pool on `close()` and is destroyed on communication failure.
