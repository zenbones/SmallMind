== Abstract

*Quorum* is a Java concurrency and resource-management toolkit that provides
four independent but conceptually related frameworks for managing shared
resources in a multi-threaded application. It answers a family of closely
related questions:

* *How do I throttle access to a resource?* -- `quorum.bucket` (token-bucket rate limiting).
* *How do I spread load across a group of interchangeable resources and survive the failure of any of them?* -- `quorum.juggler` (load balancing + failover + blacklist).
* *How do I reuse expensive-to-create resources safely?* -- `quorum.pool` (object pooling, simple and complex).
* *How do I expose arbitrary directory-style storage behind the standard JNDI API?* -- `quorum.namespace` (custom `java:` URL context provider).

The common thread across all four is the careful, thread-safe orchestration
of a *finite set of resources* across an *unbounded set of callers*.  Quorum's
name is apt: every one of its frameworks is about deciding, at any given
moment, which members of a group are available to serve the next request.

== Why Quorum Exists

Many server-side Java applications eventually confront
a set of common problems in clustering and distribution:

. A downstream service (database, LDAP directory, message broker, remote API)
  is expensive to connect to, so connections must be reused.
. That downstream service can fail, degrade, or be replaced by redundant
  siblings, so clients must load-balance and fail over.
. The application itself must protect that service (and itself) from
  runaway traffic, so calls must be rate-limited.
. Inside the JVM, standard JNDI lookups want to resolve `java:/...` URIs
  against an arbitrary configuration store, and that store is often an
  external directory whose naming conventions do not match the JNDI scheme.

Quorum addresses all four problems with idiomatic, production-grade Java:

* Thread-safe by default (synchronized methods, atomic counters,
  `ReentrantReadWriteLock`, `ConcurrentSkipListMap`, `LinkedBlockingQueue`).
* Instrumented for production -- complex pools publish Claxon metrics and
  JMX notifications, jugglers log blacklist events, and everything emits
  Scribe log lines at appropriate levels.
* Pluggable by design -- every framework is organized around small strategy
  interfaces (factories, selectors, quantifiers, translators) so that callers
  can wire in domain-specific behavior without subclassing the core.
* Lifecycle-aware -- pools have `startup`/`shutdown`/`deconstruct` phases,
  jugglers have `initialize`/`startup`/`shutdown`/`deconstruct`, pooled
  JNDI contexts distinguish logical close from forced close.

If you are building a long-running service that must remain healthy under
load and in the face of partial failures, Quorum is the layer immediately
beneath your business logic.

== The Pool Framework
// ==========================================================================

=== Why It Exists

Creating resources such as database connections, RPC stubs, LDAP contexts,
thread stacks, or cryptographic sessions is expensive -- in time, memory,
and in some cases in handshake cost against a remote endpoint. At the same
time, these resources are typically *safe to reuse* once the caller is
done with them. A pool captures exactly that observation: maintain a
small, bounded set of live resources, hand them out to callers one at a
time, and take them back when the caller is finished.

Quorum's pool framework ships with two implementations sharing a small
common core:

* *Simple* -- a compact, fully-synchronized pool based on two `LinkedList`
  instances. Ideal when the resource has a trivial lifecycle and you do
  not need validation, timeouts, or monitoring.
* *Complex* -- a production-grade pool with validation hooks, creation
  timeouts, lifetime fuses (lease, idle, processing), event listeners,
  JMX monitoring, Claxon metrics, and optional acquisition stack-trace
  capture for leak diagnostics.

When in doubt, start with *complex*. The cost is a few extra interfaces
to implement; the payoff is visibility and safety when your production
pool misbehaves.

=== Common Core

==== `Pool`

The abstract base class for every pool implementation. Its only
contribution is the `register()` method, which publishes the pool
instance to the per-application `PoolManager` so that callers anywhere
in the same `PerApplicationContext` can retrieve it without needing an
injected reference.

==== `PoolManager`

An application-scoped registry (not JVM-wide). Holds a single `Pool`
instance per `PerApplicationContext`. Static accessors:

[source,java]
----
PoolManager.register(myPool);
Pool pool = PoolManager.getPool();
----

==== `PoolConfig`

Abstract configuration base, using atomic fields so that reconfiguration
can happen *live* without stopping the pool. Two shared properties:

[cols="1,1,4"]
|===
| Property | Default | Meaning

| `maxPoolSize` | `10` | Cap on concurrently-held components; `0` means unbounded.
| `acquireWaitTimeMillis` | `0` | How long a caller blocks when the pool is saturated; `0` means fail immediately.
|===

All setters are fluent and return the correct subclass type (via
`getConfigurationClass()`), so subclass setters chain cleanly.

==== `ComponentPoolException`

Checked exception extending `FormattedException`. Thrown when acquisition
fails: pool closed, factory raised an error, wait timer expired, thread
interrupted, validation could not find a healthy instance.

=== Simple Pool (`pool.simple`)

==== Model

The simple pool is the shortest path from "I want object pooling" to
"I have object pooling". It tracks two lists of `PooledComponent`
instances -- `freeList` and `usedList` -- behind a method-level
`synchronized` guard.

==== Contracts

[source,java]
----
public interface PooledComponent {
  void terminate();               // Called when the pool discards the component.
}

public interface ComponentFactory<T extends PooledComponent> {
  T createComponent() throws Exception;
}
----

==== Lifecycle

. `new ComponentPool<>(factory, config)` -- sized per `SimplePoolConfig`.
. `pool.register()` -- optional, publishes to `PoolManager`.
. `pool.getComponent()` -- loops: return a free one; otherwise create
  one if below `maxPoolSize`; otherwise wait up to
  `acquireWaitTimeMillis`; otherwise throw `ComponentPoolException`.
. `pool.returnComponent(c)` -- moves from used to free.
. `pool.close()` -- waits for all borrowed components to come back,
  terminates the free list, uses a `CountDownLatch` so concurrent
  callers exit cleanly.

==== Example: A Pool of JDBC Connections

[source,java]
----
// 1. A PooledComponent that wraps a JDBC connection.
public final class ConnectionHolder implements PooledComponent {

  private final Connection connection;

  public ConnectionHolder (String url) throws SQLException {
    this.connection = DriverManager.getConnection(url);
  }

  public Connection getConnection () { return connection; }

  @Override
  public void terminate () {
    try {
      connection.close();
    } catch (SQLException ignored) {
      // Pool is discarding the holder; swallow the failure.
    }
  }
}

// 2. A factory closing over the JDBC URL.
ComponentFactory<ConnectionHolder> factory =
  () -> new ConnectionHolder("jdbc:postgresql://db/app");

// 3. A pool of up to 20 connections, willing to wait 5 seconds when full.
SimplePoolConfig config = new SimplePoolConfig()
  .setMaxPoolSize(20)
  .setAcquireWaitTimeMillis(5000);

ComponentPool<ConnectionHolder> pool = new ComponentPool<>(factory, config);
pool.register();

// 4. Use.
ConnectionHolder holder = pool.getComponent();
try {
  try (PreparedStatement ps = holder.getConnection().prepareStatement("...")) {
    // ...
  }
} finally {
  pool.returnComponent(holder);
}

// 5. On shutdown.
pool.close();
----

==== When to Use Simple

* Short-running or batch programs where a few synchronized method calls
  cost nothing in practice.
* Test harnesses.
* Resources whose health is trivially inferable from the absence of
  exceptions.
* Situations where you would rather avoid the interface surface area of
  the complex pool.

=== Complex Pool (`pool.complex`)

==== Model

The complex pool separates concerns that the simple pool conflates.
A caller deals with the *component* (`C`); the pool deals with a
*`ComponentInstance<C>`* wrapper that can validate itself, serve the
underlying component, and close cleanly; and internally the pool
attaches a *`ComponentPin<C>`* tracking record to each instance to
carry lease timers, deconstruction fuses, and existential stack
traces.

The free list is a `LinkedBlockingQueue`; the used map is a
`HashMap<ComponentInstance, ComponentPin>` guarded by a
`ReentrantReadWriteLock`. A background daemon thread
(`DeconstructionQueue`) polls once per second and ignites fuses that
have reached their scheduled time.

==== Contracts

[source,java]
----
public interface ComponentInstance<C> extends Existential {
  boolean validate();        // Called on create (if testOnCreate) and on acquire (if testOnAcquire).
  C serve() throws Exception;
  void close() throws Exception;
}

public interface ComponentInstanceFactory<C> {
  ComponentInstance<C> createInstance (ComponentPool<C> pool) throws Exception;
  void initialize() throws Exception;   // Once, before startup.
  void startup() throws Exception;      // After the pre-warm phase.
  void shutdown() throws Exception;     // Before deconstruction.
  void deconstruct() throws Exception;  // Final cleanup.
}
----

An `AbstractComponentInstanceFactory<C>` provides empty defaults for
the four lifecycle methods so that most factories need only implement
`createInstance`.

==== `ComplexPoolConfig`

Everything in `PoolConfig`, plus:

[cols="1,1,4"]
|===
| Property | Default | Meaning

| `initialPoolSize`          | `0`       | Pre-warm size at startup.
| `minPoolSize`              | `0`       | Floor: when a component is terminated below this count, a replacement is created.
| `testOnCreate`             | `false`   | Validate immediately after the factory hands back an instance.
| `testOnAcquire`            | `false`   | Validate before returning an instance to the caller; skip invalid instances.
| `creationTimeoutMillis`    | `0`       | Ceiling on how long `createInstance()` may take; `0` disables.
| `maxLeaseTimeSeconds`      | `0`       | Absolute wall-clock lifetime from creation; component is gracefully retired at the deadline.
| `maxIdleTimeSeconds`       | `0`       | How long an instance may sit on the free queue before being gracefully retired.
| `maxProcessingTimeSeconds` | `0`       | How long a caller may hold an instance before it is *prejudicially* yanked back.
| `reportLeaseTimeNanos`     | `false`   | Fire a `LeaseTimeReportingComponentPoolEvent` each time an instance is returned.
| `existentiallyAware`       | `false`   | Capture a stack trace at each acquisition, retrievable for leak diagnostics.
|===

All fields are `AtomicInteger` / `AtomicLong` / `AtomicBoolean`; setters
take effect on the next pool operation.

==== Deconstruction Fuses

The three timeout properties each install a subclass of
`DeconstructionFuse` on every pin at creation. Fuses sit quietly until
their ignition time, at which point they call `pin.kaboom(withPrejudice)`:

[cols="1,1,4"]
|===
| Fuse | Prejudicial? | Trigger

| `MaxLeaseTimeDeconstructionFuse`      | No  | Ignition set once at creation. No reset on serve/free.
| `MaxIdleTimeDeconstructionFuse`       | No  | Ignition scheduled on `free`, cleared on `serve`.
| `MaxProcessingTimeDeconstructionFuse` | Yes | Ignition scheduled on `serve`, cleared on `free`. Uses a generation counter to ignore stale ignitions and logs the existential stack trace when it fires.
|===

"Prejudicial" means the pool forcibly removes the pin even though the
caller still holds its component. It is the mechanism by which a
long-running request cannot permanently pin a pool slot.

The `DeconstructionCoordinator` bundles all fuses attached to a given
pin and ensures that only the first fuse to fire wins via an
`AtomicBoolean`.

==== Events and JMX

Listeners registered via `addComponentPoolEventListener(...)` receive:

* `ErrorReportingComponentPoolEvent<C>` -- a creation, validation, or
  fuse-driven error occurred.
* `LeaseTimeReportingComponentPoolEvent<C>` -- a component was just
  returned; exposes `getLeaseTimeNanos()`.

`ComponentPoolMonitor` implements the `ComponentPoolMonitorMXBean`
interface. Registering one with the platform MBean server exposes every
configuration property, the current free/processing/total sizes, and
broadcasts typed `Notification` instances
(`CreationErrorOccurredNotification`,
`ComponentLeaseTimeNotification`) so that a standard JMX client can
observe pool behavior live.

==== Metrics (Claxon)

If a Claxon registry is wired up, the complex pool emits the following,
each tagged with the pool's name:

[cols="1,3"]
|===
| Tag | Meaning

| `processing` | Current checked-out count (tachometer).
| `free`       | Current idle count (tachometer).
| `timeout`    | Acquisition-wait timeouts (counter).
| `released`   | Lease duration on return (speedometer).
| `waited`     | Time spent blocked in `getComponent()` (speedometer).
|===

==== Example: A Complex Pool of Remote Service Stubs

[source,java]
----
// 1. A ComponentInstance wrapping a remote stub.
public final class RemoteServiceInstance implements ComponentInstance<RemoteService> {

  private final RemoteService stub;
  private StackTraceElement[] stackTrace;

  public RemoteServiceInstance () throws Exception {
    this.stub = RemoteServiceFactory.connect();
  }

  @Override
  public boolean validate () {
    try { stub.ping(); return true; } catch (Exception e) { return false; }
  }

  @Override
  public RemoteService serve () {
    return stub;
  }

  @Override
  public void close () throws Exception {
    stub.disconnect();
  }

  @Override
  public StackTraceElement[] getExistentialStackTrace () { return stackTrace; }

  @Override
  public void setExistentialStackTrace (StackTraceElement[] stackTrace) {
    this.stackTrace = stackTrace;
  }
}

// 2. A factory.
final class RemoteServiceInstanceFactory
  extends AbstractComponentInstanceFactory<RemoteService> {

  @Override
  public ComponentInstance<RemoteService> createInstance (ComponentPool<RemoteService> pool)
    throws Exception {
    return new RemoteServiceInstance();
  }
}

// 3. A production-grade configuration.
ComplexPoolConfig config = new ComplexPoolConfig()
  .setMaxPoolSize(50)
  .setMinPoolSize(10)
  .setInitialPoolSize(10)
  .setAcquireWaitTimeMillis(10_000L)
  .setTestOnCreate(true)
  .setTestOnAcquire(true)
  .setCreationTimeoutMillis(5_000L)
  .setMaxLeaseTimeSeconds(300)
  .setMaxIdleTimeSeconds(60)
  .setMaxProcessingTimeSeconds(120)
  .setExistentiallyAware(true)
  .setReportLeaseTimeNanos(true);

// 4. The pool.
ComponentPool<RemoteService> pool = new ComponentPool<>(
  "RemoteServicePool", new RemoteServiceInstanceFactory(), config);

pool.addComponentPoolEventListener(new ComponentPoolEventListener() {

  @Override
  public void reportErrorOccurred (ErrorReportingComponentPoolEvent<?> event) {
    log.error("Pool error", event.getException());
  }

  @Override
  public void reportLeaseTime (LeaseTimeReportingComponentPoolEvent<?> event) {
    metrics.observe(event.getLeaseTimeNanos());
  }
});

// 5. Optional JMX registration.
ManagementFactory.getPlatformMBeanServer().registerMBean(
  new ComponentPoolMonitor(pool),
  new ObjectName("com.example:type=ComponentPool,name=RemoteService"));

// 6. Start, use, stop.
pool.startup();
try {
  RemoteService service = pool.getComponent();
  try {
    service.doWork();
  } finally {
    pool.returnInstance(service);
  }
} finally {
  pool.shutdown();
}
----

==== Leak Diagnostics

When `existentiallyAware` is `true`, `pool.getExistentialStackTraces()`
returns an array of `StackTraceElement[]`, one per currently
checked-out instance, each captured at the moment of acquisition. This
is the single most useful tool in the framework when the pool looks
"stuck": dump all stacks, and you immediately see which caller forgot
to return a component.

=== Choosing Simple vs. Complex

[cols="1,2,2"]
|===
| Concern | Simple | Complex

| Validation          | You roll your own.                     | `testOnCreate` + `testOnAcquire`.
| Timeouts            | Only `acquireWaitTimeMillis`.          | Lease, idle, processing, creation.
| Pre-warming         | None.                                  | `initialPoolSize`, `minPoolSize`.
| Monitoring          | None.                                  | JMX notifications + Claxon metrics.
| Leak detection      | None.                                  | Existential stack traces.
| Events              | None.                                  | Per-error and per-lease listeners.
| Thread model        | Method-level `synchronized`.           | Read/write lock + lock-free queues.
| Typical use         | Tests, scripts, trivial resources.     | Servers, databases, remote endpoints.
|===

== The Juggler Framework
// ==========================================================================

=== Why It Exists

A pool reuses *one* provider's resources. A juggler spreads *many*
providers' resources across a stream of requests, automatically
quarantines any provider that starts throwing, and optionally validates
that a blacklisted provider has recovered before returning it to
service.

If you have three database replicas, four LDAP servers, five message
brokers, or any other set of *interchangeable* resource producers, a
juggler is the right abstraction.

The metaphor in the name is literal: the juggler keeps N items in
motion, alternating among them in a predictable pattern, and if one
starts to fail, the juggler drops it rather than disrupting the rest.

=== The Load-Balancing Algorithm

A juggler holds two lists of pins -- `sourcePins` and `targetPins`. The
rotation rule is:

. On each `pickResource()`, choose a pin at random from `sourcePins`.
. Move that pin from `sourcePins` to `targetPins`.
. When `sourcePins` is empty, swap the list references.
. If the chosen pin throws while yielding its resource, blacklist it
  and try the next one.
. If both lists are exhausted, throw
  `NoAvailableJugglerResourceException` with each preceding failure
  attached as a suppressed exception.

This is "uniform random selection without replacement per rotation".
Over any one rotation, every active pin is used exactly once; the
per-rotation ordering is random, so adjacent rotations do not produce
identical sequences.

=== Anatomy

==== `JugglingPin<R>`

A pin wraps *one* provider instance and yields a resource of type `R`.

[source,java]
----
public interface JugglingPin<R> {
  void start (Method method, Object... args) throws JugglerResourceException;
  void stop (Method method, Object... args) throws JugglerResourceException;
  void close (Method method, Object... args) throws JugglerResourceException;
  R obtain() throws JugglerResourceException;
  boolean recover();
  String describe();
}
----

The `method`/`args` pair allows the caller to pass a reflective hook
invoked on the underlying resource during each lifecycle stage -- for
example, a `getConnection()` method to pre-warm a database connection.
Almost all callers pass `null`, for which `AbstractJugglingPin` supplies
zero-argument overloads:

[source,java]
----
public abstract class AbstractJugglingPin<R> implements JugglingPin<R> {

  public void start () throws JugglerResourceException         { start(null); }
  public void stop () throws JugglerResourceException          { stop(null); }
  public void close () throws JugglerResourceException         { close(null); }

  // Default no-op implementations; override what you need.
  public void start (Method method, Object... args) throws JugglerResourceException { }
  public void stop (Method method, Object... args) throws JugglerResourceException { }
  public void close (Method method, Object... args) throws JugglerResourceException { }
}
----

==== `JugglingPinFactory<P, R>`

Strategy for constructing a pin from a provider. The juggler calls this
once per provider during `initialize()`:

[source,java]
----
public interface JugglingPinFactory<P, R> {
  JugglingPin<R> createJugglingPin (P provider, Class<R> resourceClass)
    throws JugglerResourceCreationException;
}
----

==== `BlackList<R>` and `BlacklistEntry<R>`

The juggler itself implements `BlackList<R>`. A pin that detects an
unrecoverable error may hold a reference to the juggler-as-blacklist
and call `addToBlackList(new BlacklistEntry<>(this, cause))` to
quarantine itself proactively, rather than waiting for the juggler to
discover the failure during the next `pickResource()`.

`BlacklistEntry<R>` is a record pairing the failed pin with the
causing `Throwable`. The juggler stores entries in a
`ConcurrentSkipListMap<Long, BlacklistEntry<R>>` keyed by
`System.currentTimeMillis()` at failure time, which gives the recovery
worker an ordered view of failures.

==== Exceptions

[cols="1,4"]
|===
| Type | Meaning

| `JugglerResourceException`         | Base checked exception; extends `FormattedException`.
| `JugglerResourceCreationException` | The factory could not instantiate a pin or its resource. Thrown from `initialize()`.
| `NoAvailableJugglerResourceException` | All pins are blacklisted. The most recent failure is the primary cause; every earlier failure is attached as a suppressed exception so that a caller can see the entire failure chain at once.
|===

=== Lifecycle

[cols="1,4"]
|===
| Phase | Semantics

| `initialize()`  | Allocate pin lists, construct one pin per provider via the factory, shuffle them into random order.
| `startup()`     | Call `start(...)` on every pin. A pin that throws is immediately blacklisted. If `recoveryCheckSeconds > 0`, launch the recovery worker daemon.
| `pickResource()`| Normal operation: draw a pin, call `obtain()`, return the resource. Retry on failure.
| `shutdown()`    | Call `stop(...)` on every pin. Signal the recovery worker to exit and wait for it.
| `deconstruct()` | Call `close(...)` on every pin. Reset the juggler so that a subsequent `initialize()` starts fresh.
|===

Each lifecycle method has a no-argument overload and a
`(Method, Object... args)` overload mirroring `JugglingPin`.

=== Recovery

When `recoveryCheckSeconds > 0`, a daemon worker thread
(`ProviderRecoveryWorker`) wakes up at most every three seconds,
scans the blacklist map for entries older than the configured
recovery window, and calls `pin.recover()`. If recovery returns
`true`, the pin is removed from the blacklist and appended to the
`targetPins` list under the juggler lock; the next rotation cycle
will include it once more. If recovery returns `false`, the pin stays
blacklisted and will be re-checked on the next worker iteration.

`recoveryCheckSeconds = 0` disables the worker entirely; blacklisted
pins stay blacklisted for the lifetime of the juggler.

=== Example: Load-Balanced Database Connection Pooling

[source,java]
----
// Three ConnectionPoolDataSource replicas.
ConnectionPoolDataSource ds1 = openReplica("db-1");
ConnectionPoolDataSource ds2 = openReplica("db-2");
ConnectionPoolDataSource ds3 = openReplica("db-3");

JugglingPinFactory<ConnectionPoolDataSource, PooledConnection> factory =
  new PooledConnectionJugglingPinFactory(); // application-supplied

Juggler<ConnectionPoolDataSource, PooledConnection> juggler =
  new Juggler<>(
    ConnectionPoolDataSource.class,
    PooledConnection.class,
    30,              // check for recovery every 30s
    factory,
    ds1, ds2, ds3);

juggler.initialize();
juggler.startup();

try {
  PooledConnection pooled = juggler.pickResource();
  try (Connection conn = pooled.getConnection()) {
    // ...
  }
} catch (NoAvailableJugglerResourceException e) {
  log.error("All replicas unavailable", e);
  for (Throwable t : e.getSuppressed()) {
    log.error("  cascaded failure", t);
  }
}

juggler.shutdown();
juggler.deconstruct();
----

=== Example: A Custom Pin for a Remote HTTP Service

[source,java]
----
public final class ApiPin extends AbstractJugglingPin<ApiClient> {

  private final String baseUrl;
  private ApiClient client;

  public ApiPin (String baseUrl) {
    this.baseUrl = baseUrl;
  }

  @Override
  public void start (Method method, Object... args) throws JugglerResourceException {
    try {
      client = new ApiClient(baseUrl);
    } catch (Exception e) {
      throw new JugglerResourceException(e);
    }
  }

  @Override
  public ApiClient obtain () throws JugglerResourceException {
    try {
      client.healthCheck();
      return client;
    } catch (Exception e) {
      throw new JugglerResourceException(e, "health check failed at %s", baseUrl);
    }
  }

  @Override
  public boolean recover () {
    try { client.healthCheck(); return true; } catch (Exception e) { return false; }
  }

  @Override
  public void close (Method method, Object... args) throws JugglerResourceException {
    client.close();
  }

  @Override
  public String describe () {
    return "ApiPin[" + baseUrl + "]";
  }
}

public final class ApiPinFactory implements JugglingPinFactory<String, ApiClient> {

  @Override
  public JugglingPin<ApiClient> createJugglingPin (String url, Class<ApiClient> resourceClass) {
    return new ApiPin(url);
  }
}

Juggler<String, ApiClient> juggler = new Juggler<>(
  String.class, ApiClient.class, 60, new ApiPinFactory(),
  "https://api-1.example.com",
  "https://api-2.example.com",
  "https://api-3.example.com");

juggler.initialize();
juggler.startup();
----

=== Threading and Performance Notes

* Every public method of `Juggler` is `synchronized`; `pickResource()`
  is therefore effectively a critical section. If you expect very high
  request rates, keep `obtain()` fast (a pin should *not* block on
  network I/O; it should return a prepared resource).
* The recovery worker is the only thread that modifies the blacklist
  without holding the juggler lock, and it does so through a
  concurrent map; promotion of a recovered pin back to the list *does*
  take the juggler lock.
* The daemon worker is always a daemon thread, so it will not keep a
  JVM alive past `shutdown()`; the `abort()` path uses a
  `CountDownLatch` to signal and await clean exit.

== The Bucket Framework
// ==========================================================================

=== Why It Exists

Rate limiting is the act of deciding, at call time, whether a request
should be admitted or rejected based on how many requests have come
before it and how fast they are arriving. A naive counter-per-second
approach is unfair (an "edge effect" where a client can double its
burst rate by straddling the second boundary) and rigid (there is no
notion of allowing a brief spike above the long-term rate).

The *token-bucket algorithm* solves both problems with a single
metaphor. A bucket holds up to `limit` tokens. Tokens refill at a
constant rate, up to the capacity ceiling. Each request consumes one
token (or more, depending on its size). A request is admitted if and
only if there are enough tokens to cover its cost. The long-term
admission rate matches the refill rate; bursts are admitted up to the
bucket's capacity; tokens spent now are not reissued until the refill
period recharges them.

Quorum's implementation adds two refinements on top of the textbook
algorithm:

* *Variable cost* -- an input can cost any non-negative number of
  tokens, not just one. This lets you rate-limit by message size,
  query complexity, or any other resource-meaningful metric.
* *Hierarchy* -- a bucket can own child buckets, each indexed by a
  key. A request must pass *both* the parent bucket and the applicable
  child bucket. This is how you enforce a global limit and a
  per-user/per-tenant limit in one pass.

=== Components

==== `TokenBucket<T>`

The main type. Thread-safe via method-level `synchronized`.
Construction:

[source,java]
----
public TokenBucket (
  BucketQuantifier<T> quantifier,  // how many tokens does this input cost?
  BucketSelector<T> selector,      // which child bucket should this input also consult?
  double limit,                    // maximum token capacity (also the starting balance).
  double refillQuantity,           // number of tokens added each refillRate.
  Stint refillRate)                // duration of one refill period.
----

Operation:

* `boolean allowed(T input)` -- refills based on elapsed time, then
  checks token cost against the available balance and, if sufficient,
  recursively defers to the child bucket identified by
  `selector.selection(input)`. On success, tokens are deducted; on
  failure, the balance is unchanged and `false` is returned.
* `void add(BucketKey<T> key, BucketFactory<T> factory)` -- registers
  a child bucket. The factory is invoked lazily; if `key` is already
  registered, the call is a no-op.

==== `BucketQuantifier<T>`

[source,java]
----
public interface BucketQuantifier<T> {
  double quantity (T input);
}
----

The strategy that maps an input to its token cost. A return of `0.0`
is free; a value greater than or equal to `limit` can never be
admitted.

==== `BucketSelector<T>`

[source,java]
----
public interface BucketSelector<T> {
  BucketKey<T> selection (T input);
}
----

Picks the child bucket key that also governs the input. Returning
`null` means "no child applies" -- the input is judged by the parent
alone.

==== `BucketKey<T>`

A marker interface; implementations must override `equals()` and
`hashCode()` because the key is stored in a `HashMap`.

==== `BucketFactory<T>`

A lazy factory for child buckets:

[source,java]
----
public interface BucketFactory<T> {
  TokenBucket<T> create();
}
----

=== Example: Global Rate Limiting

[source,java]
----
// 100 requests per second, no burst beyond the steady rate.
TokenBucket<Request> bucket = new TokenBucket<>(
  request -> 1.0,                 // every request costs one token
  request -> null,                // no child bucket
  100.0,                          // capacity
  100.0,                          // refill amount
  new Stint(1, TimeUnit.SECONDS));// per refill period

if (bucket.allowed(incoming)) {
  handle(incoming);
} else {
  reject(incoming, "too many requests");
}
----

=== Example: Bursty, Size-Weighted Rate Limiting

[source,java]
----
// 1 MB/s sustained throughput, but tolerate bursts up to 10 MB.
TokenBucket<Payload> bucket = new TokenBucket<>(
  payload -> (double)payload.sizeInBytes(),
  payload -> null,
  10_000_000.0,                        // 10 MB capacity -> burst
  1_000_000.0,                         // 1 MB added per...
  new Stint(1, TimeUnit.SECONDS));     // ...one second
----

=== Example: Global Cap Plus Per-Tenant Cap

[source,java]
----
record TenantKey(String tenantId) implements BucketKey<Request> { }

// Parent: 1000 rps across all tenants.
TokenBucket<Request> parent = new TokenBucket<>(
  req -> 1.0,
  req -> new TenantKey(req.tenantId()),
  1000.0, 1000.0, new Stint(1, TimeUnit.SECONDS));

// For every tenant, register a 100 rps child.
for (String tenantId : tenants) {
  parent.add(new TenantKey(tenantId),
    () -> new TokenBucket<>(
      req -> 1.0,
      req -> null,
      100.0, 100.0, new Stint(1, TimeUnit.SECONDS)));
}

// At call time: an input has to pass both levels.
if (parent.allowed(request)) {
  handle(request);
}
----

=== Notes

* The bucket uses `System.nanoTime()` for all refill calculations, so
  wall-clock changes do not perturb it.
* Each call to `allowed()` takes one `System.nanoTime()` reading and
  reuses it for the entire hierarchy so that parent and child see a
  consistent clock.
* Children are created lazily through the factory; an unused key does
  not cost a bucket instance.
* There is no background thread: refills are computed on demand.
* Because `allowed()` is `synchronized`, contention on a very hot
  bucket can become measurable. Prefer a shallow hierarchy and
  selectors that return `null` when no child is needed.

== The Namespace Framework
// ==========================================================================

=== Why It Exists

Enterprise Java uses JNDI (Java Naming and Directory Interface) as the
standard lookup mechanism for bound objects: a data source, an EJB,
a configuration value. Applications do `new InitialContext().lookup("java:/...")`
and expect to get back an object. But JNDI does not tell you *where*
the bindings actually live; that is the job of a *context factory*.

Most backing stores that administrators actually want -- LDAP, Active
Directory, a custom database -- do not use the `java:/foo/bar/baz`
naming convention natively. LDAP, for example, uses distinguished names
like `cn=baz,cn=bar,cn=foo,dc=example,dc=com`. Something has to
translate between the two.

`quorum.namespace` is that something. It provides:

* A custom `ObjectFactory` (`javaURLContextFactory`) registered for the
  `java:` URL scheme.
* A `DirContext` implementation (`JavaContext`) that translates names
  on every operation before delegating to a backing-store context.
* An extensible backing-store SPI (`ContextCreator`, `NameTranslator`)
  with an LDAP implementation out of the box.
* A pooled variant (`PooledJavaContext`) that fires lifecycle events
  so that a `ComponentPool` can reuse and recycle JNDI contexts.

=== How JNDI Finds Us

Register the package as a URL context factory prefix:

[source,java]
----
Hashtable<String, Object> env = new Hashtable<>();
env.put(Context.URL_PKG_PREFIXES, "org.smallmind.quorum.namespace");
env.put(JavaContext.CONTEXT_STORE,        "ldap");
env.put(JavaContext.CONNECTION_DETAILS,   new NamingConnectionDetails(
  "ldap.example.com", 389, false,
  "dc=example,dc=com",
  "cn=admin,dc=example,dc=com", "secret"));
env.put(JavaContext.CONTEXT_MODIFIABLE,   "true");
env.put(JavaContext.POOLED_CONNECTION,    "true");

InitialContext root = new InitialContext(env);
----

When the application calls `root.lookup("java:/comp/env/jdbc/myds")`,
JNDI finds `javaURLContextFactory` by the `java:` prefix, calls
`getObjectInstance(null, ...)`, which in turn locates the backing-store
classes by naming convention:

[cols="1,3"]
|===
| Key | Looked up as

| `CONTEXT_STORE = "ldap"` | `org.smallmind.quorum.namespace.backingStore.ldap.LdapContextCreator`
|                          | `org.smallmind.quorum.namespace.backingStore.ldap.LdapNameTranslator`
|===

Both classes are instantiated reflectively with single-argument
constructors taking a `NamingConnectionDetails`.

=== Name Translation

A `JavaName` is a list of components parsed from a slash-delimited
string. `JavaNameParser` treats the leading `java:` token as one
component and the remaining `/`-delimited tokens as the rest:

 "java:/comp/env/jdbc"   ==>  ["java:", "comp", "env", "jdbc"]

`LdapNameTranslator` converts such a name into an LDAP DN with the
components in *reverse* order (most specific first) and `cn=` prefix on
each:

 ["comp", "env", "jdbc"]  ==>  "cn=jdbc,cn=env,cn=comp"

and concatenates the configured root DN from
`NamingConnectionDetails.getRootNamespace()` on lookup.

Every `JavaContext` operation flows through the translator, so the
caller sees a `java:`-style namespace but the backing store sees its
native format. Enumerations go through `JavaNamingEnumeration`, which
translates each element on-the-fly; bound `DirContext` values are
transparently wrapped in another `JavaContext` so that recursive
lookups keep the illusion intact.

=== Read-Only vs. Modifiable

By default `JavaContext` is read-only: `bind`, `rebind`, `unbind`,
`modifyAttributes`, `createSubcontext`, `destroySubcontext`, and
`rename` all throw `OperationNotSupportedException`. Setting the
environment property `CONTEXT_MODIFIABLE = "true"` unlocks them.

This matters because JNDI is often used as a configuration-delivery
mechanism; keeping contexts read-only by default prevents accidental
mutation of a shared directory.

=== Pooling

`PooledJavaContext` is a `JavaContext` that:

* Wraps every operation in a `try/catch` that catches
  `CommunicationException` and fires a `contextAborted` event.
* Overrides `close()` to fire `contextClosed` *without* physically
  closing the underlying context. This is the "return to pool"
  semantics.
* Provides `close(boolean forced)` that either fires the event (for
  `false`) or physically closes the context (for `true`).

`JavaContextListener` subscribers receive:

[source,java]
----
public interface JavaContextListener {
  void contextClosed (JavaContextEvent event);
  void contextAborted (JavaContextEvent event);
}
----

`JavaContextComponentInstance` is the bridge between that event
interface and the complex pool's `ComponentInstance<PooledJavaContext>`
contract: on `contextClosed` it calls `pool.returnInstance(this)`; on
`contextAborted` it calls `pool.terminateInstance(this)` and
`pool.reportErrorOccurred(...)`.

=== Example: JNDI Lookup Against LDAP

[source,java]
----
Hashtable<String, Object> env = new Hashtable<>();
env.put(Context.URL_PKG_PREFIXES, "org.smallmind.quorum.namespace");
env.put(JavaContext.CONTEXT_STORE, "ldap");
env.put(JavaContext.CONNECTION_DETAILS, new NamingConnectionDetails(
  "ldap.corp", 636, true,
  "dc=corp,dc=example,dc=com",
  "cn=apps,dc=corp,dc=example,dc=com", "secret"));

InitialContext ctx = new InitialContext(env);

DataSource ds = (DataSource)ctx.lookup("java:/comp/env/jdbc/apps");

SearchControls controls = new SearchControls();
controls.setSearchScope(SearchControls.SUBTREE_SCOPE);
NamingEnumeration<SearchResult> results = ctx.search(
  new JavaName(null /* parser-provided */),
  "(cn=*datasource*)", controls);
----

=== Example: A Pool of JNDI Contexts

[source,java]
----
PooledJavaContextComponentInstanceFactory factory =
  new PooledJavaContextComponentInstanceFactory(
    StorageType.LDAP,
    "java:/comp/env",        // path to look up once a context is opened
    "ldap.corp", 636, true,
    "dc=corp,dc=example,dc=com",
    "cn=apps,dc=corp,dc=example,dc=com", "secret");

ComplexPoolConfig config = new ComplexPoolConfig()
  .setMaxPoolSize(20)
  .setMinPoolSize(2)
  .setTestOnAcquire(true);

ComponentPool<PooledJavaContext> pool = new ComponentPool<>(
  "JndiContextPool", factory, config);

pool.startup();

PooledJavaContext ctx = pool.getComponent();
try {
  DataSource ds = (DataSource)ctx.lookup("jdbc/apps");
  // ...
} finally {
  ctx.close();            // logical close -> returned to pool
}

pool.shutdown();
----

A `CommunicationException` during the middle of that `lookup` fires
`contextAborted`, which terminates the instance and triggers the pool
to refill back to `minPoolSize` on the next acquisition.

=== Extending: A New Backing Store

To support a new backing store, implement two classes in
`org.smallmind.quorum.namespace.backingStore.<yourStore>`:

. `<YourStore>ContextCreator extends ContextCreator`, opening an
  `InitialDirContext` against your service in `getInitialContext()`.
. `<YourStore>NameTranslator extends NameTranslator`, translating
  between the `java:/...` form and your service's native naming.

Then add an entry to `StorageType` and pass your key through
`JavaContext.CONTEXT_STORE`. The factory locates the classes by
convention; no registration step is required.

=== Threading

`JavaContext` and `JavaName` instances are *not* thread-safe; each
thread that needs JNDI access should hold its own context (typically
via a pool). The listener list inside `PooledJavaContext` is protected
by a `WeakEventListenerList`. The pool-side
`JavaContextComponentInstance` uses an `AtomicBoolean` to make
`close()` idempotent.

== Putting It All Together
// ==========================================================================

The four subpackages are independent, but they compose naturally.

=== Pattern: Pooled, Load-Balanced, Rate-Limited Database Access

A common production wiring combines all four frameworks to protect a
relational database cluster:

. *Multiple replicas* -- each `ConnectionPoolDataSource` is wrapped in
  a `JugglingPin`, and a `Juggler` spreads load across them with
  failover.
. *Pooled connections per replica* -- the pin hands its pool's
  connections through to callers. Alternatively, a single `ComponentPool`
  sits over the juggler, acquiring pooled connections for reuse.
. *Rate limiting at the edge* -- a `TokenBucket` in front of the pool
  enforces global and per-tenant request caps, so a runaway client
  cannot drain the entire connection pool.
. *JNDI-published configuration* -- `JavaContext` looks up the data
  source definitions from a central LDAP directory, so the application
  does not hard-code host names.

Each layer is independently testable, observable (through Claxon and
JMX), and swappable.

=== Pattern: A Thin Rate-Limiter In Front of a Complex Pool

When you just want to keep callers from drinking all the pool's
resources at once:

[source,java]
----
TokenBucket<Void> gate = new TokenBucket<>(
  v -> 1.0, v -> null, 50.0, 50.0, new Stint(1, TimeUnit.SECONDS));

ComponentPool<RemoteService> pool = /* ... as earlier ... */;

public RemoteService acquire () throws Exception {
  if (!gate.allowed(null)) {
    throw new RejectedException("rate limit exceeded");
  }
  return pool.getComponent();
}
----

The gate fails fast; the pool's `acquireWaitTimeMillis` protects only
the callers that got past the gate.

== Lifecycle Summary

[cols="1,2,2"]
|===
| Framework | Startup | Shutdown

| `bucket`    | `new TokenBucket(...)`; optional `add(key, factory)` calls. | Garbage collection; no resources to release.
| `juggler`   | `initialize()`, `startup()`.                                | `shutdown()`, `deconstruct()`.
| `pool.simple`  | `new ComponentPool(factory, config)`; optional `register()`. | `close()` (blocks until all borrowed components return).
| `pool.complex` | `new ComponentPool(...)`, `startup()`; optional event/JMX registration. | `shutdown()` (runs factory shutdown, terminates pins, deconstructs factory).
| `namespace` | Register `URL_PKG_PREFIXES`; open `InitialContext`. Pool via `PooledJavaContextComponentInstanceFactory` if desired. | Close the root context; shut down the pool.
|===
