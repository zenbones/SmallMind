[[claxon, Claxon]]
= Claxon

[partintro]
Claxon is Smallmind's metrics and instrumentation library. It keeps metric collection, naming, tagging, and backend emission separate so libraries and applications can add measurements while backend choice, runtime container, and framework selection stay with the caller.

At the center of the module are `org.smallmind.claxon.registry.ClaxonRegistry`, `Instrument`, `MeterBuilder`, and `Emitter`. Meters accumulate values in-process. A registry-owned background worker collects those values on a fixed cadence and forwards the resulting quantities to every bound emitter.

Claxon is designed around a few explicit rules:

* meter naming is caller-owned; if no naming rule matches, the meter becomes a no-op instead of being auto-named
* registry lifecycle is explicit; constructing a registry starts collection and `stop()` ends it
* emitters are optional and pluggable; reusable libraries can stay on the neutral registry API while applications choose the backend
* direct API instrumentation and annotation-based instrumentation share the same meter model

Audience: developers instrumenting reusable libraries or applications while keeping metric backend choice and runtime ownership explicit.
Versioning: examples use `${smallmind.version}` for Smallmind artifacts; keep related modules on the same version unless you intentionally need to split them.

[[claxon-why-use, Why Use Claxon]]
== Why Use Claxon

Use Claxon when you want instrumentation that stays explicit and library-safe.

Advantages over more application-centric approaches:

* libraries can depend on the neutral registry API without choosing a backend vendor
* applications decide how names are generated, which emitters are bound, and how pull endpoints are exposed
* meters keep their aggregation behavior in-process, so the same instrumentation code can feed both push and pull emitters
* metric collection runs on an explicit cadence controlled by `ClaxonConfiguration`
* `Instrument.with(...)` is safe to call from shared code because it becomes a no-op when no registry has been registered

=== Comparison To Micrometer

Micrometer is often used as an application observability facade with strong backend registry concepts and common framework integrations.
Claxon stays narrower:

* the registry and emitters are separate objects with explicit binding
* unmatched callers are dropped instead of auto-named
* metric semantics live in the meter implementation rather than in per-backend registries
* pull exposure lives in the optional `claxon-http` bridge, which keeps web-runtime choice separate from metric collection

Choose Claxon when you want explicit integration seams more than convention-heavy application ownership.

[[claxon-installation, Installation]]
== Installation

=== Reusable Libraries

Reusable libraries should normally depend only on `claxon-registry`.

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>claxon-registry</artifactId>
  <version>${smallmind.version}</version>
</dependency>
----

Reusable libraries usually stop at `claxon-registry` so applications can choose the emitters they want.

=== Applications

Applications choose the runtime surface explicitly:

* always include `claxon-registry`
* add one or more `claxon-emitter-*` modules
* add `claxon-http` if a pull emitter must be exposed over JAX-RS
* add `claxon-exotic` only when you want the built-in extra features

Push-oriented example:

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>claxon-registry</artifactId>
  <version>${smallmind.version}</version>
</dependency>

<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>claxon-emitter-message</artifactId>
  <version>${smallmind.version}</version>
</dependency>
----

Pull-oriented example:

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>claxon-registry</artifactId>
  <version>${smallmind.version}</version>
</dependency>

<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>claxon-emitter-prometheus</artifactId>
  <version>${smallmind.version}</version>
</dependency>

<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>claxon-http</artifactId>
  <version>${smallmind.version}</version>
</dependency>
----

Optional feature module:

[source,xml]
----
<dependency>
  <groupId>org.smallmind</groupId>
  <artifactId>claxon-exotic</artifactId>
  <version>${smallmind.version}</version>
</dependency>
----

[IMPORTANT]
====
`ClaxonConfiguration` defaults to `new ImpliedNamingStrategy()` with no prefix map.
That default names nothing.
If you create a registry and never replace the naming strategy, meter registrations from `Instrument.with(...)` or `ClaxonRegistry.register(...)` become no-ops.
====

=== Annotation-Based Instrumentation

If you use `@Instrumented`, Claxon provides the annotation, aspect, and JSON parser classes, but it does not enable AspectJ for you.

You must supply:

* `aspectjrt` at runtime
* a weaving strategy such as compile-time or post-compile weaving

This repository uses the AspectJ Maven plugin during `process-classes` and `process-test-classes`, but Claxon does not require that exact build shape.
If you do not weave `InstrumentedAspect`, the annotation is only metadata and no metrics are recorded.

[[claxon-quick-start, Quick Start]]
== Quick Start

The smallest useful setup is:

1. create a `ClaxonConfiguration`
2. set a naming strategy that actually matches your callers
3. construct a `ClaxonRegistry`
4. bind at least one emitter
5. call `initializeInstrumentation()` if your code uses `Instrument.with(...)`

[source,java]
----
import java.util.HashMap;
import java.util.concurrent.TimeUnit;
import org.smallmind.claxon.emitter.message.MessageEmitter;
import org.smallmind.claxon.registry.ClaxonConfiguration;
import org.smallmind.claxon.registry.ClaxonRegistry;
import org.smallmind.claxon.registry.ImpliedNamingStrategy;
import org.smallmind.claxon.registry.Instrument;
import org.smallmind.claxon.registry.Tag;
import org.smallmind.claxon.registry.meter.SpeedometerBuilder;
import org.smallmind.nutsnbolts.time.Stint;
import org.smallmind.nutsnbolts.util.DotNotation;

ClaxonConfiguration configuration = new ClaxonConfiguration();
configuration.setCollectionStint(new Stint(2, TimeUnit.SECONDS));
configuration.setRegistryTags(new Tag[] {new Tag("service", "orders")});

HashMap<DotNotation, String> prefixMap = new HashMap<>();
prefixMap.put(new DotNotation("com.mycompany.orders.*"), "com.mycompany.orders");
prefixMap.put(new DotNotation("*"), "com.mycompany.unknown");
configuration.setNamingStrategy(new ImpliedNamingStrategy().setPrefixMap(prefixMap));

ClaxonRegistry registry = new ClaxonRegistry(configuration);
registry.bind("console", new MessageEmitter());
registry.initializeInstrumentation();

Runtime.getRuntime().addShutdownHook(new Thread(() -> {
  try {
    registry.stop();
  } catch (InterruptedException interruptedException) {
    Thread.currentThread().interrupt();
  }
}));

Instrument.with(OrderService.class, new SpeedometerBuilder(), new Tag("operation", "lookup"))
    .update(18, TimeUnit.MILLISECONDS);
----

With the default `MessageEmitter`, a later collection cycle produces one line per quantity:

[source,text]
----
com.mycompany.orders[service=orders, operation=lookup].minimum=18.0
com.mycompany.orders[service=orders, operation=lookup].maximum=18.0
com.mycompany.orders[service=orders, operation=lookup].count=1.0
com.mycompany.orders[service=orders, operation=lookup].rate=0.5
----

The `rate` value above is `0.5` because `SpeedometerBuilder` defaults to a one-second resolution while the registry is collecting every two seconds.

[[claxon-registry-lifecycle, Registry Lifecycle]]
== Registry Lifecycle

`org.smallmind.claxon.registry.ClaxonRegistry` is the runtime center of Claxon.

=== Construction And Shutdown

Constructing a registry immediately starts a daemon collection worker.

[source,java]
----
ClaxonRegistry registry = new ClaxonRegistry(configuration);
----

The registry owns only that worker thread.
Call `stop()` when your application shuts down:

[source,java]
----
try {
  registry.stop();
} catch (InterruptedException interruptedException) {
  Thread.currentThread().interrupt();
}
----

=== Binding Emitters

Use `bind(name, emitter)` to add or replace emitters.

[source,java]
----
registry.bind("console", new MessageEmitter());
registry.bind("prometheus", new PrometheusEmitter());
----

The binding name matters when:

* you later call `getEmitter(name)`
* you expose a pull emitter through `EmitterResource`
* you intentionally rebind a different emitter under the same logical name

Every bound emitter receives the same quantities on each collection cycle.

=== How Collection Works

On each `collectionStint`, the worker does three things in order:

1. sweeps tracked objects and updates their meters
2. records configured `Feature` instances
3. records every registered meter and forwards the resulting quantities to all bound emitters

Emitter failures are logged through Scribe and do not stop the collection loop.
A bad emitter can lose its own export path without taking down the rest of the registry.

=== `Instrument` Registration

`Instrument.with(...)` looks up the current registry through `Instrument.getRegistry()`.

If your code uses `Instrument.with(...)`, call:

[source,java]
----
registry.initializeInstrumentation();
----

You only need that step for the static `Instrument` API.
If you interact with the registry directly through `register(...)`, `track(...)`, or `bind(...)`, no global registration step is required.

[[claxon-naming, Naming And Meter Identity]]
== Naming And Meter Identity

The most important Claxon configuration is not the emitter.
It is the naming rule.

=== Naming Strategy

`ClaxonRegistry.register(...)` asks the configured `NamingStrategy` for a meter name based on the caller class.

If the strategy returns `null`, the registry returns `NoOpMeter.instance()` and drops future updates for that caller/tag combination.

There are two built-in strategies.

==== `ObviousNamingStrategy`

`ObviousNamingStrategy` returns the caller's fully-qualified class name when the caller matches a whitelist of `DotNotation` patterns.

Use it when you want one metric name per caller class.

[source,java]
----
import java.util.HashSet;
import org.smallmind.claxon.registry.ObviousNamingStrategy;
import org.smallmind.nutsnbolts.util.DotNotation;

HashSet<DotNotation> whiteListSet = new HashSet<>();
whiteListSet.add(new DotNotation("com.mycompany.orders.*"));

configuration.setNamingStrategy(
    new ObviousNamingStrategy().setWhiteListSet(whiteListSet));
----

A call from `com.mycompany.orders.OrderService` then uses the meter name `com.mycompany.orders.OrderService`.

==== `ImpliedNamingStrategy`

`ImpliedNamingStrategy` chooses the strongest matching `DotNotation` entry from a `Map<DotNotation, String>` and returns the mapped string.

Use it when several callers should contribute to the same logical meter name.

[source,java]
----
import java.util.HashMap;
import org.smallmind.claxon.registry.ImpliedNamingStrategy;
import org.smallmind.nutsnbolts.util.DotNotation;

HashMap<DotNotation, String> prefixMap = new HashMap<>();
prefixMap.put(new DotNotation("com.mycompany.orders.http.*"), "com.mycompany.orders.http");
prefixMap.put(new DotNotation("com.mycompany.orders.*"), "com.mycompany.orders");
prefixMap.put(new DotNotation("*"), "com.mycompany.other");

configuration.setNamingStrategy(
    new ImpliedNamingStrategy().setPrefixMap(prefixMap));
----

In that configuration:

* `com.mycompany.orders.http.OrderResource` becomes `com.mycompany.orders.http`
* `com.mycompany.orders.service.OrderService` becomes `com.mycompany.orders`
* everything else falls back to `com.mycompany.other`

[IMPORTANT]
====
`ImpliedNamingStrategy` does not append the caller class name.
It returns only the mapped string.
If several callers match the same map entry, they emit under the same meter name unless tags distinguish them.
====

=== Tag Layers

Claxon merges tags from three places:

1. registry-wide tags from `ClaxonConfiguration.getRegistryTags()`
2. meter-name-specific tags from `ClaxonConfiguration.getMeterTags()`
3. instance tags supplied to `register(...)` or `Instrument.with(...)`

The merge order is registry tags, then meter tags, then instance tags.

[source,java]
----
HashMap<String, Tag[]> meterTags = new HashMap<>();
meterTags.put("com.mycompany.orders", new Tag[] {new Tag("component", "api")});

configuration.setRegistryTags(new Tag[] {new Tag("service", "orders")});
configuration.setMeterTags(meterTags);

Instrument.with(OrderService.class, new SpeedometerBuilder(), new Tag("operation", "lookup"));
----

That meter is emitted with:

* `service=orders`
* `component=api`
* `operation=lookup`

`meterTags` keys are looked up by emitted meter name, not by caller class.

=== Meter Identity

Registry identity is determined by:

* caller class
* ordered `Tag...` array

The `MeterBuilder` is not part of the key.

That has three practical consequences:

* changing tag order creates a different meter even when the key/value pairs are the same
* reusing the same caller and tag order with a different builder returns the first meter that was registered
* two different callers can still emit the same meter name if the naming strategy maps them to the same string

Use stable tag ordering and a deliberate naming strategy.

[NOTE]
====
If a caller/tag combination misses the naming strategy once, the registry caches that combination as a no-op for the life of that registry instance.
Use a catch-all rule such as `*` when silent drops are not acceptable.
====

[[claxon-meters, Meter Semantics]]
== Meter Semantics

A Claxon `Meter` accepts `update(long value)` calls and later emits one or more `Quantity` objects during collection.

Claxon quantities are numeric only.
There is no backend-neutral unit field for milliseconds, bytes, rows, or percentages.
Keep units consistent in your update calls and make them obvious through naming and conventions.

[cols="1,1,2,2,2",options="header"]
|===
|Meter |Builder |Typical Use |Quantities |Collection Behavior

|`Gauge`
|`GaugeBuilder`
|Current sample values such as queue size or open sessions
|`minimum`, `maximum`, `average`
|min/max/average reset every collection cycle

|`Histogram`
|`HistogramBuilder`
|Latency or size distributions
|`count`, `rate`, `minimum`, `maximum`, `mean`, configured percentiles
|interval histogram resets every collection cycle

|`Speedometer`
|`SpeedometerBuilder`
|One event per call plus observed value, such as request durations
|`minimum`, `maximum`, `count`, `rate`
|min/max/count/rate reset every collection cycle

|`Tachometer`
|`TachometerBuilder`
|Pure event rates when the update payload itself is irrelevant
|`count`, `rate`
|count/rate reset every collection cycle

|`Tally`
|`TallyBuilder`
|Running totals such as in-flight requests or queue depth adjustments
|`count`
|cumulative; does not reset during collection

|`Trace`
|`TraceBuilder`
|Exponentially weighted moving averages over configured windows
|one quantity per configured window
|moving averages continue across collection cycles
|===

[NOTE]
====
`Speedometer` counts one event per `update(...)` call and uses the supplied value only for interval minimum and maximum.
`Tachometer` ignores the supplied value and also counts one event per `update(...)` call.
If you need the numeric delta itself to be added or subtracted, use `Tally`.
====

=== Builder Defaults

Built-in builders ship with these defaults:

* `HistogramBuilder`: resolution `1 SECOND`, lowest discernible value `1`, highest trackable value `3600000`, significant digits `2`, percentiles `p75`, `p95`, `p98`, `p99`, `p999`
* `SpeedometerBuilder`: resolution `1 SECOND`
* `TachometerBuilder`: resolution `1 SECOND`
* `TraceBuilder`: windows `m1`, `m5`, `m15` with `MINUTES` as the window time unit

Builder setter methods mutate the concrete builder instance.
Configure the builder first, then pass it into `Instrument.with(...)` or `register(...)`.

[source,java]
----
HistogramBuilder histogramBuilder = new HistogramBuilder();
histogramBuilder.resolution(new Stint(30, TimeUnit.SECONDS));
histogramBuilder.percentiles(
    new Percentile("p50", 50.0),
    new Percentile("p95", 95.0),
    new Percentile("p99", 99.0));

Instrument.with(OrderService.class, histogramBuilder, new Tag("operation", "lookup"))
    .update(42);
----

=== Choosing The Right Meter

Use these rules of thumb:

* choose `Gauge` when each update is a sampled value and interval min/max/average are useful
* choose `Histogram` when you need distribution shape and percentiles
* choose `Speedometer` when each update represents one event with an attached measured value
* choose `Tachometer` when only event throughput matters
* choose `Tally` when you need a running counter that should not reset
* choose `Trace` when you want smoothed moving averages rather than interval snapshots

[[claxon-direct-api, Direct Instrumentation API]]
== Direct Instrumentation API

The direct API is normally the best choice for shared libraries and explicit application code.

=== `Instrument.with(...)`

`Instrument.with(caller, builder, tags)` returns an `Instrumentation`.

If a registry has been registered through `initializeInstrumentation()` or `Instrument.register(registry)`, the returned object is active.
Otherwise Claxon returns a no-op instrumentation that simply executes the work.

=== Recording Raw Values

[source,java]
----
Instrument.with(OrderService.class, new TallyBuilder(), new Tag("state", "inflight"))
    .update(1);

Instrument.with(OrderService.class, new TallyBuilder(), new Tag("state", "inflight"))
    .update(-1);
----

This produces a cumulative `count` quantity.

=== Timing Work

`Instrumentation.on(...)` executes the supplied callback and records the elapsed monotonic time.
The default output unit is milliseconds.
Use `as(TimeUnit)` when you want a different output unit.

[source,java]
----
public Order loadOrder (String orderId)
  throws Throwable {

  return Instrument.with(OrderService.class, new SpeedometerBuilder(), new Tag("operation", "load"))
           .as(TimeUnit.MILLISECONDS)
           .on(() -> orderRepository.fetch(orderId));
}
----

You can also record a value measured elsewhere:

[source,java]
----
long started = System.nanoTime();
Order order = orderRepository.fetch(orderId);
long elapsed = System.nanoTime() - started;

Instrument.with(OrderService.class, new HistogramBuilder(), new Tag("operation", "load"))
    .as(TimeUnit.MILLISECONDS)
    .update(elapsed, TimeUnit.NANOSECONDS);
----

=== Tracking Live Objects

`track(...)` is useful when the registry should sample a live object on every collection cycle.

[source,java]
----
import java.util.concurrent.BlockingQueue;
import java.util.concurrent.LinkedBlockingQueue;
import org.smallmind.claxon.registry.meter.GaugeBuilder;

BlockingQueue<Runnable> queue =
    Instrument.with(WorkerPool.class, new GaugeBuilder(), new Tag("queue", "notifications"))
        .track(new LinkedBlockingQueue<>(), BlockingQueue::size);
----

Tracked objects are held through weak references.
If the application drops its own strong reference, Claxon eventually unregisters the associated meter.

=== Registering Directly Against The Registry

If you already own the registry object, you can bypass `Instrument` entirely.

[source,java]
----
Meter meter = registry.register(HttpServer.class, new TachometerBuilder(), new Tag("route", "/orders"));
meter.update(1);
----

That style is explicit but lower-level.
`Instrument.with(...)` is usually easier when code should stay decoupled from registry ownership.

=== Quantity Types

`QuantityType` currently has two values:

* `NONE`
* `COUNT`

Built-in meters mark their count-like quantities as `COUNT`.
Among the built-in emitters, Datadog is the one that materially changes behavior based on that type.

[[claxon-annotations, Annotation-Based Instrumentation]]
== Annotation-Based Instrumentation

`org.smallmind.claxon.registry.aop.Instrumented` lets you attach meter definitions to methods or constructors and apply them through `InstrumentedAspect`.

[IMPORTANT]
====
This section only works when `InstrumentedAspect` is actually woven into your code.
Claxon does not auto-enable AspectJ.
====

=== Annotation Members

`@Instrumented` exposes these controls:

[cols="1,3",options="header"]
|===
|Member |Meaning

|`parser`
|Required `InstrumentedParser` class used to build the meter

|`json`
|JSON passed to the parser; defaults to `{}`

|`caller`
|Optional caller class used for naming; defaults to the declaring type

|`constants`
|Static tags added to every recorded measurement

|`parameters`
|Tags resolved from method parameters

|`timeUnit`
|Output unit for `on(...)` timing; defaults to `MILLISECONDS`

|`active`
|Shortcut to disable the annotation without removing it
|===

=== Built-In Parsers

[cols="1,1,2",options="header"]
|===
|Parser |Meter |Recognized JSON Fields

|`GaugeParser`
|`Gauge`
|none

|`HistogramParser`
|`Histogram`
|`resolutionStint`, `percentiles`, `lowestDiscernibleValue`, `highestTrackableValue`, `numberOfSignificantValueDigits`

|`SpeedometerParser`
|`Speedometer`
|`resolutionStint`

|`TachometerParser`
|`Tachometer`
|`resolutionStint`

|`TallyParser`
|`Tally`
|none

|`TraceParser`
|`Trace`
|`windowTimeUnit`, `windows`
|===

`resolutionStint` JSON uses the shape:

[source,json]
----
{
  "time": 30,
  "timeUnit": "SECONDS"
}
----

`windows` values use the shape:

[source,json]
----
[
  {"name": "m1", "value": 1},
  {"name": "m5", "value": 5},
  {"name": "m15", "value": 15}
]
----

=== Example

[source,java]
----
import java.util.concurrent.TimeUnit;
import org.smallmind.claxon.registry.aop.ConstantTag;
import org.smallmind.claxon.registry.aop.Instrumented;
import org.smallmind.claxon.registry.aop.ParameterTag;
import org.smallmind.claxon.registry.json.HistogramParser;

@Instrumented(
    parser = HistogramParser.class,
    json = "{\"resolutionStint\":{\"time\":30,\"timeUnit\":\"SECONDS\"},"
         + "\"percentiles\":[{\"name\":\"p95\",\"value\":95.0},{\"name\":\"p99\",\"value\":99.0}]}",
    constants = {@ConstantTag(key = "operation", constant = "load")},
    parameters = {@ParameterTag(key = "tenant", parameter = "tenantId")},
    timeUnit = TimeUnit.MILLISECONDS)
public Order load (String tenantId, String orderId) {
  return orderRepository.fetch(orderId);
}
----

Parameter tags are resolved by parameter name.
Nested property access is also supported through dotted getter paths such as `customer.id`.

If parser creation, JSON parsing, or parameter resolution fails, the exception propagates out of the advised call.

[[claxon-features, Features And `claxon-exotic`]]
== Features And `claxon-exotic`

A `Feature` is a registry-polled metric source that is not tied to caller-based meter registration.

`Feature` exposes three methods:

* `getName()`
* `getTags()`
* `record()`

Register features through `ClaxonConfiguration.setFeatures(...)`.

[source,java]
----
import java.net.UnknownHostException;
import org.smallmind.claxon.exotic.jvm.ProfileFeature;
import org.smallmind.claxon.registry.feature.Feature;

try {
  configuration.setFeatures(new Feature[] {
      new ProfileFeature("com.mycompany.jvm", 5000L, true, new Tag("role", "api"))
  });
} catch (UnknownHostException unknownHostException) {
  throw new IllegalStateException("Unable to resolve host name", unknownHostException);
}
----

Every collection cycle, the registry records each feature and sends its quantities through every bound emitter just like regular meters.

=== `ProfileFeature`

`claxon-exotic` currently ships `org.smallmind.claxon.exotic.jvm.ProfileFeature`.

Constructor shape:

[source,java]
----
new ProfileFeature(String name, Long minimumRecordingDelayMilliseconds, boolean addHostNameTag, Tag... tags)
----

Behavior to know:

* `name` is the emitted meter name and does not use the registry naming strategy
* `minimumRecordingDelayMilliseconds` can throttle feature emission independently of `collectionStint`
* `addHostNameTag` prepends a `host=<local host name>` tag and can throw `UnknownHostException`
* emitted quantities come from JVM and OS MXBeans and cover heap, memory, GC, and compilation metrics

[[claxon-emitters, Emitters]]
== Emitters

Emitters translate collected quantities into backend-specific output.
Claxon ships both push and pull emitters.

[[claxon-emitter-caveat-table, Emitter Caveat Table]]
=== Emitter Caveat Table

[cols="1,1,2,3",options="header"]
|===
|Artifact |Method |Name Shape |Important Caveats

|`claxon-emitter-message`
|Push
|`meterName[tags].quantityName=value`
|Default constructor writes to stdout; one line is emitted per quantity

|`claxon-emitter-noop`
|Push
|none
|Drops everything intentionally

|`claxon-emitter-datadog`
|Push
|`meterName.quantityName`
|Defaults to `localhost:8125`; `COUNT` quantities become counters when `countAsCount` is `true`

|`claxon-emitter-aws`
|Push
|quantity name only
|`meterName` is ignored; CloudWatch unit is always `StandardUnit.NONE`

|`claxon-emitter-jmx`
|Push
|JMX `ObjectName` domain plus quantity attributes
|`meterName` must be a valid JMX domain; tags become object name properties

|`claxon-emitter-prometheus`
|Pull
|`meterName:quantityName` after Prometheus-safe normalization
|Each scrape drains the stored values; scrapes that are too fast can be empty and scrapes that are too slow collapse intermediate collections
|===

=== Message Emitter

`MessageEmitter` is the easiest way to see what Claxon is producing.

[source,java]
----
registry.bind("console", new MessageEmitter());
----

Or send the formatted lines somewhere else:

[source,java]
----
registry.bind("log",
    new MessageEmitter(new ScribeLoggerConsumer(OrdersMetrics.class, Level.INFO)));
----

This emitter is ideal for smoke tests, local verification, and piping metrics into an existing logging flow.

=== No-op Emitter

`NoopEmitter` keeps the registry active while discarding export output.

[source,java]
----
registry.bind("discard", new NoopEmitter());
----

It is useful when you want instrumentation code to remain active but you do not want a backend export path in a specific runtime.

=== Datadog Emitter

`DataDogEmitter` wraps the Datadog StatsD client.

Default constructor:

[source,java]
----
registry.bind("datadog", new DataDogEmitter());
----

That uses:

* host `localhost`
* port `8125`
* `countAsCount = true`

Explicit configuration:

[source,java]
----
registry.bind("datadog",
    new DataDogEmitter(
        "orders",
        "statsd.internal",
        8125,
        true,
        new Tag("service", "orders"),
        new Tag("env", "prod")));
----

Behavior to know:

* tags are translated to `key:value`
* `COUNT` quantities are emitted as counters when `countAsCount` is `true`
* everything else is emitted as a gauge
* metric names are `meterName.quantityName`

=== AWS CloudWatch Emitter

`AWSEmitter` requires a `CloudWatchClient` and a CloudWatch namespace.

[source,java]
----
CloudWatchClient client = CloudWatchClient.builder()
    .region(Region.US_WEST_2)
    .credentialsProvider(credentialsProvider)
    .build();

registry.bind("cloudwatch", new AWSEmitter(client, "Orders"));
----

Behavior to know:

* the built-in emitter publishes one `MetricDatum` per quantity
* the published CloudWatch metric name is the quantity name, not `meterName`
* tags become CloudWatch dimensions
* the built-in emitter always uses `StandardUnit.NONE`

If two different meters emit the same quantity names with the same dimensions into the same namespace, CloudWatch sees them as the same metric series.

=== JMX Emitter

`JMXEmitter` exposes each meter as a dynamic MBean.

[source,java]
----
registry.bind("jmx",
    new JMXEmitter(ManagementFactory.getPlatformMBeanServer()));
----

Behavior to know:

* `meterName` becomes the `ObjectName` domain
* tags become `ObjectName` key properties
* quantities become readable attributes
* attribute values are stored as doubles

Choose meter names and tag keys that are valid for JMX object names.

[[claxon-prometheus, Prometheus And HTTP Exposure]]
=== Prometheus And HTTP Exposure

`PrometheusEmitter` is a pull emitter.
Bind it like any other emitter:

[source,java]
----
PrometheusEmitter prometheusEmitter = new PrometheusEmitter();

registry.bind("prometheus", prometheusEmitter);
----

Then expose it through `claxon-http`:

[source,java]
----
EmitterResource resource = new EmitterResource(registry);
// register this JAX-RS resource with Jersey, RESTEasy, or another JAX-RS runtime
----

The resource path is fixed:

[source,text]
----
GET /org/smallmind/claxon/emitter/{name}
----

If `{name}` is not bound, `EmitterResource` throws `UnknownEmitterException`.
If `{name}` resolves to a push emitter, it throws `InvalidEmitterException`.

==== Prometheus Output Behavior

`PrometheusEmitter` stores only the latest value per `meterName` + tags + quantity name.
`emit()` swaps its buffers, renders the current contents, appends `# EOF`, and clears the emitted buffer.

That means:

* if you scrape faster than the registry collects, some scrapes can return only `# EOF`
* if you scrape slower than the registry collects, intermediate collection cycles are overwritten by the latest value for each series
* the most stable setup is to align scrape cadence closely with `collectionStint`

A typical line looks like this:

[source,text]
----
com_mycompany_orders:count{service="orders",operation="lookup"} 1.0
----

Name normalization rules:

* punctuation becomes `_`
* leading digits are prefixed with `_`
* upper/lower transitions can insert `_`
* tag keys are normalized the same way

Tag values are written directly inside quoted label values.
Keep Prometheus tag values simple and avoid quotes, backslashes, and newlines.

[[claxon-spring-support, Spring Support]]
== Spring Support

`claxon-registry` includes optional Spring beans support.
The central type is `ClaxonRegistryFactoryBean`.

`ClaxonRegistryFactoryBean`:

* constructs the registry after properties are set
* binds every emitter from `emitterMap`
* calls `initializeInstrumentation()`
* calls `stop()` during bean destruction

Minimal XML example:

[source,xml]
----
<bean id="claxonConfiguration" class="org.smallmind.claxon.registry.ClaxonConfiguration">
  <property name="collectionStint">
    <bean class="org.smallmind.claxon.registry.spring.StintFactoryBean">
      <property name="time" value="2"/>
      <property name="timeUnit" value="SECONDS"/>
    </bean>
  </property>
  <property name="namingStrategy">
    <bean class="org.smallmind.claxon.registry.ImpliedNamingStrategy">
      <property name="prefixMap">
        <map>
          <entry value="com.mycompany.orders">
            <key>
              <bean class="org.smallmind.nutsnbolts.util.DotNotation">
                <property name="notation" value="com.mycompany.orders.*"/>
              </bean>
            </key>
          </entry>
          <entry value="com.mycompany.unknown">
            <key>
              <bean class="org.smallmind.nutsnbolts.util.DotNotation">
                <property name="notation" value="*"/>
              </bean>
            </key>
          </entry>
        </map>
      </property>
    </bean>
  </property>
</bean>

<bean id="prometheusEmitter" class="org.smallmind.claxon.emitter.prometheus.PrometheusEmitter"/>

<bean id="claxonRegistry" class="org.smallmind.claxon.registry.spring.ClaxonRegistryFactoryBean">
  <property name="configuration" ref="claxonConfiguration"/>
  <property name="emitterMap">
    <map>
      <entry key="prometheus" value-ref="prometheusEmitter"/>
    </map>
  </property>
</bean>

<bean id="claxonEmitterResource" class="org.smallmind.claxon.http.EmitterResource">
  <property name="registry" ref="claxonRegistry"/>
</bean>
----

Other built-in Spring helpers:

[cols="1,3",options="header"]
|===
|Type |Purpose

|`StintFactoryBean`
|Builds a singleton `Stint` from `time` and `timeUnit`

|`TagFactoryBean`
|Builds a singleton `Tag` from `key` and `value`

|`DataDogEmitterFactoryBean`
|Builds a singleton `DataDogEmitter` with configurable host, port, prefix, constant tags, and `countAsCount`

|`MessageEmitterFactoryBean`
|Builds a singleton `MessageEmitter` from a supplied `Consumer<String>`

|`ScribeLoggerConsumerFactoryBean`
|Builds a `ScribeLoggerConsumer` for message-emitter integration with Scribe

|`NoopEmitterFactoryBean`
|Builds a singleton `NoopEmitter`

|`CloudWatchClientFactory`
|Builds a singleton `CloudWatchClient` from explicit AWS access key, secret key, and region
|===

[NOTE]
====
`MessageEmitterFactoryBean` expects you to supply a `Consumer<String>`.
If you want the default stdout behavior, instantiate `MessageEmitter` directly.
====

[[claxon-testing, Testing And Verification]]
== Testing And Verification

The easiest verification loop is:

1. bind `MessageEmitter` to an in-memory consumer
2. drive the instrumented code
3. wait long enough for one collection cycle
4. assert on the emitted lines

[source,java]
----
import java.util.List;
import java.util.concurrent.CopyOnWriteArrayList;
import org.smallmind.claxon.emitter.message.MessageEmitter;
import org.smallmind.claxon.registry.Tag;
import org.smallmind.claxon.registry.meter.TallyBuilder;

List<String> lines = new CopyOnWriteArrayList<>();

registry.bind("capture", new MessageEmitter(lines::add));

Instrument.with(TestService.class, new TallyBuilder(), new Tag("case", "smoke"))
    .update(1);

Thread.sleep(2100L);

boolean sawCount = lines.stream().anyMatch(line -> line.contains(".count=1.0"));
----

Focused verification checklist:

* confirm the naming strategy matches the caller you are instrumenting
* confirm `initializeInstrumentation()` ran before code used `Instrument.with(...)`
* confirm at least one emitter is bound
* for interval meters, wait at least one `collectionStint`
* for tracked objects, keep a strong reference alive during the test
* for pull emitters, request the same emitter name used in `bind(name, emitter)`

[[claxon-troubleshooting, Troubleshooting]]
== Troubleshooting

[cols="1,2,3",options="header"]
|===
|Symptom |Likely Cause |What To Check

|No metrics are emitted at all
|No naming strategy match, no emitter, or `Instrument` was never initialized
|Check `ClaxonConfiguration.getNamingStrategy()`, add a catch-all `DotNotation("*")`, bind an emitter, and call `initializeInstrumentation()`

|Only some calls show up
|Tag order changed, or the same caller/tag tuple is being reused inconsistently
|Keep tag arrays in a stable order and do not reuse the same caller/tag tuple for different meter types

|A method annotated with `@Instrumented` never records metrics
|Aspect was not woven
|Add `aspectjrt` and verify your weaving setup includes `InstrumentedAspect`

|`@ParameterTag` fails at runtime
|Parameter name or nested getter path is wrong
|Make sure the `parameter` member matches a real method parameter name and that dotted access paths resolve through getters

|The HTTP pull endpoint throws `UnknownEmitterException`
|The emitter name in the path was never bound
|Use the exact name passed to `registry.bind(name, emitter)`

|The HTTP pull endpoint throws `InvalidEmitterException`
|The named emitter is a push emitter
|Expose only pull emitters such as `PrometheusEmitter` through `EmitterResource`

|Prometheus scrapes are intermittently empty
|Scrape cadence is faster than collection cadence
|Align scrape interval closely with `collectionStint`

|Prometheus misses intermediate values
|Scrape cadence is slower than collection cadence
|Reduce scrape interval or accept that only the latest per-series value is retained between scrapes

|CloudWatch metrics collide unexpectedly
|AWS emitter ignores `meterName`
|Use namespace and dimensions deliberately, or distinguish series through tags

|JMX registration fails with object-name errors
|`meterName` or tags are not valid for `ObjectName`
|Use JMX-safe meter names and tag keys

|Gauge `average` is `NaN`
|No samples were updated during that collection interval
|Update the gauge every cycle or use `track(...)` on a live object
|===
