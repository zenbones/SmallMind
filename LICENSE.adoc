== Overview

The `smallmind-license-maven-plugin` is a Maven plugin that maintains *license
notice headers in source files* and *license text files in build output* for
a reactor of Maven modules. It is not a library. It has one artifact, two
goals, and exactly enough configuration surface to describe the file types,
comment styles, and notice texts that a multi-module project tends to need.

The two goals are:

* **`generate-notice-headers`** â€” bound by default to the
  `process-sources` lifecycle phase. For each configured rule, it walks the
  module's source, script-source, and (optionally) resource directories and
  rewrites every file whose name matches the rule's glob patterns so that
  the top-of-file notice is exactly the text of a chosen notice file,
  wrapped in a chosen comment-block style.
* **`install-license-files`** â€” bound by default to the `prepare-package`
  phase. It copies one or more license text files (typically `LICENSE`,
  `COPYRIGHT`, `NOTICE`, `APACHE`, `AGPL`, etc.) into the module's build
  output directory so they become part of the packaged artifact.

The plugin is deliberately small: a handful of Maven configuration beans,
a stencil abstraction for the comment-block layout, and a filesystem walker
that rewrites files atomically via a temporary-file-plus-rename sequence.
There is no network dependency, no SCM awareness, no year-range templating,
no YAML/JSON config file â€” the project's `pom.xml` is the only input beyond
the notice files themselves.

=== Why This Package Exists

Open-source and commercial software alike carry a recurring paperwork
obligation: every source file must begin with a copyright or license
notice, and every distributable artifact must include the full text of
the licenses it is offered under. These obligations are not cosmetic.
Most major licenses â€” Apache 2.0, MPL 2.0, EPL 2.0, the GNU family, the
BSD family â€” either require a notice at the head of each source file,
require the license text to be distributed with the work, or both. A
codebase that does not honor those requirements cannot be cleanly
redistributed, no matter how good its code is.

Maintaining these headers by hand does not scale. A non-trivial Java
repository is typically a Maven reactor of many modules, each containing
hundreds or thousands of files in a mix of languages: `.java`, `.xml`,
`.properties`, `.yaml`, `.sh`, `.sql`, and so on. Each language has its
own comment syntax, so a notice that works in one will not paste cleanly
into another. Copyright years roll over. License choices change when a
project relicenses or dual-licenses. Notice text evolves as legal
counsel revises it. Every such change ripples across the entire reactor,
and every file must be updated consistently or the compliance posture of
the whole project is weakened.

The typical alternatives do not handle this well:

* **Manual editing** guarantees drift. One maintainer updates the year,
  another forgets; one copies the notice from a sibling file that was
  itself out of date; a new contributor pastes whatever their IDE
  template had configured.
* **Generic search-and-replace** is dangerous in practice. A sloppy
  substitution can clip the first line of a shebang script, corrupt an
  XML declaration, damage a YAML front-matter block, or leave a stale
  notice sitting directly below the new one when the pattern fails to
  match on a single character.
* **IDE templates** only affect newly created files. They do nothing for
  existing code, and they silently diverge between contributors using
  different editors.
* **Heavyweight third-party license plugins** typically bring in
  significant dependency trees and configuration surfaces that are hard
  to reason about inside a build.

This plugin takes a deliberately narrow approach. It treats the notice
as text living in a single file at the top of the reactor, treats each
source-language comment style as a reusable *stencil*, and treats
rewriting as a safe, repeatable operation:

1. *Notice headers are regenerated deterministically from a plain-text
   notice file.* Edit the file in one place, rebuild, and every source
   file in the reactor is rewritten to match â€” in whatever comment
   style is appropriate for each file type.
2. *License text files are copied into each module's build output* as
   part of the standard `package` phase, so every jar, war, or other
   packaged artifact physically carries the license set the project
   distributes under.

The rewriting strategy is forgiving by design. The plugin treats an
existing notice as something to *recognize and consume*, not as
something that must already match byte-for-byte. As long as the previous
notice was written in the same stencil â€” that is, inside the same style
of comment block â€” the plugin will detect it, strip it, and emit the
current notice text in its place, without touching anything below. That
is what makes the plugin safe to rerun on every build, and what makes
it safe to revise the notice text without manually editing every file
that already carried a version of it.

=== What It Does, Concretely

* **Rewrites source files in place.** For each file matching a rule, the
  plugin scans the top of the file for an existing notice block, skips
  over it, emits the new notice, and then copies the rest of the file
  verbatim. The output is written to `license.temp` in the same directory
  and then atomically moved over the original via
  `java.nio.file.Files.move(..., REPLACE_EXISTING)`.
* **Detects existing notices via delimiters, not content.** The plugin
  does not remember what it previously wrote. It recognizes an existing
  notice purely by the stencil's opening delimiter, per-line prefix, and
  closing delimiter. If the opening line matches, subsequent lines that
  carry the stencil's line prefix (or the blank-line prefix) are consumed
  until the closing line is found. Any other content ends the scan and
  is preserved.
* **Supports leading-line skipping.** A stencil may declare a
  `skipLinePattern` regular expression that matches pre-notice content
  (shebangs, XML declarations, encoding pragmas). Lines that match the
  pattern are forwarded verbatim to the output before the notice is
  detected or emitted.
* **Normalizes line endings.** The `lineEndings` parameter chooses
  between a hard-coded Unix LF (`\n`) and the host platform's native
  separator. Only the newly emitted notice and its trailing newline are
  written through the terminator; existing file content below the notice
  is copied byte-for-byte through a reused 8 KiB character buffer.
* **Walks the reactor to a designated root.** Both goals walk up the
  Maven `parent` chain of the current module to locate a root project,
  against which relative notice and license paths are resolved. By
  default the walk continues until `getParent() == null`; a `<root>`
  element pins the walk to a specific `groupId`/`artifactId` instead.
* **Copies license text files into build output.** `install-license-files`
  resolves each configured license path (absolute or reactor-root-relative),
  and copies it into `${project.build.outputDirectory}` using the source
  file's base name. If the output directory does not yet exist, the goal
  is a no-op.

[[installation]]
== Installation

The plugin is published under the usual SmallMind coordinates. A typical
reactor configures it in the root `pom.xml` so every module inherits the
same rules.

[source,xml]
----
<build>
  <plugins>
    <plugin>
      <groupId>org.smallmind</groupId>
      <artifactId>smallmind-license-maven-plugin</artifactId>
      <version>${project.version}</version>  <!--1-->
      <executions>
        <execution>
          <id>generate-notice-headers</id>
          <goals>
            <goal>generate-notice-headers</goal>              <!--2-->
          </goals>
        </execution>
        <execution>
          <id>install-license-files</id>
          <goals>
            <goal>install-license-files</goal>                <!--3-->
          </goals>
        </execution>
      </executions>
      <configuration>
        <!-- see Quick Start -->
      </configuration>
    </plugin>
  </plugins>
</build>
----
<1> Use the reactor's shared version placeholder or a pinned release.
<2> Binds to `process-sources`. Runs on every build unless skipped.
<3> Binds to `prepare-package`. Quiet no-op if `target/classes` doesn't exist.

A single `<configuration>` block can feed both goals. `install-license-files`
declares the same `rules` parameter as `generate-notice-headers` (so both
goals accept the same configuration without errors) but ignores it.

=== Runtime Requirements

The plugin depends only on Maven's `maven-core`, `maven-model`,
`maven-plugin-api`, and `maven-plugin-annotations` (the first three at
`provided` scope, contributed by Maven itself). There is no runtime
classpath requirement beyond Maven and the JDK.

[[quick-start]]
== Quick Start

The configuration below is the pattern the SmallMind reactor itself uses:
every `.java` file gets a JavaDoc-style header copied from `COPYRIGHT`,
and every packaged artifact ships with `COPYRIGHT`, `AGPL`, `AGPL.LESSER`,
and `APACHE` alongside its classes.

[source,xml]
----
<plugin>
  <groupId>org.smallmind</groupId>
  <artifactId>smallmind-license-maven-plugin</artifactId>
  <version>${project.version}</version>
  <executions>
    <execution>
      <id>generate-notice-headers</id>
      <goals><goal>generate-notice-headers</goal></goals>
    </execution>
    <execution>
      <id>install-license-files</id>
      <goals><goal>install-license-files</goal></goals>
    </execution>
  </executions>
  <configuration>
    <root>                                                         <!--1-->
      <groupId>org.smallmind</groupId>
      <artifactId>smallmind</artifactId>
    </root>
    <verbose>true</verbose>
    <lineEndings>UNIX</lineEndings>                                <!--2-->
    <includeResources>true</includeResources>                      <!--3-->
    <includeTests>false</includeTests>
    <rules>
      <rule>
        <id>java-headers</id>                                      <!--4-->
        <fileTypes>
          <fileType>*.java</fileType>
        </fileTypes>
        <excludes>
          <exclude>package-info.java</exclude>                     <!--5-->
        </excludes>
        <stencilId>org.smallmind.license.stencil.JavaDocStencil</stencilId>  <!--6-->
        <notice>COPYRIGHT</notice>                                 <!--7-->
      </rule>
    </rules>
    <licenses>                                                     <!--8-->
      <license>COPYRIGHT</license>
      <license>AGPL</license>
      <license>AGPL.LESSER</license>
      <license>APACHE</license>
    </licenses>
  </configuration>
</plugin>
----
<1> Pins reactor walking to the `smallmind` parent, even if this POM is
    itself included in a larger repository.
<2> Emitted notice lines terminate with `\n` regardless of host OS. Use
    `SYSTEM` to follow the platform default.
<3> Resource directories (both main and test, when `includeTests` is also
    enabled) participate in rule evaluation.
<4> A human-readable id used in log messages and error reports; must be
    unique across rules within the same configuration block.
<5> Glob-pattern exclusions. See <<rule-excludes-caveat>> for current
    behavior.
<6> References the built-in JavaDoc stencil by its fully qualified class
    name.
<7> Notice text file, resolved against the reactor root. The file is read
    as plain lines and wrapped by the stencil at emit time.
<8> License files for `install-license-files`. Each is copied verbatim
    into `target/classes/` under its base name.

[[goals]]
== Goals

[[goal-generate-notice-headers]]
=== `generate-notice-headers`

Default lifecycle phase: `process-sources`. Implemented by
`SourceNoticeMojo`.

For every configured rule, the goal:

1. Verifies that `fileTypes` is non-empty and that a registered stencil
   matches `stencilId`.
2. Reads the notice file into a line array â€” or, if `notice` is null and
   `allowNoticeRemoval` is `true`, prepares to remove rather than replace.
3. Walks the main source directory, the script-source directory, and â€”
   depending on `includeResources` and `includeTests` â€” main and test
   resource directories.
4. For each file whose name matches the rule's glob patterns, scans the
   top for a notice block in the rule's stencil, removes it if present,
   writes the new notice if one was supplied, and copies the remainder
   of the file unchanged.

The write is performed to a sibling temp file (`license.temp`) and then
atomically moved over the original. A partial write cannot leave the
project with a half-rewritten source file; either the move completes and
the file is fully updated, or an exception propagates out of the goal as
a `MojoFailureException`.

| `stencils`
| `Stencil[]`
| empty
| Additional comment-block styles merged with the built-in

| `rules`
| `Rule[]`
| *required*
| At least one rule binding a set of file types to a stencil and notice.

| `lineEndings`
| `LineEndings`
| `UNIX`

| `allowNoticeRemoval`
| `boolean`
| `false`

| `includeResources`
| `boolean`
| `true`
| Include main (and optionally test) resource directories when walking.

| `includeTests`
| `boolean`
| `false`

| `verbose`
| `boolean`

|===

[[goal-install-license-files]]
=== `install-license-files`

Default lifecycle phase: `prepare-package`. Implemented by
`TargetLicenseMojo`.

The goal copies the list of `licenses` â€” resolved against the reactor
root (or the `<root>`-designated ancestor) for relative paths, left
absolute otherwise â€” into `${project.build.outputDirectory}`. A license
that cannot be located produces a warning; a license that can be located
but cannot be copied fails the build with `MojoExecutionException`.

If `target/classes/` does not yet exist (for example, a pure POM module
with nothing to package), the goal does nothing.

[[rules-and-stencils]]
== Rules And Stencils

A rule answers three questions:

* Which files in this module should this rule apply to? â€” `fileTypes`,
  `excludes`.
* What text should be inserted as the notice? â€” `notice`.
* How should that text be wrapped? â€” `stencilId`.

A stencil answers the fourth:

* What is the exact shape of the comment block? â€” opening delimiter,
  per-line prefix, blank-line prefix, closing delimiter, optional
  padding before/after, optional skip pattern for lines that must stay
  above the block.

The built-in `JavaDocStencil` encodes the standard Java/JavaDoc
block-comment shape:

[source,text]
----
/*
 * Copyright (c) 2007 through 2026 David Berkman
 *
 * This file is part of the SmallMind Code Project.
 * ...
 */
----

with opening line `/*`, per-line prefix `" * "`, blank-line prefix
`" *"`, and closing line `" */"`. No blank lines are inserted before or
after, and no leading file lines are skipped.

[[rule-excludes-caveat]]
=== Rule Glob Semantics

File-type and exclude patterns use a minimal glob syntax translated into
regular expressions by `FileTypeRegExTranslator`:

[cols="1,2",options="header"]
|===
| Glob | Regex translation

| `*`
| `.*` â€” zero or more characters

| `?`
| `.?` â€” zero or one character

| `$`
| literal `$`

| `.`
| literal `.`

| any other character
| copied verbatim
|===

Matching is performed against the *filename only*, not the full path,
so `*.java` matches every `.java` file regardless of depth in the
source tree. `src/main/java/**/*.java`-style multi-segment globs are
not supported â€” the glob is a filename pattern, not a path pattern.

[[custom-stencils]]
== Custom Stencils

Any file type whose comment style is not JavaDoc needs a custom stencil.
Declare one inline under `<stencils>` and then reference its `id` from
each rule that should use it.

[source,xml]
----
<stencils>
  <stencil>
    <id>xml-stencil</id>
    <firstLine><![CDATA[<!--]]></firstLine>                        <!--1-->
    <linePrefix>  </linePrefix>                                    <!--2-->
    <lastLine><![CDATA[-->]]></lastLine>
    <blankLinesAfter>1</blankLinesAfter>                           <!--3-->
  </stencil>
  <stencil>
    <id>shell-stencil</id>
    <skipLinePattern>^#!.*$</skipLinePattern>                      <!--4-->
    <linePrefix># </linePrefix>
    <blankLinePrefix>#</blankLinePrefix>
  </stencil>
</stencils>

<rules>
  <rule>
    <id>xml-headers</id>
    <fileTypes><fileType>*.xml</fileType></fileTypes>
    <stencilId>xml-stencil</stencilId>
    <notice>COPYRIGHT</notice>
  </rule>
  <rule>
    <id>shell-headers</id>
    <fileTypes>
      <fileType>*.sh</fileType>
      <fileType>*.bash</fileType>
    </fileTypes>
    <stencilId>shell-stencil</stencilId>
    <notice>COPYRIGHT</notice>
  </rule>
</rules>
----
<1> Opening delimiter line. Not prefixed; written exactly as configured.
<2> Per-line prefix for non-blank notice lines. Written before each line
    of the notice file.
<3> Requests one blank line below the block. No blank line is inserted
    above, because `blankLinesBefore` defaults to zero.
<4> Lines matching this regex (here, a shebang) are forwarded verbatim
    before notice detection begins, so they are preserved on the first
    line of the file.

[NOTE]
====
A stencil without an opening delimiter (`firstLine == null`) cannot
detect an existing notice; the scanner enters the `LAST` state
immediately. This is useful for free-form prefixes (e.g., Python-style
`# ...` headers) but means the plugin must rely on the `linePrefix` /
`blankLinePrefix` to recognize the boundary of an existing notice. Test
such a stencil against a file that already has a conforming notice
before relying on idempotent rewrites.
====

=== Stencil Parameters

[cols="1,1,3",options="header"]
|===
| Parameter | Type | Description

| `id`
| `String`
| Identifier used by `<rule><stencilId>` to select this stencil. Custom
  stencils use whatever id is configured; built-in `StaticStencil`
  subclasses use the fully qualified class name.

| `skipLinePattern`
| `String` regex
| Lines matching this pattern at the very top of the file are forwarded
  to output before any notice detection or emission. Useful for
  shebangs, XML declarations, encoding pragmas.

| `firstLine`
| `String`
| Opening delimiter. Written literally at the start of each new notice
  and also used to detect an existing notice. `null` means no explicit
  opening line.

| `lastLine`
| `String`
| Closing delimiter. Written at the end of each new notice and also
  used to detect the end of an existing notice. `null` means no
  explicit closing line â€” detection instead terminates at the first
  line that does not carry the expected line prefix.

| `linePrefix`
| `String`
| Prefix prepended to each non-blank notice line on emission, and used
  as one of the criteria for recognizing existing notice body lines.

| `blankLinePrefix`
| `String`
| What to write in place of a blank line inside the notice. Setting
  this preserves the decorated look of block comments (e.g., `" *"`
  inside a JavaDoc block). When `null`, blank notice lines are emitted
  as truly empty lines and terminate detection of an existing block.

| `blankLinesBefore`
| `int`
| Count of blank lines emitted above the new notice. Default `0`.

| `blankLinesAfter`
| `int`
| Count of blank lines emitted below the new notice. Default `0`.
|===

[[behavioral-reference]]
== Behavioral Reference

=== Notice Detection State Machine

`SourceNoticeMojo.seekNotice` advances through four states while reading
the top of each candidate file:

`FIRST`::
    Searching for the opening delimiter. Entered when the stencil
    defines a `firstLine`. Transitions to `LAST` if the first non-blank
    line equals `firstLine`; otherwise transitions to `TERMINATED`,
    meaning no existing notice was present. When the stencil has no
    `firstLine`, scanning starts directly in `LAST`.

`LAST`::
    Inside the notice body. Each line is compared against `lastLine`
    (transitions to `COMPLETED`), the `blankLinePrefix` and
    `linePrefix` (stays in `LAST`), or anything else (transitions to
    `TERMINATED`). An empty line when `blankLinePrefix` is null also
    terminates.

`COMPLETED`::
    Notice fully consumed. Any additional blank lines immediately below
    the closing delimiter are also consumed, so a rerun of the plugin
    does not accumulate blank-line padding at the top of the file.

`TERMINATED`::
    A non-notice line was encountered. That line is returned to the
    caller as the "unprocessed line" so it can be written back ahead of
    the remainder of the file â€” preserving user content that happened
    to sit at the top of an unannotated source file.

=== Skip-Line Forwarding

Lines that match the stencil's `skipLinePattern` are written straight
to the output *before* the state machine runs. This means the plugin
can safely process files where the first line must not move: a
`.xml` file whose first line is `<?xml version="1.0"?>`, a `.sh`
file whose first line is `#!/usr/bin/env bash`, and so on. Configure
a `skipLinePattern` on any stencil whose target file type has such a
requirement.

=== Idempotency

A stencil that defines both `firstLine` and `lastLine` â€” the common
case â€” can recognize any notice block it previously wrote, so running
`generate-notice-headers` repeatedly against an unchanged notice file
produces no net change beyond the atomic temp-file rewrite itself.

Rerunning after editing the notice file rewrites every matching source
file to the new text in a single pass. Old notice content is stripped
regardless of its wording, as long as its shape still matches the
stencil.

=== Atomic Rewrites

All rewrites go through a two-step sequence:

1. Open the original file for reading and a sibling `license.temp` for
   writing, both under `try-with-resources`. Stream content through an
   8 KiB character buffer.
2. On successful close, `Files.move(tempPath, licensedPath,
   REPLACE_EXISTING)`.

If any I/O step fails, the exception is wrapped in `WrappedException`,
unwrapped outside the `forEach` stream, and surfaced as
`MojoFailureException`. The temp file remains on disk; the original is
untouched.

=== Reactor-Root Resolution

Both mojos implement the same root-finding loop:

[source,java]
----
MavenProject rootProject = project;
while ((root == null)
       ? !(rootProject.getParent() == null)
       : !(root.getGroupId().equals(rootProject.getGroupId())
           && root.getArtifactId().equals(rootProject.getArtifactId()))) {
  rootProject = rootProject.getParent();
}
----

Without `<root>`, the walk continues until a project with no parent is
reached â€” typically the actual reactor root, but potentially further if
the reactor inherits from an external parent that is also loaded in the
current build. With `<root>`, the walk stops at the first matching
ancestor. The resolved project's `basedir` is what relative `notice`
and `license` paths are anchored against.

=== Line Ending Scope

The configured `lineEndings` value affects only newly written content:
notice body lines, opening and closing delimiters, blank-line padding,
and the trailing newline after the first post-notice line. The bulk of
the file â€” everything below the notice â€” is copied through an 8 KiB
character buffer without newline translation.

This means that a file written with CRLF line endings will, after the
plugin runs, have LF-terminated notice lines and CRLF-terminated
remaining lines. If uniform line endings across the whole file are a
requirement, a follow-up formatting step (editor settings, Git
`core.autocrlf`, a dedicated formatter plugin) should handle that;
this plugin does not.

[[limitations-and-known-issues]]
== Limitations And Known Issues

Rule excludes are not wired through to the filesystem walk::
    `SourceNoticeMojo.execute()` constructs an `excludeFilters` array
    from `Rule.getExcludes()` but never passes it to
    `updateNotice(...)`. The method signature declares
    `PathFilter... excludeFilters` as varargs, so every call-site is
    invoking it with an empty exclude list. In the current build,
    configuring `<excludes>` inside a rule has no effect: files that
    match an include pattern are rewritten regardless of whether they
    also match an exclude pattern. The author's `//TODO: Excludes,
    Seek/Process Optimization` comment in `SourceNoticeMojo`
    acknowledges this.

Rule `<excludes>` element must be present, even when empty::
    `Rule.getExcludes()` defaults to `null` when the element is omitted,
    and `SourceNoticeMojo.execute()` dereferences the getter
    unconditionally: `new PathFilter[rule.getExcludes().length]`. A rule
    configured without an `<excludes>` element therefore fails with
    `NullPointerException`. Workaround: always declare at least an
    empty `<excludes/>` element inside every rule until the upstream
    null guard is added.

No multi-segment path globs::
    Globs apply to the file name component only. There is no way to
    match, for example, only files under `src/main/java/com/example/`;
    all `*.java` files in the walked directories are matched.

Line-ending normalization only touches newly written content::
    See <<behavioral-reference>>. Mixed line endings are possible in
    the output file if existing content used a different style.

No dry-run mode::
    The plugin rewrites files as it walks. Review is by diff against
    version control. Idempotency makes a rerun safe, but there is no
    preview switch.

Notice file is read per rule, not cached across rules::
    Each rule independently reads its `notice` file. Reactors with
    many rules pointing at the same notice will re-read it on every
    rule.

`install-license-files` has no selective copy::
    All listed licenses are copied into every module's `target/classes`.
    There is no per-module filter. Modules that should not include
    license files at the root of their artifact should not bind this
    goal.

No stencil chaining or inheritance::
    Each rule binds to exactly one stencil. A custom stencil cannot
    extend the built-in `JavaDocStencil` via configuration â€” extension
    is only available by writing a Java subclass and putting it on the
    plugin's classpath, which typically means forking the module.

[[troubleshooting]]
== Troubleshooting

`MojoExecutionException: No notice was provided for rule(id)...`::
    A rule lacks a `<notice>` element and `allowNoticeRemoval` is
    `false`. Either add a notice file or set
    `<allowNoticeRemoval>true</allowNoticeRemoval>` if removal is the
    intent.

`MojoExecutionException: No file types were specified for rule(id)`::
    The rule's `<fileTypes>` list is empty or missing. At least one
    glob is required.

`MojoExecutionException: No stencil found with id(...) for rule(id)`::
    The rule's `stencilId` does not match any registered stencil. For
    the built-in JavaDoc stencil, use
    `org.smallmind.license.stencil.JavaDocStencil`. For a custom
    stencil, confirm that the `<stencil><id>` matches the rule's
    `<stencilId>` exactly.

`Unable to acquire the notice file(...), skipping notice updating...`::
    The notice path resolved to a file that does not exist or could
    not be read. Confirm the path is correct relative to the reactor
    root â€” or the module named in `<root>` â€” and that the file has
    read permissions. The rule is skipped rather than failing the
    build.

`Unable to acquire the license file(...), skipping license copying...`::
    Same as above, for `install-license-files`.

Existing header is not being replaced::
    The existing header probably does not match the stencil's
    `firstLine` exactly. Comparison is byte-for-byte on trimmed
    content; a difference of one character ends notice detection
    immediately. Either align the existing header with the stencil
    or run once with `allowNoticeRemoval=true` and no `<notice>` to
    strip it, then rerun with the notice configured.

Shebang or XML declaration got rewritten under the notice::
    The stencil has no `skipLinePattern`. Add one that matches the
    required leading line (e.g. `^#!.*$`, `^<\?xml.*\?>$`). Skipped
    lines are forwarded verbatim above the notice on emission.

`NullPointerException` from `SourceNoticeMojo`::
    Most often caused by a rule that omits the `<excludes>` element.
    See <<limitations-and-known-issues>>. Configure an empty
    `<excludes/>` block as a workaround.
